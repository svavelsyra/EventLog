## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

This story exists because the approved architecture now requires a generic repository-facing contract in addition to the low-level `DatabaseAdapter`. Defining that repository base early:

1. **Creates a clear repository boundary** - Higher layers depend on repository-facing behavior instead of low-level adapter mechanics
2. **Keeps repositories dialect-agnostic at the base level** - SQLite-specific details stay out of the shared contract
3. **Enables later concrete repository work** - SQLite repositories can build on a stable generic base

Without this contract, later repository stories risk mixing CRUD behavior, transaction semantics, and backend mechanics inconsistently.

### User Value
- Clear repository-facing contract for future callers
- Cleaner separation between repository logic and low-level DB mechanics
- Better foundation for later SQLite repository implementation

---

## Goal (WHAT)

**What functionality are we delivering?**

Create the generic `BaseRepository` abstract base class and define the shared repository-facing lifecycle/CRUD direction that concrete repositories will build on.

### Acceptance Criteria

- [ ] File `src/db/repositories/base_repository.py` exists
- [ ] `BaseRepository` inherits from `ABC`
- [ ] Generic repository-facing methods are defined with `@abstractmethod`:
  - `create(...)`
  - `update(...)`
  - `delete(...)`
  - `begin_transaction()`
  - `commit_transaction()`
  - `rollback_transaction()`
  - `close()`
- [ ] Contract remains dialect-agnostic and does not embed SQLite-specific behavior
- [ ] All methods have type hints for parameters and return values
- [ ] All methods have docstrings explaining purpose, args, returns, and exceptions
- [ ] Method grouping reflects shared repository responsibilities rather than entity-specific sections

- **Type hints required** - All parameters and returns typed

### Negative Constraints (MUST NOT)

- Keep docstrings focused on contract meaning rather than implementation details

## Implementation Notes

### What's Needed

Add a generic repository base contract following the approved architecture:

1. **Create Method**
   - Purpose: Define repository-facing create capability at a generic level
   - Returns: Repository-defined creation result (for later concrete repositories to refine)

2. **Update Method**
   - Purpose: Define repository-facing update capability
   - Returns: Success/status information appropriate for repository contract use

3. **Delete Method**
   - Purpose: Define repository-facing delete capability
   - Returns: Success/status information appropriate for repository contract use

4. **Transaction Methods**
   - Purpose: Define repository-facing transaction semantics that delegate downward to adapters in concrete implementations
   - Returns: No value

5. **Close Method**
   - Purpose: Define cleanup semantics for repository-facing callers
   - Returns: No value

### Pattern Consistency

- Follow the contract style established in 001.002.001
- Use `@abstractmethod` decorator with `pass` body (no implementation)
- Consistent type hints and docstring format
- Keep the contract generic enough that concrete SQLite repositories can specialize later
