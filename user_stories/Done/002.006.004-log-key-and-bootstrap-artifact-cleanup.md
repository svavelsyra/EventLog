## Purpose (WHY) ⭐ MOST IMPORTANT

Even if the active database footprint is removed correctly, `Nollställ` still leaves dangerous residual access paths if application logs or stale bootstrap selectors survive the reset. Those artifacts sit outside the main database footprint but still matter to the real local security story: logs may preserve operational traces, and stale `config.ini` state could accidentally point startup back toward a destroyed or contradictory database context.

This story exists to define and deliver cleanup of those non-database artifacts without mixing them back into the database-footprint story or the caller-mode reporting story.

Without this story, later implementation could drift into unsafe behavior such as:
- cleaning the database file but leaving rotated logs or other runtime-known log artifacts behind
- treating externally chosen key files as app-owned artifacts and deleting user files the app should not assume ownership of
- leaving `config.ini` in a stale state that reintroduces invalid bootstrap selectors after reset
- hiding the difference between unreachable artifacts and actual cleanup failures

### User Value
- Operators get a more realistic denial capability that targets the app-owned local security footprint, not only the database file.
- Startup after `Nollställ` can return to a known-safe bootstrap state instead of reopening stale or contradictory configuration.
- The reset flow can report non-database cleanup outcomes consistently without merging them into caller-specific UI behavior.

## Goal (WHAT)

Implement best-effort cleanup of runtime-known non-database app artifacts for `Nollställ`, covering application log files and explicit bootstrap-reset handling for `config.ini` or equivalent bootstrap state so stale database selectors do not survive reset, while leaving external key files outside automatic cleanup scope.

### Current Delivery Snapshot
- `src.security.reset_flow.enumerate_log_cleanup_targets(...)` and `delete_log_cleanup_targets(...)` now cover runtime-known current and rotated log artifacts from the active `config.ini`, including relative-path resolution, existing-only targeting, and rerun-safe deletion behavior.
- `src.app.run_active_context_reset(...)` now clears remembered bootstrap selectors after successful denial through `save_bootstrap_target_config(..., BootstrapTargetConfig())`, preserving creation defaults plus unrelated config sections instead of leaving stale startup selectors behind.
- The reset policy now explicitly leaves external key files alone: caller-selected key files are treated as user-owned inputs rather than reset-managed artifacts, so `Nollställ` focuses on removing app-owned data such as database artifacts, logs, and remembered bootstrap state.
- The newer caller-facing reset-report seam now keeps non-database cleanup outcomes sanitized and neutral for later interpretation: `src/core/reset_report.py` reduces follow-up handling to coarse concern categories such as `LOG_ARTIFACTS` and `BOOTSTRAP_RESET`, while `src/app.py` builds those facts from reset execution without exposing raw artifact paths or technology-specific details.
- Focused coverage now exists in `tests/unit/security/test_reset_flow.py` for runtime-known log cleanup, `tests/unit/core/test_reset_report.py` for normalized/deduplicated neutral follow-up issue assembly, and `tests/unit/gui/test_startup_dialog_controller.py` for post-reset bootstrap-state cleanup, rerun-safe log cleanup, preserved external-key-file policy, startup-only remembered-target cleanup, and sanitized caller-visible follow-up messaging.

### Closeout Result
- The delivered non-database cleanup contract is now re-verified on the current codebase.
- Runtime-known log cleanup, explicit bootstrap-selector clearing, preserved external-key-file policy, rerun-safe behavior, and sanitized neutral follow-up categories all have focused passing proof in their owning seams.
- Any broader remaining work now lives in sibling caller-outcome or wider integration stories rather than in this narrower app-owned artifact scope.

### Acceptance Criteria
- [x] The cleanup path can target runtime-known application log artifacts for the active installation context, including the configured current log file and derivable rotated log files when file logging is enabled and those artifacts exist.
- [x] External key files are not reset-managed artifacts: `Nollställ` does not delete caller-selected key files automatically, and caller-visible reset outcomes do not report missing key-file deletion as a reset failure.
- [x] Bootstrap-reset handling explicitly resolves the post-`Nollställ` `config.ini` state so stale database selectors or security bootstrap flags do not reopen contradictory startup state, while preserving only the approved safe subset of non-sensitive configuration if any is intentionally retained; reruns must treat already-safe bootstrap state as neutral rather than failed cleanup.
- [x] Focused tests verify log-target enumeration, explicit post-reset bootstrap state, safe rerun behavior for logs/bootstrap artifacts, and sanitized neutral outcomes without coupling this work to caller-mode presentation or active database-footprint deletion.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Scope only to non-database app-owned artifacts** - This story covers logs and bootstrap-reset artifacts outside the active database footprint.
- **Use runtime-known app-owned targets only** - Cleanup should act only on log files and bootstrap state the current runtime manages, not speculative external copies or user-owned files.
- **Resolve bootstrap state explicitly** - Post-reset handling must make the `config.ini` or equivalent bootstrap state safe for the next startup and aligned with `002.001`.
- **Return sanitized neutral outcomes** - Results must distinguish success, skipped, and partial-failure cases without exposing secret material or unnecessary path details to higher layers.
- **Make reruns safe** - Repeating `Nollställ` after a partial failure must not treat already-removed logs or already-safe bootstrap state as failed cleanup work.

### Negative Constraints (MUST NOT)
- **Don't absorb active database-footprint deletion** - Main database file and sidecar/journal cleanup belongs to `002.006.003`.
- **Don't redefine the backend-owned seam** - Backend-native close/lock/delete ownership belongs to `002.006.002`.
- **Don't absorb caller-mode interpretation** - Controlled-vs-emergency reporting behavior belongs to `002.006.005`.
- **Don't treat user-selected key files as app-owned reset targets** - The app must not delete arbitrary external files just because they were used during bootstrap.
- **Don't preserve stale bootstrap selectors by accident** - Post-reset config handling must not leave old database-path or security-open state active unless explicitly approved as safe.

### Soft Preferences (SHOULD)
- Prefer explicit target categories so tests can verify current log file, rotated logs, and bootstrap-reset handling separately.
- Keep logging cleanup aligned with the current config/template shape, including `log_file_path` and `log_file_backup_count` where applicable.
- Preserve only clearly safe non-database settings if config rewriting is chosen instead of full deletion.

## Implementation Notes

### Suggested Approach
1. Use the shared reset flow from `002.006.001` after immediate denial and alongside the backend-owned seam from `002.006.002`, while keeping this story focused on artifacts outside the active database footprint.
2. Enumerate runtime-known log targets from current logging/bootstrap configuration, including the configured active log file and derivable rotated files when those are part of the active installation context.
3. Define and implement the explicit post-reset `config.ini` policy: delete it, rewrite only an approved safe subset, or equivalent handling that prevents stale database bootstrap state from surviving.
4. Keep key-file handling out of the automatic cleanup path and document it as an external manual concern rather than an app-owned reset artifact.
5. Add focused tests for log enumeration and the approved post-reset bootstrap state.

### Technical Considerations
- `config.ini.template` currently includes both bootstrap-critical database selectors and non-security operational settings such as logging and application window state, so post-reset config handling should be deliberate rather than all-or-nothing by accident.
- `src/config/app_config.py` currently parses database bootstrap hints from `config.ini`, which means stale `db_type` or `db_file_path` values could influence startup if reset handling does not clear or safely rewrite them.
- Rotated logs should be treated as part of the active runtime footprint when they are derivable from the configured active log target and backup-count policy.
- Because key files may point to arbitrary user-owned files, automatic reset cleanup should not delete them even when a path is currently known.
- Retry-safe behavior should focus results on remaining non-database cleanup concerns only; artifacts already removed or state already made safe by an earlier run should be treated as absent/already-cleaned, not as fresh failures.

### Affected Components
- Non-database cleanup support under `src/security/` or adjacent reset-integration infrastructure
- Bootstrap/config helpers that determine the approved post-reset `config.ini` state
- Focused unit/integration tests for log cleanup, preserved external key files, and bootstrap-reset behavior

## Context & Background

The parent `002.006` story originally mixed database artifacts, log cleanup, key-file handling, bootstrap reset, and caller reporting in one delivery unit. Splitting those apart makes this child story the focused place to define what happens to non-database artifacts that still matter after `Nollställ`.

This story also carries the explicit bootstrap-reset decision because `002.001` established that some startup state lives in `config.ini`, and leaving those selectors untouched after reset could reopen stale or contradictory database state on next launch.
