## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

The SQLite adapter and repository skeleton provide the foundation for all database operations. Together they handle the structural responsibilities that later CRUD stories will depend on:

1. **Connection management** - SQLite adapter opens and maintains database connection
2. **Schema initialization** - SQLite adapter creates tables on first run or for tests
3. **Transaction context** - Adapter exposes transaction control for multi-step operations
4. **Repository skeleton** - Application-facing repository is created with the correct delegation boundary
5. **Row dictionary support** - SQLite adapter configures SQLite for dict-like row access

Without this skeleton, individual CRUD operations (stories 001.004-006) would each need to handle connection setup, schema initialization, execution details, and repository wiring separately, leading to duplication and unclear boundaries.

### User Value
- Database "just works" on first launch (schema auto-created)
- Tests can create fresh databases quickly (in-memory support)
- Consistent connection handling across all operations
- Clean transaction support for complex operations

## Goal (WHAT)

**What functionality are we delivering?**

Create the SQLite adapter and the first SQLite repository skeleton. This story focuses on the structural skeleton only: connection management, schema initialization, transaction control, and the repository-to-adapter delegation seam. Actual CRUD operations will be added in stories 001.004-006.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Respect approved split** - Low-level SQLite mechanics belong in `sqlite_adapter.py`, not in the repository class
- **SQLite3 only** - Use stdlib sqlite3 module
- **Row factory = sqlite3.Row** - Enable dict-like row access
- **Support :memory:** - Must work with in-memory databases for tests
- **Call schema executor** - Use schema creation from 001.001
- **Keep repository skeleton thin** - It should establish the later CRUD seam without collapsing adapter/repository responsibilities back together

### Negative Constraints (MUST NOT)
- **Don't implement CRUD operations yet** - That's stories 001.004-006
- **Don't add configuration loading** - Keep it simple for Phase 1
- **Don't add connection pooling** - Single connection is fine
- **Don't add retry logic** - Basic connection only
- **Don't add encryption** - Deferred to Epic 002
- **Don't move SQL execution responsibility back into the repository skeleton**

### Soft Preferences (SHOULD)
- Use pathlib for file path handling
- Add helpful error messages for connection failures
- Log when schema is initialized (using stdlib logging)
- Keep file/package names aligned with the approved architecture layout

## Implementation Notes

### What's Needed

1. **SQLiteAdapter Structure**
   - Purpose: Implement the low-level `DatabaseAdapter` contract for SQLite
   - Owns: Connection, schema initialization, execution/fetch helpers, transaction primitives, cleanup

2. **SQLiteAdapter Constructor**
   - Purpose: Establish database connection with proper configuration
   - Input: Database path (string, supports both file paths and `:memory:`)
   - Configuration: Set row factory to sqlite3.Row for dict-like row access
   - Stores: Connection and any needed SQLite execution state

3. **Schema Initialization**
   - Purpose: Create database tables on first use
   - Implementation: Check if key table exists before creating schema
   - Uses: Schema executor from story 001.001
   - Idempotent: Safe to call multiple times (creates only if missing)

4. **Transaction Management** (~3 methods)
   - Purpose: Support atomic multi-step operations
   - Capabilities: Begin transaction, commit changes, rollback on error
   - SQLite note: Explicit BEGIN statement recommended for clarity

5. **Connection Cleanup**
   - Purpose: Properly release database resources
   - Implementation: Close connection when done

6. **Repository Skeleton**
   - Purpose: Establish the application-facing repository class in `src/db/repositories/sqlite/`
   - Implementation: Receive or create the adapter in a way that keeps low-level SQLite mechanics delegated downward
   - CRUD methods may remain stubs/placeholders until 001.004/005/006

7. **Context Manager Support** (optional)
   - Purpose: Enable cleaner resource management for the adapter and/or repository surface
   - Implementation: `__enter__` returns self, `__exit__` handles cleanup
   - Behavior: Rollback on exception, always close connection

### Architectural Guidance

See `ai_instructions/architecture/db_architecture.md` for:
- Adapter/repository/factory responsibility boundaries
- Why SQLite execution details belong in the adapter layer

### Technical Considerations
- Check table existence before schema initialization (use sqlite_master query)
- Use relative path to locate schema SQL file
- Row factory must be set before any queries for dict-like access
- Consider whether explicit BEGIN is needed vs SQLite's implicit transactions
- Keep the repository skeleton in the approved `src/db/repositories/sqlite/` package

## Context & Background

This skeleton establishes the pattern that all CRUD operations will follow:
1. Take domain entity as input
2. Repository prepares SQL/query intent
3. Adapter executes SQL
4. Repository converts result back to domain entity
5. Return result

Stories 001.004-006 will fill in the repository-side CRUD behavior for each entity type while reusing the adapter mechanics established here.

**Why keep repository CRUD mostly deferred here**:
- This story should establish structure, not deliver entity behavior prematurely
- Later CRUD stories need a clean repository seam to extend
- Adapter mechanics can be tested independently from CRUD behavior
