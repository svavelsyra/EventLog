## Purpose (WHY) ⭐ MOST IMPORTANT

Epic `002` cannot be implemented safely if the application still treats encrypted startup as an informal convention. Before any key derivation, SQLCipher open, or GUI unlock flow can work reliably, the project needs a clear contract for what bootstrap data lives in `config.ini`, what security dependencies are explicitly allowed, and which responsibilities belong to bootstrap orchestration versus the encrypted database itself.

This story exists to define that contract in a way that prevents later security drift. Without it, later implementation stories could accidentally:
- store secret material in config or logs
- let adapter, repository, and GUI layers make conflicting bootstrap assumptions
- blur the dependency exception policy for SQLCipher and cryptographic KDF support
- mix defaults for new databases with authoritative settings for existing encrypted databases
- hardcode SQLite-specific bootstrap behavior into what should remain generic startup orchestration
- hide administrator-owned policy decisions behind hard-coded credential rules instead of making the allowed envelope explicit

### User Value
- The project gets one agreed startup/security contract before implementation spreads across multiple layers.
- Later stories can build on explicit bootstrap rules instead of inventing local assumptions.
- Reviewers can verify early that the security model stays realistic rather than pretending `config.ini` is protected.

---

## Goal (WHAT)

Define the approved bootstrap and dependency contract for encrypted database startup, including what belongs in convenience-only startup memory, what remains inside the encrypted database or SQLCipher-managed metadata, how dependency exceptions are justified for security-critical work, and where generic bootstrap stops so the selected database dialect/adapter can take over.

### Acceptance Criteria
- [x] The project documents a concrete bootstrap contract for encrypted startup covering at least `db_type`, current `database_path` naming for the remembered database target, `require_key_file`, `min_password_length`, `secure_delete_passes`, and `kdf_iterations`, including which values are remembered startup hints, which are creation-time defaults, which remain authoritative only inside the encrypted database/runtime state, and which details are current SQLite examples rather than permanent cross-backend rules.
- [x] The contract explicitly states that administrators define the allowed creation-time policy envelope for database technology and credential combinations, operators choose a concrete setup within that envelope, and the created database becomes the authoritative source for its own effective rules afterward.
- [x] The approved dependency exception for SQLCipher support and PBKDF2/KDF support is recorded clearly enough that later implementation stories can update `requirements.txt` without reopening the policy question.
- [x] The contract explicitly forbids storing passwords, derived keys, raw key-file content, or remembered key-file paths in configuration, logs, or database tables.
- [x] The documented startup contract makes clear that `config.ini` is convenience-only startup memory: missing or malformed remembered bootstrap values must not prevent application startup, and recovery must fall back to a create/select database flow where the user resolves the technology first and then sees technology-specific fields.
- [x] The documented contract distinguishes what is allowed versus what is merely recommended: the application may guide toward stronger choices, but enforcement must come from the configured/admin-defined rule set and backend support rather than from hidden hard-coded UI assumptions.
- [x] Delivery includes focused validation coverage for config parsing/default behavior and keeps the security/bootstrap documentation aligned with the current adapter/factory architecture.

### Current Delivery Snapshot
- `config.ini.template` now documents the concrete bootstrap contract in the current config shape: `db_type` in `[DEFAULT]`, remembered `database_path` and `require_key_file` in `[sqlite]`, and creation-time defaults such as `require_key_file_for_creation`, `min_password_length`, `secure_delete_passes`, and `kdf_iterations` in convenience config outside the encrypted database.
- `src/config/app_config.py` now encodes the same contract structurally through `BootstrapTargetConfig`, `DatabaseCreationDefaults`, and `DatabaseConfig`, keeping remembered startup hints separate from creation-time defaults and explicitly documenting that existing encrypted databases remain authoritative for their own effective runtime security values.
- The current config contract now rejects older naming/section assumptions instead of silently reviving them: the parser reads the current `database_path` option, does not reuse the removed `db_file_path` name, and ignores the older `[Database]` / `[Security]` section split.
- The current parsing path is recovery-first rather than startup-blocking: missing files return no config, malformed boolean/integer values fall back to safe defaults with warnings, and partial remembered bootstrap state is preserved so higher layers can drop into create/select recovery instead of treating convenience memory as mandatory authority.
- The contract now explicitly forbids secrets in convenience storage. `config.ini.template` states that passwords, derived keys, raw key-file content, and remembered key-file paths must never be stored in config, logs, or database tables, and focused serializer tests confirm remembered-target persistence does not introduce `key_file_path` into config state.
- The approved dependency exception is now recorded in `docs/DEPENDENCY_PHILOSOPHY.md`: `sqlcipher3` is documented as the approved application dependency for encrypted SQLite support, while PBKDF2 key derivation stays on stdlib `hashlib.pbkdf2_hmac`.
- Focused coverage now exists in `tests/unit/db/test_repository_factory.py` and `tests/unit/db/test_bootstrap_target_serializer.py` for config parsing defaults, malformed-value recovery, remembered-target persistence/clearing, preservation of unrelated sections, separation of remembered unlock hints from create-time policy, and the current backend/factory responsibility boundary.
- The allowed-vs-recommended policy boundary is now explicit in the durable contract itself: `config.ini.template` treats `require_key_file_for_creation` and `min_password_length` as administrator-owned creation defaults, while `docs/design/root_design.md` states that low-level helpers may reject invalid or abusive inputs but must not silently enforce narrow recommended ranges where policy belongs to the caller/administrator. In other words, weak-but-allowed setups may be guided against without becoming hidden universal blocks.
- With that wording now explicit, this story can be treated as complete for the current Epic `002` scope rather than left open for a purely documentary nuance.

### Closeout Note
- No further story-local gap is currently known in the bootstrap/dependency contract itself. Later stories may still build on this contract, but they no longer need this story to stay open for the allowed-versus-recommended policy wording.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep bootstrap minimal** - Only values required before encrypted open, defaults for new databases, or non-security operational conveniences belong outside the encrypted database.
- **Stay realistic about local files** - `config.ini` and Python source are readable/modifiable by anyone with device access, so they cannot be treated as secret or tamper-proof.
- **Respect the current architecture** - Bootstrap/factory code owns orchestration and profile resolution; adapter code owns backend-specific readiness after construction.
- **Follow the dependency philosophy** - Third-party dependencies remain exceptions, and the justification must be explicit and security-focused.
- **Keep bootstrap backend-agnostic at the top level** - Startup should read only enough shared config to resolve the requested database target/profile, then let the selected adapter/dialect read and validate the rest of what it needs.
- **Keep startup recovery-capable** - Missing or malformed remembered bootstrap memory may block automatic reuse of the last target, but must not block the application from reaching a usable startup/create/select UI.
- **Make policy ownership explicit** - Admin-defined policy determines what can be created next; already-created databases own their effective rules afterward.

### Negative Constraints (MUST NOT)
- **Don't add actual encryption code yet** - This story defines the contract; helper and adapter implementation belong to later `002.x` stories.
- **Don't introduce general authentication or account management** - Epic `002` protects the database, not application identity.
- **Don't store secret material outside the encrypted database** - No remembered key paths, passwords, or derived keys.
- **Don't treat config defaults as authoritative for already-created databases** - Existing encrypted databases must not silently inherit changed defaults from a later `config.ini` edit.
- **Don't hardcode SQLite-only assumptions into the generic bootstrap layer** - File-existence checks, SQLite header rules, and SQLCipher-specific open behavior belong to the SQLite path, not to the long-term bootstrap architecture.
- **Don't let convenience memory become startup authority** - Remembered config may prefill or speed up startup, but must not be the only route to choosing or recovering a database target.
- **Don't hardcode one credential workflow as universal** - No-password/no-key-file, password-only, key-file-only, and combined modes may all be valid if allowed by admin policy and backend support.

### Soft Preferences (SHOULD)
- Prefer short, explicit bootstrap rules over large narrative docs.
- Keep naming and section boundaries aligned with `security_architecture.md`, `security_design.md`, and `db_architecture.md`.
- Make the dependency exception easy to reference from later implementation stories.

### Technical Considerations
- `require_key_file` is bootstrap information for the current database instance, not a general user preference.
- Generic bootstrap should read enough config to decide the backend/dialect (`db_type` today, potentially richer target/profile input later), then hand off to the selected adapter path for backend-specific readiness.
- SQLite is the only supported backend today, but the contract should leave room for future technologies that may need different bootstrap fields such as URLs, provider names, or backend-specific authentication settings.
- `kdf_iterations` in config is a creation-time default; authoritative iteration settings belong with the encrypted database/SQLCipher state.
- Startup is technology-first: remembered config may prefill the choice, but the startup UI must resolve the backend first and then show the fields relevant to that technology.
- Current SQLite/file-path/key-file behavior must be documented as a current implementation example, not as the permanent generic startup rule.
- The contract must stay compatible with the single-database security model.
- Allowed credential combinations are policy-driven: depending on admin-defined creation policy and backend support, creation may allow no credentials, password only, key file only, or password + key file.
- The create flow should distinguish between what is allowed and what is recommended; weak-but-allowed setups may warn without being blocked.
- Creation-time policy inputs in `config.ini` may shape what can be created on the current machine, but once a database exists, later local config changes are not authoritative for that database.

## Context & Background

Epic `002` is the point where EventLog stops being a plaintext SQLite application and starts requiring a secure bootstrap story. That creates a special planning need: some startup memory may help the UI recover the last-used target, but most security-sensitive values must never live outside the encrypted boundary and startup must still remain usable when that memory is missing or stale.

At the same time, this story should not accidentally freeze the architecture into a permanent SQLite-only bootstrap model. Current implementation work is still SQLite/SQLCipher-specific, but the bootstrap contract should reflect the database architecture rule that startup/profile resolution is generic while deeper readiness and backend-specific config handling belong to the selected adapter/dialect path.

The practical consequence is that `config.ini` is convenience-only startup memory, not authority: it may prefill a remembered technology/target, but malformed remembered values must degrade into a recovery-capable create/select flow instead of blocking startup.

This story also now needs to protect a second planning boundary: the application is meant to guide rather than dictate. Administrators define which technologies and credential combinations are allowed for new database creation, operators choose within that allowed envelope, and the created database then owns its effective rules afterward. That keeps the tool realistic for both stricter deployments and simplified exercise/training setups without pretending local config can enforce security after the fact.

This story handles that planning boundary first so later stories can focus on implementation details instead of reopening the question of what belongs in config, in SQLCipher metadata, or nowhere at all.
