## Purpose (WHY) ⭐ MOST IMPORTANT

`Nollställ` cannot be trusted as an emergency-denial capability unless the application first stops access quickly and consistently before it starts any slower best-effort deletion work. That immediate denial step is also the part that must stay shared across all reset entry points, because controlled cleanup and emergency cleanup differ mainly in caller priorities, not in the need to invalidate active access.

This story exists to carve out that shared foundation so later cleanup stories do not each invent their own reset sequence, ordering, or result shape.

Without this story, later implementation could drift into unsafe behavior such as:
- deleting some files before the running process has actually stopped normal database use
- letting startup and in-app reset triggers build parallel reset flows that diverge over time
- mixing caller-facing reporting rules into low-level denial sequencing
- making later backend cleanup work guess when access was truly blocked

### User Value
- Operators get a reset flow that prioritizes immediate denial of access before slower cleanup attempts.
- Later `Nollställ` entry points can share one coordinated reset contract instead of duplicating sequence logic.
- The rest of the cleanup work can plug into a stable top-level flow without redefining ordering each time.

## Goal (WHAT)

Define the shared top-level `Nollställ` reset flow for the active secured database context, including immediate access denial as the first phase and a neutral structured outcome contract that later cleanup stories can build on.

### Acceptance Criteria
- [x] A shared reset coordinator/service contract exists for `Nollställ` so callers use one underlying reset flow for the active database context instead of inventing trigger-specific cleanup sequences.
- [x] The first phase of that flow is immediate access denial: the shared flow attempts the configured denial callback before any slower cleanup work, and entry-point/runtime integration can clear sensitive state plus block or invalidate normal repository/database use as quickly as the active technology allows before slower artifact cleanup begins.
- [x] The shared flow exposes a structured, caller-safe reset outcome contract that records at least whether immediate denial succeeded, whether later cleanup phases were started/completed, and sanitized failure categories without exposing secret material or overly specific attack-helpful details.
- [x] Focused tests verify that immediate denial happens before slower cleanup work, that the flow behaves safely when no active database context is loaded, and that repeated or interrupted invocation does not reopen normal access by accident.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep one shared reset flow** - Controlled and emergency scenarios must share the same core reset sequence.
- **Prioritize denial before destruction** - Immediate access denial must happen before slower best-effort artifact cleanup.
- **Scope to the active database context only** - Epic `002` still assumes one active secured database instance at a time.
- **Return structured neutral outcomes** - The result contract must be reusable by later caller-mode handling without forcing UI wording into this story.
- **Stay technology-aware at the boundary** - The coordinator may depend on backend-owned close/lock behavior, but it must not hardcode all cleanup mechanics itself.

### Negative Constraints (MUST NOT)
- **Don't absorb trigger-specific UI behavior** - Startup dialog layout, toolbar placement, and confirmation UX belong outside this story.
- **Don't implement full artifact cleanup here** - Database files, sidecars, logs, key files, and bootstrap artifacts are broken into later child stories.
- **Don't leak secrets through results** - Structured outcomes must avoid credential material, precise key-file paths when unnecessary, or detailed attack-oracle failure reporting.
- **Don't split controlled vs emergency into separate reset implementations** - Mode differences belong above or alongside the shared flow, not as duplicate sequence logic.

### Soft Preferences (SHOULD)
- Prefer explicit phase names so later cleanup stories can plug into the flow predictably.
- Keep the top-level reset contract small enough that future entry points can reuse it without presenter-specific assumptions.
- Make safe shutdown or restart recommendations available in the neutral outcome contract when relevant, without deciding caller presentation here.

## Implementation Notes

### Suggested Approach
1. Define a reset coordinator/service for `Nollställ` in security/bootstrap-facing infrastructure rather than inside a presenter.
2. Model the flow in explicit phases, with immediate access denial first and later cleanup phases invoked only after denial is attempted.
3. Define a neutral result object or equivalent contract that can accumulate phase outcomes without embedding scenario-specific UI wording.
4. Add focused tests around sequencing, missing active context, and repeated/interrupted invocation behavior before broader cleanup stories attach more artifact-specific work.

### Technical Considerations
- This story is the sequencing foundation for the rest of the `002.006` split.
- The coordinator should be able to call into later backend-owned cleanup work without knowing every artifact type itself.
- Immediate denial may include invalidating active repository/session access, entry-point-specific sensitive-state clearing, and closing handles, but the shared coordinator itself should not overclaim ownership of all caller/runtime cleanup behavior or memory-forensics guarantees.
- The result contract should be neutral enough that controlled and emergency caller handling can differ later without changing the underlying flow.

### Affected Components
- New or updated reset-flow/coordinator support under `src/security/` or adjacent bootstrap/security infrastructure
- Runtime/bootstrap integration points that trigger the shared reset flow
- Focused unit/integration tests for sequencing and safe denial behavior

## Context & Background

The parent `002.006` story became too broad because it mixed sequencing, backend ownership, artifact classes, bootstrap-reset handling, and caller-mode reporting into one delivery unit.

This child story intentionally isolates the first thing that must be true for every later cleanup path: once `Nollställ` is triggered, the application should deny continued normal access as quickly as possible and do so through one shared flow.
