## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

The database schema is the foundation of all data persistence in EventLog. Without a properly designed schema, we cannot store communication logs, event records, or personnel tracking data. The schema must be created programmatically so that:

1. **New installations work out-of-the-box** - First-time users get a working database automatically
2. **Tests can create fresh databases** - In-memory test databases need schema creation
3. **Schema is version-controlled** - SQL files in source control ensure consistency
4. **Future migrations have a baseline** - Initial schema is the starting point for all future changes

### User Value
- Users don't need to manually set up databases
- Application "just works" on first launch
- Developers can test against consistent schema
- Database structure is documented and reviewable

Create the initial database schema SQL script and Python helper code to execute it. This includes the three core entity tables that Epic 001 focuses on plus the minimal `settings` table needed for repository/runtime configuration.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **SQLite3 syntax only** - Must work with Python sqlite3 module
- **Core entity tables plus required runtime settings** - communication_entries, event_entries, personnel_entries, and the minimal `settings` support required by the current design
- **Match design exactly** - All fields from `db_design.md` must be included
- **ISO 8601 for datetime** - Store as TEXT in format "YYYY-MM-DDTHH:MM:SS"
- **Boolean as INTEGER** - 0=false, 1=true with CHECK constraints
- **JSON as TEXT** - system_capabilities stored as valid JSON TEXT

### Negative Constraints (MUST NOT)
- **Don't create structured_reports table yet** - Deferred to Epic 004
- **Don't create broader configuration tables yet** - Deferred to Epic 003 (communication_systems, channel_designations, etc.)
- **Don't create file_attachments table yet** - File support deferred
- **Don't create user_preferences table yet** - Settings deferred
- **Don't add foreign keys yet** - Only self-contained entity tables for now
- **Don't add full-text search** - Basic schema only

### Soft Preferences (SHOULD)
- Clear comments explaining field purposes
- Grouped field definitions (identity, content, metadata)
- Consistent formatting and indentation
- Index names follow convention: `idx_<table>_<field>`

## Implementation Notes

### Suggested Approach

**File structure**:
```
src/db/schema/
├── sqlite/
│   └── initial_schema.sql
└── schema_executor.py
```

**SQL File Format**:
```sql
-- EventLog Database Schema
-- Database Version: 0.1.0 (Epic 001 - Core entities only)
-- Last Updated: 2026-04-23
--
-- This is the INITIAL schema for NEW databases.
-- New installations run this file and skip all migrations.
-- When cleaning up old migrations, this file is updated to remain current.
--
-- Note: db_version and app_version are separate.
-- Both start at 0.1.0 for initial development.

-- ========== COMMUNICATION ENTRIES ==========
CREATE TABLE IF NOT EXISTS communication_entries (
    -- Identity & Timestamps
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    event_time TEXT NULL,           -- When communication occurred (ISO 8601)
    logged_time TEXT NOT NULL,      -- When logged into system (ISO 8601)
    
    -- Communication Content
    message_content TEXT NOT NULL CHECK (message_content != ''),
    from_field TEXT NULL,
    to_field TEXT NULL,
    
    -- ... etc
);

-- Indexes for communication_entries
CREATE INDEX IF NOT EXISTS idx_comm_event_time ON communication_entries(event_time);
CREATE INDEX IF NOT EXISTS idx_comm_logged_time ON communication_entries(logged_time);
-- ... etc
```

**Note**: Use `IF NOT EXISTS` to make schema idempotent (safe to run multiple times, useful for testing).

**Python Helper**:
```python
import sqlite3

def execute_schema_file(connection: sqlite3.Connection, sql_file_path: str) -> None:
    """Execute SQL schema file against database connection."""
    with open(sql_file_path, 'r', encoding='utf-8') as f:
        schema_sql = f.read()
    connection.executescript(schema_sql)
    connection.commit()
```

### Technical Considerations
- Use `executescript()` to run multiple SQL statements
- Handle comments in SQL (sqlite3 supports SQL comments)
- Ensure path handling works across platforms (use pathlib)
- Consider adding schema version comment in SQL file

### Affected Components
- **New files**:
  - `src/db/schema/sqlite/initial_schema.sql`
  - `src/db/schema/schema_executor.py`
- **No existing files modified** (this is the first database story)

**Design decisions already made**:
- Three specialized tables (not one generic table) - from architecture docs
- Field names and types - from `db_design.md`
- Index strategy - from `db_design.md`

### Initial Schema vs Migrations Philosophy

**Critical distinction**:
- **initial_schema.sql** - Complete schema for NEW databases (never runs migrations)
- **migrations/** - Incremental changes for EXISTING databases (upgrade path)

**New database initialization**:
1. Check if database is empty (no tables)
2. Run `initial_schema.sql` (complete current schema)
3. Ensure required default settings rows exist
4. Set `db_version` to current version
5. Skip all migrations (new DB has everything already)

**Existing database upgrade**:
1. Check current `db_version`
2. Run migrations newer than current version
3. Update `db_version` after each migration

**Migration cleanup strategy**:
- When no users have version < X, delete old migrations < X
- Update `initial_schema.sql` to include changes from deleted migrations
- Initial schema always represents "complete current schema for new installations"
- Never have to carry migration files forever
- Example: When all users reach 0.5.0, can delete migrations 001-020, update initial_schema.sql

**Versioning note**:
- **app_version** - Application code version (e.g., 0.1.0, 0.2.0, ..., 1.0.0 when production-ready)
- **db_version** - Database schema version (e.g., 0.1.0, 0.2.0, can differ from app version)
- Both start at 0.1.0 for initial development
- Version 1.0.0 reserved for actual production release

**Why .sql file instead of Python code**:
- Easier to review and diff (it's just SQL, not Python strings)
- Cleaner separation of schema definition from execution code
- Easier to update when cleaning up old migrations
- Version control friendly
