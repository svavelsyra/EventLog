## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

Operators need to retrieve and filter historical communications for operational awareness and record-keeping. Read operations enable:

1. **Retrieve specific communications** - Look up entry by ID
2. **List all communications** - View complete communication log
3. **Filter communications** - Find entries by operator, system, method, date range, etc.
4. **Build UI views** - Communication tab can display filtered lists

Without read operations, logged communications are write-only - operators cannot view what they've logged.

### User Value
- View communication history
- Filter by operator, system, method, date
- Find specific communications quickly
- Build functional GUI views

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Implement in the SQLite repository layer** - Replace placeholder behavior using the approved repository package layout
- **Delegate low-level query execution downward** - Repository handles filtering/query intent and mapping; adapter handles low-level execution/fetch behavior
- **Use parameterized queries** - Prevent SQL injection
- **Reuse row mapper** - Use conversion helper from 001.004.001
- **Sort by event time DESC** - Most recent first
- **Return empty list** - Not None, if no matches

### Negative Constraints (MUST NOT)
- **Don't implement update/delete yet** - That's story 001.004.003
- **Don't implement search yet** - That's story 001.004.003
- **Don't add full-text search indexes** - Basic queries only
- **Don't implement pagination** - Not needed for Phase 1

### Soft Preferences (SHOULD)
- Build SQL query dynamically based on filters present
- Clear variable names for query components

## Context & Background

Read operations reuse the row-to-entity conversion helper from story 001.004.001, demonstrating the value of separating the mapper.

Under the approved architecture, the repository prepares the query and maps results, while the adapter handles the low-level fetch behavior.

**Filter pattern**:
- Filter dict keys are field names
- Filter values are exact matches (or date range bounds)
- Multiple filters combine with AND logic
- No filters = get all entries

This pattern will be reused for EventEntry and PersonnelEntry.
