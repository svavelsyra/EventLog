## Purpose (WHY) ⭐ MOST IMPORTANT

Once the portability contract is defined, operators still need a real export path that can capture approved communication configuration into a stable bundle before reset, redeployment, or migration. Without that export step, the portability boundary remains theoretical.

### User Value
- Communication configuration can actually be preserved before `Nollställ` or machine replacement.
- The project gets one stable export path instead of ad hoc data dumping.
- Export behavior stays reviewable independently from import/apply complexity.

---

## Goal (WHAT)

Implement communication configuration export into the approved portable bundle shape.

### Acceptance Criteria
- [x] Export support exists to serialize the allowlisted communication runtime metadata into the approved portable bundle contract from `003.006.001`.
- [x] Export behavior reads communication systems, configured options, qualifier definitions/behavior, and other allowlisted communication metadata through approved repository/runtime seams rather than direct ad hoc SQL scattered across callers.
- [x] Export output is deterministic enough for review/testing, including stable ordering where relevant.
- [x] Focused tests verify that export includes the approved communication metadata and excludes out-of-scope data.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Implement the approved contract, not a new one** - Export must follow `003.006.001`.
- **Use approved read seams** - Export should consume repository/runtime data rather than inventing separate table-reading logic where possible.
- **Keep output communication-only** - Event metadata, preferences, secrets, and records remain out of scope.

### Negative Constraints (MUST NOT)
- **Do not implement import/apply here** - This story is export only.
- **Do not widen the allowlist opportunistically** - Exporting extra data because it is convenient is out of scope.
