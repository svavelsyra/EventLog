## Purpose (WHY) ⭐ MOST IMPORTANT

The encrypted database cannot be opened safely until EventLog has a deterministic, well-bounded way to turn operator credentials into the exact key material expected by SQLCipher. That logic is security-critical and easy to get wrong if it is improvised inside GUI callbacks or repository construction code.

This story exists to isolate the credential-processing layer into focused helpers that can be tested independently. It gives later startup and adapter stories a stable contract for:
- validating password input
- validating that an optional key file is usable
- deriving a consistent encryption key for whatever credential combinations are allowed by admin policy and supported by the selected backend
- surfacing clear errors without leaking secrets into logs or configuration
- avoiding accidental leakage about which part of the credential material was wrong when unlock fails

### User Value
- Operators get predictable unlock behavior across stricter deployments and simplified exercise/training modes.
- The project can test credential handling directly instead of only through full GUI startup flows.
- Later stories can depend on one trusted helper layer instead of duplicating crypto-related logic.

## Goal (WHAT)

Implement the security helper functions needed to validate password/key-file input and derive backend-ready encryption key material according to the approved Epic `002` design, with clear separation between creation-time policy inputs, existing-database authoritative inputs, and backend-owned credential rules.

### Acceptance Criteria
- [x] A security helper module exists for password validation, key-file validation/opening, and encryption-key derivation, with stable typed entry points for later bootstrap and GUI stories.
- [x] Password validation consumes administrator-controlled minimum-length policy rather than hard-coding one universal password rule, and key-file validation enforces generic file usability plus the documented abuse/performance bounds before derivation proceeds.
- [x] Key derivation implements the approved PBKDF2-HMAC-SHA256 primitive contract, returns deterministic bytes for the same exact inputs, and accepts caller/backend-supplied salt, iteration, and output-length inputs from approved non-UI sources such as creation-time policy defaults or authoritative existing-database metadata.
- [x] The helper contract is compatible with policy-driven credential combinations where backend support exists. In the current SQLite/SQLCipher scope, that explicitly includes password-only, key-file-only, and password+key-file encrypted modes, while no-password/no-key-file is deliberately not treated as a supported encrypted helper mode because it produces no backend-ready key material.
- [x] Delivery includes focused tests for same-input consistency, different-input divergence, caller-visible failure cases such as missing files, unreadable files, and invalid iteration settings, and confirms that helper/failure contracts do not require the unlock flow to reveal whether password, key file, or iteration inputs were the specific cause of failure.

### Current Delivery Snapshot
- `src/security/helpers.py` now provides the shared credential-helper surface for this story through `validate_password(...)`, `load_key_file_bytes(...)`, and `derive_encryption_key(...)`, and those typed entry points are re-exported from `src/security/__init__.py` for later startup and GUI callers.
- Password validation now consumes caller/admin-supplied minimum-length policy instead of hard-coding one universal rule, including support for empty passwords when policy explicitly permits them.
- Generic key-file validation/opening now covers the expected shared concerns before derivation proceeds: existence, file-vs-directory checks, stat/read failures, lower-bound and upper-bound size checks, and caller-visible validation failures that later layers can map without opening the database.
- The shared derivation primitive now uses PBKDF2-HMAC-SHA256 with caller-supplied salt, iteration count, and output length. It remains deterministic for the same exact inputs and rejects structurally invalid salt/iteration/length values through explicit helper errors.
- SQLite-specific KDF rules now stay in the backend-owned wrapper `src/db/sqlite_key_preparer.py`, where password-only mode uses the SQLite-owned default salt and key-file mode hashes the supplied key-file bytes before calling the shared derivation helper. That split keeps backend policy out of the generic security helper layer.
- Focused coverage now exists in `tests/unit/security/test_security_helpers.py` for policy-driven password validation, key-file validation/opening failures, deterministic PBKDF2 derivation, divergent-input behavior, invalid iteration/output settings, and generic higher-layer invalid-credential handling. `tests/unit/db/test_sqlite_key_preparer.py` now covers password-only, password+key-file, and key-file-only SQLite-owned preparation behavior.
- The current no-password/no-key-file path is now explicitly understood as outside the encrypted helper contract for the current backend scope: `src/db/repositories/startup_bootstrap.py` returns no prepared key when both inputs are absent, and `src/db/sqlite_adapter.py` treats `encryption_key=None` as plaintext mode rather than encrypted startup. That means the story should not keep pretending this combination is an unproven encrypted mode for current SQLite/SQLCipher behavior.

### Closeout Note
- No further story-local gap is currently known in the shared helper layer for the current Epic `002` scope. The important closeout clarification is that policy flexibility still exists at the helper boundary, but the current SQLite/SQLCipher encrypted path does not support a no-password/no-key-file mode because that would yield no encryption key at all.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Use approved cryptographic primitives** - Do not implement custom cryptography; use approved stdlib or explicitly approved dependency-backed primitives.
- **Keep helper APIs explicit** - Later callers should not need GUI widgets, repository objects, or global state to use these helpers.
- **Support policy-driven credential combinations** - The helper layer must not hard-code one universal credential workflow if admin policy and backend support allow alternatives.
- **Fail clearly** - Error conditions must be surfaced in a way later bootstrap/UI code can map to user-visible messages.
- **Respect authoritative iteration sources** - Creation may use approved defaults, but existing-database unlock should use the authoritative stored/backend-provided iteration value automatically when available instead of treating iterations as operator-entered data.

### Negative Constraints (MUST NOT)
- **Don't open the database here** - SQLCipher connection/readiness belongs to `002.003`.
- **Don't persist secrets or remembered key paths** - Helpers must not write password, file content, derived keys, or key-file paths to config/logs.
- **Don't turn this into a full startup flow** - GUI orchestration belongs to `002.004` and `002.005`.
- **Don't silently inject backend-owned policy into generic helpers** - Salt rules, output-length rules, and credential-combination rules that exist only because one backend needs them belong with that backend.
- **Don't expose credential-component mismatch details in normal unlock behavior** - Later callers should not need to tell the user whether password, key file, or iteration selection was specifically wrong.

### Soft Preferences (SHOULD)
- Keep the helper module under a dedicated security package such as `src/security/`.
- Prefer pure function boundaries where practical to make testing easier.
- Reuse approved Swedish error-message wording where user-facing validation text is already designed.

## Implementation Notes

### Suggested Approach
1. Create a focused security helper module for password validation, key-file validation/opening, and derivation.
2. Keep the file-validation helper responsible for existence, readability, type, and size checks.
3. Keep derivation separate from UI and backend-specific wrappers so startup/bootstrap code can call the shared primitive through explicit inputs.
4. Add unit tests for deterministic output and failure cases before wiring the helpers into the factory/UI layers.

### Technical Considerations
- The shared primitive uses PBKDF2-HMAC-SHA256, while backend-owned wrappers decide salt rules and requested output length.
- For existing databases, iteration count should come from the authoritative backend/database metadata when available; it should not be a manual unlock field.
- SQLite currently uses a backend-owned default salt for password-only mode and a hash of key-file contents for key-file mode, but the generic helper contract should not hard-code those rules as universal.
- Unicode passwords should be handled consistently through UTF-8 encoding.
- Callers may need readable validation errors without exposing sensitive file details in logs.
- Empty password may be allowed when admin policy and backend support allow a credential combination that does not require password material.
- Key-file helpers should distinguish between recommended lower bounds and hard abuse/performance caps rather than turning every recommendation into a protocol rule.
- Helper contracts should support generic invalid-credentials handling at higher layers rather than encouraging separate user-facing errors for password-vs-key-file-vs-iteration mismatch.

### Affected Components
- New security helper module(s) under `src/security/`
- Unit tests under `tests/unit/security/`
- Later startup/factory and GUI stories that consume these helpers

## Context & Background

Epic `002` deliberately separates credential processing from encrypted database open. That separation matters because a wrong password, a missing key file, and a wrong SQLCipher key are related but not identical failure points. If all of that logic is mixed together too early, later debugging and testing become much harder.

This story should also avoid creating an accidental information leak: even if different credential inputs contribute to key derivation, the normal unlock experience should not become a diagnostic oracle that tells an attacker which piece was wrong.

The story now also needs to respect the newer policy philosophy: the application should guide, administrators define what credential combinations are allowed for creation, operators choose within that envelope, and the generic helper layer should support that flexibility without pretending one fixed credential workflow is always correct.

This story gives the project the smallest reusable crypto-adjacent helper layer needed before encrypted adapter readiness is implemented.

## Testing Strategy

### Key Test Scenarios
- [x] Same password and same key file produce the same derived key every time.
- [x] Different passwords or different key files produce different derived keys.
- [x] Password-only derivation remains deterministic for the same input.
- [x] Empty-password and no-password combinations behave according to admin policy and backend support rather than hidden universal assumptions; in the current SQLite/SQLCipher encrypted scope, empty-password key-file-only mode is supported, while no-password/no-key-file is explicitly not treated as encrypted startup behavior.
- [x] Invalid password length, missing files, unreadable files, directories, and structurally invalid iteration/output settings fail clearly.
- [x] Existing-database unlock paths can consume authoritative iteration values automatically without requiring a manual iteration input from the operator.
- [x] Higher-layer failure handling can remain generic for invalid credential material instead of exposing which credential component was wrong.

### Integration Points
- `002.003` consumes the derived key for SQLCipher adapter readiness.
- `002.004` uses these helpers during startup/bootstrap orchestration.
- `002.005` uses the validation helpers to drive create/unlock dialog behavior.
