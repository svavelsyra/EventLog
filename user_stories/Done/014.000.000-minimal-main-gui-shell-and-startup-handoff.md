## Purpose (WHY) ⭐ MOST IMPORTANT

EventLog now has substantial startup, bootstrap, repository, reset, and runtime-configuration foundation work, but it still stops before a real operator-facing application shell exists. That creates a practical gap: it is difficult to visualize the real application, difficult to judge whether the current architecture feels right in actual use, and difficult to start plugging later workflows into a production-shaped GUI instead of continuing to reason only from documents and isolated foundations.

This epic exists to deliver the smallest real main-window shell that can host later workflows without pretending to finish them early. The purpose is to ensure that EventLog can:
- own one persistent Tk application root at the app-shell level instead of treating the startup dialog as the whole lifetime of the GUI
- run the existing startup flow as a child/modal step and then continue into a real main window on success
- provide a production-owned place to attach later communication, event, and personnel workflows
- expose the real top-level layout early enough that UX, screen-space, and ownership problems are discovered before the later tab epics build on the wrong shell
- keep the first GUI shell small enough that it validates structure without dragging in the full scope of the later tab/workflow epics

The deeper reason is architectural honesty. A minimal real shell gives the project a truthful GUI seam to build on. Without it, later tab epics risk either inventing their own shell assumptions or being delayed by top-level GUI questions that should be answered once, centrally, and early.

### Success Criteria
- A real production-owned main GUI shell exists after successful startup, using one persistent app-shell-owned Tk root
- The startup dialog remains usable, but it is no longer the long-term owner of GUI lifetime once the main shell exists
- The main shell exposes the top-level layout needed by later GUI epics without claiming to deliver the actual tab workflows yet
- Epic 014 stays small enough to review as shell/lifecycle work rather than expanding into communication, event, personnel, preferences, or polish scope
- The epic is clear enough to split into a few small delivery stories rather than one large GUI implementation block

## Goal (WHAT)

Define and deliver the minimal production main-window shell for EventLog, including startup-to-shell handoff and the smallest persistent top-level GUI structure needed for later work.

This epic covers the end-to-end contract for:
1. app-shell ownership of `tk.Tk()` and overall GUI lifetime
2. launching the existing startup dialog from that shell as a startup step rather than a standalone app lifetime
3. transitioning from successful startup into a real main application window
4. rendering the minimal top-level main-window structure such as toolbar, tab container, and status area in a production-owned seam
5. keeping reset/close/active-context lifecycle behavior compatible with the existing startup/bootstrap cleanup contracts

### Key Deliverables
 - [x] App-shell/root-lifetime contract for startup dialog handoff into the real main window
 - [x] Production main-window shell seam under `src/gui/` with a minimal toolbar, notebook/tab container, and status area
 - [x] Placeholder or shell-level content for Communication, Event, and Personnel tabs that proves layout ownership without claiming workflow completion
 - [x] Explicit integration boundary between the shell and the already-existing startup success result (`repository`, invalidation, cleanup, remembered target)
 - [x] Reset/close/exit behavior at the shell level that reuses the existing active-context cleanup contracts instead of inventing separate GUI cleanup logic
- [x] Clear separation between this shell epic and the later feature epics that deliver actual tab workflows and UI-state persistence

## Limitations (CONSTRAINTS & REQUIREMENTS)

**What boundaries must we work within? What must we avoid?**

### Hard Constraints (MUST)
- **Keep the epic shell-focused** - Epic 014 must deliver the top-level main-window shell and startup handoff only, not full communication/event/personnel workflows
- **Move root ownership upward deliberately** - Once a real shell exists, the persistent Tk root should be owned by the app shell/main-window lifetime rather than remaining hidden inside the startup-only controller lifecycle
- **Preserve the existing startup contract where practical** - The current startup result shape and bootstrap/reset contracts should remain the integration seam unless there is a strong reason to change them
- **Keep GUI responsibilities layered** - Top-level lifecycle orchestration belongs in the app/shell layer, while views remain widget-focused and later presenters own tab/workflow logic
- **Provide a stable host for later epics** - The shell must expose a clean top-level structure that later Epics 006, 007, and 008 can plug into without redefining the app lifetime model
- **Support the documented compact tabbed application shape** - The shell should align with the documented top-level GUI architecture and design for toolbar, tab container, and status area
- **Test the shell as delivered** - Each later `014.x` story should include focused tests for lifecycle and GUI seams rather than pushing verification into a vague later testing task

### Negative Constraints (MUST NOT)
- **Do not trust the demo as architecture** - `Demo/demo_app.py` may provide loose visual inspiration, but it must not define production ownership, contracts, or implementation structure
- **Do not absorb full tab workflows** - Create/edit/filter/detail behavior for Communication, Event, and Personnel remain in Epics 006, 007, and 008
- **Do not absorb preference persistence** - Operator context, window-state persistence, column configuration, and other non-secret UI-state storage remain under Epic 005
- **Do not require polished final UX before the shell exists** - This epic should prove the real shell shape first, not block on every later UX refinement
- **Do not bypass existing reset/bootstrap contracts with GUI-local shortcuts** - Shell reset/close behavior must reuse the established invalidation and backend-cleanup seams
- **Do not let placeholder tabs become disguised workflow implementations** - Placeholder content should validate ownership and navigation only, not quietly grow into half-finished feature tabs

### Soft Constraints (SHOULD)
- Prefer a very small story split, likely three stories or fewer
- Keep the first visible shell read-only or placeholder-based unless a later approved story deliberately widens scope
- Keep operator-facing wording and top-level terminology aligned with the GUI design documents
- Favor shell structure that makes later status-bar logging and operator-context integration easy without making them prerequisites now

## Context & Background

### Why This Epic Exists Before the Tab Workflow Epics
The currently planned tab epics already assume that a real main application window exists. In practice, however, the current application entry point still ends after the startup dialog. That means there is still a missing architectural slice between successful bootstrap and the later workflow epics.

Epic 014 fills that gap by creating the real application shell first, so later GUI epics can focus on their own tab workflows instead of simultaneously deciding:
- who owns `tk.Tk()`
- how startup transitions into the main window
- where toolbar/status-area concerns live
- how app-level reset/exit behavior interacts with the active repository context

### Why This Epic Must Stay Small
This is not the place to deliver real communication/event/personnel behavior. Those already have their own epic boundaries. The value of Epic 014 is precisely that it solves the app-shell question once without reopening all later feature scope at the same time.

### Relationship to the Existing Startup Dialog
The startup dialog is already real and useful. This epic should not throw it away. Instead, it should re-home that existing startup flow under a persistent shell-owned root lifetime so the dialog becomes the opening step of the application rather than the whole application.

### Trust Boundary for the Demo
The project has an early demonstration app that was useful for showing concept and direction, but it is not the production contract. Epic 014 should therefore use architecture/design documents and the actual runtime seams as its source of truth, treating the demo only as optional loose inspiration for visual ideas.

## Implementation Notes

### Current Implementation State
- `src/gui/app_shell.py` now owns the persistent Tk root lifetime for startup plus the visible shell handoff, while `src/gui/startup_dialog_controller.py` supports the caller-owned root model instead of treating startup as the whole application lifetime.
- `src/gui/views/main_window_shell_view.py` now provides the production-owned main window shell scaffold with toolbar, notebook, status area, placeholder Communication/Event/Personnel tabs, and shell-level reset/close entry points.
- `src/app.py` now completes the startup-to-shell handoff, persists remembered bootstrap state after successful startup, and routes shell reset/close actions through the existing active-context invalidation and cleanup seams instead of GUI-local cleanup logic.
- Focused validation now exists across `tests/unit/gui/test_startup_dialog_controller.py`, `tests/unit/gui/views/test_main_window_shell_view.py`, and `tests/integration/security/test_reset_ui_slice.py`, which together cover startup handoff, visible shell structure, shell reset/close behavior, and key lifecycle failure branches for the current Epic `014` scope.

### Rough Story Sizes (Estimated)
- App-shell root ownership and startup handoff: ~80-160 lines
- Minimal main-window shell and placeholders: ~120-220 lines
- Shell-level reset/close integration: ~80-160 lines

**Total estimated**: ~280-540 lines across a few small stories, depending on how much current startup wiring can be reused directly.

### Key Risks
- **Lifecycle confusion** - root, dialog, and shell ownership could become muddled if startup and main-window responsibilities are not separated cleanly
- **Scope drift** - placeholder tabs could quietly turn into partial feature implementations and duplicate later epic work
- **Shell/controller bloat** - too much workflow logic could collect in the top-level shell if presenter boundaries are not held early
- **Reset integration mistakes** - shell close/reset behavior could accidentally bypass the existing invalidation and backend-cleanup seams

### Success Metrics
- Epic 014 is small enough to review and implement as a shell/lifecycle epic rather than a broad GUI build-out
- A later implementation pass can produce a real main window without forcing decisions about full tab workflows yet
- Later feature epics can clearly depend on an existing production-owned shell instead of re-solving root ownership
- The project gains a truthful GUI host for further work without overcommitting to final UX too early
