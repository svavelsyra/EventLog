## Purpose (WHY) ⭐ MOST IMPORTANT

EventLog's security story is incomplete if operators can encrypt the database during normal use but cannot deny access when the information is no longer needed or when the device, storage media, or related local security footprint is about to be captured. `Nollställ` therefore needs to support two related user scenarios through one shared reset flow:
- **controlled cleanup** after an operation, when there is time to shut things down cleanly, attempt fuller cleanup, and review what failed
- **emergency cleanup** when rapid denial matters more than completeness because the device may be about to be overrun or captured

This story exists to define and deliver the destructive cleanup contract for the currently active secured database footprint. It must make it possible to deny immediate access first, then attempt broader artifact destruction in a technology-aware way without pretending to offer impossible guarantees. The cleanup flow should stay the same at its core; what changes between the two scenarios is mostly how much completion and reporting the caller/operator cares about.

Without this story, later implementation could easily drift into unsafe or misleading behavior such as:
- deleting only database rows while leaving readable file artifacts behind
- forgetting SQLite sidecar files, rotated logs, bootstrap-reset state, or other reset-relevant artifacts the active runtime/backend actually owns
- exposing sensitive details about cleanup targets in logs or UI messages
- implying guaranteed forensic erasure even though file systems and flash media limit what software alone can prove

### User Value
- Operators get a real emergency-denial capability, not just encrypted storage during normal operation.
- The application defines clearly what `Nollställ` tries to remove and what “best effort” means.
- Later UI and bootstrap flows can call one cleanup contract instead of inventing parallel reset behavior.

## Goal (WHAT)

Implement the infrastructure-level cleanup contract for `Nollställ`, using one shared reset flow that serves both controlled and emergency cleanup needs for the active encrypted database footprint, related runtime artifacts, and caller-visible best-effort outcome handling.

### Acceptance Criteria
- [ ] A dedicated cleanup path exists for the currently active database footprint and app-owned runtime artifacts, covering at least the active database file, SQLite sidecar/journal artifacts when present, application log files, and bootstrap-reset state without deleting arbitrary external key files.
- [ ] The cleanup contract distinguishes immediate-access denial from slower artifact destruction, so callers can first clear sensitive UI state and block normal database use before proceeding to broader best-effort cleanup.
- [ ] The cleanup contract defines a clear best-effort execution order for secure-delete attempts, normal deletion fallback where overwrite is not possible, and structured caller-visible results without exposing secret material or turning failures into a detailed attack oracle.
- [ ] The same underlying reset flow works for both scenarios, but caller-visible result handling stays on one shared contract: controlled use can surface meaningful partial-failure reporting, while emergency use can prioritize immediate denial and tolerate the operator leaving before detailed reporting is consumed.
- [ ] The story explicitly resolves how bootstrap-reset artifacts are handled after `Nollställ` for the active database context, including whether `config.ini` is deleted, rewritten, or preserved with only safe bootstrap state retained.
- [ ] The story defines ownership for cleanup mechanics so the active loaded persistence technology is responsible for its own backend-specific lock/close/delete behavior, whether through the loaded repository, a specialized security repository, or an equivalent technology-owned cleanup contract.
- [ ] Delivery includes focused tests for missing files, partial cleanup failures, sidecar/log cleanup, and caller-visible success/partial-failure outcomes while keeping the story explicit that secure deletion is best effort rather than guaranteed forensic erasure.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Scope cleanup to the active database context** - Epic `002` still assumes one active database instance at a time.
- **Treat secure deletion as best effort** - File-system journaling, SSD wear leveling, flash storage, and inaccessible media can limit guarantees.
- **Cover the real local security footprint** - Database file, related sidecars, logs, bootstrap-reset state, and any backend-owned active artifacts matter more than only clearing rows inside the database.
- **Return structured outcomes to callers** - Startup/UI layers need to know whether cleanup fully succeeded, partially succeeded, or could not target some artifacts.
- **Keep ownership technology-aware** - Different persistence technologies may require different locking, closing, and artifact-deletion behavior, so the active loaded database technology must own those backend-specific steps.
- **Keep one shared reset contract** - Controlled cleanup and emergency cleanup should not become separate implementations that drift apart; they are different caller priorities on top of the same cleanup sequence.

### Negative Constraints (MUST NOT)
- **Don't turn this into the startup dialog story** - Trigger placement, button layout, and confirmation UX belong to the UI flow around this contract, not inside the cleanup implementation itself.
- **Don't support multi-database wiping** - The story remains limited to the current active database target and its known artifacts.
- **Don't promise impossible guarantees** - No wording should imply the application can prove complete forensic erasure on all storage types.
- **Don't leak sensitive cleanup details** - Avoid logging secret material, remembered key locations, or highly specific failure details that reveal more than the caller needs.
- **Don't absorb later attachment/export cleanup scope** - Attachment, portability, and other future artifact classes remain separate work unless they are already part of the active secured database footprint defined today.
- **Don't hardcode one backend's cleanup mechanics as the long-term architecture** - SQLite-specific file handling is today's concrete case, not the permanent ownership model for all future database technologies.

### Soft Preferences (SHOULD)
- Prefer one reusable cleanup service/helper that different entry points can call.
- Keep Swedish success/failure wording aligned with existing security design where it has already been specified.
- Make the cleanup result contract explicit enough that future callers can choose whether to exit, restart, or refresh UI without re-implementing deletion logic.

## Implementation Notes

### Suggested Approach
1. Define a cleanup coordinator/service for `Nollställ` rather than embedding deletion logic inside presenters.
2. Make the first cleanup phase immediate access denial: clear sensitive UI state, stop normal repository use, and close or lock active persistence handles as quickly as the technology allows.
3. Let the active loaded persistence technology provide backend-specific cleanup behavior for its own artifacts and locking/close requirements.
4. Enumerate which artifact classes belong to the current `Nollställ` scope: database file, SQLite sidecars, logs, backend-owned active artifacts when the active technology owns them, in-memory/runtime cleanup, and bootstrap-reset artifact handling, while keeping caller-selected external key files outside automatic reset cleanup.
5. Implement best-effort secure-delete helpers with explicit fallback behavior when overwrite/delete steps cannot fully complete.
6. Return a structured outcome object or equivalent caller contract so startup/UI flows can react consistently, including whether application shutdown is recommended or required after cleanup and how much of the result should be surfaced in controlled versus emergency use.
7. Add focused tests for success, missing-file, and partial-failure paths before wiring more UI entry points.

### Technical Considerations
- Current docs contain a design tension between the older logging-architecture `Nollställ` flow and the newer security-design startup emergency flow; this story should resolve the cleanup contract itself even if different entry points later wrap it differently.
- Controlled cleanup and emergency cleanup should share one contract and one core sequence, but use different caller priorities: controlled cleanup optimizes completeness and readable reporting, emergency cleanup optimizes immediate denial first and may never present or preserve detailed failure feedback.
- SQLite-specific sidecar files such as WAL, SHM, and journal files should be considered part of the active database footprint when present.
- Cleanup may need to clear in-memory key material held by the active runtime in addition to file artifacts, but should not pretend Python can guarantee memory-forensics resistance.
- If a backend owns cleanup of a reachable active key artifact, that ownership belongs beneath the backend seam; caller-selected external key files should not be treated as app-owned reset targets.
- The bootstrap-reset outcome must align with `002.001` so post-`Nollställ` startup does not reopen the database with stale or contradictory config state.
- Closing the application after cleanup may be the safest default for the emergency path because it reduces the chance that sensitive state remains in process memory or that stale handles keep artifacts open; the story should make this an explicit caller-contract decision rather than an accidental side effect.
- Emergency use has an interruption edge case: the operator may shut the laptop immediately after pressing `Nollställ`, which may trigger OS sleep before best-effort cleanup finishes. The story should therefore prioritize quick access denial first and explicitly consider whether the application should try to postpone sleep, request shutdown, or otherwise favor the fastest practical destructive sequence.
- Ownership should align with the active loaded technology: a SQLite implementation may clean up files directly, while future technologies may need different delete/lock semantics exposed through a specialized security repository or comparable backend-owned cleanup interface.

### Affected Components
- New security cleanup module such as `src/security/secure_delete.py`
- Runtime/bootstrap integration points that invoke emergency cleanup
- Focused unit/integration tests for cleanup behavior
- Possibly config/bootstrap helpers if bootstrap-reset handling is implemented here

## Context & Background

Epic `002` kept emergency `Nollställ` inside the secure-bootstrap epic because it is tightly coupled to encrypted database lifecycle and the local security footprint around that database.

The project already has two related strands of guidance:
- logging architecture describes a fast destructive reset of database and logs from the running app
- security design adds the stronger emergency path that can be triggered from the unlock screen without login and extends the target set to key material and bootstrap reset artifacts

This story should deliver the shared cleanup contract beneath those flows, while keeping trigger-specific UI behavior outside the cleanup implementation itself and making technology ownership explicit enough for future non-SQLite backends.
