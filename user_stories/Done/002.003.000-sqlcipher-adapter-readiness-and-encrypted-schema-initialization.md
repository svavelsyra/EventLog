## Purpose (WHY) ⭐ MOST IMPORTANT

The application only becomes meaningfully secure when the low-level database path can open or create an encrypted database without changing how the rest of the repository layer behaves. That boundary belongs at the adapter/readiness level, not in Core entities and not as ad hoc logic scattered across repositories.

This story exists to make encrypted database access a real infrastructure capability. It provides the backend-specific open/unlock behavior that lets higher layers keep using normal repository CRUD/query flows while ensuring that:
- missing databases are created as encrypted databases from the start
- existing encrypted databases are unlocked and validated before repository use
- wrong keys fail clearly
- plaintext fallback never happens silently

### User Value
- Operators can trust that stored data is actually protected at rest.
- The rest of the application can keep using the repository layer without encryption-specific branching.
- Wrong credentials fail fast instead of corrupting the startup flow or opening an unsafe path.

---

## Goal (WHAT)

Implement the SQLCipher-capable adapter readiness path that opens existing encrypted databases, creates new encrypted databases, initializes schema, and validates backend identity/version state before repositories begin normal work.

### Acceptance Criteria
- [x] The SQLite adapter path gains an encrypted/SQLCipher-capable readiness flow that accepts the derived key, applies it immediately after connection, and verifies the database can be read before repository construction continues.
- [x] Adapter readiness handles both startup cases: missing database file results in encrypted creation plus schema initialization and metadata stamping; existing database file results in unlock plus identity/version validation.
- [x] Wrong keys, unreadable/corrupt databases, or adapter/profile mismatches fail with explicit adapter-level exceptions and never fall back to a plaintext SQLite open path.
- [x] Delivery includes focused adapter/integration tests for encrypted create, encrypted reopen, and wrong-key failure behavior using the approved dependency path.

### Current Delivery Snapshot
- `src/db/sqlite_adapter.py` now owns the encrypted SQLite readiness path directly: when an `encryption_key` is supplied, `SQLiteAdapter.connect()` resolves the SQLCipher DB-API module, opens the connection, applies `PRAGMA key` immediately, and verifies encrypted readability before repository use continues.
- `SQLiteAdapter.initialize_schema()` now cleanly separates the two startup cases described by this story: a new encrypted target runs schema initialization plus authoritative profile metadata stamping, while an existing encrypted target runs application-id and user-version validation before normal repository work begins.
- Explicit encrypted adapter failure types now exist under `src/db/database_adapter.py`, and the SQLite adapter classifies wrong-key, unreadable/corrupt, migration-needed, newer-database, and profile-mismatch cases without silently falling back to a plaintext open path.
- The repository/bootstrap seam now consumes this capability through the backend-owned repository creator in `src/db/repositories/bootstrap_backend_policy.py`, keeping encryption-specific open/create behavior in infrastructure rather than pushing it into repository CRUD logic or higher layers.
- Focused coverage now exists in `tests/unit/db/test_sqlite_adapter_encryption.py` for SQLCipher key application, encrypted-readiness verification, wrong-key classification, and profile/version validation, together with file-backed create/reopen coverage in `tests/unit/db/test_sqlite_repository_skeleton.py`.
- A focused closeout verification run now also passed for `tests/unit/db/test_sqlite_adapter_encryption.py` and `tests/unit/db/test_sqlite_repository_skeleton.py`, which confirms that this story is not just theoretically aligned on paper but still behaves as expected in the current environment.

### Closeout Note
- No further story-local gap is currently known in the encrypted adapter-readiness slice for the current Epic `002` scope. The current SQLite/SQLCipher implementation, explicit adapter exception taxonomy, and focused create/reopen/failure tests are enough to treat this story as complete.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep encryption in infrastructure** - Core models and repository callers remain encryption-agnostic after readiness succeeds.
- **Follow current bootstrap architecture** - The adapter owns backend-specific readiness; bootstrap/factory code decides when to invoke it.
- **Preserve Epic `001` repository behavior** - Existing CRUD/query patterns should continue once a ready adapter is supplied.
- **Use established SQLCipher support** - No custom encrypted file format or Python-layer field encryption workaround.

### Negative Constraints (MUST NOT)
- **Don't move credential derivation into the adapter** - The adapter consumes a derived key; it does not own password or key-file rules.
- **Don't let repository code own startup identity/version decisions** - Those stay in adapter readiness/bootstrap.
- **Don't build migration tooling here** - This story handles initial encrypted readiness, not a legacy plaintext-to-encrypted conversion workflow.
- **Don't add multi-database management** - Epic `002` stays on the single-database security model.

### Soft Preferences (SHOULD)
- Keep the encrypted adapter path as close as practical to the existing `DatabaseAdapter`/`SQLiteAdapter` boundaries.
- Reuse current schema initialization and backend-metadata patterns where they remain architecturally valid.
- Make wrong-key failure easy for higher layers to distinguish from unrelated I/O problems.

## Implementation Notes

### Suggested Approach
1. Extend or adapt the current SQLite adapter path so it can perform encrypted connection readiness with SQLCipher.
2. Apply the key immediately after connection and verify the key by executing a simple read.
3. Route missing-file startup through encrypted creation plus schema initialization and backend metadata stamping.
4. Add integration tests for create/open/fail cases before wiring the full bootstrap flow.

### Technical Considerations
- The current architecture treats adapter-managed identity/version metadata as authoritative for startup decisions.
- New encrypted databases should be created encrypted from first write rather than created in plaintext and converted later.
- Existing file detection is a SQLite-specific concern inside the adapter path, not generic bootstrap logic.
- Tests may need conditional handling if SQLCipher support is unavailable in the local environment, but the story should still define the intended coverage.

### Affected Components
- `src/db/sqlite_adapter.py`
- Potential supporting DB/security exception definitions in `src/db/database_adapter.py`
- Integration/unit tests for encrypted adapter readiness
- Repository construction path indirectly, once a ready adapter is supplied

## Context & Background

Epic `001` gave the project a plain SQLite persistence foundation. Epic `002` does not replace that architecture; it wraps secure open/create behavior around it.

This story is the infrastructure hinge of the epic. If it is too broad, it drifts into GUI and startup orchestration. If it is too narrow, later stories must invent their own encrypted-open rules. The intended boundary is therefore: adapter readiness only.
