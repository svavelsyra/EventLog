## Purpose (WHY) ⭐ MOST IMPORTANT

Even after contract, export, and import exist, portability is not truly usable until the recovery path is tested and the guardrails are proven. This story exists to finish the portability slice with the quality and recovery behavior needed for actual use.

### User Value
- The portability feature becomes trustworthy for reset/recovery scenarios.
- Round-trip behavior is proven rather than assumed.
- Exclusion rules, malformed-bundle handling, and recovery usage stay visible as their own final review slice.

---

## Goal (WHAT)

Complete the recovery-focused guardrails and round-trip verification for communication portability.

### Acceptance Criteria
- [x] A documented recovery-oriented usage flow exists for exporting approved communication config before reset and restoring it afterward.
- [x] Focused tests verify round-trip behavior across the approved contract, export, and import/apply path.
- [x] Guardrail behavior is verified for excluded sections, malformed bundles, unsupported versions, and other contract violations.
- [x] The final communication portability slice is reviewable as a complete narrow recovery feature rather than only disconnected serializer/apply primitives.

### Bookkeeping Notes
- Recovery-oriented usage flow is documented in `docs/design/root_design.md` under `Communication Configuration Portability Recovery Flow`, including the export-before-reset and restore-after-reset sequence plus explicit exclusions.
- Round-trip and guardrail verification live in `tests/unit/core/test_communication_portability.py`, including canonical export/import/reload/re-export coverage and rejection checks for excluded top-level sections, malformed top-level bundle shapes, unsupported bundle kinds, unsupported versions, and other contract violations.
- `src/core/communication_portability.py` now normalizes malformed top-level non-mapping bundles into `CommunicationPortabilityContractError`, keeping recovery-path failures reviewable at the portability-contract boundary.
- Story status is now marked `Done` based on the user-approved documentation closeout; the file intentionally remains in `user_stories/ToDo/` because this pass does not move files.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Build on the earlier child stories** - This story verifies and finishes the slice; it should not redefine the contract, export shape, or import rules arbitrarily.
- **Stay narrow and recovery-focused** - The goal is communication-config recovery, not a broad backup/restore product.
- **Keep exclusions enforced** - Guardrails must prove that out-of-scope data stays out.

### Negative Constraints (MUST NOT)
- **Do not absorb user-preference portability** - Epic `012` remains separate.
- **Do not absorb event metadata portability** - Event metadata remains outside Epic `003`.
- **Do not turn this into generic export/report functionality** - Human-readable exports and broad reporting are separate concerns.
