## Purpose (WHY) ⭐ MOST IMPORTANT

EventLog is intended for operational use where the device, storage media, or removable key material may be lost, stolen, or captured. Epic 001 provides persistence, but plaintext persistence is not sufficient for the project’s stated security model. If the database can be copied and read directly, operational communications, events, and personnel data become exposed at exactly the moment the system most needs to protect them.

This epic exists to turn the working database foundation into a protected datastore that still behaves like an operational tool rather than a lab exercise. The application must be able to:
- protect data at rest without breaking normal SQL querying
- unlock the database in a way that supports both training and operational deployments
- bootstrap access safely before the repository layer begins normal work
- fail clearly when credentials are wrong without damaging the database
- support emergency destruction of the local security footprint when the operator must deny access quickly

The deeper purpose is not “add crypto because crypto is good.” The purpose is to preserve trust in the application under field conditions. Operators must be able to keep using EventLog for fast logging and retrieval while knowing that captured storage is not equivalent to captured information.

### Success Criteria
- Encrypted databases can be created and opened through a documented bootstrap flow
- Normal repository queries continue to work after unlock without Core-layer encryption awareness
- Both password-only and key-file-plus-password modes are supported according to the security design
- Wrong credentials fail with clear caller-visible behavior and without silent fallback to plaintext access
- Emergency reset can remove the encrypted database and related local artifacts on a best-effort basis

## Goal (WHAT)

Introduce the security layer around the database foundation by defining how EventLog creates, unlocks, and uses an encrypted database instance.

This epic covers the end-to-end contract for:
1. SQLCipher-backed encrypted database access
2. key derivation from password and optional key file
3. bootstrap configuration needed before the database can be opened
4. repository/factory integration for encrypted connections
5. startup credential flow for first-run creation and later unlock
6. best-effort emergency destruction of database-related security artifacts

## Limitations (CONSTRAINTS & REQUIREMENTS)

**What boundaries must we work within? What must we avoid?**

### Hard Constraints (MUST)
- **Preserve the existing architecture** - Encryption remains an infrastructure/database concern; Core entities and business logic stay encryption-agnostic
- **Use established security libraries** - Do not invent custom database encryption or custom cryptographic primitives; use the documented SQLCipher + KDF approach
- **Support offline-only operation** - No network dependency, licensing service, or cloud key retrieval
- **Honor the single-database security model** - Epic 002 assumes the application manages one database instance at a time
- **Keep bootstrap data outside the encrypted database only when necessary** - Settings required to know how to open the database belong in `config.ini`; protected operational/security values belong in the encrypted database or SQLCipher-managed metadata
- **Support both deployment modes** - Password-only mode for simpler/training use and key-file-plus-password mode for stricter operational use
- **Maintain repository-level transparency after unlock** - Once the repository is created successfully, normal CRUD and query behavior from Epic 001 should continue without caller-specific encryption branching
- **Keep testing with the delivery work** - Each later 002.x story should carry its own tests rather than deferring quality to a broad standalone testing story

### Negative Constraints (MUST NOT)
- **Do not add general user authentication or role management** - This epic is about database protection and secure access bootstrap, not multi-user identity
- **Do not support multiple simultaneously managed databases** - The security design explicitly avoids that model
- **Do not store key-file paths or secret material in configuration, logs, or database tables** - Avoid leaking sensitive operational details
- **Do not add plaintext fallback behavior** - Failed unlock attempts must fail visibly instead of silently opening an unencrypted path
- **Do not design a full plaintext-to-encrypted migration workflow here** - Migration remains a later concern if it becomes necessary
- **Do not over-specify Phase 2 polish** - Password-strength indicators, advanced rotation tooling, and other later enhancements should not bloat the first Epic 002 scope
- **Do not rewrite Epic 001 domain/repository semantics** - This epic extends secure opening and lifecycle behavior around the database foundation rather than redefining entity contracts

### Soft Constraints (SHOULD)
- Keep startup/unlock behavior operationally simple and predictable
- Keep user-facing failure behavior explicit enough for both implementation and tests
- Minimize additional dependencies beyond the documented security exception
- Prefer narrowly scoped follow-up stories that isolate config/bootstrap, crypto helpers, repository integration, GUI flow, and emergency reset behavior

## Context & Background

### Architectural Context
Epic 001 established the baseline repository pattern, schema initialization path, and SQLite-centric persistence model. Epic 002 adds the security boundary around that baseline.

The security architecture already defines several important decisions:
- **SQLCipher** is the chosen database-at-rest approach because it preserves normal SQL behavior
- **Key derivation** combines password input with an optional key file for stronger operational mode
- **Bootstrap problem** is real: the app must know database path and whether a key file is required before it can open the encrypted database
- **Single-database model** is intentional because emergency destruction must be able to target the currently used database safely
- **Secure deletion is best effort** because file-system behavior can limit guarantees, especially on journaled file systems and flash media

### Why This Is Its Own Epic
Epic 001 explicitly deferred encryption so the project could first establish a plain, testable persistence foundation. That separation still makes sense: repository CRUD and schema work are now distinct from credential handling, encrypted connection bootstrapping, dependency exceptions, and security-focused destruction behavior.

### What This Epic Excludes
- application-wide login/identity features
- configuration-table work from Epic 003
- structured report work from Epic 004
- broad migration tooling for legacy plaintext databases
- future password-change/key-rotation tooling beyond what is needed to establish the initial secure access model
