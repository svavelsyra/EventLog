# User Story: 014.001 - App-Shell Root Ownership and Startup-Dialog Hosting

**Story Number**: 014.001  
**Created**: 2026-05-06  
**Status**: Done  
**Epic**: [014 - Minimal Main GUI Shell and Startup Handoff](014.000.000-minimal-main-gui-shell-and-startup-handoff.md)  
**Owner**: Developer

**Filename**: `014.001.000-app-shell-root-ownership-and-startup-dialog-hosting.md`

---

## Purpose (WHY) ⭐ MOST IMPORTANT

Epic `014` only becomes durable if the application first solves who owns the Tk lifetime. Right now the startup dialog effectively owns the GUI lifetime because it creates and destroys the root while the application still ends after startup. That is acceptable only as long as there is no real post-startup shell. Once a real main window exists, that ownership model becomes the wrong center of gravity.

This story exists to move Tk lifetime ownership upward into an app-shell seam before visible shell widgets arrive. Without that move, later work risks:
- awkward root destruction and recreation when transitioning from startup into the main window
- top-level GUI logic leaking into the startup controller because it still owns the whole application lifetime
- confusion about where shutdown, later reset integration, and startup success handoff actually belong
- avoidable rework in `014.002` when the main-window shell appears

### User Value
- The application gains a stable GUI lifetime model before visible shell UI is added.
- Later main-window work can focus on layout and navigation instead of reopening root-ownership questions.
- Reviewers can validate the core lifecycle seam separately from visible GUI structure.

---

## Goal (WHAT)

Create the app-shell lifecycle seam that owns the single persistent `tk.Tk()` root and hosts the existing startup dialog beneath it.

### Acceptance Criteria
 - [x] A new app-shell or equivalent top-level orchestration seam owns the single persistent `tk.Tk()` instance for the application lifetime instead of `StartupDialogController` creating and destroying its own root.
 - [x] The startup dialog continues to run as a blocking child/modal startup step under that app-shell-owned root and still returns `StartupDialogSuccess | None` as the caller-visible result.
 - [x] `src/app.py` or the equivalent top-level entry wiring now reaches startup through the app-shell seam while preserving the existing remembered-target save behavior after successful startup and clean exit behavior on cancel.
 - [x] Focused unit tests cover the root-ownership/handoff contract, including successful startup returning the existing success result and cancellation exiting without leaving root-lifetime ambiguity behind.

### Current Delivery Snapshot
- `src/gui/app_shell.py` now owns the single persistent Tk root and hosts the startup dialog beneath that app-level lifetime instead of leaving root creation/destruction inside the startup-only controller path.
- `src/gui/startup_dialog_controller.py` now supports caller-owned root execution so startup can remain a blocking startup step without owning the whole application lifetime.
- `src/app.py` now enters startup through the app-shell seam while preserving the existing remembered-target persistence behavior on successful startup and clean exit behavior on cancel.
- Focused lifecycle coverage in `tests/unit/gui/test_startup_dialog_controller.py` now proves successful startup handoff, cancellation cleanup, startup-runner failure cleanup, and the main `run_app()` entry wiring for this story scope.

### Closeout Note
- No story-local gap is currently known in the app-shell root-ownership and startup-dialog hosting slice for Epic `014`. The root-lifetime handoff contract, caller-owned startup execution path, and focused lifecycle tests are sufficient to treat this story as complete.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep this story invisible-first** - This story is about lifetime ownership and startup hosting, not about rendering the real main-window shell yet.
- **Preserve the current startup success contract where practical** - `StartupDialogSuccess` and the existing startup outcome shape should remain the handoff seam for later shell work unless a strong reason emerges during implementation.
- **Use one Tk root for the app lifetime** - The result should establish one persistent root owned by the app-shell lifetime rather than normalizing multi-root handoff patterns.
- **Keep ownership boundaries clear** - App-level lifecycle orchestration belongs above the startup controller; widget rendering remains in the view layer.
- **Prepare later shell work without requiring it now** - The implementation should leave a clean place for `014.002` to attach the minimal visible main window.

### Negative Constraints (MUST NOT)
- **Do not build the toolbar, notebook, status area, or placeholder tabs here** - Those belong to `014.002`.
- **Do not absorb reset/close active-context cleanup UX here** - Shell-level reset/close integration belongs to `014.003`.
- **Do not redesign the startup presenter contract unnecessarily** - This story should host the existing startup flow, not reopen bootstrap/business rules without need.
- **Do not treat `Demo/demo_app.py` as architectural guidance** - Demo structure remains low-trust and must not drive the lifetime design.
- **Do not pull in tab presenters, preferences, or workflow behavior** - Those remain later work under Epic `014`, Epic `005`, and Epics `006`-`008`.

### Soft Preferences (SHOULD)
- Prefer a small app-shell seam that is easy to test without immediately coupling it to the visible main-window widgets.
- Favor a design where the root can stay hidden or minimal until the later shell story decides how the first visible main window should appear.
- Keep the startup controller reusable as a startup-step coordinator rather than collapsing it into `src/app.py`.

---

## Implementation Notes

### Suggested Approach
1. Introduce an app-shell orchestration seam, likely under `src/gui/`, that owns `tk.Tk()` and overall GUI lifetime.
2. Refactor the startup controller so it can host the existing startup dialog under a caller-supplied root instead of manufacturing and destroying the root internally.
3. Keep the startup dialog result contract intact and return it through the new app-shell seam.
4. Update `src/app.py` to enter startup through that seam while preserving remembered-target persistence on success.
5. Add focused lifecycle tests around startup success and cancellation.

### Affected Components
- `src/app.py`
- New app-shell orchestration module under `src/gui/`
- `src/gui/startup_dialog_controller.py`
- Focused unit tests for GUI/app-shell startup lifecycle

## Context & Background

This story intentionally solves the least visible but most foundational part of Epic `014` first. It should make later visible shell work smaller, not postpone it indefinitely. The success condition is not that the operator sees a finished main window yet; the success condition is that the application lifetime model is now correct for one to exist.

## Testing Strategy

### Key Test Scenarios
- Successful startup under the app-shell-owned root returns the same startup result shape expected by the rest of the app.
- Cancellation exits cleanly through the app-shell seam.
- The startup controller no longer needs to create/destroy the Tk root itself to run the dialog.

### Integration Points
- `src/app.py` entry wiring into the new app-shell seam
- `StartupDialogController` integration with caller-owned root lifetime
