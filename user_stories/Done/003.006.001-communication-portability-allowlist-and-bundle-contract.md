## Purpose (WHY) ⭐ MOST IMPORTANT

Before export/import code is written, the project needs one explicit answer for what communication configuration is portable and what must stay out. Without that contract, later portability code will guess from table shape, include too much data, or quietly create a second configuration authority.

### User Value
- Reviewers get one explicit allowlist before serializer/import code exists.
- Later export/import stories can implement against a stable bundle shape instead of guessing.
- Communication portability stays narrow, auditable, and separate from user preferences or event metadata.

---

## Goal (WHAT)

Define the communication portability allowlist, exclusions, and portable bundle contract.

### Acceptance Criteria
- [x] An explicit allowlist exists for which communication configuration data is portable under Epic `003`, covering only the settled communication runtime metadata needed to rebuild communication systems, configured options, and qualifier behavior.
- [x] Explicit exclusions exist for secrets, bootstrap/security material, user preferences, event metadata, structured-report templates, logs, entry records, and any other non-communication data.
- [x] The portable bundle shape is defined clearly, including format, version field, and section layout for communication configuration.
- [x] Focused tests or validation checks verify that the contract rejects unsupported/missing version information and does not silently widen scope.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep scope allowlisted** - Only approved communication configuration may be portable.
- **Keep the database as runtime authority** - The bundle contract is for export/import, not for live runtime queries.
- **Stay communication-only** - Event metadata belongs to Epic `013`; user preferences belong to Epic `012`.
- **Remain inspectable and versionable** - The contract must be understandable without hidden binary structure.

### Negative Constraints (MUST NOT)
- **Do not implement full export/import here** - This story defines the contract only.
- **Do not infer scope from storage shape alone** - Database presence is not enough to make a field portable.
- **Do not include secrets or bootstrap state** - Security-sensitive data remains outside scope.
