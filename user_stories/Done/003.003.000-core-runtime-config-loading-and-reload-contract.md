## Purpose (WHY) ⭐ MOST IMPORTANT

Even with schema and repository reads in place, Epic `003` still fails if every validator, presenter, and workflow loads its own configuration in its own way. Communication configuration only becomes trustworthy when higher layers can depend on one core-owned runtime contract for what systems, paths, qualifiers, and related rules are currently valid.

This story exists to define that runtime-loading seam. Without it, later feature work could drift into:
- duplicated repository lookups spread across presenters and validators
- accidental hardcoded fallback lists when a caller does not know how to load config properly
- inconsistent caching or stale reads after targeted config updates
- DB-shaped data leaking upward instead of stable core configuration objects
- callers reconstructing communication meaning differently because no shared runtime object defines systems, recursive option paths, and qualifier behavior consistently

### User Value
- Validators and presenters can consume clear runtime config objects instead of raw repository data.
- Configuration-driven behavior becomes consistent across communication workflows.
- Future targeted config changes can trigger one known reload path instead of ad hoc refresh logic.

---

## Goal (WHAT)

Create the core-layer configuration objects and loading contract for Epic `003` runtime operational metadata.

### Acceptance Criteria
- [x] Core-owned runtime configuration models and loading support exist for the communication metadata needed by later workflows, including communication system configuration, recursive communication-option/path support, and top-level qualifier behavior.
- [x] The loading contract uses repository reads from `003.002` and presents stable core-facing APIs that validators and presenters can consume without database knowledge.
- [x] Cache and reload expectations are explicit: the story defines whether config is memoized, when a reload is required after targeted updates, and how callers can request a fresh view without inventing their own global state.
- [x] Focused unit tests verify initial load, stable object shape, and reload behavior after underlying configuration changes.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep the core layer database-agnostic** - Core code may depend on repository interfaces/results, not SQLite details.
- **Expose configuration as domain/runtime objects** - Higher layers should receive explicit config objects, not raw rows.
- **Define reload behavior deliberately** - The contract must say how fresh config is obtained after changes rather than leaving cache invalidation implicit.
- **Preserve the clarified communication contract** - Core runtime objects must model communication as a top-level system/way plus recursive configured options and top-level qualifiers, rather than reintroducing a fixed `system/method/channel` hierarchy.

### Negative Constraints (MUST NOT)
- **Do not put SQL or adapter logic in core** - Repository access remains below this layer.
- **Do not let each presenter load config independently** - This story exists to centralize the runtime-loading path.
- **Do not absorb GUI quick-add/update wiring** - Mutation triggers are handled in `003.004`; this story defines the reload contract they consume.
- **Do not fold report-template runtime loading into this story** - Structured-report templates remain Epic `004` territory.

### Soft Preferences (SHOULD)
- Prefer immutable or read-mostly config objects once loaded.
- Keep naming aligned with the architecture docs that already reference `SystemConfig`.
- Favor explicit refresh calls over hidden module-level mutable globals.

## Implementation Notes

### Suggested Approach
1. Define core configuration models for communication systems, options, and qualifiers.
2. Add a core service/manager that builds those objects from repository reads.
3. Make cache/reload behavior explicit and small enough for later GUI/runtime use.
4. Add unit tests around load and refresh behavior.

### Technical Considerations
- The architecture docs already assume configuration-driven validators, so this story should turn that assumption into a concrete API.
- Communication config should expose a top-level communication system plus recursive child-option structure and qualifier behavior in a way validators/presenters can consume without caring about storage shape.
- Reload behavior matters because later quick-add/update flows should not force a full application redesign just to pick up a newly added communication option, label change, or qualifier-behavior adjustment.

### Affected Components
- New or updated core configuration model/loading modules under `src/core/`
- Repository-facing seams used to assemble runtime config objects
- Focused unit tests for core config loading/reload behavior

## Context & Background

Epic `003` explicitly calls out runtime loading and caching expectations as one of its key deliverables. This story is where that contract becomes concrete.

It is also the story that protects later GUI work from a common failure mode: reaching down into repository methods directly because no reusable core configuration object exists yet.
