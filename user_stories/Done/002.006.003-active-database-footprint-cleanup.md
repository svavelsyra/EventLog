## Purpose (WHY) ⭐ MOST IMPORTANT

`Nollställ` is not meaningful as a database-denial capability if the active database file remains on disk or if backend-generated sidecar artifacts such as WAL, SHM, or journal files are left behind. Once the shared reset flow has denied immediate access and the backend-owned cleanup seam has released backend-native locks and handles, the next focused responsibility is to target the active database footprint itself.

This story exists to define and deliver that footprint-specific cleanup slice without mixing it with broader artifact classes such as logs, key files, or bootstrap-reset state.

Without this story, later implementation could drift into unsafe or misleading behavior such as:
- deleting only the main database file while leaving SQLite sidecars or journals readable on disk
- hardcoding a database-file target list outside the backend-aware cleanup seam
- treating missing files as fatal even though best-effort cleanup should continue across partial states
- implying guaranteed forensic erasure even though file systems and storage media can limit what overwrite/delete can prove

### User Value
- Operators get a real best-effort wipe of the active database footprint rather than partial cleanup of only the main file.
- The reset flow can target the backend-known persistence footprint consistently for the currently loaded database context.
- Later non-database artifact cleanup can remain separate and simpler because the database-footprint scope is already defined here.

## Goal (WHAT)

Implement best-effort cleanup of the active database footprint after immediate denial and backend-owned cleanup preparation, covering the current database file and backend-native sidecar or journal artifacts that belong to the active database context.

### Current Delivery Snapshot
- The shared adapter seam now includes `DatabaseAdapter.get_cleanup_metadata()` plus `BackendCleanupMetadata` / `BackendCleanupArtifactMetadata`, so backend-owned cleanup metadata can describe the active database footprint without pushing backend knowledge into bootstrap orchestration.
- The current SQLite backend now owns the active on-disk database footprint for the loaded target through `SQLiteAdapter.get_cleanup_metadata()`, including the main database file plus present `-wal`, `-shm`, and `-journal` sidecars together with backend-provided `artifact_kind` and `allow_secure_overwrite` metadata.
- `src/db/repositories/startup_bootstrap.py` now consumes that backend-owned cleanup metadata during `backend_cleanup()` after access denial/release instead of inferring SQLite cleanup meaning from filenames or suffixes locally.
- Cleanup now performs a best-effort overwrite-before-delete step only for artifacts whose backend-owned metadata allows secure overwrite, then falls back to normal `unlink()` when overwrite cannot be completed.
- Already-missing targets remain neutral because backend cleanup only enumerates paths that presently exist for the active database context.
- Partial artifact cleanup is now explicit in the backend contract through `BackendCleanupStatus.PARTIAL` plus `BackendCleanupError.outcome`, and the current cleanup seam now preserves earlier progress by continuing across later final-deletion failures before raising one final sanitized error. `BackendCleanupOutcome.report` now carries backend-provided artifact kinds together with release/enumeration/removal/failure counts as sanitized outcome data.
- Focused coverage now exists in `tests/unit/db/test_startup_bootstrap.py` for overwrite-before-delete ordering, configured secure-delete pass plumbing, overwrite-failure fallback, completed cleanup, repeated cleanup, a three-artifact continuation case where a locked middle artifact is left behind while a later artifact is still removed before the final partial-cleanup error is raised, a retry-after-partial-failure case where a later rerun targets only the still-present artifact and treats already-removed artifacts as neutral, and a metadata-first case where bootstrap passes through backend-owned artifact kinds and overwrite policy. `tests/unit/db/test_sqlite_adapter_encryption.py` now covers the SQLite-owned cleanup metadata seam directly, and `tests/integration/security/test_reset_flow.py` covers both completed reset cleanup and explicit partial-cleanup behavior through the shared reset flow.
- A fresh closeout verification pass on 2026-05-05 first exposed five failing shared reset-flow scenarios in `tests/integration/security/test_reset_flow.py`, then a focused repair restored the active-context contract by making post-invalidation SQLite access fail with a stable closed-database error instead of a `NoneType` crash; the focused unit slice and the shared reset-flow integration proof are now both green again.

### Closeout Result
- The backend-owned active-footprint cleanup seam remains delivered and the shared reset-flow proof is now green again on the current codebase.
- The repair did not move cleanup ownership out of the backend seam; it restored a stable post-invalidation closed-state error at the SQLite adapter boundary so the denial-first reset flow can verify access denial cleanly before backend cleanup runs.
- Any later need for more granular caller-mode interpretation or richer backend-specific metadata should still build on the current backend-owned seam rather than reintroducing backend inference into bootstrap orchestration.

### Acceptance Criteria
- [x] The cleanup path can identify the active database footprint for the currently loaded backend context, including at minimum the main database file and SQLite sidecar or journal artifacts such as WAL, SHM, and rollback journal files when they exist for the active target.
- [x] Cleanup of those targets follows an explicit best-effort sequence: attempt secure overwrite/delete where supported, fall back to normal deletion when overwrite cannot be completed, and do not claim guaranteed forensic erasure.
- [x] Missing, already-removed, or partially present database-footprint artifacts do not crash the cleanup flow, and the result records richer sanitized backend-owned artifact categories plus outcome-level phase/count data that higher layers can interpret without exposing unnecessary sensitive detail.
- [x] Focused tests verify main-file plus sidecar handling, overwrite-before-delete ordering, overwrite-failure fallback, continuation across later delete failures after earlier cleanup progress, safe rerun behavior after a previous partial cleanup, and execution through the shared reset flow/backend-owned seam rather than through trigger-specific UI code.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Scope only to the active database footprint** - This story covers the currently loaded database file and backend-native sidecar/journal artifacts for that same target.
- **Execute beneath the approved seams** - `002.006.001` owns sequencing and `002.006.002` owns backend-native cleanup responsibility; this story delivers artifact cleanup within that structure.
- **Treat secure deletion as best effort** - Overwrite and deletion behavior must acknowledge journaling file systems, SSD wear leveling, flash storage, and other real-world limits.
- **Use runtime-known targets only** - Cleanup should act on artifacts that belong to the active database context and are derivable from the loaded backend/database target, not speculative unrelated copies.
- **Return caller-safe outcome data** - Results must be sanitized enough for later reporting without exposing secret material or turning cleanup failures into a detailed attack oracle.
- **Make reruns safe** - Repeating `Nollställ` after a partial failure must not treat already-removed active-footprint artifacts as failed cleanup work.

### Negative Constraints (MUST NOT)
- **Don't absorb logs, key files, or bootstrap artifacts** - Those belong to `002.006.004`.
- **Don't absorb caller-mode interpretation** - Controlled-vs-emergency result handling belongs to `002.006.005`.
- **Don't redefine immediate denial sequencing** - That belongs to `002.006.001`.
- **Don't bypass backend ownership** - Top-level code should not hardcode database-footprint cleanup outside the backend-aware structure defined in `002.006.002`.
- **Don't promise complete forensic erasure** - Wording and behavior must stay honest about best-effort limits.

### Soft Preferences (SHOULD)
- Prefer explicit target categories so tests can verify main database files and sidecar/journal artifacts separately.
- Keep SQLite sidecar handling concrete today while preserving room for future backends to define their own active-footprint artifacts through the same seam.
- Record enough neutral detail that later callers can distinguish full success, partial cleanup, and no-target cases without exposing more than necessary.

## Implementation Notes

### Suggested Approach
1. Use the shared reset flow from `002.006.001` and the backend-owned seam from `002.006.002` so database-footprint cleanup happens only after immediate denial and backend-native handle release/preparation.
2. Let the backend expose cleanup targets and cleanup metadata through a backend-owned seam such as `get_cleanup_metadata()`, covering at minimum the main database file and SQLite WAL/SHM/journal artifacts when present.
3. Apply best-effort secure-delete helpers according to backend-provided cleanup metadata, with normal delete fallback when overwrite cannot be completed or is unsupported.
4. Continue through partial failures while returning neutral, sanitized cleanup results for the active footprint, including backend-provided artifact kinds without inferring them in bootstrap.
5. Add focused tests for present, missing, and partially failing target combinations.

### Technical Considerations
- For SQLite, the active footprint should include the main database file plus common sidecar artifacts such as `-wal`, `-shm`, and rollback journal files when present, but that artifact knowledge should stay in `SQLiteAdapter.get_cleanup_metadata()` rather than being inferred in bootstrap orchestration.
- This story should stay on the database-footprint boundary only; logs, key files, and `config.ini` remain separate so the split stays useful.
- Runtime-known target enumeration is important because speculative backup copies or unrelated files should not be swept up accidentally.
- Retry-safe behavior should focus follow-up results on remaining artifacts only; targets already removed by a previous run should be treated as already-cleaned or absent, not as fresh failures.
- Best-effort overwrite/delete behavior should remain honest about storage and file-system limitations while still raising the bar for recovery.
- The current overwrite stage is intentionally conservative and local to the backend cleanup seam: it uses configured pass counts when backend-owned artifact metadata allows secure overwrite, treats overwrite-stage filesystem problems as fallback-to-delete conditions, and still treats final deletion success/failure as the authoritative cleanup outcome.
- The current delivered contract already distinguishes neutral completed cleanup from explicit partial cleanup through `BackendCleanupOutcome` plus `BackendCleanupError.outcome`; later caller/reporting stories should build on backend-provided artifact kinds and outcome data instead of re-infering partial state from filesystem side effects or technology-specific filenames.

### Affected Components
- `src/db/database_adapter.py` cleanup metadata contract (`BackendCleanupMetadata` / `BackendCleanupArtifactMetadata`)
- Current SQLite cleanup path for main database and sidecar/journal targets under `src/db/sqlite_adapter.py`
- `src/db/repositories/startup_bootstrap.py` as the backend-agnostic cleanup orchestrator/pass-through consumer of backend metadata
- Focused unit/integration tests for target enumeration, backend-owned metadata, best-effort deletion ordering, and partial-failure handling

## Context & Background

The parent `002.006` story originally lumped database deletion together with logs, key material, bootstrap reset, and caller reporting. Splitting those concerns makes this child story the focused place where the project defines what counts as the active database footprint and how that footprint is cleaned up honestly.

This aligns with the approved design direction: deny access first, let the active backend own backend-native preparation semantics, then target the active database artifacts with best-effort cleanup rather than pretending software can prove complete erasure.
