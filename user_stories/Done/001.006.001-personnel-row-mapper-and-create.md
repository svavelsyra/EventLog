## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

Personnel tracking is essential for operational safety. Operators must log who is where, their status, and when they last checked in. The Create operation enables:

1. **Safety** - Track who is out on patrol/mission
2. **Accountability** - Log all personnel movements and status
3. **Check-in management** - Set expected check-in times for safety
4. **Historical record** - Full history of each person/unit's status changes

The row mapper is the most complex of the three entities - it handles 3 datetime fields, multiple boolean flags, and supersedes relationships.

### User Value
- Operators can track personnel locations and status
- Check-in alarms can be set for safety
- Historical tracking for each person/unit
- Foundation for personnel safety features

**What functionality are we delivering?**

Implement the row-to-entity mapper and the create operation for `PersonnelEntry` in the SQLite repository layer under the approved package layout.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Implement in the SQLite repository layer** - Replace the placeholder behavior using the approved repository package layout
- **Delegate low-level DB execution downward** - Repository handles mapping/query intent; adapter handles low-level execution/transaction mechanics
- **Use parameterized queries** - Prevent SQL injection
- **Set logged_time automatically** - Use the current timestamp for new records
- **Default last_contact_time** - Set to logged_time if not provided
- **Default active=1** - New entries are active
- **Validate alarm fields** - If alarm_enabled=1, expected_checkin_time must not be None

### Negative Constraints (MUST NOT)
- **Don't implement read operations yet** - That's story 001.006.002
- **Don't implement update/delete yet** - That's story 001.006.003
- **Don't implement specialized queries yet** - That's story 001.006.004
- **Don't implement status update workflow** - GUI manages active flag changes

### Soft Preferences (SHOULD)
- Reuse repository patterns already established for CommunicationEntry and EventEntry
- Keep mapper behavior centralized so later personnel reads stay consistent
- Make alarm-related validation behavior easy to understand and verify in tests

## Implementation Notes

### What's Needed

1. **Row-to-Entity Conversion Capability**
   - Provide a private helper that transforms personnel rows into `PersonnelEntry` objects for reuse across later repository read paths
   - Ensure the helper handles multiple time fields, multiple boolean flags, and optional values consistently

2. **Create Operation Capability**
   - Persist new personnel records with the required defaults for logged time, contact time, and active state
   - Enforce the alarm-related validity rule before storing invalid personnel alarm data
   - Return the database-generated identifier after successful creation
   - Use adapter-backed execution while respecting the transaction boundary from 001.003

3. **Testing Capability**
   - Add focused tests that verify round-trip mapping, default handling, type conversions, and alarm validation behavior

### Architectural Guidance

See `ai_instructions/architecture/db_architecture.md` for:
- Repository pattern CRUD implementation
- Repository-vs-adapter delegation boundaries
- Helper method reusability patterns

### Technical Considerations
- PersonnelEntry has more time- and state-oriented fields than the other entry types, so consistent conversion behavior matters more here
- Defaults for contact time and active state are part of the expected create behavior for newly logged personnel records
- Alarm-enabled entries require enough information to support later overdue-check workflows
- `supersedes` remains stored data at this layer; relationship workflow handling stays outside this story

## Context & Background

PersonnelEntry is the most complex of the three entities:
- **Most datetime fields**: 3 (vs 2 for Comm/Event)
- **Most boolean fields**: 4 (vs 2 for Comm/Event)
- **Historical tracking**: active flag, supersedes relationships
- **Alarm management**: alarm_enabled, expected_checkin_time, alarm_triggered

Despite that complexity, this story still follows the same repository pattern established by the earlier CommunicationEntry and EventEntry create stories:
- Shared row-to-entity conversion handles database-to-domain translation
- Create logic stores the entity via the adapter boundary and returns the assigned identifier
