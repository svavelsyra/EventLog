## Purpose (WHY) ⭐ MOST IMPORTANT

`Nollställ` should mean one button, one trigger, one cleanup flow, and one execution order regardless of whether the operator is reacting to an emergency or has time to stay and review the result. The earlier controlled-versus-emergency discussion was about two operator scenarios around the same reset behavior, not about maintaining two different reset modes, screens, or procedures.

This story exists to define the caller-visible result contract for that one shared flow: after the cleanup sequence runs, the caller should receive a coarse safe result such as `NOLLSTÄLLD` or `MISSLYCKADES`, plus sanitized follow-up hints when cleanup was incomplete. The same result contract should be produced every time; the difference between the urgent and non-critical scenarios is only whether the operator later consumes that information. The underlying sequence should still reflect emergency precedence by doing the fastest highest-value denial/cleanup work first and only then moving on to slower lower-gain cleanup steps.

Without this story, later implementation could drift into unsafe or inconsistent behavior such as:
- turning one reset flow into separate controlled and emergency implementations
- forcing later callers to invent their own meaning for partial cleanup outcomes
- leaking overly specific cleanup failures to callers in ways that become an attack oracle
- failing to tell a non-critical-use operator what category of cleanup still needs manual attention

### User Value
- All reset entry points can rely on one shared cleanup flow and one shared result contract.
- Operators in a non-critical setting can get sanitized hints about what still needs manual cleanup.
- Operators in an emergency can still use the same button and same flow without depending on detailed result review.

## Goal (WHAT)

Define the caller-visible result contract for the one shared `Nollställ` reset flow, including a coarse final `NOLLSTÄLLD` / `MISSLYCKADES` outcome and sanitized follow-up hints for incomplete cleanup, while explicitly acknowledging two operator scenarios around that same procedure without changing the underlying cleanup order, trigger, or reset flow.

### Acceptance Criteria
- [x] The same shared `Nollställ` button, flow, and execution order is used in both operator scenarios, with emergency-priority sequencing preserved from `002.006.001`: fastest highest-value denial/cleanup work happens first, while slower lower-gain cleanup steps may continue afterward without becoming a separate reset procedure.
- [x] The caller-visible result contract produces a coarse safe final outcome of `NOLLSTÄLLD` or `MISSLYCKADES` based on the shared cleanup results, without requiring callers to interpret low-level phase details themselves.
- [x] When the final outcome is `MISSLYCKADES`, the contract can provide sanitized actionable hints grouped by cleanup concern such as database artifacts, logs, or bootstrap-reset follow-up, so the same result remains useful for manual cleanup or pressing `Nollställ` again without exposing secret material or precise attack-helpful details.
- [x] Focused tests verify practically useful but sanitized follow-up, safe interruption/shutdown tolerance when detailed hints are never consumed, and retry-safe behavior where rerunning `Nollställ` does not turn already completed cleanup into a new reported failure.
- [x] If multiple reset entry points consume this contract, focused tests verify they map the shared reset outcome consistently rather than inventing entry-point-specific meanings for success or failure.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep one underlying reset button, flow, and order** - The urgent and non-critical scenarios must use the same cleanup sequence from `002.006.001`.
- **Preserve emergency-priority ordering inside that one flow** - Fast high-value denial/cleanup work must happen first, with slower lower-value cleanup attempted afterward in the same procedure.
- **Return a coarse caller-safe final outcome** - Callers should be able to treat the result as a simple `NOLLSTÄLLD` or `MISSLYCKADES` conclusion without parsing low-level cleanup internals.
- **Allow sanitized follow-up hints on `MISSLYCKADES`** - The same result contract must be able to name which cleanup concern still needs manual attention, such as logs or bootstrap-reset follow-up, without exposing secret material or detailed attack-helpful specifics.
- **Support interruption as a normal emergency case** - The reset remains valid even if the operator leaves immediately and never consumes follow-up hints.
- **Stay entry-point reusable** - Different reset triggers may use the same result contract without inventing separate meanings for success or failure.
- **Allow retry** - The caller contract should not preclude the operator from triggering `Nollställ` again if they choose to do so after reviewing follow-up hints.

### Negative Constraints (MUST NOT)
- **Don't redefine reset sequencing** - Immediate denial ordering and shared flow behavior belong to `002.006.001`.
- **Don't absorb backend, database, or non-database cleanup implementation** - Those concerns belong to `002.006.002`, `002.006.003`, and `002.006.004`.
- **Don't turn this into the startup-dialog layout story** - Button placement, widget text polish, and confirmation UX remain with the relevant GUI flow such as `002.005`.
- **Don't require separate scenario-specific implementations** - The situations differ in urgency and in how much result detail is consumed, not in what button or reset flow runs.
- **Don't leak an attack oracle** - Caller-visible messages must not expose precise cleanup weaknesses, secret locations, or unnecessary artifact-level detail.

### Soft Preferences (SHOULD)
- Prefer a very small result vocabulary such as `NOLLSTÄLLD`, `MISSLYCKADES`, and a limited set of sanitized follow-up hint categories.
- Keep non-critical-use reporting readable enough for post-run operator understanding without becoming a forensic cleanup transcript.
- Allow callers to ignore detailed hints entirely when immediate exit is safer or more realistic.

## Implementation Notes

### Suggested Approach
1. Reuse the neutral outcome contract from `002.006.001` and the sanitized phase results supplied by sibling cleanup stories.
2. Define a caller-side mapping layer that translates the shared cleanup outcome into a coarse final result of `NOLLSTÄLLD` or `MISSLYCKADES` plus optional sanitized follow-up hints.
3. Always produce the same result contract; operators may continue manual cleanup, retry `Nollställ`, or ignore the details entirely depending on what the situation allows.
4. Let the urgent scenario tolerate immediate exit, sleep, or shutdown without assuming the operator will read or confirm anything, while still using the same trigger, the same underlying procedure, and the same result contract.
5. Add focused tests for shared outcome mapping, interruption tolerance, and sanitized follow-up behavior.

### Technical Considerations
- The important boundary here is caller interpretation, not cleanup execution.
- This story should not introduce an alternate "emergency flow"; emergency priority is expressed through the shared ordering, not through separate buttons or branching procedures.
- The key distinction is two operator scenarios around one procedure, not two technical reset modes.
- A useful failure contract may group follow-up hints by cleanup concern such as database footprint, logs, or bootstrap-reset handling rather than exposing raw per-step internals.
- Entry points from startup, toolbar, or later reset surfaces should be able to share the same shared-flow outcome handling rules even if their surrounding UI differs, but the story should not over-claim that reuse until a second real entry point exists and is verified.
- Retry should be treated as a normal caller option: if the operator triggers `Nollställ` again, already-removed artifacts should not be surfaced as new failures, and follow-up hints should focus on remaining cleanup concerns.
- This story should preserve the project's realism principle: best effort can be reported honestly without implying guaranteed forensic erasure or exposing precise cleanup gaps.

### Affected Components
- Caller-side reset outcome handling in startup or in-app reset integration layers
- Presenter/controller logic that reacts to one shared reset outcome contract without branching into separate emergency/non-emergency reporting modes
- Focused unit/integration tests for shared-flow reporting and interruption handling

## Context & Background

The parent `002.006` story originally mixed low-level cleanup mechanics with caller-visible behavior. Splitting those concerns makes this child story the place where the project defines what callers see after the shared reset flow runs, without causing the shared cleanup contract itself to fork.

This also resolves the approved urgent-scenario edge case explicitly: the operator may trigger `Nollställ` and immediately shut the laptop, accept sleep, or otherwise abandon the session before detailed cleanup reporting is consumed. That behavior should still be considered valid because the same reset flow has already started; the difference is only whether any follow-up hint is later consumed.

In other words, the project has one reset procedure with emergency-priority ordering, not an emergency procedure and a separate normal procedure.
