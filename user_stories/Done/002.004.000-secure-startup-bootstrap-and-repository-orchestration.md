## Purpose (WHY) ⭐ MOST IMPORTANT

Even with good KDF helpers and a working SQLCipher adapter, the application still needs one coherent startup path that decides what kind of database target is being opened, gathers the approved bootstrap inputs, constructs the right adapter, and returns a ready-to-use repository only after secure readiness succeeds.

This story exists to keep startup orchestration from leaking across GUI, adapter, and repository layers. Without it, the project risks creating half-ready repositories, duplicating unlock logic in presenters, or burying backend-selection assumptions inside the wrong layer.

### User Value
- Application startup becomes predictable and reviewable.
- Higher layers receive a ready repository instead of managing low-level encrypted-open details.
- Training and operational deployments can follow the same startup pipeline with different approved inputs.

---

## Goal (WHAT)

Implement the bootstrap/orchestration path that reads the approved encrypted-startup config, gathers credential inputs from the caller, derives the encryption key, runs adapter readiness, and only then constructs and returns the repository used by the rest of the application.

### Acceptance Criteria
- [x] A startup/bootstrap orchestration path exists that resolves the current database target/profile from approved config input, validates whether key-file mode is required, and gathers the exact inputs needed for encrypted open/create.
- [x] The orchestration path uses the `002.002` helper layer for validation/derivation and the `002.003` adapter readiness flow before constructing the repository/factory result.
- [x] The factory/bootstrap boundary guarantees callers receive either a ready repository or an explicit failure outcome, never a partially initialized repository with deferred encrypted-open work.
- [x] Delivery includes focused tests for password-only versus key-file-required startup paths, plus clear failure surfaces for bad credentials, missing bootstrap data, and unsupported backend/profile combinations.

### Current Delivery Snapshot
- `src/db/repositories/startup_bootstrap.py` now provides the non-GUI orchestration seam for secure startup through `BootstrapRepositoryRequest`, `BootstrapRepositoryResult`, and `bootstrap_repository(...)`, keeping credential handling and startup failure mapping out of presenters and widget code.
- The orchestration path now validates approved bootstrap target inputs before repository construction begins: it rejects incomplete remembered targets, unsupported dialects, and missing required key-file input through explicit `BootstrapFailure` outcomes instead of letting partially valid startup state leak downward.
- Credential preparation now flows through the approved shared/helper boundary from `002.002`: `validate_password(...)` and `load_key_file_bytes(...)` handle generic validation, while backend-owned key preparation is resolved through `resolve_startup_key_preparer(...)` and run before repository construction.
- Repository creation now happens only after the bootstrap layer has validated inputs and prepared backend-ready key material. `RepositoryFactory.create_event_log_repository(...)` remains the construction seam, and adapter-readiness failures from `002.003` are mapped to explicit caller-visible startup failure codes such as invalid credentials, profile mismatch, migration needed, newer database, or repository-open failure.
- The success path now returns a ready repository together with reset-support callbacks, while failure paths return an explicit `BootstrapFailure` and no repository, preserving the story's intended “ready repository or explicit failure” contract.
- Focused coverage now exists in `tests/unit/db/test_startup_bootstrap.py` for incomplete bootstrap targets, unsupported dialects, required key-file mode, password validation failures, password-only and key-file-assisted key preparation, invalid prepared-key handling, wrong-credential/profile/version failure mapping, and successful repository handoff into the factory seam.
- A focused closeout verification run now also passed for `tests/unit/db/test_startup_bootstrap.py`, which confirms that the startup-bootstrap seam still behaves as expected in the current environment rather than only matching the story text structurally.

### Closeout Note
- No further story-local gap is currently known in the non-GUI startup/bootstrap orchestration slice for the current Epic `002` scope. The explicit request/result contract, backend-policy handoff, ready-repository-or-failure boundary, and focused startup-bootstrap tests are enough to treat this story as complete.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Respect architecture boundaries** - Bootstrap/factory code orchestrates; adapter code performs backend readiness; repository code stays focused on persistence workflows.
- **Keep caller contract explicit** - Startup code must make it clear when the app should create a new encrypted database versus unlock an existing one.
- **Preserve single-database assumptions** - The flow operates on one active database target at a time.
- **Remain compatible with Epic `001` construction patterns** - The repository returned from the factory should still fit the existing application architecture.

### Negative Constraints (MUST NOT)
- **Don't bury GUI widget logic here** - Dialog state and controls belong to `002.005`.
- **Don't redo key derivation here** - This story consumes the approved helper layer rather than duplicating it.
- **Don't let repository code trigger its own encrypted-open decisions** - That would undermine the bootstrap separation.
- **Don't add authentication, session management, or attempt-limiting policy** - Epic `002` is about database bootstrap security, not user-account security.

### Soft Preferences (SHOULD)
- Prefer a narrow startup service or factory-facing helper over startup logic spread across `main.py` and presenters.
- Make failure outcomes easy for GUI code to map to retryable versus terminal startup states.
- Keep the orchestration API small enough that it can be tested without launching the GUI.

## Implementation Notes

### Suggested Approach
1. Define the startup orchestration entry point and the inputs it expects from config/UI.
2. Use the `002.002` validation and derivation helpers to normalize credentials.
3. Invoke the encrypted adapter readiness path from `002.003`.
4. Construct the repository/factory result only after readiness succeeds.
5. Add tests that cover mode selection and failure handling without requiring the full GUI.

### Technical Considerations
- The flow must distinguish bootstrap defaults for new database creation from authoritative settings for existing encrypted databases.
- Startup logic should not assume every future backend is file-based, even if the current SQLite path is.
- Caller-visible outcomes may need to distinguish wrong credentials, missing files, unsupported configuration, and corrupt databases.
- This story is the natural seam between non-GUI startup policy and the later dialog/presenter story.

### Affected Components
- Startup/bootstrap module(s)
- `src/db/repositories/repository_factory.py` or adjacent factory/service code
- Config/bootstrap parsing integration
- Tests for startup orchestration and failure outcomes

## Context & Background

`db_architecture.md` explicitly separates generic bootstrap orchestration from backend-specific readiness. Epic `002` is the first place that distinction becomes operationally important, because encrypted startup adds several failure modes that higher layers should not have to re-implement.

This story takes the non-GUI half of that startup problem so `002.005` can focus on user interaction rather than architecture decisions.
