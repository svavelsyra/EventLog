## Purpose (WHY) ⭐ MOST IMPORTANT

Operational communication configuration is not static. Nets move, labels change, qualifier behavior may need adjustment, and the operator may need to add or adjust live communication options during use. Epic `003` therefore needs more than read-only configuration tables: it needs a disciplined targeted-update seam so later communication workflows can add or adjust approved communication metadata without bypassing the architecture.

This story exists to provide that targeted mutation path. Without it, later work could drift into:
- GUI dialogs writing directly to configuration tables
- new channel/tier options or qualifier changes appearing only in temporary UI state instead of the database source of truth
- configuration changes rewriting saved communication entries instead of affecting only future choices
- ad hoc reload behavior after updates, leaving some callers stale and others refreshed

### User Value
- Operators can later add or adjust communication configuration quickly when operational reality changes.
- New communication options can become available to the form immediately through an approved update path.
- Historical communication entries remain accurate even when live config is updated.

---

## Goal (WHAT)

Add the repository/core update support needed for later quick-add and focused communication-configuration management flows.

### Acceptance Criteria
- [x] Targeted write operations exist for the communication-configuration areas that later workflows need to change in practice, especially tier options/labels where operators make live selections and any related qualifier definitions or override behavior that must support limited operational updates.
- [x] The update contract supports a quick-add style flow where a newly added active communication option can be persisted and then selected by the communication workflow without requiring a full admin screen.
- [x] Update behavior preserves historical log integrity by changing configuration rows for future use rather than rewriting saved `CommunicationEntry` values that were already logged.
- [x] Focused tests verify add/update/deactivate behavior, duplicate/conflict handling, and the expected runtime reload or refresh interaction after a successful change.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep mutation ownership below the GUI** - Later dialogs/forms must call repository/core update seams rather than issue SQL directly.
- **Support fast operational additions** - At minimum, the contract must handle practical quick-add needs such as a new selectable tier option during live use.
- **Preserve history** - Updating config must not retroactively change the values stored on historical communication entries.
- **Integrate with the reload contract** - Successful updates must fit the runtime refresh behavior defined by `003.003`.
- **Respect the refined communication shape** - Update support must work with neutral tiers and top-level qualifiers instead of assuming one radio-only system/method/channel stack.

### Negative Constraints (MUST NOT)
- **Do not require a broad configuration-management UI** - This story creates backend/core support for later focused flows, not a whole settings screen.
- **Do not expand into general import/export** - Bulk portability is handled in `003.006`.
- **Do not bypass repository/core boundaries with view-side mutation helpers** - The architecture seam must stay reviewable.
- **Do not absorb event category/priority cleanup** - That belongs to Epic `013`.
- **Do not force every qualifier rule into live operator editing in Phase 1** - Some qualifiers may be fixed or hidden per system, and this story should not overbuild a general rule editor unless later scope explicitly demands it.

### Soft Preferences (SHOULD)
- Prefer non-destructive deactivate/update patterns over hard deletes when practical.
- Keep duplicate-handling rules explicit enough that a quick-add dialog can surface a clear outcome later.
- Favor small targeted methods over one giant "save all configuration" mutation entry point.

---

## Implementation Notes

### Suggested Approach
1. Add focused repository write methods for the communication-config options that later operator flows must change in practice, starting with tier options and adjacent metadata.
2. Expose a core-facing service or helper if later presenters should not call repository writes directly.
3. Define duplicate/update/deactivate behavior explicitly.
4. Verify that successful writes cooperate with the runtime-config reload contract.

### Technical Considerations
- The GUI design already assumes later fast-add communication updates; this story creates the backend/core seam those flows will need.
- Not every communication-config table needs broad CRUD in Phase 1. The story should stay focused on the practical targeted updates later workflows actually need.
- If qualifier-definition updates or deeper per-path rules become too large for this story once implementation begins, they are candidates for further split under `003.004.xxx`.

### Affected Components
- Repository update methods for communication configuration
- Core helpers/services coordinating targeted updates and reload behavior
- Focused DB/core tests for quick-add/update flows

## Context & Background

Epic `003` explicitly mentions repository access patterns for targeted updates and the communication discussion has now made clear that the update seam must be broader than channel-designation rows alone. That makes this story the bridge between static seeded config and real operational adaptability.

It is intentionally narrower than a full administration feature: the point is to enable practical quick changes without letting every later GUI slice invent its own persistence shortcut.

- [x] Duplicate or conflicting communication-option adds return a defined caller-visible outcome.
- [x] Deactivating or updating live communication config affects future runtime choices without corrupting historical entry data.
- [x] Successful targeted updates cooperate with the defined runtime reload/refresh behavior.
