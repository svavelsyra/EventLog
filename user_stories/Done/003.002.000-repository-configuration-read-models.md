## Purpose (WHY) ⭐ MOST IMPORTANT

Once Epic `003` configuration tables exist, the application still needs one disciplined way to read them. If presenters, validators, and startup code each begin querying operational-config tables directly or inventing parallel helper functions, the project will recreate the same coupling Epic `003` is supposed to prevent.

This story exists to define repository-owned read access for operational configuration so higher layers can consume typed results instead of SQL knowledge. Without it, later work could drift into:
- GUI or core code knowing table names and join details
- inconsistent JSON parsing for tier options, qualifier definitions, and per-system override behavior
- repeated ad hoc filtering of inactive versus active configuration rows
- multiple callers interpreting ordering/default rules differently

### User Value
- Configuration-backed workflows can rely on one repository contract instead of ad hoc SQL.
- Later core and GUI stories get cleaner typed inputs for systems, configured options, and qualifiers.
- Reviewers can verify config-read behavior in one layer instead of hunting across presenters and validators.

---

## Goal (WHAT)

Add repository-level read models and query methods for Epic `003` operational configuration data.

### Acceptance Criteria
- [x] Repository APIs exist to read the active communication configuration needed by later workflows, including communication systems/ways, recursive configured options by system, and top-level qualifier definitions plus per-system override behavior.
- [x] Repository row-mapping and serialization logic converts storage details such as JSON arrays, booleans, and ordering fields into stable caller-facing Python shapes without leaking SQL/storage mechanics upward.
- [x] Repository behavior defines consistent handling for active-only reads, deterministic ordering, and missing/inactive records so later callers do not invent their own filtering rules.
- [x] Focused tests verify the repository can read seeded communication configuration correctly, including multi-table communication config assembly for both radio and non-radio examples.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep SQL ownership in the repository layer** - Higher layers must not need to know table shapes or joins.
- **Return stable caller-facing structures** - Read methods must produce predictable Python data shapes suitable for core/runtime loading.
- **Support refined communication-shape assembly** - The repository contract must make it practical to reconstruct a communication system/way with its tiers, selectable options, and top-level qualifier behavior without repeated caller-side stitching.
- **Handle active/inactive state consistently** - The repository defines what the normal runtime read returns by default.

### Negative Constraints (MUST NOT)
- **Do not absorb core caching/reload policy** - That belongs to `003.003`.
- **Do not absorb quick-add/update mutation flows** - Targeted writes belong to `003.004`.
- **Do not let presenters or validators query raw tables directly** - This story exists specifically to avoid that drift.
- **Do not redefine report-template reads** - `report_templates` remain Epic `004` territory.
- **Do not collapse the model back into radio-only assumptions** - Repository read shapes must remain usable for non-radio examples such as `Kurir`, even when radio systems dominate the first seed set.

### Soft Preferences (SHOULD)
- Prefer typed dataclasses or similarly explicit read shapes over loose nested dictionaries when practical.
- Keep method names and return contracts aligned with future `SystemConfig` loading needs.
- Favor read methods that are easy to test against in-memory SQLite.

## Implementation Notes

### Suggested Approach
1. Add focused repository methods for each operational-config domain.
2. Create private row-mapping/JSON-conversion helpers as needed.
3. Define clear default behavior for ordering and active filtering.
4. Add DB tests that assemble full configuration from seeded data.

### Technical Considerations
- The communication configuration path likely needs more than one simple table read because systems, tier options, qualifier definitions, and override rules are related but may be stored separately.
- The repository contract should stay practical for the current single public repository shape, even if internal extraction into smaller repository files happens later.

### Affected Components
- `src/db/repositories/sqlite/` repository implementation
- Any shared repository-facing config read models/helpers
- Focused DB tests for operational configuration reads

## Context & Background

Epic `003` is only valuable if later runtime code can treat operational configuration as a normal dependency instead of a special-case database spelunking exercise. This story is the DB-layer seam that makes that possible.

It also protects the architecture boundary from a common failure mode: "temporary" config lookup code in presenters that becomes permanent because no stable repository contract existed early enough.
