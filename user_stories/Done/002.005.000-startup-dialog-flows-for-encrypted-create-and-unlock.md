## Purpose (WHY) ⭐ MOST IMPORTANT

The secure bootstrap flow only helps operators if it is presented through a startup interaction that is clear under field conditions. Users need an obvious way to create the first encrypted database, unlock an existing one, recover from bad credential entry, and trigger emergency destruction when necessary.

This story exists to turn the lower-level Epic `002` contracts into a user-facing workflow that remains operationally simple. It keeps the startup UI focused on collecting and validating the right inputs, not on re-implementing security logic.

### User Value
- Operators can create or unlock the secured database without needing to understand internal crypto details.
- Wrong credentials and missing key files lead to clear retryable behavior.
- Emergency `Nollställ` remains reachable at startup even when normal unlock cannot proceed.

---

## Goal (WHAT)

Implement the GUI/presenter startup dialog flows for first-run encrypted database creation, later unlock, optional versus required key-file mode, and handoff into the secure bootstrap orchestration path.

### Acceptance Criteria
- [x] The startup UI supports at least the approved create and unlock cases: first-run encrypted database creation, existing password-only unlock, and existing key-file-plus-password unlock when `require_key_file` is true.
- [x] The create flow validates password confirmation, minimum password length, and optional key-file rules before invoking the bootstrap/orchestration layer, and persists only the approved bootstrap fields for the newly created database.
- [x] The unlock flow shows clear retryable errors for wrong password/key-file combinations, does not impose a fake attempt limit, and cleanly supports cancel/exit behavior when the user chooses not to continue.
- [x] Delivery includes focused presenter or GUI-facing tests for the main dialog states, validation failures, and successful handoff into the non-GUI startup/bootstrap path.

### Current Delivery Snapshot
- `src/gui/presenters/startup_dialog_presenter.py` now owns the main create/unlock dialog behavior for this story: it computes create-versus-unlock state from the selected/remembered target, applies local validation before bootstrap handoff, maps coarse bootstrap failures into presenter-visible retry behavior, and returns only approved remembered bootstrap state through `StartupDialogSuccess.remembered_target`.
- `src/gui/startup_dialog_controller.py` now keeps the GUI layer thin: it wires the presenter into the Tk view, supports create/unlock target selection, browse actions, cancel/exit handling, retry behavior, and a dedicated migration action when bootstrap reports that migration is required before unlock can continue.
- `src/gui/views/startup_dialog_view.py` now renders presenter-owned startup state without embedding security/bootstrap policy in widget code. It shows/hides dialect, target, password-confirmation, and optional/required key-file fields according to presenter state and preserves entered values across rerenders where that helps a safe retry.
- Successful startup continues to persist only approved remembered bootstrap state through `StartupDialogSuccess.remembered_target` and the centralized DB-owned `src.db.repositories.bootstrap_backend_policy.save_bootstrap_target_config(...)` seam, without remembering key-file paths for convenience.
- Focused coverage now exists in `tests/unit/gui/presenters/test_startup_dialog_presenter.py`, `tests/unit/gui/test_startup_dialog_controller.py`, and `tests/unit/gui/views/test_startup_dialog_view.py` for initial create/unlock state selection, create-time password confirmation/minimum-length validation, password-only versus key-file-required unlock behavior, retryable invalid-credential handling, cancel flow, migration-needed UI handling, browse/selection rerender behavior, and successful handoff into the non-GUI startup/bootstrap seam.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep GUI and logic separated** - Presenters or equivalent dialog logic should own decisions; widgets should not embed security policy.
- **Match the approved bootstrap contract** - The dialog only collects and persists fields approved by `002.001`.
- **Support both deployment modes** - Training/password-only and operational/key-file-required flows must both remain usable.
- **Stay operationally simple** - Startup interactions should minimize clutter and ambiguity on constrained field hardware.

### Negative Constraints (MUST NOT)
- **Don't derive keys or open SQLCipher directly in widget code** - Use the lower-level orchestration path from `002.004`.
- **Don't remember key-file paths for convenience** - That would leak sensitive operational information.
- **Don't add password-strength meters or Phase 2 polish here** - Keep the first startup UI focused on required secure access behavior.
- **Don't turn this into a general settings/preferences UI** - It is a startup access flow, not a broader configuration screen.

### Soft Preferences (SHOULD)
- Prefer explicit labels and states over clever UI shortcuts.
- Keep Swedish user-facing messages consistent with existing security design wording where already specified.
- Make the key-file UI disappear cleanly when the active mode does not require it.

## Implementation Notes

### Suggested Approach
1. Implement or extend a dedicated startup dialog/presenter for encrypted create and unlock flows.
2. Keep field validation and state transitions in the presenter layer where possible.
3. Call the non-GUI bootstrap/orchestration path from `002.004` after local validation succeeds.
4. Add focused tests for presenter state and dialog outcomes instead of relying only on manual startup testing.

### Technical Considerations
- First-run create behavior differs from unlock because it needs password confirmation and initial mode selection.
- `require_key_file` affects whether key-file UI is visible/required during unlock.
- Wrong-credential behavior should clear or preserve only the fields that help a safe retry.
- `Nollställ` entry should be visible from the unlock context but implemented as its own cleanup story.

### Affected Components
- New or updated startup dialog/view under `src/gui/`
- Presenter/controller logic for startup state and validation
- GUI/presenter tests
- Integration with the secure bootstrap/orchestration path

## Context & Background

The security design already defines three startup cases: create, unlock, and emergency `Nollställ`. This story intentionally covers only the create/unlock UI behavior and handoff. The destructive cleanup implementation stays in `002.006` so the GUI story does not also absorb file-deletion logic.

That split keeps the startup interface reviewable while still leaving the operator with a complete secure-access workflow.
