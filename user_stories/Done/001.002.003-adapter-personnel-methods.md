## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

This story exists because some repository-facing behaviors do not fit neatly into generic CRUD methods. Personnel tracking especially needs specialized query capabilities, but under the approved architecture those belong in repository contracts and concrete repositories, not in the low-level adapter.

1. **Places specialized queries at the correct layer** - Convenience queries stay in repository-facing design
2. **Preserves room for non-generic behavior** - Repository APIs can expose useful operations beyond bare CRUD
3. **Supports personnel safety features later** - Overdue alarm and history-style queries still have an architectural home

Without this direction, later stories risk either forcing everything into generic CRUD or incorrectly pushing entity-specific queries back into the low-level adapter.

### User Value
- Cleaner repository API design for specialized workflows
- Personnel safety and history queries still supported in the correct layer
- Less architectural drift between generic CRUD and specialized repository behavior

## Goal (WHAT)

**What functionality are we delivering?**

Define the repository-facing direction for specialized query methods that sit alongside generic CRUD, with personnel tracking as the primary motivating example.

### Acceptance Criteria

- [ ] Repository contract/design direction explicitly allows specialized query methods in addition to generic CRUD
- [ ] Specialized query examples are documented as repository-facing behavior, not low-level adapter behavior
- [ ] Personnel-oriented repository methods are identified as examples of valid specialized queries, such as:
  - retrieve active personnel entries
  - retrieve personnel history for a person/unit
  - retrieve overdue alarm entries
- [ ] Any contract guidance produced by this story stays dialect-agnostic and avoids SQLite-specific query details
- [ ] All methods have type hints for parameters and return values
- [ ] All methods have docstrings explaining purpose, args, returns, and exceptions
- [ ] Method grouping distinguishes generic CRUD direction from specialized repository query direction

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep specialized queries in the repository layer** - Do not add them to `DatabaseAdapter`
- **Follow approved split** - Generic CRUD direction stays in the base repository; specialized queries are repository-facing extensions
- **Type hints required** - All parameters and returns typed
- **Keep contract/design dialect-agnostic** - No SQLite-specific behavior here

### Negative Constraints (MUST NOT)
- **Don't implement any methods** - Abstract only
- **Don't add specialized queries to `DatabaseAdapter`** - That architecture direction has been superseded
- **Don't implement SQLite query logic here** - Concrete repository stories handle that later
- **Don't add transaction methods here** - That's story 001.002.004

### Soft Preferences (SHOULD)
- Explain clearly why specialized repository queries are valid even when a generic CRUD base exists
- Keep examples focused on personnel/history/alarm use cases without over-specifying implementation details

## Implementation Notes

### What's Needed

Add repository-facing specialized query direction for later concrete repositories:

1. **Active Entries Query Direction**
   - Purpose: Support repository-facing retrieval of currently active personnel entries
   - Returns: Collection of active personnel-oriented records/entities

2. **History Query Direction**
   - Purpose: Support repository-facing retrieval of complete history for a person/unit
   - Returns: Collection of historical records/entities for the requested subject

3. **Overdue Alarm Query Direction**
   - Purpose: Support repository-facing retrieval of overdue alarm entries for safety workflows
   - Returns: Collection of overdue alarm records/entities

4. **Search/Filter Extension Direction**
   - Purpose: Allow concrete repositories to expose specialized search/filter methods when generic CRUD alone is insufficient
   - Returns: Repository-defined search/filter results appropriate for the entity/workflow

## Context & Background

Personnel-oriented workflows are the clearest example of why a repository layer sometimes needs specialized queries beyond a generic CRUD base:
- **Active vs inactive** views
- **Historical tracking** for one person/unit
- **Alarm management** for overdue check-ins

This story keeps those needs visible while placing them in the correct architectural layer.
