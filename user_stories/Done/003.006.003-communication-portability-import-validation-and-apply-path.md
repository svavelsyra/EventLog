## Purpose (WHY) ⭐ MOST IMPORTANT

Export alone is not enough; the communication configuration must also be restorable into a clean or recreated database through an approved import/apply path. Without that, the bundle cannot complete the recovery workflow.

### User Value
- A reset or fresh setup can actually restore communication configuration.
- Import validation and apply behavior stay reviewable separately from export and broader recovery UX.
- The database remains the post-import runtime authority instead of the bundle becoming a live config source.

---

## Goal (WHAT)

Implement import validation and database apply behavior for the approved communication portability bundle.

### Acceptance Criteria
- [x] Import support validates the portable bundle against the approved contract from `003.006.001`, including version and required communication sections.
- [x] Approved communication configuration can be restored into the database through repository/bootstrap/update seams rather than bypassing normal ownership boundaries.
- [x] Import behavior defines how existing communication config is updated, replaced, merged, or deactivated so the result is deterministic and reviewable.
- [x] Focused tests verify successful apply behavior, version rejection, malformed-bundle rejection, and exclusion enforcement.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Validate before apply** - Unsupported or malformed bundles must fail safely before partial import changes are committed.
- **Keep the database as final authority** - After import, normal runtime reads should come from the database, not the file.
- **Respect communication update/reload seams** - Import should fit the communication update/runtime contracts instead of introducing a side channel.

### Negative Constraints (MUST NOT)
- **Do not absorb export generation here** - Export belongs to `003.006.002`.
- **Do not silently import excluded data** - Out-of-scope sections must be rejected or ignored by explicit contract, not accidentally applied.
- **Do not turn this into full setup UX** - Recovery usage flow polish belongs to the later recovery/guardrail story.
