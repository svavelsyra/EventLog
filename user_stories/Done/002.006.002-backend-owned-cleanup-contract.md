## Purpose (WHY) ⭐ MOST IMPORTANT

The shared `Nollställ` flow in `002.006.001` can only stay backend-agnostic if backend-native close, lock, and delete mechanics are owned by the active loaded persistence technology rather than hardcoded into the top-level reset coordinator. That ownership boundary matters because different persistence technologies may need very different shutdown, lock release, handle invalidation, or deletion preparation behavior even when they all participate in the same reset flow.

This story exists to define that seam clearly now, while SQLite is still the only concrete backend, so later work can add concrete cleanup behavior without baking SQLite assumptions into the permanent architecture.

Without this story, later implementation could drift into unsafe or brittle behavior such as:
- putting SQLite-specific close/delete logic directly inside the shared reset coordinator
- forcing future backends to pretend they behave like SQLite just to fit reset handling
- making artifact-cleanup stories guess who owns backend-native locking or handle-release behavior
- leaking concrete backend types into caller-facing reset contracts

### User Value
- Operators get a safer reset path because backend-native shutdown/lock semantics are owned where the real persistence mechanics live.
- The shared `Nollställ` flow can stay stable even if future database technologies need different cleanup mechanics.
- Later cleanup stories can build on one explicit backend seam instead of inventing ad-hoc backend hooks.

## Goal (WHAT)

Define the backend-owned cleanup capability that the shared `Nollställ` reset flow invokes after immediate denial, making the active persistence technology responsible for its own backend-native close/lock/delete preparation semantics while keeping the top-level contract neutral.

### Acceptance Criteria
- [x] A backend-owned cleanup contract exists beneath the shared reset flow so the active loaded persistence technology can perform its own backend-native close, lock, handle-release, and deletion-preparation steps after immediate denial without requiring the top-level coordinator to hardcode backend specifics.
- [x] The contract returns sanitized backend-phase results in a neutral shape that the shared reset flow can record without exposing concrete backend internals, secret material, or highly specific attack-helpful failure details.
- [x] SQLite satisfies the contract as the current concrete backend case, but the contract boundary does not require the shared reset flow or callers to branch on SQLite types/names as the long-term architecture.
- [x] Focused tests verify that the shared reset flow can invoke the backend-owned capability after denial, that missing active backend/capability state reports safely instead of crashing, and that backend-phase outcomes remain neutral enough for later caller-mode handling.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep ownership with the active backend** - Backend-native close/lock/delete preparation semantics belong to the loaded persistence technology.
- **Stay beneath the shared reset flow** - This story defines the seam used by `002.006.001`, not a parallel top-level reset implementation.
- **Keep the contract neutral** - Result/state data must be reusable by the shared flow and later caller-mode handling without leaking backend-specific presentation into higher layers.
- **Use SQLite only as today's proof point** - SQLite may demonstrate the seam now, but must not become the permanent architecture assumption.
- **Report unsupported or missing backend state safely** - The contract must allow caller-safe reporting when no active backend-owned cleanup capability is available.

### Negative Constraints (MUST NOT)
- **Don't implement database-footprint target lists here** - Concrete database file, WAL, SHM, journal, or other artifact classes belong to sibling cleanup stories.
- **Don't absorb log, key-file, or bootstrap cleanup here** - Those concerns belong to `002.006.004`.
- **Don't absorb caller-mode behavior** - Controlled-vs-emergency interpretation and reporting belongs to `002.006.005`.
- **Don't require top-level code to branch on SQLite** - The coordinator and callers must not need `if sqlite` logic to use this contract.
- **Don't over-prescribe one repository-only shape** - The seam may live on an adapter, specialized security repository, or equivalent backend-owned contract as long as ownership stays with the active persistence technology.

### Soft Preferences (SHOULD)
- Prefer a narrow interface that is easy for the shared reset coordinator to invoke once denial has completed.
- Keep backend-phase result categories explicit enough that later sibling stories can distinguish unsupported, skipped, partial, and successful backend cleanup preparation outcomes.
- Reuse existing persistence boundaries where practical instead of creating a parallel hierarchy with overlapping responsibility.

## Implementation Notes

### Suggested Approach
1. Define a backend-owned cleanup capability that can be invoked by the shared reset coordinator after immediate denial succeeds or is attempted.
2. Place that capability on the persistence side of the architecture, such as an adapter-owned method, a specialized security repository contract, or a closely related backend-owned seam.
3. Have the current SQLite path implement the capability as the first concrete proof that backend ownership works without leaking SQLite branching into the top-level flow.
4. Return a neutral backend-phase result that higher layers can record while leaving artifact-specific deletion and caller-mode interpretation to sibling stories.
5. Add focused tests for invocation order, safe missing-backend behavior, and neutral result reporting.

### Technical Considerations
- Existing persistence boundaries already live around `src/db/database_adapter.py`, `src/db/sqlite_adapter.py`, and `src/db/repositories/repository_factory.py`; this story should align with that direction instead of bypassing it.
- The important design decision is ownership, not whether the seam lands on an adapter versus a specialized backend-facing repository helper.
- Backend-owned cleanup may need to prepare later file deletion by releasing handles, closing connections, invalidating state, or exposing backend-known target information, but the actual artifact lists belong in sibling stories.
- Missing active backend context should surface as an explicit safe outcome category rather than a crash or silent hidden assumption.

### Affected Components
- Persistence-side cleanup contract near `src/db/database_adapter.py` or equivalent backend-owned seam
- Current SQLite implementation path such as `src/db/sqlite_adapter.py` and/or closely related repository integration
- Shared reset integration points that invoke the backend-owned capability after `002.006.001` denial sequencing
- Focused unit/integration tests for backend-owned invocation and neutral phase results

## Context & Background

The parent `002.006` story mixed both ownership and artifact scope. Splitting those apart keeps this story focused on one architectural rule: the active backend owns its backend-native cleanup mechanics, while the shared reset flow owns orchestration.

This matches the approved direction from the parent story: SQLite is today's concrete case, but future technologies may need different close/lock/delete semantics and should not be forced through a SQLite-shaped top-level API.
