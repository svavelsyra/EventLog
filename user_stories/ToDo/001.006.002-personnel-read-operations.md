## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

Operators need to retrieve and filter personnel tracking information:

1. **View specific personnel** - Look up current status by ID
2. **List all personnel** - See complete personnel roster
3. **Filter by name/callsign** - Find specific person/unit
4. **Filter by active status** - View only active deployments or historical entries
5. **Build UI views** - Personnel tab can display filtered lists

Without read operations, personnel tracking is write-only - operators cannot see who is where.

### User Value
- View current personnel status
- Filter by who, operator, active status
- Find specific personnel quickly
- Build functional GUI personnel views

---

## Goal (WHAT)

**What functionality are we delivering?**

Implement read and list operations for `PersonnelEntry` in the SQLite repository layer under the approved package layout.

### Acceptance Criteria

- [ ] **Get Single Entry by ID**:
  - Can retrieve a specific personnel entry by identifier
  - Reuses row-to-entity conversion from `001.006.001`
  - Returns `None` when the requested identifier is not found
- [ ] **Get All Entries with Filtering**:
  - Can retrieve all personnel entries when no filters are provided
  - Supports filtering by relevant personnel fields and time boundaries needed for personnel review workflows
  - Returns results in recency-first order and an empty list when nothing matches
- [ ] **Unit Tests Pass**:
  - Retrieve-by-ID behavior is verified, including the not-found case
  - Unfiltered list retrieval returns all expected entries
  - Individual filters and combined filters behave correctly
  - Ordering behavior is verified

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Implement in the SQLite repository layer** - Replace placeholder behavior using the approved repository package layout
- **Delegate low-level query execution downward** - Repository handles filtering/query intent and mapping; adapter handles low-level execution/fetch behavior
- **Use parameterized queries** - Prevent SQL injection
- **Follow Comm/Event pattern** - Same structure as previous read stories
- **Use _row_to_personnel_entry helper** - Reuse from 001.006.001
- **Sort by logged_time DESC** - Most recent status changes first

### Negative Constraints (MUST NOT)
- **Don't implement update/delete yet** - That's story 001.006.003
- **Don't implement specialized queries yet** - That's story 001.006.004 (get_active, get_history, get_overdue_alarms)

### Soft Preferences (SHOULD)
- Use a filtering approach consistent with other repository read operations
- Keep personnel-specific filters easy to extend and reason about
- Handle active-state filtering in a clear, testable way

## Implementation Notes

### What's Needed

1. **Single-Entry Retrieval Capability**
   - Retrieve one personnel entry by identifier and return `None` when it does not exist

## Context & Background

PersonnelEntry read operations should follow the same repository read pattern used for CommunicationEntry and EventEntry while adapting the supported filters to personnel tracking needs.

Under the approved architecture, the repository prepares the query and maps results, while the adapter handles the low-level fetch behavior.

This story covers:
- Single-entry lookup with not-found handling
- Filtered list retrieval for personnel review workflows
- Consistent descending ordering by logged status time

**Note**: Specialized queries such as active-only views, full personnel history, and overdue alarms are intentionally deferred to `001.006.004`.
