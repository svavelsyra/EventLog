## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

Transaction management is essential for data integrity in multi-step operations. These methods enable:

1. **Atomic operations** - Multiple database changes succeed or fail together
2. **Test isolation** - Tests can rollback changes for clean state
3. **Error recovery** - Failed operations can be rolled back without partial data corruption
4. **Resource management** - Connections can be closed cleanly

Example use cases:
- Personnel status update (create new entry + mark old inactive) - both must succeed
- Bulk operations - all entries or none
- Test setup - rollback after each test

Without transaction control, the application cannot guarantee data integrity for complex operations.

### User Value
- Data integrity guaranteed for multi-step operations
- Tests can run in isolation (rollback after each test)
- Failed operations don't leave partial data
- Clean shutdown (connections closed properly)

## Implementation Notes

### What's Needed

Four transaction/connection control capabilities to complete the adapter interface:

1. **Start Transaction**
   - Purpose: Begin explicit transaction for multi-step atomic operations
   - Must be followed by commit or rollback
   - Example use: Personnel status update (create new entry + mark old inactive = both must succeed)

2. **Commit Transaction**
   - Purpose: Make all changes since transaction start permanent
   - Should document potential failure modes (DatabaseError)

3. **Rollback Transaction**
   - Purpose: Discard all changes since transaction start
   - Use cases: Error recovery, test cleanup (rollback after each test)

4. **Close Connection**
   - Purpose: Clean resource shutdown
   - Use cases: Application shutdown, after test suite

### Transaction Patterns

**Without explicit transaction**: Individual CRUD operations work independently (may auto-commit)

**With explicit transaction**: 
- Start transaction
- Execute multiple CRUD operations
- Commit (success) or Rollback (failure/cleanup)

**Test isolation pattern**:
- Start transaction before test
- Execute test operations
- Rollback after test (cleanup, no permanent changes)
