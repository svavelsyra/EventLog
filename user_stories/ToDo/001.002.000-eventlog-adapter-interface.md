## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

The database contract layer is key to making EventLog's persistence layer testable and flexible. By separating low-level adapter responsibilities from application-facing repository responsibilities, we:

1. **Enable testing without mocking** - Swap real SQLite for in-memory database in tests
2. **Decouple business logic from database details** - Core and GUI layers work against interface, not concrete implementation
3. **Support future database backends** - Could add PostgreSQL, MySQL without changing calling code
4. **Enforce API contracts** - Adapter and repository layers each expose a clear boundary

Without this abstraction, the application would be tightly coupled to SQLite specifics, making testing painful and future changes risky.

### User Value
- More reliable application (better testability)
- Faster test execution (in-memory database)
- Future-proof architecture (easy to swap databases if needed)
- Clear API contract for developers

## Goal (WHAT)

**What functionality are we delivering?**

Define the foundational persistence contracts for Epic 001:
- a low-level `DatabaseAdapter` abstract base class for connection, schema, execution, fetch, and transaction behavior
- a generic repository contract/base that application-facing repositories can build on

These contracts will be used by the SQLite adapter and SQLite repository work in story 001.003.

  - Initialize schema
  - Execute SQL
  - Fetch many rows

  - Begin transaction
  - Commit transaction
  - Roll back transaction
  - Close connection/resources
- [ ] Generic repository contract or base is defined in the repository layer
- [ ] Repository contract direction is dialect-agnostic and does not embed SQLite-specific behavior
- [ ] Contract definitions align with `ai_instructions/architecture/db_architecture.md`
- [ ] All methods have type hints (Python 3.14 syntax)
- [ ] All methods have docstrings explaining purpose, parameters, and return values

## Context & Background

This contract story is inspired by layered architecture and Ports & Adapters. The low-level `DatabaseAdapter` defines backend-facing mechanics, while repositories define the application-facing persistence API above it.

**Why not put CRUD into the adapter contract**:
- CRUD/query behavior belongs to repositories, not low-level database adapters
- SQLite-specific execution details should not leak into higher-layer persistence APIs
- Future backends are easier to add when execution primitives and repository workflows are separated

**Design decisions from architecture docs**:
- Low-level adapter owns schema/execute/fetch/transaction mechanics
- Generic repository base stays dialect-agnostic
- SQLite-specific repositories live under `src/db/repositories/sqlite/`
