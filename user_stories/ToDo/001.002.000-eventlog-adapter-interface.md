## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

The Adapter pattern is the key to making EventLog's database layer testable and flexible. By defining an abstract interface, we:

1. **Enable testing without mocking** - Swap real SQLite for in-memory database in tests
2. **Decouple business logic from database details** - Core and GUI layers work against interface, not concrete implementation
3. **Support future database backends** - Could add PostgreSQL, MySQL without changing calling code
4. **Enforce API contract** - All implementations must provide the same methods

Without this abstraction, the application would be tightly coupled to SQLite specifics, making testing painful and future changes risky.

### User Value
- More reliable application (better testability)
- Faster test execution (in-memory database)
- Future-proof architecture (easy to swap databases if needed)
- Clear API contract for developers

## Goal (WHAT)

**What functionality are we delivering?**

Create an abstract base class `EventLogAdapter` that defines all database operations as abstract methods. This interface will be implemented by SQLiteEventLogRepository in story 001.003.

### Acceptance Criteria
- [ ] File `src/db/adapters/event_log_adapter.py` exists
- [ ] `EventLogAdapter` class inherits from `ABC` (abstract base class)
- [ ] CommunicationEntry operations defined as abstract methods:
  - Can create new entry (returns database ID)
  - Can retrieve single entry by ID (returns None if not found)
  - Can retrieve all entries with optional filtering
  - Can update existing entry (returns success status)
  - Can delete entry by ID (returns success status)
  - Can search entries by text with optional filtering
- [ ] EventEntry operations defined as abstract methods:
  - Can create new entry (returns database ID)
  - Can retrieve single entry by ID (returns None if not found)
  - Can retrieve all entries with optional filtering
  - Can update existing entry (returns success status)
  - Can delete entry by ID (returns success status)
  - Can search entries by text with optional filtering
- [ ] PersonnelEntry operations defined as abstract methods:
  - Can create new entry (returns database ID)
  - Can retrieve single entry by ID (returns None if not found)
  - Can retrieve all entries with optional filtering
  - Can update existing entry (returns success status)
  - Can delete entry by ID (returns success status)
  - Can retrieve only active personnel entries
  - Can retrieve personnel history for specific person
  - Can retrieve personnel entries with overdue alarms
- [ ] Transaction management capabilities defined:
  - Can start database transaction
  - Can commit current transaction
  - Can rollback current transaction
- [ ] Connection management:
  - Can close database connection and cleanup resources
- [ ] All methods have type hints (Python 3.14 syntax)
- [ ] All methods have docstrings explaining purpose, parameters, and return values

## Context & Background

This interface is inspired by the Repository pattern and Hexagonal Architecture (Ports & Adapters). The abstract base class acts as a "port" that defines the contract, while concrete implementations like SQLiteEventLogRepository are "adapters".

**Why single adapter (not three separate adapters)**:
- Simpler factory pattern (one repository to create)
- Shared connection/transaction context across entity types
- Easier to coordinate multi-entity operations
- Less boilerplate

**Design decisions from architecture docs**:
- Methods organized by entity type (clear grouping)
- Convenience methods for common queries (get_active_personnel_entries, get_overdue_alarms)
- Filter dict for flexible querying without method explosion
