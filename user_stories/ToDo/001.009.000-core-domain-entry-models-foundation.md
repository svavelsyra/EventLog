## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

Epic `001` currently assumes the existence of core domain entities such as `CommunicationEntry`, `EventEntry`, and `PersonnelEntry`, but those symbols are not yet implemented in `src/core/`. That creates a practical gap in the implementation sequence: the database adapter and repository stories are designed to accept and return real domain objects, but there is nothing stable for them to type against yet.

This story exists to close that gap with the smallest useful step. It provides the foundational entry-model classes that let the database layer translate rows into domain objects without pulling database or GUI concerns upward into the Core layer.

Without these models:
- adapter interfaces cannot reference concrete project types cleanly
- repository methods cannot define stable row-to-entity mappings
- CRUD stories risk inventing entity shapes ad hoc during implementation

### User Value
- The project gets a stable shared contract for the three primary log entry types.
- Later database work can proceed without guessing field names, defaults, or ownership boundaries.
- Core remains the authoritative home for domain data shape instead of letting persistence code define it implicitly.

---

## Goal (WHAT)

**What functionality are we delivering?**

Create the initial Core-layer domain models for `CommunicationEntry`, `EventEntry`, and `PersonnelEntry` so Epic `001` database stories have real project entities to import, construct, and return.

### Acceptance Criteria
- [ ] Core entry model classes exist for `CommunicationEntry`, `EventEntry`, and `PersonnelEntry` under `src/core/` with stable import paths for later stories.
- [ ] Each class exposes the Epic `001` persistence-facing fields needed by the current database schema and repository stories, using type hints aligned with `ai_instructions/design/core_design.md`.
- [ ] The initial model implementations support unsaved and saved instances cleanly, including optional database `id` values and sensible defaultable flags/timestamps where the design already calls for defaults.
- [ ] Delivery includes tests that verify import stability plus basic object construction/default behavior without requiring GUI or database imports.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Core layer only** - Models belong in `src/core/` and must not depend on GUI or database implementation code.
- **Use Python stdlib only** - No third-party model libraries.
- **Match current design intent** - Field names and types should follow `ai_instructions/design/core_design.md` closely enough to support upcoming database stories.
- **Support typed DB contracts** - The resulting symbols must be suitable for adapter/repository type hints and row-mapping code.

### Negative Constraints (MUST NOT)
- **Don't implement repository logic** - No SQL, row mapping, schema execution, or persistence behavior.
- **Don't implement GUI/presenter behavior** - No view state, widgets, or presenter helpers.
- **Don't bundle validators into this story** - Validation logic remains separate work unless already trivial and unavoidable.
- **Don't broaden scope to `StructuredReport`** - Keep this story limited to the three primary entry models needed by current Epic `001` work.
- **Don't force later-epic attachment behavior** - Attachment/report-reference fields that depend on later epics should stay deferred or minimally neutral if unavoidable.

### Soft Preferences (SHOULD)
- Prefer simple dataclass-style models unless a stronger reason emerges.
- Keep module layout obvious and import paths stable for later stories.
- Default behavior should reflect the documented auto-fill intent without baking in presenter or repository responsibilities that belong elsewhere.

---

## Implementation Notes

### Suggested Approach

1. Add the three entry model classes under `src/core/` in a layout that can stay stable as the project grows.
2. Represent the fields needed by the current Epic `001` schema and repository stories first.
3. Export the public entry-model symbols from `src/core/__init__.py` so later layers can import them consistently.
4. Add focused tests for construction and defaults rather than trying to deliver full validation behavior in the same story.

### Technical Considerations
- `logged_time`, `event_time`, and `last_contact_time` need a clear representation that will round-trip cleanly through the database layer later.
- Boolean workflow fields such as `confirmed`, `edited`, `active`, `alarm_enabled`, and `alarm_triggered` should be represented now so the repository contract does not need to change shape later.
- Event and communication attachment/report concepts exist in the design, but the related storage and lifecycle epics are still deferred.
- Keep the classes lightweight enough that repository mapping code can construct them directly.

### Affected Components
- **New/updated Core files**: entry model modules and package exports under `src/core/`
- **Tests**: unit tests for model construction/import behavior
- **Downstream integration**: adapter and repository stories in Epic `001`

## Context & Background

This story is intentionally numbered at the next free slot in Epic `001` rather than inserted earlier in the numbering sequence. The numbering records planning order, not required implementation order.

The need for this story became visible when reviewing the actual workspace state against the existing database stories:
- `001.002.x` expects adapter methods typed against Core entities
- `001.003.000` expects repository stubs and later CRUD methods to accept and return those same entities
- `src/core/` currently does not appear to provide those models yet

This story gives Epic `001` a cleaner internal dependency chain without changing the architecture principle that Core should remain independent from database and GUI implementation details.
