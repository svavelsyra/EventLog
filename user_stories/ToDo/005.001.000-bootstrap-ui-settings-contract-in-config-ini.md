# User Story: 005.001 - Bootstrap UI and Startup User Settings Contract in `config.ini`

**Created**: 2026-05-06

---

## Purpose (WHY) ⭐ MOST IMPORTANT

Epic `005` cannot stay structurally clean if startup-needed UI state and normal runtime preferences are still mixed together. The application already implies that some values must exist before the database is opened, while other values belong to the later runtime preference layer. If this boundary stays loose, later GUI stories may each make their own local decision about where settings live.

This story exists to lock down the bootstrap-side UI settings contract first. Without it, later work risks:
- window geometry/state being saved in multiple places with unclear authority
- runtime presenters treating `config.ini` as a general-purpose preference store
- bootstrap-sensitive settings such as language or startup-needed shell state drifting into the database too early
- repeated re-argument about what the app must know before repository startup succeeds

### User Value
- The application gets one explicit startup-time home for UI settings needed before normal runtime preferences are available.
- Later preference stories can build on a stable boundary instead of inventing ad hoc ownership rules.
- Reviewers can validate the bootstrap/runtime split separately before database-backed preference work begins.

---

## Goal (WHAT)

Define the `config.ini` contract for startup-needed app/window settings and the small bootstrap-side user context that must exist before the database-backed preference layer becomes available.

### Acceptance Criteria
- [ ] The story defines the authoritative bootstrap-side ownership for main-window geometry (`window_width`, `window_height`, `window_x`, `window_y`), supported main-window state (`normal` or `zoomed`), `language`, and `last_operator`, with `last_operator` stored in a dedicated `[User]` section instead of being mixed into unrelated bootstrap keys.
- [ ] The startup dialog uses `last_operator` only as a startup prefill value while explicitly not persisting its own geometry or window state.
- [ ] The story defines code-authoritative load/save behavior where `config.ini` overrides code defaults only within safe limits, malformed values are logged and repaired/fallen-back silently, geometry is clamped to the current display bounds, and saves occur on normal app-owned close/shutdown.
- [ ] Focused config/bootstrap tests are required for missing or malformed values, invalid/off-screen geometry recovery, and unsupported window-state fallback.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep this story on the bootstrap side only** - This story defines `config.ini` ownership and startup-time behavior, not the database-backed `user_preferences` layer.
- **Support pre-database startup needs** - Any setting kept here must be justifiable as needed before normal repository-backed preference loading.
- **Keep the contract explicit** - The result must make it clear which settings belong in `config.ini` and which do not.
- **Preserve non-secret semantics** - These values remain convenience/application-state data, not security material.
- **Use code defaults as the behavioral authority** - Runtime code defines the true defaults; `config.ini` overrides them only when present and valid enough after safe validation/clamping.
- **Persist only main-window state in this story** - The startup dialog remains fixed/predictable and does not get its own persisted geometry/state contract here.
- **Limit persisted window-state support deliberately** - Only `normal` and `zoomed` are approved persisted states for now.
- **Favor deterministic fallback behavior** - Missing or malformed bootstrap UI settings must degrade to defined defaults.

### Negative Constraints (MUST NOT)
- **Do not absorb the `user_preferences` repository/helper contract** - That belongs to `005.002`.
- **Do not absorb broader active-operator workflow behavior** - Storing/loading/prefilling `last_operator` belongs here, but broader app-wide operator management and later in-app operator switching remain later work.
- **Do not absorb per-tab column/filter/display persistence** - That belongs to `005.004`.
- **Do not absorb multi-monitor screen affinity** - Display-affinity persistence is deferred to a later separate story.
- **Do not treat `config.ini.template` as runtime authority** - The template is reference/convenience only, not the source of truth for live defaults.
- **Do not turn `config.ini` into a catch-all preference bucket** - This story must narrow ownership, not broaden it.

### Soft Preferences (SHOULD)
- Prefer a small set of clearly named bootstrap UI keys over loosely related application-state accumulation.
- Keep the contract easy to document in `config.ini.template` and related startup/config helpers.
- Favor a boundary that later reset/default behavior can explain simply.

---

## Implementation Notes

### Suggested Approach
1. Review the current `config.ini.template` and startup/config loading code for already implied shell/application-state settings.
2. Define the bootstrap-side key ownership for main-window state, `language`, and `last_operator`, including the dedicated `[User]` section.
3. Tighten the load/apply/save contract so defaults come from code, saves happen on app-owned close, and invalid geometry/state is safely clamped or repaired.
4. Ensure the startup dialog can prefill operator input from `last_operator` without adding startup-window persistence.
5. Add focused tests for default and malformed bootstrap-state handling.

### Affected Components
- `config.ini.template`
- App config/bootstrap loading helpers under `src/`
- Startup dialog bootstrap-prefill integration under `src/gui/`
- Focused tests for bootstrap UI-setting ownership and fallback behavior
