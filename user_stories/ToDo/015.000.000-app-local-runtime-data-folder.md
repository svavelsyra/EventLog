## Purpose (WHY) ⭐ MOST IMPORTANT

EventLog is intentionally an offline, portable Python application. The project has already decided that generated app data should stay close to the application instead of being scattered into OS-specific per-user folders. That decision exists for both operational and trust reasons:
- one machine should not silently accumulate one EventLog data set per Windows user profile
- an administrator should be able to place the app in a writable folder, including a USB stick or a normal machine folder, and know where all generated state lives
- operators and maintainers should be able to back up, inspect, move, or remove the full EventLog working set without hunting through hidden profile directories
- moving the application to another machine should be as simple as moving the application folder when that deployment model is desired

This epic exists to turn that product decision into an explicit implementation contract: EventLog should have one app-local `data/` folder that becomes the home for generated runtime artifacts such as the active database, live config, logs, and similar app-owned state.

### Success Criteria
- EventLog has one clearly defined app-local runtime data root under the application folder
- Default generated paths no longer imply one data location per Windows user profile or depend on process working-directory accidents
- Runtime-generated artifacts that belong to the app have a predictable home such as `data/config.ini`, `data/eventlog.db`, and `data/logs/...`
- The implementation remains portable: moving the app folder also moves the app-owned runtime data set
- The work stays small and focused on path ownership/defaults rather than expanding into unrelated bootstrap or deployment redesign

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep generated app data under the application folder** - The approved default model is an app-local `data/` directory, not per-user profile storage.
- **Treat writable install location as an operational responsibility** - The design assumes the administrator places EventLog where write access is available.
- **Preserve portability** - The chosen path model must continue to support moving the app folder, including USB-style deployment, without inventing a second mandatory data location.
- **Centralize path ownership** - Runtime paths should come from shared helpers/constants rather than repeated ad hoc joins in unrelated modules.
- **Keep the work small** - This epic should mainly reshape path defaults, path helpers, direct dependents, and focused tests.
- **Stay stdlib-only for app code** - No third-party dependency should be introduced for path/config handling.

### Negative Constraints (MUST NOT)
- **Do not move generated runtime data into OS-specific per-user folders** such as roaming/local profile data by default.
- **Do not split the approved design into separate install-root versus runtime-data-root models** for this slice.
- **Do not turn this epic into a full backup/sync/export feature** - It is about default storage location, not broad portability tooling.
- **Do not quietly widen scope into startup UX redesign or database security redesign** - those concerns already have other epic boundaries.
- **Do not leave multiple competing default roots in code** - old root-level or per-user defaults should not remain as parallel authorities.

### Soft Constraints (SHOULD)
- Prefer a small number of well-named path helpers over many special-purpose constants.
- Keep the `data/` layout easy to explain in docs and comments.
- Favor a structure that lets reset/cleanup and logging reuse the same root cleanly.

## Context & Background

Recent AI/user discussion settled the storage direction explicitly:
- earlier startup work moved toward one fixed app-owned SQLite location rather than arbitrary operator-chosen paths
- later discussion refined that into a stronger product decision: keep config, database, logs, and similar generated state together under one application-local `data/` folder
- the reason for rejecting per-user storage was not just convenience; separate Windows-user data roots would create multiple EventLog copies on one machine and make `Nollställ` semantics less clear

This epic is therefore not inventing a new direction. It is capturing an already chosen product decision so implementation can follow one explicit storage contract.
