## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

The repository skeleton provides the foundation for all database operations. It handles core responsibilities that all CRUD operations will depend on:

1. **Connection management** - Opens and maintains database connection
2. **Schema initialization** - Creates tables on first run or for tests
3. **Transaction context** - Provides transaction control for multi-step operations
4. **Row dictionary support** - Configures SQLite for dict-like row access

Without this skeleton, individual CRUD operations (stories 001.004-006) would each need to handle connections, schema setup, and configuration separately, leading to duplication and inconsistency.

### User Value
- Database "just works" on first launch (schema auto-created)
- Tests can create fresh databases quickly (in-memory support)
- Consistent connection handling across all operations
- Clean transaction support for complex operations

---

## Goal (WHAT)

**What functionality are we delivering?**

Create the SQLiteEventLogRepository class that implements EventLogAdapter. This story focuses on the structural skeleton - connection management, initialization, and transaction control. Actual CRUD operations will be added in stories 001.004-006.

### Acceptance Criteria
- [ ] File `src/db/repositories/sqlite_event_log_repository.py` exists
- [ ] `SQLiteEventLogRepository` class inherits from `EventLogAdapter`
- [ ] Constructor capabilities:
  - Accepts database path (file path or `:memory:` for in-memory)
  - Creates SQLite connection with row factory for dict-like access
  - Initializes connection and cursor as instance variables
- [ ] Schema initialization capability:
  - Can create database schema if tables don't exist
  - Safe to call multiple times (idempotent)
  - Works for both file-based and in-memory databases
  - Uses schema executor from story 001.001
- [ ] Transaction management capabilities:
  - Can start database transaction
  - Can commit current transaction
  - Can rollback current transaction
- [ ] Connection cleanup:
  - Can close database connection and release resources
- [ ] Adapter interface satisfaction:
  - All abstract methods from EventLogAdapter implemented as stubs
  - Each stub raises NotImplementedError with story reference
  - All stubs have proper type hints matching interface
- [ ] Context manager support (optional enhancement):
  - Supports `with` statement usage for automatic cleanup
  - Automatically rolls back on exception and closes connection

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Implement EventLogAdapter** - Must inherit and satisfy abstract interface
- **SQLite3 only** - Use stdlib sqlite3 module
- **Row factory = sqlite3.Row** - Enable dict-like row access
- **Support :memory:** - Must work with in-memory databases for tests
- **Call schema executor** - Use schema creation from 001.001
- **Stub all CRUD methods** - Raise NotImplementedError with story reference

### Negative Constraints (MUST NOT)
- **Don't implement CRUD operations yet** - That's stories 001.004-006
- **Don't add configuration loading** - Keep it simple for Phase 1
- **Don't add connection pooling** - Single connection is fine
- **Don't add retry logic** - Basic connection only
- **Don't add encryption** - Deferred to Epic 002

### Soft Preferences (SHOULD)
- Use pathlib for file path handling
- Add helpful error messages for connection failures
- Log when schema is initialized (using stdlib logging)
- Include usage example in docstring

## Implementation Notes

### What's Needed

1. **Class Structure**
   - Purpose: Implement EventLogAdapter interface for SQLite
   - Inherits from: EventLogAdapter abstract base class
   - Instance variables: Database connection, cursor, path

2. **Constructor**
   - Purpose: Establish database connection with proper configuration
   - Input: Database path (string, supports both file paths and `:memory:`)
   - Configuration: Set row factory to sqlite3.Row for dict-like row access
   - Stores: Connection and cursor as instance variables

3. **Schema Initialization**
   - Purpose: Create database tables on first use
   - Implementation: Check if key table exists before creating schema
   - Uses: Schema executor from story 001.001

4. **Transaction Management** (~3 methods)
   - Purpose: Support atomic multi-step operations
   - Capabilities: Begin transaction, commit changes, rollback on error
   - SQLite note: Explicit BEGIN statement recommended for clarity

5. **Connection Cleanup**
   - Purpose: Properly release database resources
   - Implementation: Close connection when done

6. **CRUD Method Stubs** (~23 methods total)
   - Purpose: Satisfy abstract interface requirements
   - Implementation: Each raises NotImplementedError with story reference
   - Organization: Group by entity type (Communication, Event, Personnel)
   - Story references: "To be implemented in story 001.004/005/006"

7. **Context Manager Support** (optional)
   - Purpose: Enable `with` statement usage for cleaner resource management
   - Implementation: `__enter__` returns self, `__exit__` handles cleanup
   - Behavior: Rollback on exception, always close connection

## Context & Background

This skeleton establishes the pattern that all CRUD operations will follow:
1. Take domain entity as input
2. Convert to database row format
3. Execute SQL
4. Convert result back to domain entity
5. Return result

Stories 001.004-006 will fill in the "convert and execute SQL" parts for each entity type.

**Why stub CRUD methods**:
- Satisfies abstract interface (class is instantiable)
- Clear TODO markers for next stories
- Can test skeleton independently (connection, schema, transactions)
