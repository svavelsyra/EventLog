## Purpose (WHY) ⭐ MOST IMPORTANT

EventLog needs to support structured military reports such as 7S and later formats without hardcoding every report form directly into GUI code. Structured reports are valuable because they capture repeatable operational information in a consistent shape, improve reporting speed, and allow the application to generate readable summaries from disciplined input instead of free-text guesswork.

This epic exists to give EventLog a maintainable structured-report system with two important qualities:
- built-in report templates remain persistently available as part of the application package
- runtime report behavior still comes from a database-backed template catalog rather than scattered file reads and hardcoded form logic

The user also expects technically capable operators or support personnel to be able to study existing template JSON and create additional templates later. That makes it useful to keep built-in templates in a dedicated JSON template folder while still loading them into the database for normal application use.

The deeper reason is long-term flexibility without runtime confusion. New report types should be introducible without rewriting every presenter, but the application should still have a single runtime source of truth for template selection, validation, summary generation, and UI rendering.

### Success Criteria
- Structured reports are defined as template-driven attachments to `EventEntry` rather than hardcoded one-off forms
- Built-in report templates exist as JSON files in a dedicated application template folder
- Startup logic can load or sync built-in templates from JSON into the database catalog
- The database remains the runtime source of truth for report template selection, validation, form generation, and summary behavior
- The epic leaves room for later user-provided JSON templates without requiring live file watching or a second runtime authority outside the database
- Epic 004 stays focused on structured reports and their template lifecycle, without drifting into general configuration management, secure bootstrap, or full file-based runtime configuration

## Goal (WHAT)

Define and deliver the structured-report capability for EventLog, including both the report entity behavior and the template lifecycle that makes multiple report types possible.

This epic covers the end-to-end contract for:
1. storing structured reports as separate entities attached to events
2. defining built-in structured report templates as JSON assets in a dedicated folder
3. syncing those built-in templates into the database at startup or first initialization
4. using the database template catalog for runtime validation, form generation, and summary generation
5. leaving room for later user-added JSON templates without requiring runtime file monitoring

### Key Deliverables
- [ ] Structured-report persistence contract for attaching reports to `EventEntry`
- [ ] Dedicated application folder for built-in structured report template JSON files
- [ ] Startup template sync/import contract from JSON assets into the database template catalog
- [ ] Clear runtime rule that report templates are read from the database during normal application use
- [ ] Validation and summary-generation contract based on database-backed template configuration
- [ ] Boundary for later support of user-provided template JSON without turning the filesystem into a second live runtime authority

## Limitations (CONSTRAINTS & REQUIREMENTS)

**What boundaries must we work within? What must we avoid?**

### Hard Constraints (MUST)
- **Keep structured reports separate from `EventEntry` core fields** - Reports remain attached entities, not inline schema expansion of the event table
- **Use template-driven behavior** - Report forms, validation rules, and auto-summary behavior must come from template configuration rather than per-report hardcoded GUI logic
- **Keep the database as runtime source of truth** - Built-in JSON files are packaged defaults and sync/import sources; normal runtime selection and validation must use the database catalog
- **Support a dedicated built-in template folder** - Application-shipped report templates should live in a clear folder under source/resources so they are persistent, reviewable, and usable as examples
- **Startup-only template discovery is sufficient** - The first implementation only needs to check for new or changed template formats during startup/initialization; no live file watching is required
- **Do not include template editing UI in this epic** - Template creation/editing inside the application is outside the Epic 004 boundary
- **Allow room for later user-authored JSON templates** - The epic should support the idea that technically competent users can create additional template JSON later, even if exact user workflow is deferred to a later story
- **Preserve offline operation** - Template loading and use must work entirely offline
- **Keep testing with delivery stories** - Later 004.x stories should include their own repository/core/GUI tests rather than relying on a broad standalone testing story

### Negative Constraints (MUST NOT)
- **Do not move runtime report-template authority out of the database** - Filesystem JSON must not become a parallel live runtime source that presenters and validators query directly during normal use
- **Do not absorb general operational configuration management** - Communication systems, categories, priorities, and non-report config remain Epic 003 territory
- **Do not absorb secure bootstrap, encryption, or `Nollställ` behavior** - Those remain Epic 002 territory
- **Do not require file watchers or hot reload** - Restart/startup sync is enough for the first implementation
- **Do not confuse built-in templates with arbitrary full data import/export** - This epic is about report-template lifecycle, not general backup/restore
- **Do not force user-created templates to live in the source tree at runtime** - The system may later support user-supplied JSON, but runtime usage should still flow through the database catalog

### Soft Constraints (SHOULD)
- Prefer a small, explicit JSON schema for template files so competent users can inspect and copy existing templates safely
- Keep template storage paths and loading rules predictable and documented
- Favor startup sync behavior that is deterministic and easy to test
- Keep room for later template provenance/version metadata if built-in and user-provided templates need to coexist cleanly

## Context & Background

### Architectural Context
Epic 001 established the database foundation. Epic 002 adds secure database access and startup bootstrap. Epic 003 defines the broader operational configuration model. Epic 004 builds on those layers to deliver structured reports as a template-driven feature.

Current design assumptions already point in this direction:
- `StructuredReport` is a separate entity attached to `EventEntry`
- `report_templates` already exist conceptually as database-backed configuration
- event workflows open a report-type selection dialog and then render fields from template configuration
- auto-summary generation depends on template-defined summary behavior

### Template Source Decision for This Epic
This epic follows the chosen structured-report template model:
- built-in structured report templates live as JSON files in a dedicated application folder
- startup logic imports or syncs those templates into the database
- the database copy is the one used at runtime

This gives the project persistent built-in templates and example JSON that advanced users can study, while avoiding a split runtime model where some template state is read from files and some from the database.

### User-Provided Template Context
The project expects some users to be technically competent enough to inspect existing template JSON and create new templates later. Epic 004 therefore needs to leave room for later stories that define:
- how user-provided JSON templates are discovered or imported
- how conflicts with built-in templates are handled
- how template validity/versioning is checked

Those details are intentionally deferred. What matters at epic level is that user-provided JSON is compatible with the template model and still ends up in the database-backed runtime catalog.

### What This Epic Excludes
- general communication/event configuration tables from Epic 003
- full template-management UI for creating/editing arbitrary templates in-app
- live file monitoring for template changes while the app is running
- general data import/export unrelated to structured report templates
- encryption/bootstrap/reset behavior from Epic 002

### Key Risks
- **Dual-source confusion** - file templates and database templates must not both act as live runtime authorities
- **Template schema drift** - JSON template structure needs to stay explicit enough for both built-in maintenance and later user-authored templates
- **Scope creep** - template editing UI, file watching, and general import/export can easily bloat the epic if not deferred deliberately
- **Startup sync ambiguity** - the later story must define how new, changed, or conflicting built-in templates are handled
- **Validation inconsistency** - GUI and core validation must be driven by the same template catalog rules

### Success Metrics
- Epic 004 is clear enough to split into small stories without mixing in unrelated configuration or security concerns
- Built-in template JSON files can exist persistently without undermining the database-first runtime model
- Later stories can implement startup-only template sync and user-provided template compatibility without re-arguing the architectural boundary
