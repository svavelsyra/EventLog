## Purpose (WHY) ⭐ MOST IMPORTANT

EventLog is not only a message log; it also needs to capture what happened, who was affected, how important it was, and whether a structured operational report was attached. If the operator cannot quickly log observations, incidents, movements, and contact reports in the main UI, the application still lacks one of its central operational workflows.

This epic exists to deliver the Event tab as a real operator-facing workflow that turns the earlier database, configuration, structured-report, and preference work into practical field value. The purpose is to ensure that EventLog can:
- let an operator create and review operational event entries rapidly in the main tabbed UI
- use configuration-backed categories and priorities instead of hardcoded event-classification rules
- attach structured reports such as 7S through the approved template-driven report system without turning the event tab into template-management UI
- show recent events in a readable table that preserves both quick operator notes and generated structured-report summaries
- support correction, review, and filtering without losing the historical clarity of edited records
- reuse the same MVP, repository, configuration, and preference patterns proven by the Communication tab instead of inventing a parallel architecture

The deeper reason is operational understanding. Communications describe what was said; the Event tab is where the staff captures what actually happened. If that workflow is weak, fragmented, or report handling is awkward, the application fails to support shared situational awareness.

### Success Criteria
- A complete Event tab workflow exists for normal Phase 1 use, from entering an event through seeing it appear in the event log view
- Event categories and priorities are sourced from the runtime configuration model rather than scattered hardcoded constants
- Structured reports can be attached, reviewed, and edited through the event workflow while still using the database-backed report-template/runtime model from Epic 004
- The Event tab stays within the MVP boundary, with view, presenter, validation, and repository responsibilities remaining structurally clear
- Epic 007 stays focused on event-tab delivery, without absorbing report-template lifecycle management, attachment/document handling, or unrelated personnel/communication work
- The epic is clear enough to split into small stories for view scaffolding, presenter behavior, structured-report integration, table behavior, and edit/detail flows

## Goal (WHAT)

Define and deliver the Event tab as the second end-to-end GUI feature workflow in the EventLog application.

This epic covers the end-to-end contract for:
1. rendering the event entry form and event log table in the main application UI
2. loading configuration-backed event priorities and categories for form use and filtering
3. creating, validating, saving, listing, filtering, and editing `EventEntry` records through MVP interactions
4. integrating the structured-report workflow from Epic 004 into event creation, review, and later correction flows
5. presenting saved events in a table that combines event description and report auto-summary cleanly while respecting preference-backed column behavior

### Key Deliverables
- [ ] Event tab view contract with top/bottom layout, compact entry form, filter bar, and scrollable event log table
- [ ] Event presenter contract for configuration loading, form interaction, validation handling, save/edit flow, and table refresh behavior
- [ ] Runtime category/priority consumption contract driven by Epic 003 instead of hardcoded event-classification logic
- [ ] Event entry create/read/update/delete workflow integrated with repository methods from Epic 001 and secure startup assumptions from Epic 002
- [ ] Structured-report attach/review/edit interaction that consumes the template/runtime model from Epic 004 without redefining it
- [ ] Event log table behavior contract for default sort order, truncation, filtering, detail/edit access, and edited-state visibility
- [ ] Column-configuration and tab-state integration that consumes the Epic 005 preference contract without inventing event-specific persistence rules
- [ ] Explicit boundary between event-tab delivery and future concerns such as file attachments, broad report-template administration, and cross-tab polish

## Limitations (CONSTRAINTS & REQUIREMENTS)

**What boundaries must we work within? What must we avoid?**

### Hard Constraints (MUST)
- **Use MVP responsibilities cleanly** - The Event tab must follow the documented Tkinter View + Presenter pattern, keeping widget rendering in the view and coordination/validation flow in the presenter
- **Build on existing foundations instead of bypassing them** - Event data must flow through the repository/adapter path from Epic 001 and use the secured/configured runtime assumptions established by later foundational epics
- **Consume configuration-driven event metadata** - Valid priorities and categories must come from the operational configuration model established by Epic 003 rather than being redefined locally in the tab
- **Consume the structured-report contract from Epic 004** - Report selection, report editing, validation, and auto-summary behavior must use the approved template-driven model rather than hardcoded one-off report forms in the event tab
- **Support the documented top/bottom event layout** - The tab must preserve the fast-entry form plus recent-log overview pattern already defined in the GUI design
- **Support create and edit flows** - Operators must be able to save new events and correct existing ones, with edited-state behavior visible in the log view
- **Support practical table usage on small screens** - Sorting, truncation, scrollbars, and column visibility behavior must align with the netbook-sized UI constraints documented for the project
- **Consume preference-backed UI state where applicable** - Column configuration and related tab UI state should use the Epic 005 preference contract rather than inventing event-tab-specific persistence rules
- **Keep testing with delivery stories** - Later `007.x` stories should include their own presenter/gui/core/repository tests instead of creating a broad standalone testing story

### Negative Constraints (MUST NOT)
- **Do not absorb report-template lifecycle management** - Built-in template JSON, startup sync/import, template schema, and runtime template authority remain Epic 004 territory
- **Do not absorb file attachments or document handling** - Generic file attachments remain a later dedicated epic, even though the event design leaves room for them
- **Do not absorb communication or personnel workflows** - Communication and personnel tabs remain separate feature epics with their own delivery boundaries
- **Do not absorb broad event-configuration administration UI** - Managing categories, priorities, or report templates broadly belongs to configuration/report follow-on work, not this tab-delivery epic
- **Do not require advanced analytics or export features** - Phase 1 event filtering and review should stay focused on practical operational log use rather than broader reporting/export scope
- **Do not bypass validators or repository formatting rules with view-side logic** - Views must not implement business rules, database access, or silent data transformation beyond UI display concerns
- **Do not hardcode demo/example values as runtime truth** - Demo categories, priorities, or example event text are visual references only and must not become the implementation contract

### Soft Constraints (SHOULD)
- Prefer small follow-up stories that separate tab/view scaffolding, presenter save/load behavior, structured-report integration, and table/filter/detail behavior
- Keep Swedish UI labels and operator-facing terminology aligned with the design documents and demo reference
- Favor direct consumption of configured categories/priorities and report-template services before adding convenience shortcuts or broader admin affordances
- Keep report-selection and report-edit interactions reviewable before adding extra workflow polish or advanced shortcuts

## Context & Background

### Architectural Context
Epic 001 created the event-entry persistence path. Epic 003 then defined the operational configuration layer needed for runtime categories and valid priorities. Epic 004 defined the structured-report template/runtime model that the event workflow must consume. Epic 005 defined the preference boundary needed for column configuration and other per-tab UI persistence.

That means Epic 007 is the feature epic that pulls these pieces together into a usable event workflow:
- repository-backed `EventEntry` CRUD
- configuration-backed categories and priorities
- structured-report selection and attachment through the approved report system
- presenter/view coordination
- preference-backed column behavior
- practical event-log review on small screens

### Why Event Comes After Communication
The Communication tab was the cleanest first GUI feature slice because it exercised the MVP/repository/configuration/preference path without structured-report complexity. The Event tab is the next logical step because it reuses much of the same tab and table pattern while adding the first real feature integration point with the report system from Epic 004.

That makes Epic 007 valuable not just as a user-facing feature, but also as a proof that:
- tab-level GUI patterns can be reused across feature areas
- configuration-driven choices work beyond the communication workflow
- template-driven structured reports can be consumed cleanly inside a real operator workflow without collapsing boundaries

### Structural Boundary for Structured Reports
The event design includes adding and editing structured reports from the event workflow. That is necessary for a usable Event tab, but it must not reopen the report-template lifecycle decisions already made in Epic 004.

For that reason, Epic 007 should treat structured reports as a dependency to consume:
- the Event tab can launch report selection and report-entry interactions
- the Event tab can show auto-summary output and report presence
- the Event tab must not turn into template-management, startup-sync, or template-authoring scope

### What This Epic Excludes
This epic is intentionally about **event-tab delivery**, not about:
- communication logging workflows
- personnel tracking workflows
- generic file-attachment workflows
- template editing or report-template import management
- broad configuration-management UI for categories, priorities, or other settings
- cross-application polish unrelated to the event tab's core workflow

### Key Risks
- **Boundary drift** - event delivery could easily absorb report-template management, generic attachments, or broader admin-config scope if not held tightly
- **Report/workflow duplication** - the event tab could accidentally reimplement report-template rules locally instead of consuming Epic 004's contract
- **Presenter bloat** - event form logic, report coordination, validation, and display formatting could collect into one large presenter if story splitting is weak
- **Small-screen modal complexity** - report dialogs and event detail flows can become cramped if the scrollable-dialog patterns are not applied carefully
- **Summary display ambiguity** - later implementation must define clearly how `event_description` and report `auto_summary` appear together without confusing operators

### Success Metrics
- Epic 007 is clear enough to split into small stories without mixing in unrelated communication, personnel, attachment, or report-lifecycle scope
- A later implementation pass can deliver a genuinely usable event workflow rather than only a static tab mockup
- The Event tab consumes configuration and report-template contracts instead of bypassing them with hardcoded shortcuts
- The project gains a reusable pattern for integrating a tab workflow with a separate template-driven subsystem
