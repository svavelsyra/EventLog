## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

Operators need to correct event logs and search for specific content:

1. **Update** - Fix typos, add whom information, change priority/category
2. **Delete** - Remove duplicate or erroneous events
3. **Search** - Find events containing specific text (locations, equipment, keywords)

Without update/delete, event logs cannot be corrected. Without search, finding specific events in large logs is tedious.

### User Value
- Correct mistakes in logged events
- Remove duplicate entries
- Search for specific text in event descriptions
- Reclassify events (change priority/category)
- Maintain accurate operational log

---

## Goal (WHAT)

**What functionality are we delivering?**

Implement update, delete, and search operations for `EventEntry` in the SQLite repository layer under the approved package layout.

### Acceptance Criteria

- [ ] **Update Operation**:
  - Can modify an existing event entry by identifier
  - Updates all modifiable fields
  - Applies the edited-flag business rule using the configured grace-period setting rather than a hardcoded immediate flag change
  - Preserves original logged timestamp
  - Returns success status (true if updated, false if not found)
- [ ] **Delete Operation**:
  - Can permanently remove an event entry from the database
  - Returns success status (true if deleted, false if not found)
  - Hard delete (row removed, not soft delete)
- [ ] **Search Operation**:
  - Can search entries by text in event descriptions
  - Search is case-insensitive
  - Can combine text search with supported event filters
  - Returns empty list if no matches
- [ ] **Unit Tests Pass**:
  - Update changes fields correctly
  - Update sets edited flag automatically
  - Update preserves logged timestamp
  - Update non-existent ID returns False
  - Delete removes entry
  - Delete non-existent ID returns False
  - Search finds matching entries
  - Search with filters combines correctly

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Implement in the SQLite repository layer** - Replace placeholder behavior using the approved repository package layout
- **Delegate low-level DB mechanics downward** - Repository prepares query/update/search intent; adapter handles low-level execution/transaction mechanics
- **Use parameterized queries** - Prevent SQL injection
- **Follow CommunicationEntry pattern** - Same structure as 001.004.003
- **Apply configured edited-flag behavior on update** - Automatic rule, but based on the configured grace-period setting rather than an unconditional immediate flag
- **Never change logged_time** - Preserve original timestamp
- **Case-insensitive search** - Search should not depend on exact letter case

### Negative Constraints (MUST NOT)
- **Don't implement soft delete** - Hard delete for Phase 1
- **Don't implement StructuredReport cascade** - Reports deferred to Epic 004

### Soft Preferences (SHOULD)
- Check operation success in a clear, reliable way
- Keep update/delete behavior consistent with other repository write operations
- Reuse established event filtering patterns where helpful

## Implementation Notes

### What's Needed

1. **Update Operation Capability**
   - Purpose: Modify an existing event entry in persistent storage
   - Input: EventEntry object with identifier set
   - Output: Boolean success status
   - Key considerations: Update all editable fields, apply the configured edited-flag business rule, preserve original logged timestamp, apply required type conversions, report not-found cases cleanly, adapter-backed execution, transaction behavior

2. **Delete Operation Capability**
   - Purpose: Permanently remove an event entry from storage
   - Input: Integer identifier
   - Output: Boolean success status
   - Key considerations: Hard delete behavior, no effect on unrelated entries, report not-found cases cleanly, adapter-backed execution, transaction behavior

3. **Search Operation Capability**
   - Purpose: Find event entries containing relevant text
   - Input: Search text string, optional filter criteria dictionary
   - Output: List of EventEntry objects matching the criteria
   - Key considerations: Case-insensitive text matching, combine text search with supported filters, preserve repository read ordering conventions, reuse row-to-entity conversion, adapter-backed fetch behavior

### Architectural Guidance

See `ai_instructions/architecture/db_architecture.md` for:
- Repository pattern update/delete operations
- Repository-vs-adapter delegation boundaries
- Business rule enforcement in database layer
- Search query patterns

### Technical Considerations
- Pattern identical to CommunicationEntry update/delete/search
- Update should enforce the edited flag as a repository-level business rule using the configured grace period from the database settings table
- Update should preserve the original logged timestamp as immutable history
- Search should target the primary event text content and support common review workflows
- Search should remain compatible with the filtering approach established by event read operations
- All operations should respect the adapter/repository transaction boundary (commit after execution unless in explicit transaction)
