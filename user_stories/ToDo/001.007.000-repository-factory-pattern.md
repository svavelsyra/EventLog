## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

The factory pattern decouples application code from specific repository implementations. This provides critical benefits:

1. **Configuration-driven behavior** - Database choice comes from configuration, not hardcoded application logic
2. **Testing flexibility** - Tests can easily use in-memory databases
3. **Future extensibility** - Adding new database backends requires no changes to calling code
4. **Centralized instantiation** - One place to create repositories correctly

Without a factory, every part of the application that needs a repository would need to know:
- Which concrete class to instantiate
- How to configure it
- When to use in-memory vs file-based behavior

The factory encapsulates that knowledge, making the rest of the codebase simpler and more maintainable.

### User Value
- Application works with configured database automatically
- Tests run fast with in-memory databases
- Future database changes don't break existing code
- Cleaner application code with fewer database details scattered around

## Goal (WHAT)

**What functionality are we delivering?**

Create reusable shared configuration loading support plus a RepositoryFactory that returns properly initialized `EventLogAdapter` instances based on configuration.

### Acceptance Criteria

- [ ] **Reusable Config Loading Capability**:
  - Can load database settings from application configuration
  - Uses shared parsing infrastructure that can be reused by future configuration-related work
  - Supports create-or-update handling for the shared config loader depending on whether shared infrastructure already exists
- [ ] **Repository Factory Capability**:
  - Can create the correct repository implementation from supported database configuration
  - Returns initialized repositories through the `EventLogAdapter` interface
  - Ensures schema initialization happens before the repository is returned for use
- [ ] **Convenience and Validation Capability**:
  - Supports convenient in-memory repository creation for tests or similar workflows
  - Reports invalid or unsupported configuration clearly
  - Includes focused validation coverage for configuration loading and factory behavior

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Return `EventLogAdapter` type** - Calling code works against the interface, not a concrete implementation
- **Use stdlib configuration parsing** - No third-party dependencies
- **Support in-memory configuration for tests** - Recognize the in-memory database mode required by testing workflows
- **Initialize schema automatically** - Returned repositories must be ready to use
- **Keep config parsing reusable** - If shared parser support is created or updated, it must be structured for future configuration work too
- **Keep config parsing reusable** - If shared config-loader support is created or updated, it must be structured for future configuration work too

### Negative Constraints (MUST NOT)
- **Don't add other database types yet** - Only SQLite and SQLite memory for Phase 1
- **Don't add connection pooling** - Simple single-connection behavior is sufficient
- **Don't add encryption configuration** - Deferred to later work
- **Don't implement migration workflows** - This story covers repository creation, not schema migration strategy

### Soft Preferences (SHOULD)
- Keep error messages clear when configuration is missing or invalid
- Keep factory behavior simple for application callers and tests
- Prefer split responsibilities that are easy to test independently
- Keep configuration-loading infrastructure positioned for reuse by later stories

## Implementation Notes

### What's Needed

1. **Reusable Configuration Loading Capability**
   - Provide shared support for reading database settings from configuration input
   - Handle the case where a reusable shared config loader may already exist and should be extended rather than duplicated

2. **Repository Selection and Initialization Capability**
   - Choose the correct repository implementation from supported database settings
   - Initialize schema before handing the repository back to callers

3. **Convenience and Validation Capability**
   - Support test-friendly in-memory creation paths
   - Add focused tests for configuration handling, selection logic, and invalid configuration cases

### Technical Considerations
- This story sits at the boundary between configuration handling and DB infrastructure creation
- The reusable parser concern is broader than just repository creation, which is one reason this story was split
- The shared config loader belongs with broader bootstrap/application concerns rather than inside `src/db/`
- Tests and future application bootstrap both depend on predictable repository creation behavior
- Keeping the parent story high-level helps the sub-stories carry the more detailed implementation breakdown

### Affected Components
- **Modified/New files**:
  - `src/db/repository_factory.py`
  - Potential shared config loader support under `src/config/`
  - `tests/unit/db/test_repository_factory.py`

## Context & Background

This story provides the bridge between configuration and concrete repository setup.

**Pattern consistency**:
- Calling code depends on `EventLogAdapter`, not concrete repository types
- Configuration concerns should be centralized rather than scattered across bootstrap or tests
- Shared config loading should live with application/bootstrap concerns, not as a DB-owned helper
- Test-friendly in-memory creation should be easy to access without duplicating setup logic
