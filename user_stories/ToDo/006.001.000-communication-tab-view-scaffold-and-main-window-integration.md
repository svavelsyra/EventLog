## Purpose (WHY) ⭐ MOST IMPORTANT

Epic `006` only becomes tractable if the project first replaces the current Communication-tab placeholder with a real production-owned view seam that matches the intended operator workflow shape. Without that scaffold, later presenter, configuration, validation, and repository stories have no stable GUI surface to target, and the first full workflow risks turning into one oversized patch that mixes layout, interaction logic, persistence, and dynamic field behavior all at once.

This story exists to create the smallest useful structural slice of the Communication tab: a real tab in the main window with the documented top/bottom pattern, visible form/filter/log regions, and stable widget ownership on the view side. That gives later `006.x` stories a concrete UI contract to build on while keeping this first step small enough to review cleanly.

Without this story, later work risks:
- presenter and repository stories inventing their own temporary widget seams against a placeholder tab
- the first implementation pass mixing view layout decisions with save/edit/filter behavior and becoming too large
- demo-only layout ideas being copied ad hoc instead of translated into a production-owned GUI structure
- avoidable rework when later stories need to split the Communication tab into form, filter, and log regions after logic has already been wired in

### User Value
- Operators move one step closer to a usable Communication workflow instead of a placeholder tab.
- Reviewers can validate the communication-tab layout and shell placement before behavior-heavy stories begin.
- Later `006.x` stories get a stable view seam for presenter/configuration/persistence work.

---

## Goal (WHAT)

Replace the current Communication-tab placeholder in the main application window with a real view scaffold that reflects the approved Communication-tab layout: compact entry area at the top, filter area above the log, and a scroll-ready log/table region below.

This story is structural only. It delivers visible Communication-tab regions and their main-window integration, but it does not yet deliver save behavior, edit flows, runtime configuration loading, repository access, or preference-backed table state.

### Acceptance Criteria
- [x] The main application shell renders a real Communication-tab view scaffold instead of a placeholder-only tab body.
- [x] The Communication tab exposes distinct visible regions for a compact entry form, a filter bar, and a bottom log/table area aligned with the documented top/bottom layout.
- [x] The scaffold uses production-owned Swedish labels and layout structure without treating demo/example values as runtime truth.
- [x] The story leaves presenter, configuration, repository, filtering, sorting, edit/detail, and preference behavior outside this slice, with focused view-level tests covering render/integration behavior for the new scaffold.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Stay view-first and structural** - This story must only establish the Communication tab's GUI scaffold and its placement in the main window.
- **Respect MVP ownership** - Widget construction, layout, and passive callback seams belong in the view layer; presenter/business/repository logic must remain for later stories.
- **Follow the documented Communication layout** - The tab should reflect the approved top entry area + lower log overview pattern from the GUI design.
- **Keep Swedish operator-facing text** - Labels and visible text should follow the project's Swedish GUI language.
- **Preserve main-window shell stability** - Event and Personnel placeholders, toolbar/menu behavior, and status bar behavior should remain intact while the Communication tab is upgraded.
- **Include focused tests with the story** - View/shell tests should verify that the Communication scaffold renders and integrates without breaking the existing shell structure.

### Negative Constraints (MUST NOT)
- **Do not implement presenter behavior yet** - No save, clear, selection, validation, or table-refresh logic belongs to this story.
- **Do not absorb runtime configuration loading** - System/method/channel/capability population belongs to later `006` stories.
- **Do not absorb communication CRUD** - Repository-backed create/read/update/delete behavior remains outside this slice.
- **Do not absorb real filtering/sorting/detail/edit flows** - The filter/log regions may exist structurally, but not with full behavior yet.
- **Do not treat demo values as required defaults** - Example values from `Demo/` remain visual references only.
- **Do not re-decide column preference persistence** - Epic `005` already defined that seam; this story should not duplicate it.

### Soft Preferences (SHOULD)
- Prefer visible, reviewable placeholder widgets inside the new regions over an almost-empty frame so reviewers can assess the intended shape.
- Prefer a Communication-tab view seam that later presenter stories can attach to cleanly without restructuring the whole tab again.
- Keep the initial scaffold small enough that implementation should fit in one focused session with a modest test update.

---

## Implementation Notes

### Suggested Approach
1. Replace the current Communication placeholder with a dedicated Communication-tab view seam under `src/gui/views/` or an equivalently clear shell-owned structure.
2. Render visible grouped regions for:
   - entry/form controls
   - filter controls
   - log/table host area
3. Keep controls passive or minimally wired for shell ownership only, without introducing presenter logic prematurely.
4. Add focused GUI tests that confirm the new scaffold is present and that shell integration still preserves the rest of the main window.

### Affected Components
- Main-window shell view under `src/gui/views/`
- Potential Communication-tab-specific view module under `src/gui/views/`
- Shell wiring under `src/gui/` only if needed for clean ownership
- Focused GUI tests for shell/view rendering
