## Purpose (WHY) ⭐ MOST IMPORTANT

By the time the Communication tab has a working table, selection behavior, correction workflow, and detail browsing, operators still need one more practical usability seam: the ability to shape the table columns to fit their screen and workflow without the Communication tab inventing its own private persistence format. Epic `005` already established the shared runtime preference contract for per-tab UI state so later tabs would not each redefine column visibility, order, or widths differently. This story exists to make the Communication tab actually consume that contract.

Without this story, later work risks:
- the Communication tab hardcoding one fixed table layout even on small screens where operators need to hide or reorder less critical columns
- Communication-specific code inventing a private persistence shape instead of using the approved `ui.tab.communication.columns` runtime preference
- future Event and Personnel stories losing confidence that Epic `005`'s shared UI-state contract is practical in a real tab
- avoidable rework when column layout preferences need to be reset, reloaded, or validated against malformed stored data

### User Value
- Operators can tailor the Communication log table to fit their screen and workflow.
- Reviewers can validate that the Communication tab is consuming the shared preference contract rather than redefining it.
- The project proves the Epic `005` per-tab column-state seam in the first real operator-facing tab workflow.

---

## Goal (WHAT)

Add Communication-tab column configuration behavior that consumes the already-approved runtime preference contract for tab column state.

For this story, that means:
- applying stored Communication column visibility, order, and width preferences to the log table
- providing a small Communication-local way to adjust those column settings
- saving updated column state back through the approved runtime preference seam
- honoring the existing fallback/reset/malformed-data rules established earlier instead of redefining them in the Communication tab

This story is about column configuration only. It does not add filter persistence, sort persistence, broader display toggles, or a generic settings framework.

### Acceptance Criteria
- [ ] The Communication table consumes the approved runtime preference contract for Communication-tab column state, using the existing shared key/shape rules for column visibility, order, and widths rather than a tab-local format.
- [ ] The Communication workflow provides a small operator-facing seam for adjusting Communication table columns and applying the resulting visibility/order/width state to the current table.
- [ ] Updated Communication column state is persisted and later restored through the approved runtime preference seam, while fallback/reset behavior for missing or malformed stored values continues to follow the earlier Epic `005` contract.
- [ ] Focused tests cover representative round-trip column preference behavior, sane fallback for missing or malformed stored data, and any required always-visible/default-column rules chosen for the Communication table without absorbing sort/filter persistence or generic settings scope.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Consume the existing shared preference contract** - This story must use the approved runtime preference definition for Communication-tab columns rather than inventing a new key or structure.
- **Stay column-focused** - The scope is limited to column visibility, order, and widths for the Communication table.
- **Build on the existing Communication table contract** - This story should apply preferences on top of the stable table behavior from `006.004` rather than redesigning the table itself.
- **Honor earlier fallback/reset rules** - Missing, cleared, or malformed stored values should fall back according to the already-approved Epic `005` preference behavior.
- **Preserve MVP ownership** - The presenter coordinates preference load/save/application behavior; the view renders the table and any small column-configuration surface.
- **Include focused tests with the story** - Tests should cover round-trip persistence, fallback behavior, and table-application behavior for column state.

### Negative Constraints (MUST NOT)
- **Do not absorb filter persistence** - Filter defaults/state remain outside this story.
- **Do not absorb sort persistence** - Sort-state storage belongs to the broader display-state preference seam, not this column-only slice.
- **Do not build a generic cross-tab settings framework here** - This story is a Communication-tab consumer of the contract, not a new shared-admin system.
- **Do not redesign detail/edit/navigation behavior** - `006.005` and `006.007` remain responsible for those flows.
- **Do not bypass the shared runtime preference key registry** - No hidden presenter-local key schemes or ad hoc storage shapes.
- **Do not over-polish the configuration UI** - A small, practical Communication-local column-adjustment seam is enough.

### Soft Preferences (SHOULD)
- Prefer a simple `Kolumner...` interaction that is easy to review over a heavy configuration dialog framework.
- Prefer a minimal always-visible-column rule if one is needed, rather than turning this story into a table-usability redesign.
- Keep the stored and restored column behavior inspectable and predictable so later tabs can follow the same pattern.

## Implementation Notes

### Suggested Approach
1. Reuse the approved Communication column runtime preference definition from the shared runtime preference module.
2. Define the Communication table's default column state in a way that can be expressed through the shared `visible` / `order` / `widths` structure.
3. Add a small Communication-local column-configuration seam for showing/hiding columns, reordering them, adjusting widths as needed, and resetting to defaults.
4. Apply saved column state when the Communication tab loads and persist updated state through the existing runtime preference access seam.
5. Add focused tests for round-trip persistence, fallback/reset behavior, and application of saved Communication column state to the table.

### Affected Components
- Communication presenter under `src/gui/presenters/`
- Communication view seam under `src/gui/views/`
- Runtime user-preference consumption under `src/core/` and the existing repository-backed preference seam
- Focused presenter/view/preference tests for Communication column state
