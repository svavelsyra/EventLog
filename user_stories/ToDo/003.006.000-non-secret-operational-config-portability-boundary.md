## Purpose (WHY) ⭐ MOST IMPORTANT

`Nollställ` is supposed to destroy the database and related local artifacts quickly. That remains correct, but it means all database-resident operational configuration disappears too. If the project never defines a safe portability path for approved non-secret communication operational configuration, every reset, redeployment, or fresh setup forces the unit to rebuild systems, communication options, and qualifier behavior by hand.

This story exists to define and deliver that operational-config portability boundary without turning Epic `003` into full backup/restore. Without it, later work could drift into:
- vague "export config" ideas that accidentally include secrets, preferences, or unrelated data
- different import/export formats invented independently by future tools or GUI flows
- config restore paths that bypass the database-first runtime model
- overlap confusion with Epic `012`, which is about user-preference portability rather than mission configuration

### User Value
- Approved unit configuration can later survive reset and redeployment without treating the whole database as portable.
- The project gets one clear allowlist for what operational config may move between databases/machines.
- Recovery planning stays separate from user preferences and security-sensitive bootstrap material.

---

## Goal (WHAT)

Define the portable-scope contract and first implementation boundary for export/import of approved non-secret communication operational configuration used by Epic `003`.

### Acceptance Criteria
- [ ] An explicit allowlist exists for which communication-config data is portable under Epic `003`, covering only the approved communication runtime metadata already settled by earlier `003.x` stories while excluding secrets, bootstrap/security material, user preferences, structured-report templates, event metadata, log data, and entry records.
- [ ] The story defines a stable portable bundle contract for that approved configuration, including format/schema/version expectations and how it maps back into the database-backed runtime model after import.
- [ ] Import/export behavior restores operational configuration through approved repository/bootstrap/update paths rather than creating a second live runtime authority outside the database.
- [ ] Focused tests verify round-trip handling for allowed operational configuration and confirm that excluded data categories are neither exported nor silently accepted on import.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep portability scope allowlisted** - Only approved non-secret operational configuration may move through this feature.
- **Preserve the database-first runtime model** - Portable data is for recovery/reseeding, not a second live configuration authority.
- **Keep the boundary separate from user preferences** - Operational-config portability belongs here; non-secret preference portability belongs to Epic `012`.
- **Support reset/fresh-setup recovery** - The contract must make sense for rebuilding a new database after `Nollställ` or redeployment.
- **Remain stdlib-friendly and offline** - The portability approach must fit the project environment.
- **Export only settled communication authorities** - This story must portability-wrap the communication configuration sources that earlier `003.x` stories already defined, not revive ambiguity by guessing among half-settled or duplicate config sources.

### Negative Constraints (MUST NOT)
- **Do not export secrets or secure-bootstrap state** - Key material, passwords, key-file references, and other sensitive data remain outside scope.
- **Do not absorb user-preference export/import** - Operator/UI-state portability is Epic `012`.
- **Do not absorb report-template lifecycle or template JSON portability** - That remains Epic `004` territory.
- **Do not turn this into full backup/restore of logs or entry data** - This is configuration recovery only.
- **Do not require the application to read portable files continuously at runtime** - Import/export is explicit, not a live watched-config model.
- **Do not guess portability scope from table shape alone** - Being stored in the database or looking configuration-like is not sufficient reason to include data unless the allowlist explicitly approves it.
- **Do not absorb event metadata portability implicitly** - Event categories, priorities, and their runtime authority no longer belong to Epic `003` and must not be swept in here automatically.

### Soft Preferences (SHOULD)
- Prefer a portable format that is inspectable and versionable.
- Keep the allowlist small and explicit enough that reviewers can audit it easily.
- Favor import behavior that reseeds or updates the database cleanly instead of bypassing normal repository/config loading paths.

---

## Implementation Notes

### Suggested Approach
1. Define the allowlisted operational-config domains that may be exported/imported.
2. Make that allowlist reference the settled Epic `003` communication runtime authorities so communication metadata is exported from one known source.
3. Create serializer/deserializer support for that bundle shape.
4. Route import back through approved persistence/update seams so the database remains the runtime source afterward.
5. Add focused round-trip and exclusion tests.

### Technical Considerations
- Epic `003` already reserves portability for approved non-secret operational configuration; this story turns that reservation into a concrete contract.
- The precise portable format may be JSON or another explicit package shape, but it must stay inspectable, version-aware, and separate from live runtime authority.
- This story should align with `003.004` so imported communication config can be restored coherently.
- The communication portion should reflect the clarified Epic `003` model already settled elsewhere: top-level communication systems, recursive configured options, and top-level qualifier definitions/behavior.

### Affected Components
- Import/export or serialization support for operational configuration
- Repository/core seams used to restore imported configuration into the database
- Focused tests for allowlisted round-trip behavior and exclusion enforcement

## Context & Background

Epic `003` deliberately expanded to include a future-safe recovery path because `Nollställ` destroys the database that holds operational configuration. That makes portability part of the epic boundary, but only in a narrow, disciplined form.

This story is intentionally distinct from Epic `012`:
- Epic `003` = portable communication operational configuration
- Epic `012` = portable non-secret user preferences and UI state

Keeping those separate prevents import/export from becoming one giant unfocused feature.

## Testing Strategy

### Key Test Scenarios
- [ ] Allowed operational configuration can be exported and re-imported in a stable bundle shape.
- [ ] Excluded data classes such as secrets, user preferences, templates, and entry records are not included in the export.
- [ ] Import restores configuration into the database-backed runtime source rather than leaving callers dependent on the portable file.
- [ ] Version or schema validation fails safely for malformed or unsupported portable bundles.

### Integration Points
- `003.001` defines the underlying operational-config tables/data domains this story may move.
- `003.004` defines update/runtime behavior the import path should respect.
- Epic `012` handles preference portability and should remain a distinct recovery path.
