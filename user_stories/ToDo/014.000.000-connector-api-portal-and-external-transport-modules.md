## Purpose (WHY) ⭐ MOST IMPORTANT

EventLog currently treats communication traffic as operator-entered records inside the local application. In the long run, operators may need to connect EventLog to real transport systems that can send or receive messages, such as Slack, email gateways, or radio/data-capable endpoints. Without a clear connector API boundary, each future integration risks becoming a one-off implementation that leaks transport-specific logic into GUI/presenter/core workflows and makes maintenance unsafe.

This epic exists to define a stable API portal between EventLog and optional connector modules so transport integrations can be added incrementally without redesigning the core application each time. The deeper purpose is to preserve operational reliability while enabling future expansion: EventLog must remain usable as a standalone local logger even when no connector is configured, while still supporting controlled send/receive integration when an approved module is present.

### Success Criteria
- EventLog has a clear connector API contract that future modules can implement for send/receive workflows.
- Core and GUI layers remain transport-agnostic; connector-specific logic stays behind the portal seam.
- The application continues to function in offline/local-only mode with zero connectors installed.
- Connector failures degrade gracefully and do not block normal local logging.
- The epic boundary is clear enough to split into small stories for API contract, module lifecycle, message mapping, and connector-specific adapters.

---

## Goal (WHAT)

Define and deliver the first connector integration boundary for EventLog so future external transport modules can plug into the application through one stable API portal.

This epic covers the first end-to-end contract for:
1. a connector interface/portal that exposes approved send/receive operations
2. connector lifecycle and registration/loading behavior
3. canonical message mapping between EventLog communication entries and connector payloads
4. error/isolation behavior so connector failures do not break local workflows
5. a first validation path that proves the portal can host real connector modules later

### Key Deliverables
- [ ] Connector API/portal contract with explicit send/receive operation semantics
- [ ] Connector capability contract (for example: send-only, receive-only, bidirectional)
- [ ] Connector lifecycle contract (register, initialize, health state, disable/fallback)
- [ ] Canonical message mapping contract between `CommunicationEntry` workflows and external payload shape
- [ ] Failure-handling contract for timeout/error/retry boundaries and user-visible fallback behavior
- [ ] One small validation path proving the portal seam can host external modules without coupling GUI/core directly to transport specifics

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep architecture boundaries clean** - GUI, presenter, and core communication workflows must depend on the connector portal contract, not on connector-specific implementations.
- **Preserve offline-first operation** - EventLog must remain fully usable for local/manual logging with no connector configured.
- **Use optional connector ownership** - Connectors are pluggable modules behind the API portal; the app must not require a specific connector to start.
- **Define deterministic failure behavior** - Connector errors/timeouts must resolve into explicit fallback states rather than undefined blocking behavior.
- **Use stdlib-only app constraints** - App-level portal design must respect the project's zero third-party app dependency constraint.
- **Keep security boundaries explicit** - Credential/secret handling for connectors must remain outside GUI ad hoc logic and follow approved config/runtime ownership.

### Negative Constraints (MUST NOT)
- **Do not hardcode one transport vendor as runtime truth** - Slack/email/radio examples are targets, not core architecture identity.
- **Do not couple Communication tab directly to connector SDK calls** - No direct transport API calls in views/presenters.
- **Do not make network availability a startup requirement** - Connector connectivity loss must not break local startup/logging.
- **Do not absorb broad sync/orchestration platform scope** - This epic is about a connector API portal, not a full distributed integration framework.
- **Do not bypass existing repository and validation seams** - External send/receive behavior must integrate through approved workflows, not side-channel writes.
- **Do not over-design speculative connectors** - Start with a bounded contract that supports incremental module additions.

### Soft Constraints (SHOULD)
- Prefer a capability-based connector contract so one module can implement only what it needs.
- Prefer testable pure-Python interfaces and fakes for connector contract validation.
- Prefer explicit mapping helpers between internal communication fields and connector payload fields.
- Keep first connector lifecycle behavior small and reviewable before adding advanced retry/queue policy.

## Context & Background

EventLog has recently focused on local communication workflow delivery in Epic `006`. That work improves operator speed for manual/local logging, but it does not yet provide a portal for real transport integration.

This new epic is a side track for future-proofing: it defines how optional connectors can attach to EventLog without contaminating the core architecture. The target examples include Slack, email, and radio/data-capable systems, but the first requirement is architectural: one stable portal seam that can host any approved connector module later.

Because the project remains offline-first and stdlib-only at the app level, connector support must be optional and isolated. If connector infrastructure is unavailable, EventLog should degrade to normal local operation with clear status, not failure.
