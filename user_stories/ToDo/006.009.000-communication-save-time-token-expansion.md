## Purpose (WHY) ⭐ MOST IMPORTANT

Saved templates and rapid follow-up entry only solve part of the operator-tempo problem if commonly repeated message text still has to be edited manually every time. In recurring traffic, operators often want to keep a stable message pattern while letting a few dynamic values be filled in from the current form state or the final save-time context. If the application does not support that explicitly, templates become less useful and operators will either retype repetitive text fragments or use inconsistent manual substitutions under stress.

This story exists to define the first save-time token-expansion contract for Communication entries. The goal is to let an operator write message text containing a small approved token vocabulary such as `<tid>`, `<mottagare>`, and `<sändare>`, then have those values resolved when the entry is finally saved. This must work both for ordinary manual form entry and for form content loaded from a saved template, while keeping the behavior bounded, deterministic, and presenter-owned.

Without this story, later work risks:
- saved templates still requiring repetitive manual edits for common message patterns
- ad hoc string replacement rules appearing in the presenter or view without a clear contract
- disagreement about whether tokens resolve on template load, on keypress, or at final save
- uncontrolled expansion from a few useful placeholders into an implicit scripting language

### User Value
- Operators can prepare recurring message patterns once and let the application fill in current values when saving.
- Saved templates become more practical for repeated traffic patterns without hardcoding one-off message text.
- Reviewers can validate one bounded token-expansion contract before broader template or shortcut polish continues.

---

## Goal (WHAT)

Add a bounded save-time token-expansion contract for Communication entries.

For this story, that means:
- defining an initial approved token vocabulary for save-time expansion
- resolving those tokens when a Communication entry is finally saved
- applying that same expansion rule whether the current form content was typed manually or loaded from a saved template
- keeping unresolved/unknown token behavior explicit and deterministic

This story is about save-time expansion only. It does not add live preview, arbitrary scripting, user-defined formulas, or a broad token system across the whole application.

### Acceptance Criteria
- [ ] The Communication workflow defines an explicit first token vocabulary for save-time expansion, with `message_content` as the required first target field and tokens such as `<tid>`, `<mottagare>`, and `<sändare>` covered by the initial contract.
- [ ] On final save, the presenter/save path resolves approved tokens deterministically using the current effective Communication form values and save-time context, regardless of whether the form content was entered manually or loaded from a saved template.
- [ ] The story defines explicit behavior for unknown or unsupported tokens so save-time expansion stays bounded and predictable rather than silently turning into a broader scripting system.
- [ ] Focused tests cover token expansion for ordinary entry save and template-backed save, token resolution against the effective saved values, and the chosen behavior for unsupported tokens without absorbing template-storage or generic macro-language scope.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Resolve tokens at final save only** - Token expansion belongs to the save pipeline, not template load, live typing, or ordinary display.
- **Keep presenter/save-path ownership clear** - The presenter or save-path seam must own expansion behavior; the view must not become the token engine.
- **Start with a bounded token vocabulary** - The first story should explicitly name and limit the supported tokens rather than implying open-ended substitution rules.
- **Anchor token values to effective saved data** - Tokens such as `<tid>` should resolve from the same effective values that the final save uses, not from a parallel informal source.
- **Work with and without templates** - The token-expansion contract must apply equally to manual form entry and template-loaded form content.
- **Include focused tests with the story** - Tests should cover supported-token resolution, unknown-token behavior, and save-path consistency.

### Negative Constraints (MUST NOT)
- **Do not absorb template storage/recall behavior** - Saved-template definition and persistence remain with `006.008`.
- **Do not absorb arbitrary scripting or formula evaluation** - This story must not become a macro language.
- **Do not promise token support in every field yet** - `message_content` is the required first field; any later field expansion should be explicitly approved later.
- **Do not absorb live preview or editor assistance** - Preview/help surfaces can be later polish if needed.
- **Do not absorb cross-tab token infrastructure** - This is a Communication workflow story, not a whole-application token engine.
- **Do not silently ignore contract decisions** - Unknown-token behavior must be explicit rather than accidental.

### Soft Preferences (SHOULD)
- Prefer token names and behavior that match Swedish operator-facing terminology and the actual saved Communication form fields.
- Prefer expansion behavior that is easy to explain operationally: what you see in the form may still contain tokens, but what is stored after save is resolved text.
- Keep the first token set small enough that later expansion to a sensible subset of other fields can be reviewed separately.

## Implementation Notes

### Suggested Approach
1. Define the first approved token list and what source each token reads from.
2. Extend the Communication save flow from `006.002` so token expansion happens immediately before persistence.
3. Require token expansion to use the final effective save values, including any defaulted time behavior already approved in `006.002`.
4. Keep template recall from `006.008` unchanged so templates refill raw token-bearing text and expansion happens only later at save.
5. Add focused tests for supported tokens, unknown tokens, manual-entry save, and template-backed save.

### Affected Components
- Communication presenter under `src/gui/presenters/`
- Communication save-path logic under `src/`
- Possibly a small token-expansion helper under `src/core/` if that keeps the contract clean
- Focused presenter/save-path tests for Communication token behavior

## Context & Background

This story is intentionally separate from `006.008`. Saved templates answer the question "how do we refill the form later?" Token expansion answers the different question "when and how do placeholders in the current form content turn into final saved text?"

The current agreed direction is:
- templates refill the form
- token expansion happens on final save
- token support must at least cover the message field
- other fields may later allow a sensible subset of tokens, but that should be expanded deliberately rather than implied now
