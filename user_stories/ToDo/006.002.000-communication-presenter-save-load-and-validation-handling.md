## Purpose (WHY) ⭐ MOST IMPORTANT

Once the Communication tab has a real view scaffold, the next architectural risk is leaving it as a static shell with no approved behavior seam for reading operator input, validating it, saving entries, and refreshing the visible log. If those first behavior decisions are delayed too long, later work tends to leak repository calls into the view, invent ad hoc callback chains, or mix save logic with the more complex configuration-driven system/channel behavior that belongs to a separate story.

This story exists to establish the first real MVP interaction loop for the Communication tab: the presenter reads Communication-form input from the view, performs the first approved validation and save flow, loads communication rows from the repository, and updates the view with success/error outcomes. It also needs to define the first operator-tempo rules for rapid follow-up traffic so the tab behaves like a usable message stream rather than a one-shot blank form after every save. That creates the smallest useful behavior slice while preserving room for later stories to add dynamic communication configuration, table behavior, and edit/detail flows without having to redesign the ownership split.

Without this story, later work risks:
- the view taking on save or validation responsibilities just to make the tab feel alive
- Communication-tab behavior being blocked behind the more complex configuration-driven field story
- table reload behavior, success feedback, and validation handling emerging inconsistently across future patches
- presenter ownership for the Communication tab remaining theoretical instead of proven in the first vertical slice
- rapid back-and-forth communication traffic remaining too slow because the post-save form behavior, time rules, and common sender/receiver actions were never decided explicitly

### User Value
- Operators gain the first real ability to submit Communication data through the main tab workflow.
- Reviewers can validate the MVP boundary before the tab grows more dynamic.
- Later `006.x` stories can build on an approved presenter seam instead of inventing one during larger feature work.

---

## Goal (WHAT)

Add the first Communication-tab presenter flow that can:
- read structured form input from the Communication view
- apply the first approved save-first review rules so technically storable entries are still saved even when some fields need later correction
- apply the first approved time-entry rules, including blank-as-now behavior, `DDHHMM` shorthand parsing, exact datetime parsing, and safe fallback when operator-entered time cannot be interpreted
- create and save a new `CommunicationEntry` through the approved repository/adapter path
- load communication entries for the tab on startup and after successful save
- leave the form in a deliberate post-save state that supports rapid follow-up traffic rather than treating every save as a full reset by default
- support the first explicit sender/receiver swap interaction for rapid reply traffic
- return success and soft-warning feedback to the view without moving business logic into the GUI layer

This story is intentionally limited to the first create/save/load loop. It does not yet deliver configuration-driven system/method/channel behavior, edit/delete flows, table sorting/filtering/polish, or preference-backed column state.

### Acceptance Criteria
- [x] A Communication presenter seam exists that coordinates between the Communication view and the approved communication-entry repository path without moving database access into the view.
- [x] On initial tab load, the presenter can request communication entries and render them into the Communication log view using a simple approved display contract suitable for this early slice.
- [x] On save, the presenter persists a new `CommunicationEntry` whenever the row is technically storable, refreshes the visible communication list, and leaves the view in a clear post-save state that supports rapid follow-up messages by retaining only the approved carry-forward fields.
- [x] The story defines the first approved time-entry behavior for create flow: blank time means current local time; bare `DDHHMM` is parsed as present/past shorthand; exact forced datetime input accepts `YYYY-MM-DD HH:MM` and `YY-MM-DD HH:MM`; `YY` maps to `20YY`; and malformed operator-entered time does not block save but leaves the occurrence time unknown/review-needed.
- [x] The presenter/view supports an explicit sender/receiver swap action for rapid back-and-forth traffic without pushing field-mutation ownership down into the view.
- [x] The story's first post-save carry-forward rule is explicit: clear `time` and `message`; preserve `from`, `to`, and `system`; preserve `from`/`to` exactly as entered; never auto-swap on save.
- [x] Soft-review conditions are explicit and non-blocking in this first slice: malformed explicit time input, empty message, and empty operator trigger warning/review handling but do not block save when persistence is technically possible.
- [x] Soft-warning saves use one reusable warning-dialog shape with operator-friendly Swedish text, an OK-only acknowledgement, a generic saved-but-needs-review lead, and a per-field reason list without echoing raw entered values; the status bar also receives a generic warning via the logging-driven status channel.
- [x] Communication log rows that need later review are identified by presenter-derived row state and rendered with a whole-row visual distinction, without requiring stored warning metadata or per-column issue markers in this story.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Stay presenter-centered for behavior** - Form submission coordination, validation flow, repository calls, and post-save reload behavior must be owned by the Communication presenter, not by the view.
- **Use the approved repository/adapter path** - Communication saves and loads must go through the existing communication-entry access contract from the database layer.
- **Keep validation scope minimal and explicit** - This story should establish the first approved soft-review rule set needed for the first save loop, including the first approved time-entry rule, while leaving broader configuration-driven rules for later stories.
- **Read operator/runtime context from existing app seams** - The save flow should consume the current runtime/app state already established by earlier work instead of inventing Communication-local operator ownership.
- **Define one explicit post-save follow-up rule** - The story must say what the form retains after save for rapid follow-up traffic instead of leaving full-clear versus carry-forward behavior implicit.
- **Keep rapid reply actions presenter-owned** - Behaviors such as sender/receiver swap should flow through the same presenter/view contract as the rest of the form interaction.
- **Preserve the two-time model** - `logged_time` remains the save-time system timestamp; `event_time` remains the operator-claimed occurrence time and may be blank/unknown when the operator entered an explicit time that could not be interpreted.
- **Align persistence with approved save-first behavior** - The schema/repository contract must allow the approved soft-fail cases for this story instead of forcing placeholder values or blocking save.
- **Include focused tests with the story** - Presenter tests should cover initial load, successful save, soft-warning save, carry-forward behavior, time handling, sender/receiver swap, and review-row rendering inputs.

### Negative Constraints (MUST NOT)
- **Do not absorb dynamic system/method/channel behavior** - Loading system-specific options and capability visibility belongs to `006.003`.
- **Do not absorb edit/delete/detail behavior** - Correction and detail workflows belong to a later story.
- **Do not absorb table sorting/filtering/truncation polish** - This story only needs enough display/update behavior to support the basic load/save loop.
- **Do not move repository or validation logic into the view** - The GUI should remain passive and presenter-driven.
- **Do not redefine preference-backed table state** - Column configuration and similar persistence remain for `006.006`.
- **Do not absorb saved preset/shortcut-bank behavior** - Recalling or storing reusable prefill sets such as `F1`-`F8` presets belongs to a later dedicated story.
- **Do not add warning-popup suppression in this story** - Popup suppression can be layered in later if needed; this story only establishes the baseline warning flow.
- **Do not store separate warning metadata just for row review state** - In this first slice, review state should be derived from saved values rather than duplicated in dedicated warning fields.
- **Do not turn this into a broad Communication workflow story** - Keep it to create-only presenter behavior plus simple list refresh.

### Soft Preferences (SHOULD)
- Prefer a structured view-to-presenter form submission seam rather than many field-specific getters if the existing view shape supports it cleanly.
- Prefer simple, readable success/error feedback that can be tested without requiring heavy Tk-specific assertions.
- Prefer carry-forward behavior that supports repeated traffic between the same participants without forcing operators to re-enter obvious repeated fields.
- Prefer warning/status wording that names fields and issue types without echoing raw entered values.
- Keep the first presenter contract small enough that later stories can extend it without rewriting the basic save/load loop.

## Implementation Notes

### Suggested Approach
1. Introduce a `CommunicationPresenter` seam under `src/gui/presenters/` following the MVP rules in the GUI architecture.
2. Define or reuse a small view contract for:
   - reading the current Communication form submission
   - rendering communication log rows
   - rendering success and soft-warning feedback
   - resetting or preserving form state appropriately after save/failure
   - invoking sender/receiver swap
3. Implement create-only presenter behavior:
   - read view submission
   - collect the first approved soft-review issues
   - interpret time according to the approved blank/shorthand/exact rules
   - set `logged_time` to save-time now and set `event_time` from operator input or approved fallback behavior
   - map submission to `CommunicationEntry`
   - save through the repository/adapter path
   - restore the approved carry-forward form state for rapid follow-up traffic (`from`, `to`, `system` only)
   - reload communication rows into the view
   - on soft-warning save, show the reusable warning dialog and emit the generic warning through the logging/status-bar path
4. Align the communication-entry persistence contract with the approved save-first behavior so blank `message`/`operator` values are storable without fake placeholder content.
5. Add focused tests for presenter load/save behavior, soft-warning combinations, post-save carry-forward behavior, time handling, review-row derivation, and sender/receiver swap, with only minimal GUI seam changes needed to support the presenter contract.

### Affected Components
- Communication presenter under `src/gui/presenters/`
- Communication view seam under `src/gui/views/` only as needed to support presenter-owned behavior
- App/shell wiring only if required to attach the presenter cleanly
- Communication-entry schema/repository contract to align with approved blank-field saves
- Focused presenter and GUI seam tests

## Context & Background

This story is the first behavior-carrying slice after `006.001`. It proves the Communication tab can participate in the project's MVP architecture before the more complex runtime-configuration behavior arrives.

The main design choice here is to keep the first behavior slice create-only while still making the form practical for repeated traffic and emergency logging. That means this story should lock the basic follow-up-entry behavior now: whether repeated fields carry forward after save, how the system-recorded save time differs from the operator-claimed occurrence time, how manually entered time is interpreted or downgraded to review-needed state, and how common reply actions such as sender/receiver swap are exposed.

Configuration-driven retained fields such as subdivisions/path/qualifiers may be extended in `006.003`, but the base rapid-follow-up rule should be established here first.
