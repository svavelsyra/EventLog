## Purpose (WHY) ⭐ MOST IMPORTANT

Epic `005` exists partly to stop later tabs from inventing their own column, filter, and display-state persistence rules. The Communication, Event, and Personnel workflows will all need some form of table/layout state, but each of those epics should consume one shared preference contract rather than quietly redefining it in tab-local code.

This story exists to define that cross-tab UI-state contract before the real workflow epics deepen. Without it, later work risks:
- each tab storing column widths, order, visibility, or filters in incompatible formats
- views or presenters hardcoding tab-specific persistence keys without a shared namespace plan
- repeated arguments about which tab-state belongs in bootstrap config versus runtime preferences
- avoidable rework when `006`, `007`, and `008` try to add UI-state persistence later

### User Value
- Later tabs can remember useful layouts and filters consistently.
- Reviewers can validate one cross-tab contract before multiple workflow epics depend on it.
- The project reduces rework risk by defining tab-state persistence once.

---

## Goal (WHAT)

Define the runtime preference contract for per-tab UI state such as column configuration, filter defaults, and display options, while keeping the story generic enough that later tab epics can consume it without inheriting hidden implementation decisions.

### Acceptance Criteria
- [ ] The story defines the approved runtime-preference shape and namespace rules for per-tab UI state, covering column visibility/order/width, filter defaults, and similar display options where Epic `005` already implies persistence.
- [ ] The contract makes clear which tab-state belongs in runtime preferences versus bootstrap-side shell/application settings.
- [ ] The story leaves a reusable consumption seam for later Communication, Event, and Personnel stories without requiring those epics to re-decide storage format or ownership.
- [ ] Focused tests are required for representative per-tab UI-state round-trip behavior and sane fallback when stored tab-state data is missing or malformed.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Stay contract-first and cross-tab** - This story defines shared tab-state persistence rules, not one tab's full implementation.
- **Use the shared runtime preference seam** - Per-tab state must consume the centralized access contract from `005.002`.
- **Support structured preference values explicitly** - Column/filter/display state may require structured JSON-backed storage with clear rules.
- **Keep bootstrap/runtime boundaries explicit** - Main-window geometry and startup-needed shell state remain outside this story.
- **Prepare later workflow epics without absorbing them** - The result should help `006`-`008`, not quietly implement their tab behavior early.

### Negative Constraints (MUST NOT)
- **Do not implement the Communication/Event/Personnel tab workflows here** - Those belong to Epics `006`, `007`, and `008`.
- **Do not redefine bootstrap window/application-state ownership** - That belongs to `005.001`.
- **Do not introduce per-presenter private key schemes** - Shared tab-state persistence should not become hidden local convention.
- **Do not turn this into a general settings UI story** - This is about persistence contract, not admin surfaces.

### Soft Preferences (SHOULD)
- Prefer stable, inspectable stored shapes that can be reused across multiple tabs.
- Keep the contract flexible enough for simple Phase 1 tabs without overengineering a generic UI-schema framework.
- Favor namespacing that makes later reset/default behavior easy to explain.

## Implementation Notes

### Suggested Approach
1. Identify the tab-state categories already implied by the design docs and future tab epics.
2. Define centralized key/namespace and stored-shape rules for those categories.
3. Add the shared access/helpers needed for representative tab-state round trips.
4. Add focused tests for structured per-tab preference persistence and fallback.

### Affected Components
- Runtime preference key/shape contract under `src/`
- Shared preference access/helpers for structured UI-state values
- Focused tests for per-tab UI-state round-trip and fallback behavior
