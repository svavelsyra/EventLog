## Purpose (WHY) ⭐ MOST IMPORTANT

Preference persistence only helps if the application behaves predictably when preference data is absent, reset away, or partially broken. Epic `005` explicitly includes first-run and post-reset behavior, but those rules should not be left as scattered assumptions inside each individual preference consumer. The project needs one clear fallback/default interpretation that later stories can reuse.

This story exists to define those boundary conditions after the bootstrap-side and runtime-side preference contracts are in place. Without it, later work risks:
- different tabs reacting differently to missing or malformed preference data
- reset behavior unexpectedly wiping useful state without a documented default path
- startup becoming fragile when convenience settings are damaged or missing
- recovery logic being implemented ad hoc in presenters and views rather than at the preference boundary

### User Value
- The application remains usable on first run, after reset, and after preference corruption.
- Reviewers can validate one coherent fallback/default story instead of many local edge-case implementations.
- Later preference consumers can stay simpler because boundary behavior is already defined.

---

## Goal (WHAT)

Define the shared behavior for missing, reset-lost, defaulted, and malformed bootstrap/runtime preference data so EventLog degrades gracefully instead of blocking startup or leaving UI behavior inconsistent.

### Acceptance Criteria
- [ ] The story defines the approved fallback/default behavior for both bootstrap-side UI settings and database-backed runtime preferences when values are missing, reset-lost, or unavailable.
- [ ] Reset-related behavior is explicit about which preference data survives because it is bootstrap-side and which preference data may disappear because it is database-backed convenience state.
- [ ] Malformed or partially unreadable preference data is handled through defined recovery/default rules rather than leaving each consumer to guess.
- [ ] Focused tests are required for first-run defaults, reset-lost runtime preferences, and malformed preference data across the approved preference boundaries.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Build on the earlier `005` contracts** - This story should finalize fallback behavior after `005.001` and `005.002`, not redefine their ownership.
- **Keep startup resilient** - Broken non-secret preference data must not make the application unusable.
- **Make reset behavior honest** - The story must clearly distinguish bootstrap config that remains after reset from database-backed preferences that may be recreated gradually.
- **Centralize recovery semantics** - Missing or malformed preference behavior should be decided once at the approved boundary, not separately per presenter.
- **Preserve non-secret framing** - Recovery should favor usability defaults, not security-style lockout behavior.

### Negative Constraints (MUST NOT)
- **Do not create a backup/export feature here** - Portability and broader recovery tooling remain outside this story.
- **Do not absorb the base bootstrap or runtime preference contracts** - Those belong to `005.001` and `005.002`.
- **Do not absorb specific tab/workflow implementations** - Communication/Event/Personnel consumers remain separate later stories.
- **Do not let malformed convenience data block startup unnecessarily** - Defaults should win unless a truly bootstrap-critical parse error cannot be recovered from.

### Soft Preferences (SHOULD)
- Prefer predictable, documented defaults over silent partial behavior.
- Keep the reset explanation simple enough that future user-facing docs can mirror it easily.
- Favor test scenarios that reflect realistic first-run, post-reset, and damaged-data cases.

---

## Implementation Notes

### Suggested Approach
1. Enumerate the main preference-failure modes: first run, missing key, malformed value, and database reset/recreation.
2. Define fallback/default behavior for bootstrap-side and runtime-side preferences separately.
3. Reuse the earlier contracts to keep the recovery logic centralized.
4. Add focused tests for realistic recovery/default scenarios.

### Affected Components
- Bootstrap config/default handling under `src/`
- Runtime preference access/recovery helpers under `src/`
- Focused tests for first-run, reset, and malformed-data behavior
