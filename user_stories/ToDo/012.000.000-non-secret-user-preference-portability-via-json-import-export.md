## Purpose (WHY) ⭐ MOST IMPORTANT

EventLog is intended to be a dependable operational tool, and that includes surviving predictable local lifecycle events such as `Nollställ`, database recreation, or setting up a replacement machine. If all non-secret user preferences disappear permanently whenever the database is reset, operators must rebuild their UI layout, preferred columns, filters, operator context, and similar convenience settings by hand every time. That creates friction exactly where the application is supposed to reduce it.

This epic exists to provide a safe, explicit portability path for approved non-secret user preferences using JSON import/export. The purpose is to ensure that EventLog can:
- export a defined allowlist of non-secret user preferences to a JSON file for later restore
- re-import that preference bundle after database reset, fresh setup, or deployment to another computer
- preserve the boundary between ordinary convenience settings and security-sensitive/bootstrap material that must never be exported as portable user preferences
- support operator continuity across machines and resets without turning preference portability into a full database backup or migration system
- resolve how portable preference files interact with the existing split between `config.ini` bootstrap-side UI state and database-backed `user_preferences`

The deeper reason is resilience and speed. Operators should be able to carry forward how they use the application without carrying forward sensitive material or treating the whole database as the portability mechanism.

### Success Criteria
- A clear allowlist exists for which non-secret preferences may be exported and restored through JSON
- EventLog can restore approved preference state after database reset or fresh setup without requiring manual recreation of every UI convenience setting
- The portability format and flow support moving preferences to another computer while handling machine-specific values carefully
- Security-sensitive/bootstrap material is explicitly excluded rather than relying on vague “use judgment” rules
- The epic remains focused on user-preference portability, without absorbing operational configuration, logs, entry data, attachments, or secrets
- The epic is clear enough to split into small stories for allowlist/schema definition, export flow, import/merge flow, and reset/setup integration

## Goal (WHAT)

Define and deliver a JSON import/export capability for approved non-secret user preferences and related portable UI state.

This epic covers the end-to-end contract for:
1. defining which preference data is portable and which must remain excluded
2. exporting approved preference data to a JSON file with a stable, inspectable schema
3. importing that JSON file back into a fresh or existing setup safely
4. resolving how portable preference data interacts with both `config.ini` UI-state settings and database-backed `user_preferences`
5. supporting reset-recovery and multi-machine setup workflows without redefining the application's database-first runtime model

### Key Deliverables
- [ ] Explicit allowlist for portable non-secret user preferences and UI-state values
- [ ] JSON schema/format contract for exported preference bundles, including versioning expectations
- [ ] Export workflow for writing approved preference data to a user-selected JSON file
- [ ] Import workflow for restoring approved preference data into a fresh or existing environment with defined overwrite/merge behavior
- [ ] Clear rule for machine-specific and bootstrap-side values such as database path, window geometry, and language where portability may need filtering or translation
- [ ] Reset/setup integration boundary that explains how the exported preference bundle can be used after `Nollställ` or on another computer
- [ ] Explicit boundary between preference portability and neighboring concerns such as operational configuration export, secret/bootstrap material, full backup/restore, and attachment handling

## Limitations (CONSTRAINTS & REQUIREMENTS)

**What boundaries must we work within? What must we avoid?**

### Hard Constraints (MUST)
- **Treat export/import scope as explicit and allowlisted** - Only approved non-secret preference data may be exported or imported; the feature must not assume every setting-like value is portable
- **Preserve the bootstrap/runtime split from Epic 005** - The portability flow must define clearly which values come from `config.ini`, which come from `user_preferences`, and how they are restored without blurring ownership
- **Use JSON as the portability format** - Exported preference bundles should be human-inspectable and stable enough for later import by the same application family
- **Support reset-recovery and new-machine setup** - The epic must leave a clear path for using the exported preference file after database recreation or on another computer
- **Handle machine-specific values deliberately** - Values such as database path, window geometry, or other host-specific settings must either be excluded, transformed, or clearly flagged rather than imported blindly
- **Keep this strictly non-secret** - Passwords, encryption settings, key-file details, secure-bootstrap flags, logs, and any other sensitive/security-relevant material must stay outside the exported preference file
- **Keep testing with delivery stories** - Later `012.x` stories should include their own repository/gui/import-export tests instead of creating a broad standalone testing story
- **Keep the database as normal runtime source of truth** - JSON is a portability/recovery mechanism for approved preference data, not a second live runtime authority that the app should query continuously

### Negative Constraints (MUST NOT)
- **Do not absorb operational configuration portability** - Communication systems/options/qualifiers remain Epic `003` territory, and event metadata authority remains Epic `013` territory
- **Do not absorb security/bootstrap export** - Encryption settings, `require_key_file`, key-file paths, passwords, secure-delete settings, and related security material must never be moved through this feature
- **Do not turn this into full backup/restore of the database** - Logs, entry records, structured reports, attachments, and general application state remain outside scope
- **Do not export machine-bound paths blindly** - Database file path or attachment paths must not be copied to another machine as if they were universally valid preferences
- **Do not replace Epic 005's normal persistence model** - This epic is about backup-style portability of approved preferences, not about changing how preferences are stored during ordinary runtime
- **Do not require the preference JSON file to live inside the application source tree or be auto-watched at runtime** - This is an explicit import/export flow, not a live config-file model

### Soft Constraints (SHOULD)
- Prefer small follow-up stories that separate allowlist/schema definition, export writing, import behavior, and reset/setup integration
- Keep the JSON format explicit and inspectable so operators or support personnel can understand what is being carried across
- Favor import behavior that is predictable and reviewable before adding more advanced conflict-resolution or profile-management features
- Leave room for future format-version evolution without requiring Phase 1 to solve every long-term migration problem

## Context & Background

### Architectural Context
Epic 005 defined the baseline persistence contract for non-secret user preferences and UI state, split across bootstrap-side `config.ini` values and database-backed `user_preferences`. Epic `003` separately reserved non-secret communication operational configuration portability for a later configuration-specific path, while Epic `013` now owns event metadata authority as a separate domain concern. Epic 002 defines security-sensitive bootstrap material that must remain outside ordinary preference portability.

That means this new epic is a narrow follow-on to Epic 005 rather than a general import/export bucket:
- it moves approved non-secret preference state
- it does not move operational configuration
- it does not move secrets
- it does not move application data records

### Reset and Replacement-Machine Context
`Nollställ` and fresh-database scenarios matter here because database-backed preferences can disappear when the database is destroyed or recreated. The user also wants a fast way to set up another computer without rebuilding all non-secret UI preferences from scratch.

This epic therefore exists to give those workflows a deliberate portability path:
- export before reset or redeployment
- restore after reset or fresh setup
- carry non-secret operator/UI preferences to another machine without carrying secrets or machine-specific assumptions blindly

### Structural Boundary with Other Portability Work
The project already leaves room for portability in other areas, especially communication operational configuration in Epic `003`. That makes the boundary especially important:
- Epic 003 = portable communication operational configuration
- Epic 013 = event metadata configuration/runtime authority
- Epic 012 = portable non-secret user preferences and UI state

Keeping those separate prevents “export/import” from becoming one giant vague feature that mixes unit configuration, operator conveniences, and security-sensitive bootstrap behavior.

### What This Epic Excludes
This epic is intentionally about **non-secret user preference portability via JSON**, not about:
- operational configuration export/import
- encryption/bootstrap or secret-bearing settings
- full database backup/restore
- log/event/personnel/communication data portability
- structured report templates or template lifecycle
- file attachments or attachment-file portability

## Implementation Notes

### Rough Story Sizes (Estimated)
- Portable allowlist/schema and version contract: ~60-140 lines
- Export flow and JSON writing: ~80-160 lines
- Import flow and overwrite/merge handling: ~100-200 lines
- Reset/new-machine recovery guidance and integration: ~60-140 lines

**Total estimated**: ~300-640 lines across several small stories, depending on how much existing preference-helper infrastructure Epic 005 creates first.

### Key Risks
- **Scope drift** - preference portability could easily expand into operational-config export, full backup/restore, or secret handling if the allowlist is weak
- **Ownership confusion** - bootstrap-side and database-side preference values could get mixed unless Epic 005's split is applied rigorously
- **Machine-specific import mistakes** - paths, geometry, or environment-specific settings could behave badly if imported blindly onto another computer
- **Stale-key drift** - old JSON bundles may contain preference keys no longer recognized later, so import behavior must define how unknown keys are handled
- **Security leakage** - convenience-focused export features can accidentally scoop up more data than intended if exclusion rules are not explicit

### Success Metrics
- Epic 012 is clear enough to split into small stories without mixing in unrelated configuration, security, or backup concerns
- A later implementation pass can export and restore approved non-secret preferences predictably after reset or on a replacement machine
- The portability format is explicit enough that implementation stories do not need to guess what belongs in the JSON bundle
- The project gains a recovery/setup convenience feature without undermining the database-first runtime model or security boundaries
