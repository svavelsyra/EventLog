## Purpose (WHY) ⭐ MOST IMPORTANT

Epic `005` will fail structurally if each presenter, helper, or dialog invents its own preference keys and serialization rules. The application needs one shared access contract for runtime user preferences so later GUI stories can consume preference data without turning scattered string literals and ad hoc JSON blobs into the architecture.

This story exists to define that shared runtime preference seam. Without it, later work risks:
- presenter-local preference keys with inconsistent naming and meaning
- incompatible JSON/value formats for similar kinds of settings
- duplicated read/write/default logic across tabs and dialogs
- preference migration or reset behavior becoming hard to reason about because no centralized contract exists

### User Value
- Later GUI work gets one predictable way to read and write runtime preferences.
- Preference-backed behavior becomes easier to test because storage rules are centralized.
- Reviewers can validate the shared persistence seam before more specific preference consumers are added.

---

## Goal (WHAT)

Define the database-backed `user_preferences` access contract, including centralized key ownership, value-shape rules, and shared helper/repository seams for runtime preference reads and writes.

### Acceptance Criteria
- [ ] The story defines one approved access path for runtime preference reads/writes so presenters and views do not each talk to storage with custom one-off logic.
- [ ] Preference-key ownership is centralized, with explicit naming/namespace rules that cover scalar values and structured JSON-backed values without key sprawl.
- [ ] The contract defines how missing preferences, malformed stored values, and unknown keys are handled at the access-layer boundary.
- [ ] Focused repository/helper tests are required for round-trip reads/writes, default behavior, and malformed-value handling for the approved preference access seam.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Stay on the runtime preference side** - This story covers the database-backed `user_preferences` contract, not `config.ini` ownership.
- **Centralize preference-key meaning** - Key names and stored value shapes must be defined in one shared place.
- **Support simple and structured values** - The contract should handle both straightforward scalar preferences and explicit JSON-backed structures where needed.
- **Provide caller-friendly defaults/failures** - Consumers must not need to guess what missing or malformed preference data means.
- **Remain stdlib-only and local** - The design must fit the project's offline, stdlib-only runtime constraints.

### Negative Constraints (MUST NOT)
- **Do not absorb startup/bootstrap config ownership** - That belongs to `005.001`.
- **Do not absorb specific operator-context behavior** - `last_operator` integration belongs to `005.003`.
- **Do not absorb specific per-tab UI preference content** - Column/filter/display contracts belong to `005.004`.
- **Do not hardwire presenter-specific storage behavior into the repository contract** - This story should define a reusable seam, not a Communication/Event/Personnel-specific shortcut.

### Soft Preferences (SHOULD)
- Prefer explicit, inspectable stored value formats over opaque mixed conventions.
- Keep the contract small enough that later stories can add consumers without reopening key semantics.
- Favor helper/repository naming that matches the existing adapter/repository architecture cleanly.
