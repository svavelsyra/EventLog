## Purpose (WHY) ⭐ MOST IMPORTANT

Epic `003` cannot become real if communication configuration still exists only in design docs, demos, or hardcoded assumptions. The application needs actual database structures for communication ways/systems, their configured options, and top-level qualifier definitions before later repository, core, and GUI stories can depend on them safely.

This story exists to establish that database-backed foundation and the first practical seed data for Phase 1 use. Without it, later work could drift into:
- presenters and validators hardcoding lists because the database has nowhere stable to load from
- multiple stories inventing inconsistent seed rules for unit defaults
- unclear handling of active versus inactive configuration rows
- accidental over-normalization where historical entries depend on live config rows instead of preserving the value that was used at log time

### User Value
- The project gets a real runtime source for operational metadata instead of demo-only assumptions.
- A fresh database can start with useful unit-specific defaults for immediate field use.
- Later stories can build on explicit tables and seed behavior instead of re-arguing storage shape.

---

## Goal (WHAT)

Create the database schema and first-run seed-data contract for Epic `003` operational configuration.

### Acceptance Criteria
- [ ] The database schema contains the communication-configuration storage needed for Epic `003`, covering communication ways/systems, recursive configured options beneath them, and top-level qualifier definitions plus per-system qualifier override behavior.
- [ ] First-run or initial-schema seeding provides practical Phase 1 default configuration for the current unit context, including active rows, ordering where needed, deterministic behavior that does not duplicate seed records on repeated startup/readiness paths, and the agreed anchor examples for `RA180`, `Motorola`, `Rakel`, and `Courier`.
- [ ] The schema and seed contract preserve historical entry integrity by treating configuration rows as runtime metadata while keeping saved communication entries responsible for storing the names/designations/qualifier values that were actually used at log time.
- [ ] Focused schema/repository tests verify that a new database receives the expected config tables and default records, and that repeated initialization does not create conflicting duplicates.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep configuration as database data** - The deliverable must create or extend real database tables rather than leaving the runtime source in Python constants.
- **Seed for Phase 1 practicality** - A fresh database must be able to start with usable unit-specific operational defaults without requiring a management UI first.
- **Support soft deletion/history-friendly change** - Config rows should support active/inactive or equivalent non-destructive evolution where needed.
- **Keep runtime metadata distinct from logged entry history** - Configuration storage must not force old entries to resolve live names from foreign-key lookups at read time.
- **Store the refined communication shape, not the obsolete radio-only hierarchy** - The schema should support neutral tiers and top-level qualifiers with per-system override behavior instead of assuming every communication way is best modeled as one fixed system/method/channel stack.

### Negative Constraints (MUST NOT)
- **Do not absorb runtime loading/caching policy** - Core config objects and reload behavior belong to `003.003`.
- **Do not absorb report templates** - `report_templates` and structured-report lifecycle remain Epic `004` territory.
- **Do not absorb user preferences** - `user_preferences` and operator/UI state remain Epic `005` territory.
- **Do not require a full admin/configuration screen** - This story establishes storage and seed behavior only.
- **Do not turn seeding into import/export** - Portable config packages belong to `003.006`.
- **Do not hardcode radio-only semantics into the storage contract** - Non-radio examples such as `Courier` must remain representable without awkward fake channels or fake methods.
- **Do not turn real-world channel restrictions into a Phase 1 rule engine** - Operational voice/data restrictions may inform seed examples and comments, but this story should not overbuild enforcement logic into the schema foundation.

### Soft Preferences (SHOULD)
- Prefer seed data that mirrors the documented platoon/company communication reality closely enough to support Phase 1 workflows.
- Keep naming aligned with `db_design.md`, `gui_design.md`, and later runtime config objects.
- Favor deterministic insert-or-update seeding semantics over fragile one-off bootstrapping.

## Implementation Notes

### Suggested Approach
1. Update the SQLite schema to include the Epic `003` communication configuration tables.
2. Define the initial default rows for communication ways/systems, configured options, and top-level qualifier definitions/overrides.
3. Ensure schema initialization or seed execution can recognize already-seeded data safely.
4. Add focused tests around fresh initialization and rerun-safe seeding.

### Technical Considerations
- The current design already names communication-system and channel/config areas; this story should now make the refined tier-and-qualifier model concrete rather than leaving it conceptual.
- Seed data should reflect the project's Phase 1 platoon-oriented defaults while leaving room for later unit-specific variation.
- The first seeded communication examples should prove both radio and non-radio support, not only radio support.

### Affected Components
- `src/db/schema/sqlite/initial_schema.sql`
- SQLite schema/bootstrap support and any seed-data helpers
- Focused DB tests for schema + seed initialization

## Context & Background

The design documents already assume configuration-backed communication metadata. This story turns that assumption into actual persistence. It is the foundational database slice for the rest of Epic `003`.

Because `Nollställ` destroys the database, these defaults must also be practical enough that a recreated database is still useful even before portable config import is added later.

## Testing Strategy

### Key Test Scenarios
- [ ] Fresh database initialization creates the operational configuration tables expected by Epic `003`.
- [ ] Seeded defaults for systems, configured options, and qualifier definitions/overrides are present and readable after initialization.
- [ ] Re-running the initialization/readiness path does not duplicate or corrupt seeded configuration rows.
- [ ] Inactive/update-friendly config storage does not require historical log entries to depend on live foreign-key resolution.

### Integration Points
- `003.002` will consume these tables through repository read models.
- Later startup/reset recovery work can rely on these defaults even before `003.006` portability exists.
