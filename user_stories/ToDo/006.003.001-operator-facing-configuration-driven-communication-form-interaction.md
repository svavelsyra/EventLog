## Purpose (WHY) ⭐ MOST IMPORTANT

The Communication tab is only operationally useful if its form reacts to the actual runtime communication configuration instead of staying flat or hiding hardcoded assumptions inside the GUI layer. Operators need the form to respond to the selected system and its configured path/qualifier structure, while later Epic `006` stories need a stable contract for how that runtime configuration reaches the presenter and view.

This story exists to define and preserve the operator-facing Communication-form behavior now that the live runtime path already works through the selected database, repository, core loader, presenter, and GUI. When the operator chooses a system, the form should expose the relevant configured path selections and qualifier fields, hide irrelevant controls, and reset dependent selections predictably when upstream choices change. That gives the project a reviewable consumption contract that later table, edit, template, and token stories can build on.

This is intentionally an **operator-side runtime-consumption story**, not a setup/admin-authoring story and not a JSON import/export story. The operator-facing UI should consume prepared runtime configuration and "just work" for the current environment whether that configuration came from the seeded standard baseline or from a later imported custom JSON.

### User Value
- Operators see only the Communication selections relevant to the chosen system instead of one static or misleading form.
- Reviewers can validate that runtime configuration is truly driving the Communication UI.
- Later save/edit stories get a stable config-driven interaction seam instead of hardcoded assumptions.

---

## Goal (WHAT)

Define the operator-facing Communication-form interaction as read-only consumption of the live runtime communication configuration.

For this story, that means:
- loading the available Communication systems into the form from the existing DB-backed runtime configuration path
- subdividing the chosen top-level Communication system into its configured child path/options rather than stopping at one flat system choice
- showing the next configured path selection(s) for the chosen system within a bounded visible depth of **three visible levels** in Phase 1
- showing, hiding, or pre-filling qualifier/capability controls according to the configured qualifier definitions, including explicit shown-but-read-only handling for `forced` qualifiers
- resetting dependent selections and qualifier state when the operator changes the selected system or upstream path choice

This story is about **read-only consumption of runtime configuration in the form interaction**. It does not yet deliver JSON import/export, config-authoring flows, table/filter behavior, or broader edit/delete workflow.

### Acceptance Criteria
- [ ] The Communication presenter/view consumes the existing live runtime communication configuration to populate the available top-level Communication systems without hardcoded system lists in the GUI layer.
- [ ] When the operator changes the selected system or an upstream path choice, the form updates the visible dependent path-selection controls in a predictable config-driven way, including subdivision beneath the chosen top-level system, sane reset behavior for downstream selections, and a practical Phase 1 limit of three visible path-selection levels.
- [ ] The form renders qualifier/capability controls according to the configured qualifier definitions, including editable, hidden, and shown-but-read-only handling where the runtime contract defines `visibility_mode="forced"`.
- [ ] Focused tests cover representative config-driven interaction cases such as systems with no child options, systems with child options, bounded visible depth, and qualifier visibility/default behavior, while leaving config-source authoring and later table/edit flows outside this slice.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Consume the existing runtime config seam** - The story must read Communication form behavior from the approved runtime configuration loader/model rather than inventing a new GUI-local schema.
- **Use the live DB-backed runtime path as the current authority** - The current operator-facing contract is fed through the selected database, repository seam, and core runtime loader whether the DB currently holds the seeded standard setup or an imported custom setup.
- **Keep configuration consumption read-only** - This story is about using the configured system/path/qualifier data, not editing or administering it.
- **Stay operator-focused** - The story must shape what the operator sees and selects at runtime, not the admin/setup workflow that produced the configuration.
- **Preserve MVP ownership** - The presenter owns configuration-driven interaction decisions; the view owns widgets, rendering, and structured readback/render seams.
- **Support bounded visible depth** - The Phase 1 UI should expose at most three visible path-selection levels for now, but it must still support subdivision beneath the selected top-level system and respect the recursive runtime model rather than hardcoding one named channel branch.
- **Keep qualifier behavior configuration-driven** - Defaults, visibility, and non-editable/forced cases should come from the qualifier definitions where that contract already exists, with `forced` meaning visible to the operator but not changeable.
- **Include focused tests with the story** - Tests should cover config-driven population and dependent-field behavior, not just the happy path for one system.

### Negative Constraints (MUST NOT)
- **Do not absorb general config management** - Adding, renaming, or deactivating systems/options remains outside this story.
- **Do not absorb admin/setup authoring UX** - Equipment/rule setup and other configuration-authoring responsibilities remain outside this operator-facing slice.
- **Do not absorb JSON import/export design** - Import/export and template generation for Communication configuration belong to `006.003.002`.
- **Do not absorb save/load redesign from `006.002`** - This story should extend the existing presenter loop, not replace it with a broader workflow rewrite.
- **Do not absorb table/filter/sort behavior** - Log-view polish belongs to `006.004`.
- **Do not absorb edit/delete/detail flows** - Correction workflows belong to later stories.
- **Do not hardcode demo/example values as the runtime contract** - Demo labels are references only.
- **Do not force full recursive-depth ambition into Phase 1** - This story should keep the visible interaction bounded and reviewable.

### Soft Preferences (SHOULD)
- Prefer implementation seams that align with the current runtime model of system + option path + qualifiers, even if some older persistence field names still exist.
- Prefer predictable reset behavior when upstream choices change so operators are not left with stale downstream values.
- Keep the operator-facing interaction contract small enough that later save/edit/table stories can build on it without major presenter reshaping.

---

## Implementation Notes

### Suggested Approach
1. Reuse the existing runtime communication configuration loader/model from the core layer and treat the selected database as the current live authority.
2. Extend or preserve the Communication presenter behavior so it can:
   - load available systems
   - resolve the configured subdivisions beneath the chosen top-level system
   - respond to system/path changes
   - compute the currently visible path-selection controls (up to three visible levels in Phase 1)
   - compute qualifier control state from runtime definitions, including visible-read-only treatment for `forced` qualifiers
3. Extend the Communication view seam only as needed to render:
   - available system choices
   - one or more visible path-selection controls
   - qualifier/capability controls with their current visibility and editability/read-only state
 4. Keep the resulting form interaction compatible with the save/load seam from `006.002`, but defer config-source import/export work and broader admin/setup flows beyond what this operator-facing interaction strictly needs.

### Affected Components
- Communication presenter under `src/gui/presenters/`
- Communication view seam under `src/gui/views/`
- Core runtime communication-config consumption under `src/core/` only if small adapter/helpers are needed
- Focused presenter/view tests for config-driven interaction

## Context & Background

Epic `003` already established the runtime communication-configuration seam. Epic `006` now consumes that seam through the live database-backed runtime path already wired into the application shell and presenter.

The current core runtime model is centered on:
- Communication systems
- recursive option/path definitions
- qualifier definitions with visibility/default behavior

That means this story should describe the Communication form in those terms, even if some older storage-facing field names such as method/channel/capabilities still exist elsewhere in the codebase. The key present concern is what the operator sees and how the runtime flow behaves, while config import/export and setup/admin-authoring shape are allowed to keep evolving behind the runtime seam.

For clarity, this story assumes the operator may need to drill down beneath a chosen top-level system into configured child options or subdivisions instead of treating the system choice as the final Communication selector.
