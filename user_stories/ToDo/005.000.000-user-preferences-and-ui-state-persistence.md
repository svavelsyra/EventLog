## Purpose (WHY) ⭐ MOST IMPORTANT

EventLog is meant to support repeated operational use under time pressure, not force the operator to reconfigure the interface every time the application starts. If the system forgets who is operating the terminal, how each log view was arranged, what filters were useful last time, or how the main window was positioned on cramped hardware, the application creates friction exactly where it should be reducing it.

This epic exists to make user-preference behavior explicit, durable, and architecturally clean. The purpose is to ensure that EventLog can:
- remember non-secret operator and UI preferences across restarts
- distinguish clearly between bootstrap settings that must exist before database access and runtime preferences that belong in the database
- let presenters and views reuse a stable preference contract instead of inventing ad hoc keys and serialization patterns
- preserve operator efficiency on small screens by remembering column layouts, display choices, and other UI state that affects day-to-day use
- recover safely on first run, after reset, or after preference corruption by falling back to sensible defaults instead of blocking startup

The deeper reason is trust and tempo. Operators should feel that the application behaves like a dependable working tool that remembers how they use it, while still keeping security-sensitive bootstrap concerns separate from ordinary convenience settings.

### Success Criteria
- There is a clear source-of-truth split between bootstrap configuration and database-backed user preferences
- EventLog can persist and restore the important non-secret UI settings already implied by the design documents, including operator-related context and table/layout preferences
- Presenters, helper utilities, and dialogs can consume preference data through a defined contract rather than scattered key names and one-off JSON handling
- First-run and post-reset behavior is predictable, with documented defaults when saved preferences are missing
- Epic 005 stays focused on user preferences and UI state, without absorbing operational configuration, report-template management, or secure-bootstrap logic
- The current ambiguity around where some settings belong, especially bootstrap-vs-runtime items such as language and window/application state, is resolved clearly enough for later stories to implement consistently

## Goal (WHAT)

Define and deliver the preference-persistence layer for EventLog's operator-facing settings and UI state.

This epic covers the end-to-end contract for:
1. separating bootstrap configuration in `config.ini` from runtime user preferences stored after database startup
2. persisting non-secret operator context such as `last_operator`
3. persisting per-tab UI state such as column visibility, order, widths, filter defaults, and display options
4. defining how views and presenters load, apply, update, and reset preferences without cross-layer leakage
5. establishing safe default and fallback behavior for first launch, missing settings, and reset/recreated databases

### Key Deliverables
- [ ] Clear ownership boundary for preference data kept in `config.ini` versus the database `user_preferences` store
- [ ] Preference contract for `last_operator` and other non-secret runtime user settings
- [ ] Persistence contract for per-tab column configuration, filter defaults, and display options such as priority color-coding preferences
- [ ] Startup and shutdown/load-save expectations for window geometry and other bootstrap-needed UI state
- [ ] Repository/adapter or helper access pattern for reading and writing preferences without presenter-specific key sprawl
- [ ] Defined fallback/reset behavior when preferences are missing, invalid, or lost with a reset database
- [ ] Explicit boundary between user preferences and neighboring concerns such as operational configuration, structured report templates, and secure bootstrap

## Limitations (CONSTRAINTS & REQUIREMENTS)

**What boundaries must we work within? What must we avoid?**

### Hard Constraints (MUST)
- **Keep bootstrap and runtime responsibilities separate** - Settings needed before database access must remain available from bootstrap configuration, while normal runtime user preferences should load after repository startup
- **Treat preferences as non-secret convenience data** - Operator names, window state, column layouts, filter defaults, and display choices are usability settings, not authentication or secret material
- **Support the existing GUI design split** - Window geometry and other startup-needed UI settings must align with the documented config-file role, while `user_preferences` remains the home for post-startup preference data
- **Support existing preference use cases already present in the design docs** - At minimum, the epic must leave a clear path for `last_operator`, per-tab column configuration, filter defaults, and optional display settings such as color-coding preferences
- **Provide deterministic defaults** - Missing or malformed preference data must fall back to known defaults rather than leaving presenters and views to guess
- **Keep preference access behind a defined contract** - GUI helpers, presenters, and repositories should use shared load/save behavior rather than each component inventing its own storage format and key naming scheme
- **Preserve offline-only behavior** - Preference persistence must work entirely locally with stdlib-only application dependencies
- **Keep testing with delivery stories** - Later 005.x stories should include their own tests instead of creating a broad standalone testing story

### Negative Constraints (MUST NOT)
- **Do not absorb secure bootstrap or encryption policy** - Password prompts, key derivation, SQLCipher setup, and security-sensitive startup behavior remain Epic 002 territory
- **Do not absorb operational configuration tables** - Communication systems, channel designations, event categories, and other mission configuration remain Epic 003 territory
- **Do not absorb structured report template lifecycle** - Template JSON, startup template sync, and runtime report-template behavior remain Epic 004 territory
- **Do not require a broad settings/admin screen in Phase 1** - This epic is about preference persistence and focused UI surfaces first, not a full administration console
- **Do not treat preference storage as general backup/export scope** - Preference persistence should not quietly expand into broad import/export or full reset-recovery tooling without a later explicit decision
- **Do not let presenter-specific key names become the architecture** - Ad hoc strings such as custom per-view preference keys should not spread without a centralized contract
- **Do not block startup on damaged preference data** - Invalid non-secret preferences should degrade gracefully to defaults, not make the application unusable

### Soft Constraints (SHOULD)
- Prefer small follow-up stories that separate bootstrap config persistence, runtime preference storage, operator-context handling, and per-tab UI preference application
- Keep naming consistent between `config.ini`, database preference keys, GUI helpers, and presenter code
- Favor preference formats that are explicit and inspectable when complex values need JSON storage
- Leave room for a later lightweight preferences UI without requiring that a general settings tab be designed now

## Context & Background

### Architectural Context
Epic 001 established the database foundation and explicitly deferred user preferences to Epic 005. Epic 002 defined secure bootstrap concerns and clarified that `config.ini` is for bootstrap/default behavior rather than real security enforcement. Epics 003 and 004 then explicitly kept user preferences outside their own boundaries so that operational configuration and structured report templates would not absorb ordinary UI-state concerns.

The current design already implies a split model:
- `config.ini` stores startup-needed values such as window size, position, state, and database path
- the database `user_preferences` store holds runtime user preferences such as `last_operator`, column layouts, filter defaults, and other non-secret UI state
- presenters and helper utilities already assume preference-backed behaviors, for example column configuration persistence and showing the current operator in the status bar

### Structural Issue This Epic Should Resolve
The documentation currently points to the right general direction but still leaves some ownership questions too loose. For example, language and some application-state concerns can be read as either bootstrap configuration or later runtime preference depending on which document is consulted.

Epic 005 exists partly to remove that ambiguity before implementation starts, so later stories do not each make their own local decision about where a given setting belongs.

### Reset and First-Run Context
`Nollställ` and fresh-database scenarios matter here because database-backed preferences can disappear when the database is recreated. That is acceptable as long as the application behavior is explicit:
- startup must still succeed with defaults
- bootstrap configuration must remain sufficient to launch the app cleanly
- database-backed preferences should repopulate gradually from normal use if they are missing

This epic therefore needs to define fallback and reset behavior clearly, even though it does not turn preferences into a backup/export feature.

### What This Epic Excludes
This epic is intentionally about **user-facing preference persistence and UI state**, not about:
- mission/operational configuration data
- report-template definition or import
- security or authentication behavior
- full admin tooling for every configurable thing in the application
- broad import/export or backup strategy

## Implementation Notes

### Rough Story Sizes (Estimated)
- Bootstrap config ownership and persistence cleanup: ~80-160 lines
- Preference repository/helpers and key contract: ~100-180 lines
- Operator-context persistence and integration: ~80-160 lines
- Column/filter/display preference persistence: ~120-220 lines
- Fallback/reset/default behavior: ~60-140 lines

**Total estimated**: ~500-860 lines across several small stories, depending on how much shared helper structure is introduced for presenter/view preference handling.

### Key Risks
- **Boundary confusion** - bootstrap settings and runtime preferences could get mixed if the contract is not explicit
- **Preference-key sprawl** - each presenter or dialog may invent incompatible keys/JSON structures without a centralized pattern
- **Reset surprises** - users may lose database-backed preferences after reset unless fallback behavior is well defined
- **Scope creep** - a simple persistence epic could balloon into a full settings-management UI or import/export system if not constrained
- **Inconsistent defaults** - different tabs may apply different fallback behavior if stories do not define a shared expectation

### Success Metrics
- Epic 005 is clear enough to split into small stories without drifting into security, operational configuration, or template management
- Later GUI work has a defined place to store and restore operator context and table/display preferences
- The bootstrap-vs-runtime ownership of settings is explicit enough that implementation stories do not need to re-argue it
- Missing or invalid preference data results in predictable defaults instead of broken startup or inconsistent UI behavior
