## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

Event logging is the second core function of EventLog. Operators need to log operational events, incidents, observations, and status changes. The Create operation enables permanent storage of events, while the row mapper handles type conversions (datetime, boolean).

EventEntry differs from CommunicationEntry:
- Simpler (no JSON fields like system_capabilities)
- Priority-based (Normal, High, Critical)
- Category-tagged (7S, Kontakt, Förflyttning, etc.)

### User Value
- Operators can log operational events
- Events stored permanently with priority and category
- Foundation for event tracking and situational awareness

---

## Goal (WHAT)

**What functionality are we delivering?**

Implement the row-to-entity mapper and the create operation for EventEntry in SQLiteEventLogRepository.

### Acceptance Criteria

- [ ] **Row-to-Entity Conversion**:
  - Converts field types between database storage and domain representation (datetime, boolean, etc)
  - Handles NULL values for optional fields correctly
  - Reusable by read operations
- [ ] **Create Operation**:
  - Can persist new event entry to database
  - Converts domain types to appropriate database storage types
  - Applies sensible default for priority if not provided
  - Returns database-assigned identifier
- [ ] **Unit Tests Pass**:
  - Round-trip test validates conversions (create then retrieve)
  - Priority default behavior verified
  - NULL handling verified

## Implementation Notes

### What's Needed

1. **Row-to-Entity Conversion Capability**
   - Purpose: Transform database row data into domain entity (reusable across read operations)
   - Input: Database query result row
   - Output: EventEntry domain object
   - Key considerations: Type conversions for datetime/boolean, NULL value handling for optional fields

2. **Create Operation Capability**
   - Purpose: Persist new event entry to database storage
   - Input: EventEntry domain object
   - Output: Database-assigned identifier
   - Key considerations: Type conversions to storage format, default priority handling, auto-timestamp, parameterized query safety, transaction commit

### Technical Considerations
- EventEntry has fewer fields than CommunicationEntry (simpler)
- No JSON conversion needed
- Priority defaults to 'Normal' (sensible default for operational tempo)
- Pattern identical to CommunicationEntry mapper/create

## Context & Background

EventEntry follows the exact same pattern as CommunicationEntry (001.004.001):
- Row mapper handles type conversions
- Create operation converts entity → SQL INSERT → return ID
- Type conversions needed for datetime and boolean fields

**Simpler than CommunicationEntry**:
- No structured data fields (no JSON conversion)
- Fewer optional fields
- Default priority handling
