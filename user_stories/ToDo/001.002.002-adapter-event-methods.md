## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

EventEntry represents operational events, incidents, and observations - the second core entity type. Adding these method signatures to the adapter interface:

1. **Completes the event logging contract** - Defines how events will be stored and retrieved
2. **Follows the established pattern** - Uses same structure as CommunicationEntry (from 001.002.001)
3. **Enables event repository implementation** - Repository can implement these methods

Without these method signatures, the repository cannot implement EventEntry CRUD operations.

### User Value
- Event logging capability defined in interface
- Consistent API across entity types
- Pattern established for final entity type (PersonnelEntry)

## Implementation Notes

### What's Needed

Add standard CRUD operations for EventEntry following the pattern established in 001.002.001:

1. **Create Operation**
   - Purpose: Store new event entry in database
   - Returns: Database-assigned ID for the new entry
   - Input: EventEntry domain object

2. **Get Single Operation**
   - Purpose: Retrieve specific event entry by ID
   - Returns: EventEntry if found, None if not found
   - Input: Integer ID

3. **Get All Operation**
   - Purpose: Retrieve multiple event entries with optional filtering
   - Returns: List of EventEntry objects (empty list if none match)
   - Input: Optional filter dictionary for narrowing results
   - Filter keys: operator, priority, category, whom, date_from, date_to

4. **Update Operation**
   - Purpose: Modify existing event entry
   - Returns: Success status (true if updated, false if not found)
   - Input: EventEntry with ID set

5. **Delete Operation**
   - Purpose: Remove event entry from database
   - Returns: Success status (true if deleted, false if not found)
   - Input: Integer ID

6. **Search Operation**
   - Purpose: Full-text search across event entry fields
   - Returns: List of matching EventEntry objects
   - Input: Search text string, optional filter dictionary
   - Should search event_description field and other relevant text fields

### Pattern Consistency

- Follow same structure as CommunicationEntry methods from 001.002.001
- Import EventEntry from core layer (add to existing imports)
- Use `@abstractmethod` decorator with `pass` body (no implementation)
- Consistent type hints and docstring format
- Visual separation from CommunicationEntry methods

## Context & Background

This story adds the second entity type to the adapter interface. The pattern is identical to CommunicationEntry:
- 6 CRUD methods (create, get, get_all, update, delete, search)
- Same return types and parameter patterns
- Filter dict for flexible queries on get_all
- Search method for full-text queries

EventEntry is simpler than CommunicationEntry (no JSON fields), so the interface methods are straightforward.
