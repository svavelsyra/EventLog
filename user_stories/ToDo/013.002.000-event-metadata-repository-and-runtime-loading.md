## Purpose (WHY) ⭐ MOST IMPORTANT

Even with storage and seed data in place, event metadata still fails structurally if validators, presenters, and later event filters each load categories and priorities in their own way. Event metadata becomes trustworthy only when repository and core loading expose one stable runtime contract.

This story exists to define that repository/core seam for event metadata. Without it, later work could drift into:
- direct SQL or ad hoc lookups from presenters
- duplicated hardcoded fallback lists
- inconsistent default-priority handling across layers
- conflicting active/inactive filtering rules for categories and priorities

### User Value
- Later event features can consume one shared event-metadata contract.
- Reviewers can verify event metadata behavior in one place instead of across many callers.
- Default, ordering, and active/inactive behavior become predictable and testable.

---

## Goal (WHAT)

Add repository reads and core runtime loading support for event metadata.

### Acceptance Criteria
- [ ] Repository APIs exist to read the active event metadata needed by later workflows, including categories, priorities, ordering, and default-priority behavior.
- [ ] Core-owned runtime loading support exists for that event metadata so validators and presenters can consume stable core-facing objects without database knowledge.
- [ ] The loading contract defines consistent handling for active-only reads, deterministic ordering, and default-priority lookup so later callers do not invent their own fallback logic.
- [ ] Focused tests verify repository reads, stable runtime object shape, and predictable behavior for empty/minimal/inactive metadata states.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep SQL ownership in the repository layer** - Higher layers must not need table knowledge or SQL joins.
- **Keep the core layer database-agnostic** - Core loading may depend on repository interfaces/results, not SQLite details.
- **Use one event-metadata authority** - Runtime loading must not preserve parallel lists or fallback sources once this contract exists.
- **Support later Event-tab consumption** - The result must be practical for validators, dropdowns, filters, and default selection behavior.

### Negative Constraints (MUST NOT)
- **Do not absorb Event-tab GUI delivery** - Consuming the metadata in widgets belongs to Epic `007`.
- **Do not absorb communication configuration** - Communication runtime loading remains Epic `003` territory.
- **Do not fold report-template runtime loading into this story** - Structured-report template/runtime ownership remains Epic `004` territory.
- **Do not require broad configuration-management UI** - This story is about repository/core authority, not editing screens.

### Soft Preferences (SHOULD)
- Prefer explicit runtime objects over loose nested dictionaries when practical.
- Keep naming aligned with future event-form and validator consumption.
- Favor explicit refresh calls over hidden mutable globals.

## Implementation Notes

### Suggested Approach
1. Add focused repository methods for event metadata reads.
2. Create core runtime objects/loaders for categories, priorities, and default-priority behavior.
3. Define clear active/inactive and ordering behavior.
4. Add focused DB/core tests around read/load/default behavior.

### Affected Components
- `src/db/repositories/sqlite/` repository implementation
- New or updated core configuration model/loading modules under `src/core/`
- Focused DB/core tests for event-metadata reads and runtime loading
