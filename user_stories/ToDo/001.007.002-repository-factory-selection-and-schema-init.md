## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

Once configuration data is available, the application still needs one central place that decides which repository implementation to create and ensures that the repository is ready for use.

This story exists to:

1. **Centralize repository selection** - Callers should not need to know concrete repository classes
2. **Support configured runtime behavior** - File-based and in-memory modes should come from config rather than ad-hoc caller logic
3. **Guarantee ready-to-use repositories** - Schema initialization should happen before the repository is handed back

Without this story, application bootstrap and tests would duplicate repository-selection logic and initialization behavior.

### User Value
- Repository creation follows configuration automatically
- Callers depend on the adapter interface instead of concrete classes
- Returned repositories are ready to use immediately
- Database setup logic stays centralized and maintainable

---

## Goal (WHAT)

**What functionality are we delivering?**

Create the core `RepositoryFactory.create_repository(...)` behavior that selects the correct repository implementation from supported configuration and initializes schema before returning it.

### Acceptance Criteria

- [ ] **Core Factory Entry Point**:
  - `RepositoryFactory` exposes a factory method for creating repositories
  - The method returns the `EventLogAdapter` interface type rather than a concrete repository type
  - The method can accept explicit configuration input for caller-controlled creation
- [ ] **Repository Selection Capability**:
  - Supports the currently allowed database modes for file-based SQLite and in-memory SQLite
  - Rejects unsupported database types with a clear validation error
  - Uses configuration data rather than hardcoded caller branching
- [ ] **Schema Initialization Capability**:
  - Calls repository schema initialization before returning the created repository
  - Returns a repository instance that is ready to use immediately after creation
- [ ] **Focused Validation Coverage**:
  - Tests verify the supported repository-selection paths
  - Tests verify unsupported configuration handling and schema-initialization behavior

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Return `EventLogAdapter` type** - Callers work against the adapter interface
- **Support only current Phase 1 database modes** - File-based SQLite and in-memory SQLite only
- **Initialize schema automatically** - Returned repositories must be ready to use
- **Use configuration input rather than caller-side branching** - Factory logic stays centralized

### Negative Constraints (MUST NOT)
- **Don't re-implement configuration parsing here** - Reuse the shared parsing support from `001.007.001`
- **Don't add unsupported backends yet** - No PostgreSQL or other future database types in this story
- **Don't add migration workflows here** - Schema readiness is required, but migration strategy is outside scope
- **Don't push repository-selection details into calling code** - This story exists to centralize them

### Soft Preferences (SHOULD)
- Keep the selection logic straightforward and easy to validate
- Use clear errors for unsupported or invalid database types
- Keep the method signature simple for bootstrap and other future callers

## Implementation Notes

### What's Needed

1. **Factory Entry Capability**
   - Provide the main repository-creation entry point on `RepositoryFactory`
   - Return repositories through the adapter interface expected by the rest of the application

2. **Repository Selection Capability**
   - Use configuration data to choose between supported database modes
   - Reject unsupported configuration clearly rather than silently falling back to unexpected behavior

3. **Schema-Ready Return Capability**
   - Initialize schema before handing the repository back to the caller
   - Ensure the returned repository is ready for immediate use

4. **Testing Capability**
   - Add focused tests for file-based selection, in-memory selection, invalid type handling, and schema readiness

### Technical Considerations
- This story should stay focused on repository creation and initialization, not on parsing raw configuration files directly
- The factory is the central point where concrete repository knowledge is allowed to exist
- Schema initialization is part of the repository-readiness contract for callers
- Keeping this logic isolated makes later bootstrap and fixture code simpler

### Affected Components
- **Modified/New files**:
  - `src/db/repository_factory.py`

## Context & Background

This story is the core of the repository factory split. It bridges reusable config-loading support and actual repository use.

It keeps the following responsibilities together:
- Select the supported repository implementation
- Construct it correctly
- Return it in a ready-to-use state

That keeps later callers from needing to know concrete DB setup details.
