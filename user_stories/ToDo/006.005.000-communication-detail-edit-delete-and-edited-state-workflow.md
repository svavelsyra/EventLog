## Purpose (WHY) ⭐ MOST IMPORTANT

A Communication tab that can create entries and show them in a useful table is still incomplete if operators cannot inspect a full message, correct mistakes, or remove an erroneous entry intentionally. Operational trust depends not only on quick capture but also on controlled correction: users need to confirm what was logged, adjust mistakes without creating confusing duplicates, and see that an existing record has been edited after its original save.

This story exists to deliver the first explicit correction workflow for Communication entries. It turns passive table selection into an operator-usable detail/edit/delete path while keeping the ownership split clear: the presenter coordinates selected-entry loading, edit-mode state, update/delete actions, and table refresh; the view renders the detail/edit surfaces and shows the resulting edited-state indication. That gives the Communication workflow a complete enough lifecycle for normal Phase 1 use without yet expanding into audit history, advanced undo behavior, or broader table-preference work.

Without this story, later work risks:
- operators using duplicate new entries as a workaround for corrections because no approved edit flow exists
- the table's selected-row seam from `006.004` remaining disconnected from any real operator action
- edit behavior and edited-state presentation emerging inconsistently in later patches
- delete behavior being bolted on ad hoc without a clear confirmation and refresh contract

### User Value
- Operators can inspect the full content of a saved Communication entry instead of only truncated table text.
- Operators can correct an existing Communication entry through an approved workflow instead of creating confusing duplicates.
- Reviewers can validate edited-state presentation and delete behavior before column-preference work builds on the table contract.

---

## Goal (WHAT)

Add the first selected-entry correction workflow for the Communication tab.

For this story, that means:
- opening or rendering full details for the currently selected Communication entry
- allowing the selected entry to be opened through an explicit open action, including a keyboard trigger such as Enter on the selected row
- entering an explicit edit flow for the selected entry
- saving updates to the existing `CommunicationEntry` rather than creating a new one
- deleting a selected entry through an explicit confirmed action
- surfacing edited-state indication in the Communication log after updates

This story is about the selected-entry lifecycle only. It does not yet deliver audit history, undo/restore behavior, bulk actions, advanced conflict handling, or preference-backed column customization.

### Acceptance Criteria
- [ ] The Communication workflow provides an explicit way to inspect the currently selected entry with full message content and the key saved Communication fields beyond the truncated table view, including a clear selected-entry open trigger such as Enter from the table.
- [ ] The presenter/view supports an explicit edit mode for the selected Communication entry, clearly distinguishing update behavior from the normal create flow and allowing a safe return to normal create mode afterward.
- [ ] Saving in edit mode updates the existing `CommunicationEntry`, refreshes the visible log, and presents edited-state indication in the table according to the stored entry state rather than GUI-only inference.
- [ ] The workflow supports explicit delete of a selected entry with confirmation and predictable post-delete refresh behavior, while focused tests cover detail loading, edit save, delete confirm/cancel, and edited-state presentation without absorbing preference or audit-history scope.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Build on existing Communication seams** - This story should reuse the existing Communication view, presenter save/load flow, configuration-driven form behavior, and table-selection contract rather than redesigning them.
- **Keep presenter ownership clear** - Selected-entry loading, edit-mode state, update/delete coordination, and post-action refresh behavior belong to the presenter, not the view.
- **Keep selected-entry open behavior with the detail workflow** - Mouse and keyboard triggers for opening the selected entry should converge on the same presenter-owned detail-opening seam.
- **Update existing records explicitly** - Edit flow must update the selected `CommunicationEntry` through the approved repository/adapter path rather than faking edits as new create operations.
- **Surface edited-state from stored data** - The log should show edited-state based on the repository/domain result, not by guessing in the view.
- **Require explicit delete confirmation** - Delete must be deliberate and confirmed, unlike the emergency global reset behavior elsewhere in the shell.
- **Include focused tests with the story** - Tests should cover detail rendering/loading, edit-mode transitions, successful update, delete confirm/cancel, and edited-state refresh behavior.

### Negative Constraints (MUST NOT)
- **Do not absorb audit/history features** - Edited-by, edited-at, diffs, timelines, or revision history remain outside this story.
- **Do not absorb undo/restore/archive behavior** - This story should keep deletion simple and explicit.
- **Do not redesign the table contract from `006.004`** - Sorting/filtering/selection should only be extended as needed for the correction workflow.
- **Do not reopen configuration-management scope** - Editing a Communication entry may reuse config-driven form fields, but this story is not about managing systems/options/qualifiers.
- **Do not absorb preference-backed column customization** - Column persistence remains for `006.006`.
- **Do not turn missing-selection handling into shell-wide UX redesign** - Clear local feedback is enough for this slice.
- **Do not absorb broader keyboard-navigation policy here** - This story may include Enter/open for the selected row, but synchronized next/previous browsing remains outside this slice.

### Soft Preferences (SHOULD)
- Prefer a simple read-only detail view for full message inspection over a larger always-visible detail panel if that keeps the slice focused.
- Keep the correction workflow small enough that later column-preference work can build on it without reworking the table behavior again.

### Open Design Question (Resolve Before Implementation)

- The story should **not** silently assume the edit surface yet. Before implementation, explicitly decide whether correcting an existing Communication entry should reuse the main tab form or open through a dedicated modal/detail-edit surface.
- The story should also explicitly decide whether fields that previously triggered review/warning state in a saved entry should be highlighted when that historic entry is read back for correction.

---

## Implementation Notes

### Suggested Approach
1. Reuse the selected-entry identity from the table behavior story as the entry point for explicit actions such as detail, edit, and delete.
2. Route both UI-button/open actions and Enter/open keyboard behavior through the same selected-entry detail-opening seam.
3. Add a small presenter/view seam for loading the selected entry's full data for detail display.
4. Reuse or lightly extend the Communication form so the presenter can populate it in edit mode and distinguish update vs create behavior clearly.
5. Route update and delete through the approved repository/adapter methods, then refresh the Communication log and selection state predictably.
6. Extend the table display contract from `006.004` to include the edited-state indicator once updated entries are reloaded.

### Affected Components
- Communication presenter under `src/gui/presenters/`
- Communication view seam under `src/gui/views/`
- Communication-entry repository/adapter path for update/delete behavior
- Focused presenter/view tests for selected-entry detail/edit/delete workflow

## Context & Background

This story follows the first Communication table behavior story because explicit correction work depends on stable selected-row ownership first. It is the point where the Communication workflow becomes more complete for real daily use: operators can create entries, review them in the log, inspect the full content, correct mistakes, and remove clearly erroneous entries.

The design documents already expect:
- editable Communication entries
- full message inspection beyond truncated table display
- visible edited-state presentation in the log

This story intentionally stops short of audit-style features and preference-backed table customization so that the basic correction workflow can be reviewed on its own.

Basic Enter/open behavior belongs here because it is part of opening the selected entry, while richer in-detail next/previous browsing remains separate in `006.007`.
