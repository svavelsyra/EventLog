## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

PersonnelEntry represents personnel and group tracking - the third and final core entity type. Adding these method signatures to the adapter interface:

1. **Completes the entity coverage** - All three core entity types now have database contracts
2. **Defines specialized queries** - Personnel tracking needs convenience methods (active entries, history, overdue alarms)
3. **Enables personnel safety features** - Overdue check-in alarms are critical for operational safety

Without these method signatures, the repository cannot implement personnel tracking, which is a core safety requirement.

### User Value
- Personnel tracking capability defined in interface
- Check-in alarm system enabled
- Historical tracking for each person/unit
- Complete adapter interface ready for implementation

## Implementation Notes

### What's Needed

Add standard CRUD operations plus specialized convenience queries for PersonnelEntry:

**Basic CRUD Operations** (following pattern from 001.002.001 and 001.002.002):

1. **Create Operation**
   - Purpose: Store new personnel entry in database
   - Returns: Database-assigned ID for the new entry
   - Input: PersonnelEntry domain object

2. **Get Single Operation**
   - Purpose: Retrieve specific personnel entry by ID
   - Returns: PersonnelEntry if found, None if not found
   - Input: Integer ID

3. **Get All Operation**
   - Purpose: Retrieve multiple personnel entries with optional filtering
   - Returns: List of PersonnelEntry objects (empty list if none match)
   - Input: Optional filter dictionary for narrowing results
   - Filter keys: who, operator, active, date_from, date_to

4. **Update Operation**
   - Purpose: Modify existing personnel entry
   - Returns: Success status (true if updated, false if not found)
   - Input: PersonnelEntry with ID set

5. **Delete Operation**
   - Purpose: Remove personnel entry from database
   - Returns: Success status (true if deleted, false if not found)
   - Input: Integer ID

6. **Search Operation**
   - Purpose: Full-text search across personnel entry fields
   - Returns: List of matching PersonnelEntry objects
   - Input: Search text string, optional filter dictionary
   - Should search: who, status, location, mission_notes fields

**Convenience Queries** (specialized for personnel tracking):

7. **Get Active Entries**
   - Purpose: Retrieve only currently active personnel (active=1)
   - Returns: List of active PersonnelEntry objects
   - Sorting: Most recent contact first (last_contact_time DESC)
   - Use case: Dashboard showing current field personnel

8. **Get Personnel History**
   - Purpose: Retrieve complete history for specific person/unit (all entries, active and inactive)
   - Returns: List of PersonnelEntry objects for that person/unit
   - Input: Person/unit identifier string
   - Sorting: Most recent first (logged_time DESC)
   - Use case: Review tracking history for audit or analysis

9. **Get Overdue Alarms**
   - Purpose: Find entries with overdue check-in alarms for safety alerts
   - Returns: List of PersonnelEntry objects with overdue alarms
   - Logic: alarm_enabled=1 AND expected_checkin_time < now() AND alarm_triggered=0
   - Sorting: Most overdue first (expected_checkin_time ASC)
   - Use case: Safety monitoring, alert generation

## Context & Background

PersonnelEntry has the most complex querying needs of the three entities:
- **Active vs inactive** - Most views only show active=1 entries
- **Historical tracking** - Need to see all entries for one person
- **Alarm management** - Need to find overdue check-ins for safety

The three convenience methods address these common use cases without requiring filter dict knowledge.

**Pattern consistency with Comm/Event**:
- Same 6 basic CRUD methods (create, get, get_all, update, delete, search)
- Three additional convenience methods (get_active, get_history, get_overdue_alarms)
- Search queries multiple text fields (who, status, location, mission_notes)
