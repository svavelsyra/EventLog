## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

The Adapter pattern is the foundation of EventLog's database testability. By creating an abstract base class, we:

1. **Enable testing without mocking** - Swap real SQLite for in-memory database in tests
2. **Decouple business logic from database details** - Core and GUI work against interface, not implementation
3. **Define the contract** - All implementations must provide the same methods
4. **Establish the pattern** - First entity type (CommunicationEntry) sets the template for others

This story creates the base class structure and the first entity type methods (CommunicationEntry), establishing the pattern that EventEntry and PersonnelEntry will follow.

### User Value
- More reliable application (testable architecture)
- Clear API contract for database operations
- Pattern established for remaining entity types

## Implementation Notes

### What's Needed

Create abstract base class with standard CRUD operations for CommunicationEntry:

1. **Create Operation**
   - Purpose: Store new communication entry in database
   - Returns: Database-assigned ID for the new entry
   - Input: CommunicationEntry domain object

2. **Get Single Operation**
   - Purpose: Retrieve specific communication entry by ID
   - Returns: CommunicationEntry if found, None if not found
   - Input: Integer ID

3. **Get All Operation**
   - Purpose: Retrieve multiple communication entries with optional filtering
   - Returns: List of CommunicationEntry objects (empty list if none match)
   - Input: Optional filter dictionary for narrowing results
   - Filter keys: operator, communication_system, method_type, from_field, to_field, date_from, date_to

4. **Update Operation**
   - Purpose: Modify existing communication entry
   - Returns: Success status (true if updated, false if not found)
   - Input: CommunicationEntry with ID set

5. **Delete Operation**
   - Purpose: Remove communication entry from database
   - Returns: Success status (true if deleted, false if not found)
   - Input: Integer ID

6. **Search Operation**
   - Purpose: Full-text search across communication entry fields
   - Returns: List of matching CommunicationEntry objects
   - Input: Search text string, optional filter dictionary
   - Should search relevant text fields (subject, message content, from/to, etc.)

### Base Class Structure

- Import ABC and abstractmethod from abc module
- Import CommunicationEntry from core layer (use `from __future__ import annotations` if circular imports)
- Class docstring should explain adapter pattern purpose and list all three entity types it will support
- Establish consistent pattern for method naming, type hints, and docstrings
- All methods use `@abstractmethod` decorator with `pass` body (no implementation)
