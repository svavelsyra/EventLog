## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

The low-level database adapter contract is the foundation of EventLog's database testability and backend flexibility. By creating the base adapter contract first, we:

1. **Enable testing without mocking** - Swap real SQLite for in-memory database in tests
2. **Decouple business logic from database details** - Core and GUI work against interface, not implementation
3. **Define the low-level contract** - All backends must provide the same core database primitives
4. **Establish the pattern** - Schema and query primitives are defined before backend-specific implementation work begins

This story creates the base `DatabaseAdapter` structure and the first core non-transaction methods, establishing the pattern that the remaining contract stories will follow.

### User Value
- More reliable application (testable architecture)
- Clear API contract for low-level database operations
- Pattern established for the remaining adapter contract stories

---

## Goal (WHAT)

**What functionality are we delivering?**

Create the `DatabaseAdapter` abstract base class and define the core low-level query/schema methods that every concrete backend adapter must provide.

### Acceptance Criteria

- [ ] File `src/db/database_adapter.py` exists with `DatabaseAdapter` class
- [ ] `DatabaseAdapter` inherits from `ABC` (abstract base class)
- [ ] Core low-level adapter methods are defined with `@abstractmethod`:
  - `initialize_schema()`
  - `execute()`
  - `fetch()`
  - `fetchone()`
  - `close()`
- [ ] All methods have type hints for parameters and return values
- [ ] All methods have docstrings explaining purpose, args, returns, and exceptions
- [ ] Method grouping reflects low-level database responsibilities rather than entity-specific CRUD sections

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Abstract base class** - Use `from abc import ABC, abstractmethod`
- **Type hints required** - All parameters and returns must be typed
- **Python 3.14 syntax** - Use modern type hints (e.g., `list[T]` not `List[T]`)
- **No implementation** - This is interface only, methods are abstract
- **Respect approved split** - No entity CRUD/query methods in the low-level adapter contract

### Negative Constraints (MUST NOT)
- **Don't implement any methods** - Abstract only (implementation in 001.003+)
- **Don't add repository CRUD methods here** - Those belong to repository-facing contracts, not the low-level adapter
- **Don't add transaction methods yet** - That's story 001.002.004

### Soft Preferences (SHOULD)
- Clear class docstring explaining low-level adapter purpose
- Keep method names aligned with the architecture doc
- Group methods by low-level responsibility for readability

## Implementation Notes

### What's Needed

Create abstract base class with core low-level database methods:

1. **Schema Initialization Method**
   - Purpose: Ensure backend-specific schema exists before repository use
   - Returns: No value

2. **Execute Method**
   - Purpose: Run a database command/query that may not return rows
   - Returns: Backend-appropriate execution result/cursor object

3. **Fetch Method**
   - Purpose: Retrieve multiple rows from backend storage
   - Returns: Collection of rows/results appropriate for repository mapping

4. **Fetch One Method**
   - Purpose: Retrieve one row from backend storage
   - Returns: One row/result or `None` when no match exists

5. **Close Method**
   - Purpose: Release backend resources cleanly when done
   - Returns: No value

### Base Class Structure

- Import ABC and abstractmethod from abc module
- Class docstring should explain low-level adapter purpose and its role beneath repositories
- Establish consistent pattern for method naming, type hints, and docstrings
- All methods use `@abstractmethod` decorator with `pass` body (no implementation)

### Architectural Guidance

See `ai_instructions/architecture/db_architecture.md` for:

- Abstract interface design
- Low-level adapter vs repository responsibility boundaries
