## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

Operators make mistakes and need to correct logged communications. Additionally, they need to search historical logs for specific content:

1. **Update** - Fix typos, add missing information, mark as confirmed
2. **Delete** - Remove duplicate or erroneous entries
3. **Search** - Find communications containing specific text (call signs, keywords, etc.)

Without update/delete, logged data is immutable - operators cannot correct mistakes. Without search, finding specific communications in large logs is tedious.

### User Value
- Correct mistakes in logged communications
- Remove duplicate entries
- Search for specific text in message content
- Mark communications as confirmed (receipt acknowledged)
- Maintain data quality

---

## Goal (WHAT)

**What functionality are we delivering?**

Implement update, delete, and search operations for CommunicationEntry in SQLiteEventLogRepository.

### Acceptance Criteria

- [ ] **Update Operation**:
  - Can modify existing communication entry by identifier
  - Updates all modifiable fields
  - Sets edited flag automatically (business rule)
  - Honors 5 minute grace period for immediate corrections (bussiness rule)
  - Preserves original logged timestamp
  - Returns success status (true if updated, false if not found)
- [ ] **Delete Operation**:
  - Can permanently remove communication entry from database
  - Returns success status (true if deleted, false if not found)
  - Hard delete (row removed, not soft delete)
- [ ] **Search Operation**:
  - Can search entries by text in message content
  - Search is case-insensitive
  - Can combine text search with filters from 001.004.002
  - Returns empty list if no matches
- [ ] **Unit Tests Pass**:
  - Update changes fields correctly and sets edited flag
  - Update preserves logged timestamp
  - Update non-existent ID returns false
  - Delete removes entry from database
  - Delete non-existent ID returns false
  - Search finds matching entries (case-insensitive)
  - Search with filters combines correctly

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Implement in SQLiteEventLogRepository** - Replace NotImplementedError stubs
- **Use parameterized queries** - Prevent SQL injection
- **Set edited flag on update** - Automatic flag, don't rely on caller
- **Never change logged timestamp** - Original log time must be preserved
- **Hard delete** - Permanently remove from database (no soft delete)
- **Case-insensitive search** - Search should not be case-sensitive

### Negative Constraints (MUST NOT)
- **Don't implement soft delete** - Hard delete for Phase 1
- **Don't implement audit trail** - Tracking changes comes later
- **Don't add full-text search indexes** - Basic LIKE is sufficient for Phase 1
- **Don't validate business rules** - Repository layer only stores data

### Soft Preferences (SHOULD)
- Check operation success to determine if row was affected
- Commit after UPDATE/DELETE operations
- Search should reuse filter logic from get_all operation

## Implementation Notes

### What's Needed

1. **Update Operation Capability**
   - Purpose: Modify existing communication entry in database
   - Input: CommunicationEntry object with identifier set
   - Output: Boolean success status
   - Key considerations: Auto-set edited flag (business rule), preserve original logged timestamp, type conversions to storage format, check if row was affected, commit transaction

2. **Delete Operation Capability**
   - Purpose: Permanently remove communication entry from database
   - Input: Integer identifier
   - Output: Boolean success status
   - Key considerations: Hard delete (row removed), check if row was affected, commit transaction

3. **Search Operation Capability**
   - Purpose: Find communications containing specific text
   - Input: Search text string, optional filter criteria dictionary
   - Output: List of CommunicationEntry objects matching criteria
   - Key considerations: Case-insensitive text matching, combine text search with filters from 001.004.002, reuse row-to-entity conversion, sort by event time descending

### Architectural Guidance

See `ai_instructions/architecture/db_architecture.md` for:
- Repository pattern update/delete operations
- Business rule enforcement in database layer
- Search query patterns

### Technical Considerations
- Update: edited flag set automatically if update occurs >5 minutes after logged timestamp (grace period for immediate typo fixes)
- Update: logged timestamp not modified (preserves original timestamp)
- Update/Delete: Check operation success to determine if row was affected (0 = not found)
- Search: Case-insensitive matching for user-friendly behavior
- Search: Substring matching capability (not just exact matches)
- Search: Can reuse filter building logic from get_all operation
- All operations commit after execution (unless in explicit transaction)
