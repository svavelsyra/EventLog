## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

After the core factory behavior exists, the remaining concern is making that behavior easy to use in the most common workflows and clearly validated so later stories can build on it safely.

This story exists to:

1. **Support test-friendly creation** - Tests and fixtures should have an obvious path to in-memory repositories
2. **Improve caller ergonomics** - Common usage patterns should be straightforward for future bootstrap and fixture code
3. **Complete factory validation** - Clear documentation and focused tests reduce ambiguity for later stories

Without this story, later test-fixture and bootstrap work would either duplicate convenience behavior or rely on less explicit setup patterns.

### User Value
- Easier in-memory repository creation for tests
- Cleaner and more discoverable factory usage
- Better confidence in repository factory behavior
- Smoother path into the upcoming fixture story

---

## Goal (WHAT)

**What functionality are we delivering?**

Add the convenience-facing and validation-focused parts of the repository factory story, including a clear in-memory creation path and the remaining tests/documentation needed for reliable use.

### Acceptance Criteria

- [ ] **In-Memory Convenience Capability**:
  - Provides a clear factory-supported path for creating initialized in-memory repositories for tests or similar workflows
  - Avoids requiring every caller to handcraft the same in-memory configuration repeatedly
- [ ] **Public API Clarity**:
  - Factory-facing methods include the type-hint and documentation clarity needed for future callers to use them correctly
  - Usage expectations are clear for both config-driven creation and in-memory convenience creation
- [ ] **Validation Coverage**:
  - Focused tests cover the convenience path and the public factory usage surface
  - Validation coverage is sufficient for later stories such as `001.008` to rely on the factory confidently

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Support in-memory convenience for tests** - This story should make test-oriented repository creation straightforward
- **Build on the shared parser and core factory stories** - Do not duplicate logic already covered by `001.007.001` or `001.007.002`
- **Keep the public factory API clear** - Type hints and documentation must support future reuse

### Negative Constraints (MUST NOT)
- **Don't duplicate core selection logic here** - Repository-selection behavior belongs to `001.007.002`
- **Don't create a separate ad-hoc test setup pattern outside the factory** - Keep test convenience aligned with the factory design
- **Don't expand into full fixture infrastructure here** - That belongs to `001.008`

### Soft Preferences (SHOULD)
- Keep convenience behavior minimal and intention-revealing
- Make the convenience API easy to understand for future test-fixture work
- Keep tests focused on caller-visible behavior rather than implementation details
