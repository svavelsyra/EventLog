## Purpose (WHY) ⭐ MOST IMPORTANT

Even with carry-forward behavior after save, operators still lose time if they must repeatedly reassemble the same sender/receiver/system combinations by hand whenever traffic shifts between a few known communication patterns. In practice, recurring communication setups are not always just static copies of the last message: operators may want to save only selected fields, omit others, or include calculated values such as current time when the preset is recalled. If the application does not support this explicitly, operators will either retype the same fields over and over or rely on undocumented habits that do not survive session changes.

This story exists to add reusable saved Communication templates whose contents are user-defined and can refill the form for recurring traffic. The goal is not to create a full general settings framework or arbitrary macro system, but to provide a practical operator-tempo feature for recurring communication setups that are more flexible than simple last-value carry-forward. That lets the Communication workflow support both immediate follow-up traffic and richer recurring template traffic without blurring the two behaviors into one overloaded rule.

Without this story, later work risks:
- repeated sender/receiver/system/path combinations still taking too long to re-enter during recurring traffic patterns
- operators lacking a supported way to save and later reload a reusable Communication form template
- ad hoc shortcut logic being added later without a clear storage and ownership contract
- presenter-local or view-local template persistence shapes bypassing the shared runtime preference seam
- follow-up carry-forward behavior from `006.002` being stretched to cover a different use case than it was meant for

### User Value
- Operators can recall common Communication setups quickly through saved templates and shortcut recall.
- The project gains a deliberate way to persist reusable Communication form templates instead of relying only on last-used carry-forward behavior.
- Reviewers can validate one bounded template workflow before any broader shortcut/template ambitions appear.

---

## Goal (WHAT)

Add user-defined saved templates for the Communication form that can refill the form through explicit recall, including shortcut-based recall when useful.

For this story, that means:
- defining what Communication form data belongs in a reusable saved template
- providing a way to save or update one template definition from the current form
- recalling one saved template into the current Communication form through the presenter-owned interaction seam so the operator can modify what is needed before saving the entry
- persisting template definitions and any shortcut associations through the approved runtime preference access contract rather than a Communication-local storage rule

This story is about bounded user-defined saved templates only. It does not add token-expansion behavior itself, an arbitrary scripting language, unlimited template-management complexity, or import/export of templates.

### Acceptance Criteria
- [ ] The Communication workflow defines a bounded saved-template contract for reusable Communication form data, allowing the operator to save a useful recurring form shape rather than re-entering the same values repeatedly.
- [ ] The presenter/view supports saving or updating one user-defined template and recalling a saved template into the current form, without moving storage ownership into the view.
- [ ] Saved Communication templates and their shortcut associations are persisted and restored through the approved runtime preference seam with one centralized key/shape decision rather than presenter-local or tab-local ad hoc storage.
- [ ] Focused tests cover representative template save/update/recall behavior, fallback for missing or malformed stored template data, and interaction with the current config-driven Communication form without absorbing token-expansion rules, broad shortcut-policy, or generic settings-framework scope.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Use the shared runtime preference seam** - Preset persistence must consume the centralized runtime preference access contract and key ownership rules from Epic `005` rather than inventing local storage behavior.
- **Keep saved-template behavior bounded and reviewable** - The story may support user-defined template contents and shortcut associations, but the overall template system must stay small enough to review as one coherent feature.
- **Allow practical form-template contents** - Templates may include the parts of the Communication form that are useful to refill for recurring traffic, including message text when appropriate, while still excluding clearly transient or non-form state.
- **Preserve MVP ownership** - The presenter coordinates preset save/overwrite/recall behavior; the view only exposes the relevant actions and field rendering.
- **Stay compatible with config-driven Communication form behavior** - Recalled preset data must work with the Communication system/path/qualifier model from `006.003` rather than assuming one flat form shape.
- **Keep token expansion separate from template recall** - Template loading should refill the form; token resolution belongs to a separate save-time behavior contract.
- **Include focused tests with the story** - Tests should cover persistence, overwrite, recall, and fallback behavior for saved templates.

### Negative Constraints (MUST NOT)
- **Do not absorb full shortcut-policy design** - This story should not become a general application-wide keyboard-shortcut framework.
- **Do not absorb token-expansion behavior itself** - Placeholder calculation such as `<tid>`, `<mottagare>`, or `<sändare>` belongs to a separate save-time token-expansion story.
- **Do not absorb arbitrary scripting or formula evaluation** - Templates should not become a free-form programming surface.
- **Do not bypass the centralized preference key registry** - No hidden presenter-local keys or one-off JSON blobs.
- **Do not absorb filter/table/detail behaviors** - Those remain with their existing Communication stories.
- **Do not turn this into cross-tab preset infrastructure** - This is a Communication-tab consumer story, not a generic preset platform.

### Soft Preferences (SHOULD)
- Prefer a simple template-definition surface over a powerful but hard-to-review template editor.
- Prefer template recall behavior that merges cleanly with the current form contract instead of bypassing presenter-owned recomputation.
- Keep the saved-template model small enough that it can be reviewed and implemented without reopening the broader Epic `006` structure.

---

## Implementation Notes

### Suggested Approach
1. Define one centralized runtime preference key/shape for Communication saved-template definitions and shortcut associations through the existing shared preference seam.
2. Limit the stored template payload to the reusable Communication form fields that should be refilled for recurring traffic.
3. Extend the Communication presenter so it can:
   - capture or update one template definition from the current form state
   - recall one saved template into the current form
4. Keep template recall compatible with the config-driven Communication form so system subdivisions/path/qualifiers are re-applied through the same presenter-owned interaction rules.
5. Add focused tests for template save/update/recall and malformed-data fallback behavior.

### Affected Components
- Communication presenter under `src/gui/presenters/`
- Communication view seam under `src/gui/views/`
- Runtime preference key/shape contract under `src/core/` and the shared runtime preference access seam
- Focused presenter/view/preference tests for Communication template behavior

## Context & Background

This story is intentionally separate from the basic rapid follow-up behavior in `006.002`. Carry-forward after save and reusable template recall solve related but different operator problems:
- carry-forward helps with immediate back-and-forth traffic that continues from the current message
- saved templates help with recurring communication setups that should be recalled on demand later

This story also comes after `006.003.001` in planning terms because template recall must fit the config-driven Communication form shape rather than the old flat method/channel assumptions.

Token expansion is intentionally kept separate. A loaded template should refill the form, and any tokens in fields such as message text should be resolved later by the save-time token-expansion behavior rather than being expanded merely by loading the template.
