## Purpose (WHY) ⭐ MOST IMPORTANT

Operational logs often need supporting material beyond typed text. A scanned written order, a photo, a PDF brief, or another supporting document may be essential context for understanding what was logged later. If EventLog cannot attach and retrieve those supporting files, operators still need parallel ad hoc file handling outside the system, which weakens traceability and review value.

This epic exists to define and deliver generic file attachments as a reusable capability across the application's main entry types. The purpose is to ensure that EventLog can:
- attach supporting files to communication, event, and personnel records through one consistent attachment model
- store attachment metadata and Phase 1 attachment content inside the encrypted database boundary
- let operators review, open, and remove attachments without treating attachments as a special one-off feature of only one tab
- preserve clear boundaries between ordinary file attachments and structured reports, which remain their own entity and workflow
- keep attachment handling fully offline and stdlib-compatible, aligned with the project's local-only operating model

The deeper reason is operational context and auditability. A text entry may capture the main fact, but the supporting document often explains the exact order, image, or artifact behind it. Attachments let the log function more like a working operational record instead of an isolated text notebook.

### Success Criteria
- EventLog has a single generic attachment model that can serve communication, event, and personnel records consistently
- Attachment metadata and file storage responsibilities are clearly separated and documented
- Operators can add, list, open, and remove attachments through a defined workflow without relying on manual external bookkeeping
- Structured reports remain separate from generic file attachments even when both are conceptually attached to a parent entry
- Attachment behavior stays offline, local, and reviewable without absorbing backup/export or cloud-sync scope
- The epic is clear enough to split into small stories for storage rules, repository operations, validation/copy/remove behavior, and per-workflow GUI integration

## Goal (WHAT)

Define and deliver generic file attachments and supporting document handling for EventLog's main entry types.

This epic covers the end-to-end contract for:
1. storing attachment metadata in the database using the planned generic `file_attachments` model
2. storing and retrieving Phase 1 attachment content inside the encrypted database
3. validating supported file types and basic attachment lifecycle rules
4. listing, opening, and removing attachments for communication, event, and personnel entries
5. integrating attachment behavior into the relevant GUI workflows without turning each tab into a custom attachment subsystem

### Key Deliverables
- [ ] Generic attachment metadata contract based on the `file_attachments` table and its parent-entry relationship model
- [ ] Encrypted in-database attachment-content contract for Phase 1 storage, retrieval, and removal behavior
- [ ] Repository/adapter operations for add/list/remove/get attachment content behavior
- [ ] Attachment validation contract for allowed file types, existence/copy behavior, and parent-entry association rules
- [ ] GUI integration contract for viewing and managing attachments from communication, event, and personnel contexts
- [ ] Explicit boundary between generic file attachments and neighboring concerns such as structured reports, backup/export, and advanced document-management features

## Limitations (CONSTRAINTS & REQUIREMENTS)

**What boundaries must we work within? What must we avoid?**

### Hard Constraints (MUST)
- **Use the generic attachment model already present in the design docs** - Attachment metadata must follow the planned `file_attachments` schema using parent-table and parent-id relationships rather than inventing per-tab attachment tables
- **Keep Phase 1 attachment content inside the encrypted database** - The epic must preserve one secure-at-rest boundary for both attachment metadata and attachment content
- **Support all three main entry types** - Communication, event, and personnel records must be able to consume the same attachment capability instead of forking three separate models
- **Keep attachment behavior fully offline and stdlib-compatible** - No external storage, cloud services, or third-party document systems
- **Define attachment lifecycle rules explicitly** - Add, list, open, and remove behavior must be clear enough that later implementation does not guess about path resolution or deletion responsibilities
- **Keep parent-entry relationships explicit and reviewable** - Later stories must have a clear rule for how attachments are associated with and retrieved for a specific entry
- **Respect small-screen GUI constraints** - Attachment dialogs, list areas, and action flows must remain usable on constrained screens
- **Keep testing with delivery stories** - Later `010.x` stories should include their own repository/gui/filesystem tests instead of creating a broad standalone testing story
- **Max size** Enforce a practical maximum file size for Phase 1 attachments that can be stored in the database without risking performance or stability issues (for example, 10MB or less)

### Negative Constraints (MUST NOT)
- **Do not absorb structured reports** - Structured reports remain a separate entity and workflow from Epic 004 even if both conceptually attach to an event
- **Do not store attachment content as plaintext filesystem files** - Security-sensitive attachment content must not live in normal readable files outside the encrypted database boundary
- **Do not absorb backup/export/synchronization strategy** - Generic attachment support should not quietly become a full archive, sync, or portability project
- **Do not require cloud integration, preview generation, OCR, or rich document-management features** - Phase 1 attachment behavior should remain practical and local
- **Do not force each tab to invent its own attachment subsystem** - Attachment workflows should reuse a shared model and shared infrastructure
- **Do not silently blur deletion or reset responsibilities** - Attachment removal rules must be explicit rather than left to accidental file accumulation or unexplained data loss
- **Do not assume a large-file escape hatch is guaranteed** - A future separately encrypted external-storage path is optional future work and may be refused if it weakens the security model

### Soft Constraints (SHOULD)
- Prefer small follow-up stories that separate storage rules, repository operations, filesystem copy/remove behavior, and GUI integration by workflow
- Keep allowed-file handling explicit and easy to validate
- Favor attachment interactions that let operators understand what file is attached and where to act on it without excessive modal complexity
- Leave room for later enhancements such as richer previews or export support without requiring them in the first pass

## Context & Background

### Architectural Context
Epic 001 explicitly deferred `file_attachments` as future work even though the design already anticipated a generic attachment table. Epic 004 then claimed structured reports as a separate attachment-like concept attached to events, but deliberately kept them as their own entity and workflow. Epics 006 through 008 define the tab workflows that eventually need to consume attachment behavior rather than each inventing it locally.

That means Epic 010 is the cross-cutting document-handling capability that ties these pieces together coherently:
- repository-backed attachment metadata
- encrypted attachment-content storage within the database for Phase 1
- reusable add/list/open/remove behavior
- per-entry-type GUI integration built on one shared contract
- a clean structural split from structured-report attachments

### Why Attachments Come After the Core Tab Workflows
Attachments are important, but they are not the first proof that the application's logging workflows work. The core tab epics need to establish the primary entry, review, and edit flows first. Once those workflows exist conceptually, attachment handling can be added as a reusable supporting capability rather than distorting the first MVP of every tab.

This ordering also helps preserve clarity:
- Communication, Event, and Personnel epics define the main workflows
- File attachments then add supporting-document capability across them consistently
- The attachment epic does not need to pretend to be the owner of the main workflow in any one tab

### Structural Boundary with Structured Reports
The event design refers to both generic file attachments and structured reports. That can create confusion unless the boundary is explicit:
- structured reports are data-driven report entities with validation and summary behavior
- file attachments are generic external documents or media files with metadata and file-path handling

Epic 010 should preserve that distinction. It can define how a parent entry also has normal file attachments, but it must not collapse structured-report behavior into the generic attachment model.

### What This Epic Excludes
This epic is intentionally about **generic file attachments and document handling**, not about:
- structured-report templates or report entities
- baseline communication/event/personnel tab delivery
- backup/export/synchronization strategy
- rich document preview, OCR, or media processing
- cloud storage or external services

### Key Risks
- **Boundary drift** - attachment delivery could easily absorb export, backup, preview, or broader document-management scope if not held tightly
- **Lifecycle ambiguity** - later implementation must define clearly when files are copied, removed, or left in place so operators are not surprised
- **Parent-association mistakes** - a generic attachment model must still make it obvious which entry owns which file
- **UI inconsistency** - each tab could try to present attachments differently unless the shared contract is kept explicit
- **Structured-report confusion** - event-side attachment work could accidentally blur the line between report entities and generic files if the boundary is not preserved deliberately
- **Large-file pressure** - future requests for very large attachments could tempt the project toward weaker plaintext file storage unless the security boundary stays explicit

### Success Metrics
- Epic 010 is clear enough to split into small stories without mixing in unrelated report-template, export, or cloud/document-management scope
- A later implementation pass can add, list, open, and remove attachments consistently across all three main entry workflows
- The project gains one reusable attachment model instead of per-tab divergence
- File-handling responsibilities are explicit enough that implementation stories do not need to re-argue whether metadata and content belong inside the encrypted database or in a later explicitly approved encrypted large-object path
