## Purpose (WHY) ⭐ MOST IMPORTANT

Once operators can open a Communication entry for full inspection, they still lose tempo if they must repeatedly close the detail view, find the next row manually in the table, and reopen entries one by one while searching for the right message. In real use, operators often need to scan through nearby messages in the current result set until they find the exact one they need. That means the detail view cannot be an isolated dead end; it must participate in the same ordered list the table is currently showing.

This story exists to let an already-open Communication detail view move to the previous or next entry using the current table result order, while keeping the table selection synchronized and visible. The goal is not just keyboard convenience in isolation, but continuity of operator context: the detail view should browse the same filtered/sorted list the table is showing, and when the operator closes the detail view, the table should already be positioned on the entry they ended on.

Without this story, later work risks:
- operators losing track of where they are in the current Communication result set while reviewing messages
- detail browsing being tied to repository order or chronological assumptions instead of the actual visible table order
- the selected row in the table drifting out of sync with the entry shown in the detail view
- ad hoc keyboard shortcuts being added later without a clear presenter-owned navigation contract

### User Value
- Operators can inspect several nearby Communication entries quickly without repeated close/reopen cycles.
- The detail view stays aligned with the current sorted/filtered table context instead of behaving like an isolated popup.
- Closing the detail view returns the operator to the correct selected row in the table, already visible and ready for the next action.

---

## Goal (WHAT)

Add keyboard-driven next/previous navigation for the Communication detail view using the currently visible table result order.

For this story, that means:
- allowing the operator to move to the previous or next Communication entry while a detail view is open
- resolving that navigation against the current table sort/filter order rather than a separate repository order
- keeping the table selection synchronized with the entry currently shown in the detail view
- ensuring that when the detail view closes, the selected row remains visible in the table at the current entry

This story is about synchronized review navigation only. It does not add broader shortcut systems, multi-selection, bulk review actions, persisted detail state, or new sorting/filtering rules.

### Acceptance Criteria
- [ ] While a Communication detail view is open, the workflow supports explicit previous/next navigation using the current visible table order rather than an unrelated repository ordering.
- [ ] The presenter/view keeps the selected table entry synchronized with the entry currently shown in the detail view as navigation occurs.
- [ ] When the operator reaches the first or last available entry in the current result set, the workflow handles that boundary predictably without implying navigation beyond the available rows.
- [ ] When the detail view closes, the table remains selected on the currently displayed entry and restores that row to visible context, while focused tests cover sort/filter-order fidelity, boundary navigation, selection synchronization, and close-to-table restoration behavior.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Use current visible table order as the navigation source** - Next/previous behavior must follow the currently active table ordering and filtering, not a separate repository default order.
- **Keep presenter ownership clear** - The presenter must own neighbor resolution, boundary handling, selection synchronization, and close/restore behavior.
- **Build on existing table/detail seams** - This story should reuse the selected-row behavior from `006.004` and the detail-opening workflow from `006.005` rather than redesigning them.
- **Keep table selection synchronized during browsing** - The selected entry in the table should track the detail view's current entry, not update only after the detail view closes.
- **Restore visible table context on close** - Closing the detail view should leave the operator at the same entry in the table and ensure it is visible.
- **Include focused tests with the story** - Tests should cover current-order navigation, first/last boundary handling, synchronization, and close restoration.

### Negative Constraints (MUST NOT)
- **Do not redesign table sorting/filtering behavior** - This story consumes the existing table order rather than changing how that order is computed.
- **Do not absorb broader keyboard-shortcut policy** - Only the explicit previous/next detail navigation needed for this review workflow belongs here.
- **Do not absorb edit/delete workflow changes** - Those remain with `006.005`.
- **Do not absorb preference-backed column behavior** - Column persistence remains with `006.006`.
- **Do not introduce multi-select, bulk actions, or playlist-style review** - Keep navigation centered on one current entry.
- **Do not persist detail-view browsing state across app restarts** - This is runtime interaction only.

### Soft Preferences (SHOULD)
- Prefer navigation behavior that feels local and predictable inside the detail view rather than introducing a large cross-application shortcut layer.
- Prefer clear disabled-state or no-op behavior at list boundaries over wraparound unless a later story explicitly chooses wraparound.
- Keep the story small enough that it extends the review workflow without forcing another redesign of the table/detail contracts.

---

## Implementation Notes

### Suggested Approach
1. Reuse the current visible Communication table result list as the navigation sequence source.
2. Extend the Communication presenter so it can resolve the previous and next entries relative to the currently displayed detail entry.
3. Keep the detail view and table selection synchronized as navigation occurs.
4. Ensure close behavior scrolls or restores the current selected row into visible table context.
5. Add focused tests for filtered/sorted order fidelity, first/last entry boundaries, selection synchronization, and close restoration.

### Affected Components
- Communication presenter under `src/gui/presenters/`
- Communication view seam under `src/gui/views/`
- Detail-view interaction seam introduced or reused from `006.005`
- Focused presenter/view tests for synchronized review navigation
