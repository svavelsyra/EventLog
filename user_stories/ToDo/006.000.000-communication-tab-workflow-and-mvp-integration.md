## Purpose (WHY) ⭐ MOST IMPORTANT

EventLog only becomes operationally useful when an operator can actually log communications quickly, review what was just recorded, and correct mistakes without fighting the interface. The communication workflow is one of the application's core reasons to exist: if the operator cannot capture radio traffic, phone calls, in-person messages, and written orders at speed while still seeing recent history, the earlier database, security, configuration, template, and preference work remains foundation without field value.

This epic exists to deliver the first real operator-facing workflow on top of the completed foundations. The purpose is to ensure that EventLog can:
- let an operator create communication entries rapidly in the main tabbed UI
- use configuration-driven system/method/channel data instead of hardcoded widget logic
- show recent communications in a readable table that supports review, filtering, and editing
- preserve operator tempo on small screens through a compact top/bottom layout and practical table behavior
- connect view, presenter, validation, repository access, and preference-backed column state into one coherent communication workflow

The deeper reason is trust under pressure. Operators will judge the application less by its internal architecture and more by whether the communication tab behaves like a dependable logbook: fast to enter, easy to scan, and forgiving when a correction is needed.

### Success Criteria
- A complete Communication tab workflow exists for normal Phase 1 use, from entering a message through seeing it appear in the log view
- Communication system, method, channel, and capability UI behavior is driven by runtime configuration rather than hardcoded per-system branches
- Operators can review, filter, and edit saved communication entries without losing historical clarity such as edited-state indication
- The Communication tab respects the MVP boundary, with view, presenter, validation, and repository responsibilities staying structurally clear
- Epic 006 stays focused on the communication workflow itself, without absorbing event-tab work, personnel workflows, admin configuration screens, or unrelated attachment/report features
- The epic is clear enough to split into small delivery stories for view scaffolding, presenter behavior, table/filter behavior, and edit/detail flows

## Goal (WHAT)

Define and deliver the Communication tab as the first end-to-end GUI feature workflow in the EventLog application.

This epic covers the end-to-end contract for:
1. rendering the communication entry form and communication log table in the main application UI
2. loading configuration-backed communication systems, methods, channels, and visible capability controls
3. creating, validating, saving, listing, filtering, and editing `CommunicationEntry` records through MVP interactions
4. presenting saved entries in a table with practical formatting, sorting, truncation, and edited-state visibility
5. integrating preference-backed column configuration and other tab-level UI state where that contract already exists

### Key Deliverables
- [ ] Communication tab view contract with top/bottom layout, compact entry form, filter bar, and scrollable log table
- [ ] Communication presenter contract for configuration loading, form interaction, validation handling, save/edit/delete flow, and table refresh behavior
- [ ] Dynamic system/method/channel/capability UI contract driven by runtime configuration data from Epic 003
- [ ] Communication entry create/read/update/delete workflow integrated with repository methods from Epic 001 and secure startup assumptions from Epic 002
- [ ] Log-table behavior contract for default sort order, truncation, filtering, detail/edit access, and edited indicators
- [ ] Column-configuration and tab-state integration that consumes the preference contract defined by Epic 005 without redefining it
- [ ] Explicit boundary between the communication workflow and future concerns such as admin configuration, advanced filtering, attachments, and cross-tab polish

## Limitations (CONSTRAINTS & REQUIREMENTS)

**What boundaries must we work within? What must we avoid?**

### Hard Constraints (MUST)
- **Use MVP responsibilities cleanly** - The Communication tab must follow the documented Tkinter View + Presenter pattern, keeping widget rendering in the view and coordination/validation flow in the presenter
- **Build on existing foundations instead of bypassing them** - Communication data must flow through the repository/adapter path from Epic 001 and use the secured/configured runtime assumptions established by later foundational epics
- **Use configuration-driven communication behavior** - Systems, supported methods, channel designations, and system-capability fields must load from the operational configuration model rather than being hardcoded in the tab implementation
- **Support the documented top/bottom communication layout** - The tab must preserve the fast-entry form plus recent-log overview pattern already defined in the GUI design
- **Support create and edit flows** - Operators must be able to save new communication entries and correct existing ones, with edited-state behavior visible in the log view
- **Support practical table usage on small screens** - Sorting, truncation, scrollbars, and column visibility behavior must align with the netbook-sized UI constraints documented for the project
- **Consume preference-backed UI state where applicable** - Column configuration and related tab UI state should use the Epic 005 preference contract rather than inventing communication-tab-specific persistence rules
- **Keep testing with delivery stories** - Later `006.x` stories should include their own presenter/gui/repository tests instead of creating a broad standalone testing story

### Negative Constraints (MUST NOT)
- **Do not absorb event or personnel workflows** - Event tab work and personnel tracking remain separate epics
- **Do not absorb communication-system administration UI** - Managing systems, channels, capabilities, or other operational configuration data broadly belongs to Epic 003 follow-on work, not this tab-delivery epic
- **Do not require on-the-fly config editing to make the first communication tab usable** - The seeded standard communication setup should make ordinary use viable by default, while import/export customization and later guided setup flows remain separate concerns
- **Do not absorb file attachments or document handling** - Communication attachments remain future work outside this epic boundary
- **Do not absorb advanced query-builder behavior** - Phase 1 communication filtering should stay with the simple documented AND-based filter model rather than expanding into full advanced filter logic
- **Do not bypass validators or repository formatting rules with view-side logic** - Views must not implement business rules, database access, or silent data transformation beyond UI display concerns
- **Do not hardcode demo/example values as runtime truth** - Demo values such as call signs, systems, methods, and channels are visual references only and must not become the implementation contract

### Soft Constraints (SHOULD)
- Prefer small follow-up stories that separate tab/view scaffolding, presenter-driven configuration behavior, table/filter behavior, and edit/detail workflows
- Keep Swedish UI labels and operator-facing terminology aligned with the design documents and demo reference
- Favor read-only consumption of configuration data in the first pass, with the seeded standard baseline handling ordinary use and future config-management affordances added only when separately planned
- Keep detail-view and edit interactions simple and reviewable before adding broader keyboard-shortcut or polish work

## Context & Background

### Architectural Context
Epic 001 created the communication-entry persistence path and explicitly pointed forward to a later communication-tab implementation epic. Epic 003 then defined the operational configuration layer needed for systems, methods, channels, and capabilities. Epic 005 defined the preference boundary needed for column configuration and other per-tab UI persistence.

That means Epic 006 is the first feature epic that pulls these pieces together into something an operator can actually use in the main application window:
- repository-backed `CommunicationEntry` CRUD
- configuration-driven communication-system behavior
- presenter/view coordination
- preference-backed column behavior
- table formatting and filtering suitable for daily use

For practical Phase 1 use, the communication setup is expected to come from a good standard baseline seeded into the database, with JSON export/import available as the customization path rather than as a mandatory prerequisite for basic operation.

### Why Communication Comes Before Other GUI Feature Epics
The communication tab is the most natural first GUI delivery slice because it exercises the application's main vertical path without requiring the extra structured-report complexity of the event tab or the relationship/alarm complexity of the personnel tab.

It is therefore a good first feature epic for proving that the application can:
- render a real tabbed workflow
- talk to the repository correctly
- consume runtime configuration cleanly
- apply the MVP architecture without overcomplicating the first end-to-end delivery

### Structural Boundary for Configuration Updates
The communication tab design mentions quick-add flows for new channel designations. That is important operationally, but it also overlaps with configuration-update responsibilities already fenced into Epic 003.

For that reason, Epic 006 should treat configuration consumption as the core requirement. It may leave room for a later small story or dependent story to add quick-add behavior, but the existence of that future convenience must not blur the main boundary: Epic 006 delivers the communication workflow first, not general configuration management.

### What This Epic Excludes
This epic is intentionally about **communication-tab delivery**, not about:
- event logging workflows
- personnel tracking workflows
- structured report dialogs or template-driven event reporting
- admin/settings UI for managing operational configuration broadly
- file attachment workflows
- cross-application polish unrelated to the communication tab's core workflow

## Implementation Notes

### Rough Story Sizes (Estimated)
- Communication tab view scaffold and main-window integration: ~120-220 lines
- Presenter save/load and validation behavior: ~120-220 lines
- Dynamic configuration-driven form behavior: ~120-220 lines
- Table/filter/sort behavior: ~120-220 lines
- Detail/edit/delete workflow: ~100-180 lines
- Column-config integration: ~80-160 lines

**Total estimated**: ~660-1,220 lines across several small stories, depending on how much shared presenter/view infrastructure is already in place when implementation begins.

### Key Risks
- **Boundary drift** - communication delivery could easily absorb configuration-management, attachment, or general GUI-shell scope if not held tightly
- **Presenter bloat** - too much UI, validation, and formatting logic could collect in one class if the story split is weak
- **Configuration coupling mistakes** - hardcoded branches for specific systems could creep in instead of using the runtime config model cleanly
- **Small-screen usability issues** - filters, dynamic fields, and the table can become cramped if the tab layout is not kept disciplined
- **Edit-flow ambiguity** - loading existing entries back into the form versus using a dedicated detail/edit interaction needs deliberate later-story scoping
- **Persistence mismatch** - column configuration and operator defaults could get reimplemented ad hoc if Epic 005's contract is not actually consumed

### Success Metrics
- Epic 006 is clear enough to split into small stories without mixing in unrelated event, personnel, security, or admin-config work
- A later implementation pass can deliver a genuinely usable communication workflow rather than only a static tab mockup
- The tab consumes runtime configuration and preference contracts instead of bypassing them with hardcoded shortcuts
- The project gains a reusable pattern for later tab epics after proving one end-to-end MVP workflow in practice
