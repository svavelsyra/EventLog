## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this epic exist? What problem are we solving?**

EventLog needs persistent data storage to serve its core mission: logging operational communications, events, and personnel tracking in field conditions. Without a working database layer, the application cannot store or retrieve any information, rendering it useless for its intended purpose.

This foundation must support:
- Fast operational tempo (quick writes during combat/tactical situations)
- Offline operation (no network, no external dependencies)
- Historical record integrity (entries cannot be lost)
- Query capability (find entries by various criteria)
- Testing without file system pollution (in-memory for tests)

**The database layer bridges the gap between business logic (what to store) and physical storage (how to store it).** If this layer fails or performs poorly, operators lose trust in the system and may revert to paper logs, defeating the application's purpose.

### Success Criteria
- Application can create database and tables from scratch
- Application can store and retrieve all three entity types (Communication, Event, Personnel)
- Tests can run against in-memory database without file system access
- Database operations complete in <50ms for single entries (operational tempo requirement)
- No data loss between application restarts (persistence verified)

## Limitations (CONSTRAINTS & REQUIREMENTS)

**What boundaries must we work within? What must we avoid?**

### Hard Constraints (MUST)
- **Python stdlib only** - No third-party database dependencies (use sqlite3 from stdlib)
- **SQLite3** - Must use SQLite as the database engine (design requirement)
- **No encryption yet** - Encryption (SQLCipher) is deferred to Epic 002 (simplicity first)
- **Adapter pattern** - Must implement abstract base class for future flexibility
- **Three entity types** - Separate tables for Communication, Event, Personnel (design decision)
- **Datetime as ISO 8601 text** - Must store datetimes as sortable text strings
- **JSON storage** - System capabilities and report data stored as JSON TEXT
- **Boolean as INTEGER** - SQLite convention (0=false, 1=true)

### Negative Constraints (MUST NOT)
- **Don't implement configuration tables yet** - Defer to Epic 003 (communication_systems, channel_designations, etc.)
- **Don't implement structured_reports table yet** - Defer to Epic 004 (7S reports)
- **Don't implement file_attachments yet** - Defer to future epic
- **Don't implement encryption** - Epic 002 will add SQLCipher (security layer)
- **Don't implement migrations yet** - First run creates schema, migrations come later
- **Don't optimize prematurely** - Basic indexes only, advanced optimization later
- **Don't add full-text search** - Not required for Phase 1

### Soft Constraints (SHOULD)
- Keep it simple - Minimum viable database layer
- Test coverage >80% for repository methods
- Clear error messages on database failures
- Type hints on all methods (Python 3.14)

## Context & Background

### Architectural Context
EventLog uses a three-layer architecture:
- **GUI Layer** - Presenters call repository methods
- **Core Layer** - Domain entities (CommunicationEntry, EventEntry, PersonnelEntry)
- **Database Layer** - THIS EPIC - Adapters + Repositories persist entities

The database layer is the lowest layer, with no upward dependencies. Core entities are simple dataclasses/models with no database knowledge.

### Design Decisions Already Made
1. **Three specialized tables** - Not one generic "log_entries" table
   - Reasoning: Different entity types have different fields and business rules
2. **Adapter pattern** - Abstract base class allows swapping implementations
   - Reasoning: Enables testing with in-memory DB without mocking
3. **Single repository class** - One repository with methods for all entity types
   - Reasoning: Simpler factory, shared connection/transaction context
4. **Row dictionary pattern** - `sqlite3.Row` for dict-like access
   - Reasoning: Cleaner than tuple indexing

### What This Epic Excludes
- **Encryption**: Deferred to Epic 002 (SQLCipher integration)
- **Configuration tables**: Deferred to Epic 003 (systems, channels, templates)
- **Structured reports**: Deferred to Epic 004 (7S report table)
- **User preferences**: Deferred to Epic 005 (UI settings storage)
- **File attachments**: Deferred to future epic
- **Advanced queries**: Basic filters only, complex search later
