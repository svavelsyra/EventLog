## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

Communication logging is the primary function of EventLog. Radio messages, phone calls, and orders must be persistently stored so that:

1. **Operational continuity** - Communications survive application restarts
2. **Historical record** - Complete audit trail of all communications
3. **Search and filter** - Find specific communications by time, operator, method, etc.
4. **Edit and correct** - Fix mistakes or add late-logged entries

Without CRUD operations for CommunicationEntry, the application cannot fulfill its core mission of logging communications.

### User Value
- Operators can log radio messages and phone calls
- Communications are permanently stored (not lost on crash)
- Users can search historical communications
- Mistakes can be corrected (edit/delete capability)

## Goal (WHAT)

**What functionality are we delivering?**

Implement all CRUD (Create, Read, Update, Delete) operations for `CommunicationEntry` in the SQLite repository layer under the approved package layout. This includes basic retrieval, filtered queries, and text search, while delegating low-level SQL execution and transaction mechanics to the SQLite adapter.

### Acceptance Criteria

**Create Operation**:
- [ ] Can create new communication entry in database
- [ ] Converts all Python types to appropriate database types:
  - Datetime objects to ISO 8601 TEXT format
  - Dictionary (system_capabilities) to JSON TEXT
  - Boolean fields to INTEGER (0/1)
- [ ] Returns database-generated ID
- [ ] Sets logged_time to current timestamp
- [ ] Sets default values for edited=0, confirmed=0

**Read Operations**:
- [ ] Can retrieve single communication entry by ID
- [ ] Returns None if ID not found
- [ ] Converts database row to CommunicationEntry object with proper type conversions:
  - ISO 8601 TEXT to datetime objects
  - JSON TEXT to Python dict
  - INTEGER to boolean
- [ ] Handles NULL values appropriately for optional fields

**List & Filter**:
- [ ] Can retrieve all communication entries
- [ ] Can filter entries by supported criteria:
  - Operator (exact match)
  - Communication system (exact match)
  - Method type (exact match)
  - From field (exact match)
  - To field (exact match)
  - Date range (event_time >= date_from, event_time <= date_to)
- [ ] Results sorted by event_time descending (most recent first)
- [ ] Returns empty list if no matches

**Search**:
- [ ] Can search communication entries by text content
- [ ] Search is case-insensitive
- [ ] Can combine text search with filters
- [ ] Returns empty list if no matches

**Update Operation**:
- [ ] Can update existing communication entry
- [ ] Applies the edited-flag business rule using the configured grace-period setting rather than a hardcoded immediate flag change
- [ ] Preserves original logged_time (never modified)
- [ ] Returns success status (true if updated, false if ID not found)
- [ ] Properly converts all types for database storage

**Delete Operation**:
- [ ] Can permanently remove communication entry from database
- [ ] Returns success status (true if deleted, false if ID not found)

**Helper Methods**:
- [ ] Private helper method for converting database rows to CommunicationEntry objects
- [ ] Handles NULL values appropriately
- [ ] Converts all database types to Python types correctly

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Implement in the SQLite repository layer** - Build on the repository skeleton from 001.003 using the approved package layout
- **Delegate low-level DB mechanics downward** - Repository prepares queries and mapping; adapter handles low-level execution/transaction mechanics
- **Use parameterized queries** - Prevent SQL injection (use ? placeholders)
- **Type conversions must be correct** - datetime ↔ ISO 8601, JSON ↔ dict, INTEGER ↔ bool
- **Handle NULL values** - Optional fields can be None in database
- **Respect edited-flag policy** - Update behavior must follow the configured grace-period rule from the `settings` table
- **Preserve logged_time** - Never change logged_time on update

### Negative Constraints (MUST NOT)
- **Don't implement file attachments** - File support deferred
- **Don't validate business rules** - That's Core layer responsibility
- **Don't implement StructuredReport joins** - Reports deferred to Epic 004
- **Don't add full-text search indexes** - Basic LIKE search is sufficient for Phase 1
- **Don't implement soft deletes** - Hard delete for now

### Soft Preferences (SHOULD)
- Consistent error messages
- Use helper method for row-to-entity conversion (reusability)
- Log SQL errors with context
- Consider adding batch operations later (not required for Phase 1)

## Implementation Notes

### What's Needed

1. **Create Operation**
   - Purpose: Store new communication entry in database
   - Input: CommunicationEntry domain object
   - Returns: Database-assigned integer ID
   - Type conversions: datetime → ISO 8601 string, dict → JSON string, bool → 0/1
   - Business rule: Sets logged_time to current timestamp automatically
   - Uses parameterized SQL query with ? placeholders

2. **Read Single Entry**
   - Purpose: Retrieve specific communication entry by ID
   - Input: Integer ID
   - Returns: CommunicationEntry object if found, None if not found
   - Uses helper method for row-to-entity conversion

3. **Helper: Row-to-Entity Conversion**
   - Purpose: Convert database row to CommunicationEntry object (reusable across read operations)
   - Implementation: Private method handling all type conversions
   - Type conversions: ISO 8601 string → datetime, JSON string → dict, 0/1 → bool
   - NULL handling: Optional fields may be None

4. **List All with Filtering**
   - Purpose: Retrieve multiple entries with optional filtering
   - Input: Optional filter dictionary with supported keys
   - Returns: List of CommunicationEntry objects (empty list if no matches)
   - Filter support: operator, communication_system, method_type, from_field, to_field, date_from, date_to
   - Sorting: Results ordered by event_time descending
   - Implementation: Dynamic SQL building based on provided filters

5. **Text Search**
   - Purpose: Find entries containing specific text
   - Input: Search text string, optional filters
   - Returns: List of CommunicationEntry objects matching search criteria
   - Implementation: Case-insensitive LIKE query on message_content
   - Can combine with filters for refined results

6. **Update Operation**
   - Purpose: Modify existing communication entry
   - Input: CommunicationEntry object with ID set
   - Returns: Boolean success status
   - Business rules: Uses configured edited-flag grace-period logic and preserves original logged_time
   - Type conversions: Same as create operation
   - Returns false if entry ID not found

7. **Delete Operation**
   - Purpose: Permanently remove entry from database
   - Input: Integer ID
   - Returns: Boolean success status (true if deleted, false if not found)
   - Implementation: Hard delete (removes row)

### Technical Considerations
- Use `json.dumps()` and `json.loads()` for system_capabilities dictionary
- Use `datetime.isoformat()` and `datetime.fromisoformat()` for timestamp conversions
- Handle None values when parsing optional datetime fields
- Use `cursor.lastrowid` to get auto-generated ID after INSERT
- Delegate low-level execution and transaction handling through the adapter/repository seam established in 001.003
- Filter building: Start with "WHERE 1=1" for cleaner dynamic query construction

## Context & Background

This story establishes the pattern for all CRUD operations. EventEntry (001.005) and PersonnelEntry (001.006) will follow the same structure:
- Create: Convert entity → SQL INSERT → return ID
- Read: SQL SELECT via adapter → convert row → return entity
- Update: Convert entity → SQL UPDATE via adapter → return bool
- Delete: SQL DELETE via adapter → return bool
- Helper: Row → entity conversion

**Type conversion mappings**:
- Python datetime ↔ SQL TEXT (ISO 8601 format)
- Python dict ↔ SQL TEXT (JSON format)
- Python bool ↔ SQL INTEGER (0/1)
- Python None ↔ SQL NULL
