## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

Repository creation depends on configuration, but configuration parsing is a reusable application/bootstrap concern that should not be reinvented in every future story that needs it.

This story exists to:

1. **Centralize config loading** - One shared place to read bootstrap/application settings, including database settings
2. **Avoid duplicate parser logic** - Future stories should extend shared parsing, not create competing loaders
3. **Support future growth** - If additional config sections are needed later, the parsing support should already be structured for reuse

Without this reusable layer, repository creation would mix configuration concerns directly into factory logic and future stories could end up duplicating parser behavior.

### User Value
- One consistent way to load database configuration
- Lower maintenance cost for future config-related work
- Cleaner separation between config parsing and repository creation
- Less risk of duplicated parsing behavior across the codebase

---

## Goal (WHAT)

**What functionality are we delivering?**

Provide reusable application configuration loading support under `src/config/` that can supply normalized database settings to later factory code while remaining extendable for future configuration domains.

### Acceptance Criteria

- [ ] **Create-or-Update Shared Config Loader Support**:
  - Shared application configuration loading support exists in reusable application/bootstrap infrastructure under `src/config/`
  - If no suitable shared loader support exists, it is created
  - If suitable shared loader support already exists, it is extended rather than duplicated
- [ ] **Database Config Loading Capability**:
  - Can read the database settings needed by later repository-factory work from application configuration
  - Handles the current database configuration shape consistently with `config.ini.template`
  - Returns normalized configuration data in a form suitable for factory use
- [ ] **Validation and Reuse Coverage**:
  - Focused tests verify successful loading plus caller-visible missing or malformed configuration behavior
  - The resulting loader support remains reusable for future non-database configuration needs beyond this story

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Use stdlib parsing only** - Use Python standard library configuration support
- **Create or update shared config loader support** - Do not create duplicate config loader modules for the same concern
- **Keep loader support reusable** - Structure it so future development can add more configuration-loading behavior cleanly
- **Align with current config template** - Database parsing must match the current `config.ini.template` shape
- **Treat config loading as a shared application concern** - The loader must not be owned by the DB package just because one consumer is the repository factory

### Negative Constraints (MUST NOT)
- **Don't hardcode repository-instantiation logic here** - Repository selection belongs to `001.007.002`
- **Don't tie the loader to only this one story** - The result should be reusable infrastructure, not a one-off helper
- **Don't add unrelated configuration domains unless needed for parser structure** - Keep focus on enabling reusable database config loading

### Soft Preferences (SHOULD)
- Keep the config-loader API simple for callers such as the repository factory
- Prefer explicit normalized output over ad-hoc config access scattered through the codebase
- Make missing or malformed configuration behavior clear to callers, tests, and documentation

## Implementation Notes

### What's Needed

1. **Shared Config Loader Capability**
   - Provide or extend shared config-loading support under `src/config/` for application/bootstrap concerns, including the database settings needed by later repository-factory work
   - Support either creating a new shared module or extending an existing one, depending on what the codebase already contains at implementation time

2. **Normalized Database Settings Capability**
   - Read the database settings needed by downstream factory code from application configuration
   - Normalize those settings into a consistent caller-facing shape

3. **Validation and Reuse Capability**
   - Add focused tests for successful loading and caller-visible invalid-configuration behavior
   - Ensure the loader structure remains open for later config-related additions

### Technical Considerations
- This story intentionally separates reusable config parsing from concrete repository instantiation
- The current config template already defines a database section, so this story should align to that instead of inventing a parallel shape
- Reusable parser infrastructure may already exist by the time implementation begins, so the story must support create-or-update behavior
- The same `config.ini` also supports broader application/bootstrap concerns such as logging and window state, so the shared loader should live outside `src/db/`
- Clean separation here reduces coupling and keeps later factory stories focused on repository creation

### Affected Components
- **Modified/New files**:
  - Shared config loader support under `src/config/`
  - `tests/unit/db/test_repository_factory.py` or equivalent focused config-loading tests

## Context & Background

This story exists because config parsing is a conceptually different concern from selecting and initializing concrete repository implementations, and because `config.ini` already serves broader bootstrap/application needs beyond the database layer.

By treating parsing as reusable infrastructure:
- Later factory logic can stay focused on repository creation
- Future configuration stories can build on a shared config-loader foundation
- The project avoids multiple small, inconsistent config loaders appearing over time
