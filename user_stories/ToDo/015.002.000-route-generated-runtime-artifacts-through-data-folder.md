# User Story: 015.002 - Route Generated Runtime Artifacts and References Through the `data/` Folder

**Story Number**: 015.002  
**Created**: 2026-05-15  
**Status**: Done  
**Epic**: [015 - App-Local Runtime Data Folder](015.000.000-app-local-runtime-data-folder.md)  
**Owner**: Developer

**Filename**: `015.002.000-route-generated-runtime-artifacts-through-data-folder.md`

---

## Purpose (WHY) ⭐ MOST IMPORTANT

A shared `data/` root is only useful if the direct-dependent runtime surfaces actually use it and the project stops describing older mixed locations. If config, database, logs, reset/cleanup assumptions, comments, or templates still point to old root-level paths, reviewers and future implementers will keep reintroducing split storage behavior.

This story exists to make the chosen storage layout visible and consistent across the small set of direct dependents that own generated runtime artifacts and their default descriptions.

### User Value
- Operators and administrators can see one understandable storage layout instead of mixed old/new path assumptions.
- The default runtime artifacts end up together, making copy/move/backup/removal easier.
- Future work inherits the approved storage model from code, tests, and project-facing references.

---

## Goal (WHAT)

Align runtime-generated artifact behavior and direct project references with the app-local `data/` folder contract.

### Acceptance Criteria
- [x] Runtime-generated artifacts use the app-local `data/` folder as the canonical default home, at minimum for the live `config.ini`, managed SQLite database, and default file logs.
- [x] Direct-dependent startup, reset/cleanup, and logging flows continue to work through shared path helpers/contracts rather than retaining hard-coded assumptions about old root-level or per-user locations.
- [x] Project-facing defaults and references that describe these generated locations are updated to reflect the approved layout, including examples such as `data/config.ini`, `data/eventlog.db`, and `data/logs/...` where appropriate.
- [x] Focused regression tests cover first-run/default behavior and catch any direct-dependent path assumptions that still point outside the canonical `data/` root.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Build on the shared root from `015.001`** - This story should consume the centralized path contract rather than creating another one.
- **Stay focused on direct dependents of the storage decision** - Adjust only the code/tests/docs that directly describe or consume generated runtime artifact locations.
- **Keep generated runtime artifacts together** - This slice should reinforce the one-root model for live config, managed DB, and logs.
- **Preserve app-local portability** - The resulting layout must still support moving the app folder with its generated data.

### Negative Constraints (MUST NOT)
- **Do not reopen the storage-location decision** - The product decision is already made.
- **Do not expand into general export/import/backup tooling** - This story is about default storage and reference alignment only.
- **Do not turn every documentation file into mandatory scope** - Only direct references that would otherwise teach the wrong default path model need to change.
- **Do not bypass shared helpers with fresh string literals** - New path behavior should remain centralized.

### Soft Preferences (SHOULD)
- Prefer a small number of crisp documentation/comment updates over broad wording churn.
- Keep `.gitignore` and related examples aligned if the runtime `data/` directory is intentionally app-owned and writable.

---

## Implementation Notes

### Suggested Approach
1. Start from the shared root/path helpers introduced by `015.001`.
2. Trace the direct runtime-artifact surfaces that describe or consume those paths: startup/bootstrap, logging defaults, reset/cleanup references, and the most direct project-facing examples/comments.
3. Update those consumers to rely on the shared helpers/contracts.
4. Add or adjust focused tests so future changes cannot silently drift back to mixed path ownership.

### Technical Considerations
- `config.ini.template`, `.gitignore`, and nearby startup/config comments are likely direct references worth aligning if they still teach the old storage model.
- Reset/cleanup and startup code should consume the shared path contract rather than copy knowledge of where live config/database/log artifacts reside.
- The story should stay small; if a far-away document merely mentions config generically without teaching a wrong default path, it does not need to be dragged into scope.

### Affected Components
- Direct-dependent runtime path consumers under `src/`
- Focused regression tests under `tests/unit/` and/or `tests/integration/`
- Direct project-facing references such as `config.ini.template`, `.gitignore`, or nearby storage-layout notes if they currently teach the wrong default

## Context & Background

The user explicitly chose an app-local `data/` folder so all EventLog-generated state stays close to the application and does not fragment into one runtime-data location per Windows user. This story is the small consistency pass that makes the chosen layout true not just in one helper, but in the direct places that would otherwise keep teaching the old model.
