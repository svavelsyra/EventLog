## Purpose (WHY) ⭐ MOST IMPORTANT

Once `005.001` has made `last_operator` a bootstrap-side `config.ini` concern, the next unanswered question is not persistence itself but how the chosen operator becomes the application's active operator context after startup. Communication, event, and personnel workflows will all eventually need one approved source for "who is currently operating the terminal" rather than each feature inventing its own lookup path.

This story exists to establish that later app-side operator-context seam after the startup dialog has already been prefilling from bootstrap config. Without it, later work risks:
- feature workflows each inventing their own source for the active operator
- confusion about whether startup-prefill state and runtime operator context are the same thing
- later in-app operator-change work having no approved app-owned seam to attach to
- reviewers having to re-open operator ownership questions inside unrelated Communication/Event/Personnel stories

### User Value
- Later feature workflows can rely on one approved active-operator source instead of hidden local state.
- Reviewers can validate operator-context ownership separately from both bootstrap config persistence and later in-app operator-switching UX.
- The project gains a cleaner place to attach future operator-display or change flows without redefining the startup contract.

---

## Goal (WHAT)

Define the app-owned active-operator context seam that begins after startup dialog prefill/selection, while explicitly leaving in-app operator-change UX and richer shell display behavior for later work.

### Acceptance Criteria
- [ ] The story defines one approved app-owned seam for the active operator selected or confirmed at startup so later workflows do not need to read `config.ini` directly.
- [ ] The startup-success handoff into the main application makes clear how the prefilling/selection result becomes the current active operator for the running session.
- [ ] The story explicitly leaves in-app operator-change UX, richer status-bar presentation, and broader profile-management behavior outside its current boundary.
- [ ] Focused unit/integration tests are required for startup-to-app active-operator handoff behavior and missing-operator fallback at the app-context seam.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Build on `005.001` instead of replacing it** - Bootstrap persistence/prefill of `last_operator` remains the earlier story's responsibility.
- **Treat operator context as non-secret convenience data** - The active operator is not authentication or access control.
- **Create one app-owned runtime source** - Later features should read the current operator from one approved seam rather than from config files or view-local state.
- **Support missing-operator startup cleanly** - Lack of a remembered or confirmed operator must remain a supported state until later stories choose otherwise.
- **Keep the scope thin** - This story should establish the runtime seam, not expand into broad operator-profile management.

### Negative Constraints (MUST NOT)
- **Do not absorb bootstrap `config.ini` persistence rules** - Those belong to `005.001`.
- **Do not absorb per-tab column/filter/display persistence** - That belongs to `005.004`.
- **Do not turn operator context into security policy** - Passwords, identity verification, and protected role logic remain outside this epic.
- **Do not require in-app operator-change UI yet** - That remains later work beyond this story.
- **Do not require broad status-bar logging/polish work** - Richer status/log behavior remains later work beyond this story.

### Soft Preferences (SHOULD)
- Prefer an app-owned seam that future workflows can consume without depending on Tk widgets directly.
- Keep operator-context handoff explicit so later in-app change flows can attach cleanly.
- Favor a design that can survive reset/default fallback without special-case branching in feature presenters.

---

## Implementation Notes

### Suggested Approach
1. Reuse the startup-side operator selection/prefill result from `005.001`.
2. Define the app-owned handoff/seam that stores the active operator for the running session.
3. Keep later shell display or change actions out of scope while making the seam easy to consume.
4. Add focused tests for startup-to-app handoff and missing-operator fallback behavior.

### Affected Components
- App/shell startup-success integration under `src/`
- Any minimal app-owned active-operator context seam under `src/`
- Focused unit/integration tests for active-operator handoff behavior
