## Purpose (WHY) ⭐ MOST IMPORTANT

EventLog is not meant to be a hardcoded demo. Its communication workflows, event categorization, validation rules, and dynamic UI behavior depend on operational configuration that can vary by unit, mission, and deployment. Without a proper configuration layer, the application would either hardcode assumptions that quickly go stale or scatter config rules across presenters, validators, and repository code in ways that are difficult to review and change safely.

This epic exists to give the application a stable runtime source for operational metadata after the secure database foundation is in place. The purpose is to ensure that EventLog can:
- load communication systems, supported methods, channel designations, and capability definitions from the database
- provide a consistent source of valid event categories and priorities
- let validation and UI logic depend on configuration objects instead of hardcoded values
- preserve historical entry accuracy even when current configuration changes later
- support unit-specific default seed data without turning the application into a one-off custom build
- allow approved non-secret operational configuration to be exported and re-imported later when the database has been reset or re-created

The deeper reason is maintainability under real operational change. Nets move, channels get renamed, available systems differ by unit, and event classifications evolve. If configuration is not treated as a first-class capability, the rest of the application becomes brittle, inconsistent, and harder to trust.

### Success Criteria
- A clear database-backed source exists for the operational configuration needed by communication and event workflows
- Runtime loading contracts are clear enough that presenters and validators can consume configuration without hardcoded fallback logic
- Communication system definitions, channel designations, and capability metadata can change without corrupting historical log entries
- Event category and priority rules come from a clearly defined source rather than being split ambiguously across documents or code
- The epic leaves enough structure for a later story to define how approved non-secret configuration can be exported and restored after `Nollställ`
- Epic 003 remains focused on configuration data and runtime loading, without absorbing structured report delivery or general settings/preferences work

## Goal (WHAT)

Define and deliver the configuration-data layer that sits on top of the database foundation and secure bootstrap.

This epic covers the end-to-end contract for:
1. storing operational configuration in database tables
2. seeding initial unit-specific configuration data for Phase 1 use
3. loading configuration into repository/core-facing runtime objects
4. supporting configuration-backed validation and dynamic UI population
5. establishing how configuration changes coexist with historical entry integrity
6. defining the boundary for later import/export of approved non-secret operational configuration

### Key Deliverables
- [ ] Database-backed configuration contract for communication systems, supported methods, channel designations, and system capabilities
- [ ] Clear source of truth for event categories and valid priorities
- [ ] Seed/default-data strategy for initial operational configuration
- [ ] Repository/adapter access patterns for configuration reads and targeted updates
- [ ] Core/runtime configuration-loading contract, including caching/reload expectations where needed
- [ ] Import/export boundary for approved non-secret operational configuration, so reset databases can be re-seeded from an external config package later
- [ ] Explicit boundary between operational configuration and other concerns such as bootstrap config, structured report templates, and user preferences

## Limitations (CONSTRAINTS & REQUIREMENTS)

**What boundaries must we work within? What must we avoid?**

### Hard Constraints (MUST)
- **Build on the secured database path** - Epic 003 assumes configuration is loaded after the database foundation and secure startup path are available
- **Keep configuration as data, not code** - Communication systems, channels, capabilities, categories, and priority definitions must come from a maintainable runtime source instead of scattered hardcoded constants
- **Preserve historical log integrity** - Existing entries must remain readable and meaningful even if current configuration names, defaults, or active flags change later
- **Support the communication workflow model already designed** - System-first selection, supported methods, channel designations, and capability metadata must align with the existing communication design documents
- **Support Phase 1 seed data** - The system must be able to start with practical unit-specific defaults without requiring a full configuration-management UI first
- **Allow deliberate runtime loading** - Core validators and presenters need a defined configuration-loading path and should not each invent their own lookup logic
- **Keep the database as the source of truth** - Import/export is a portability and recovery mechanism for approved non-secret configuration, not a replacement for database-backed runtime configuration
- **Treat export/import scope as explicit and allowlisted** - A later story must define exactly which non-secret configuration is portable rather than assuming all configuration-like data should be exported
- **Keep testing with delivery stories** - Later 003.x stories should include their own tests rather than creating a broad standalone testing story

### Negative Constraints (MUST NOT)
- **Do not include secure bootstrap or encryption behavior** - That belongs to Epic 002
- **Do not absorb structured report template delivery** - Structured report forms and their template model remain Epic 004 territory for now
- **Do not absorb general user-preference storage or settings UI** - `user_preferences` and broader application settings remain outside this epic’s scope
- **Do not require a full admin/configuration screen in Phase 1** - This epic is about configuration persistence and runtime loading first, not a broad management UI
- **Do not turn this into full database backup/restore** - The planned import/export support is for approved non-secret operational configuration only
- **Do not export secrets or bootstrap/security material** - Encryption keys, key-file references, secure-bootstrap settings, passwords, and other sensitive data must remain outside this feature
- **Do not replace historical entry values with live foreign-key dependencies** - Old entries should not become incorrect or unreadable when configuration changes
- **Do not over-engineer company-level or Phase 3 multi-instance support now** - Keep the first pass aligned with the documented platoon-level Phase 1 approach while preserving room for later expansion
- **Do not leave event priority sourcing ambiguous** - Epic 003 must resolve the runtime source for valid priorities clearly enough for later stories to implement consistently

### Soft Constraints (SHOULD)
- Prefer small follow-up stories that separate schema/seed work, repository access, runtime config objects, and event-specific configuration concerns
- Keep configuration terminology consistent across database, core, and GUI layers
- Favor explicit reload/caching behavior over implicit global state
- Support on-the-fly operational updates later without requiring a complete redesign of the persistence model

## Context & Background

### Architectural Context
Epic 001 established the core persistence model. Epic 002 adds protected database access and secure bootstrap. Epic 003 is the next layer: the application now needs the operational metadata that tells the rest of the system what valid systems, methods, channels, and categories actually exist.

Several design documents already depend on this idea:
- the communication-entry UI is configuration-driven by selected system, method, channels, and capability definitions
- validators are expected to receive configuration objects rather than rely on hardcoded lists
- event workflows assume configurable categories and a defined set of valid priorities
- Phase 1 intentionally uses seeded operational defaults before any broad configuration-management UI exists

### Reset and Recovery Context
`Nollställ` is intended to destroy the current database and related local artifacts quickly. That behavior is still correct, but it means database-resident operational configuration is lost along with the rest of the database state.

For that reason, Epic 003 now also needs to leave room for a later portability feature that can export and re-import **approved non-secret configuration only**. This is not meant to weaken the database-first model. It is meant to allow a clean database to be repopulated with safe operational configuration after reset, redeployment, or fresh setup.

### Structural Decision for This Epic
This epic is intentionally scoped around **operational configuration that directly supports the base communication and event workflows**.

That means Epic 003 focuses on:
- communication systems
- channel designations
- system capabilities metadata
- event categories
- valid event priorities
- runtime config loading/caching contracts

It does **not** currently pull in:
- structured report templates, even though they are also configuration-shaped data
- user preferences such as `last_operator`, column layouts, or display preferences
- a full administrative UI for managing all configuration tables
- full backup/restore of operational logs or other application data
- secret/bootstrap/security material that should never be moved around as portable configuration

### Why This Is Its Own Epic
If operational configuration is deferred too long, later feature work either hardcodes placeholder values or builds ad hoc configuration logic into GUI and core layers. That would create the exact structural drift this planning phase is meant to catch early.

This epic exists to establish the shared configuration layer before feature epics depend on inconsistent assumptions.

- Repository/adapter configuration access: ~120-220 lines
- Runtime configuration objects/loading: ~120-220 lines
- Communication config update support: ~80-180 lines
- Event category/priority config cleanup: ~80-160 lines

**Total estimated**: ~600-1,000 lines across several small stories, depending on how much configuration-object structure is created in the core layer.

### Key Risks
- **Boundary drift** - report templates, user preferences, and admin UI can easily bloat the epic if not kept separate
- **Portability scope drift** - import/export could easily expand into full backup/restore or secret handling if the later story does not use a strict allowlist
- **Source-of-truth confusion** - categories and especially priorities must not end up half hardcoded and half database-driven
- **Historical integrity mistakes** - over-normalizing live configuration into entry records could make old logs misleading after config changes
- **Config-loading sprawl** - presenters, validators, and repositories may each start doing their own lookups if the loading contract is weak
- **Seed-data assumptions** - unit-specific defaults must be practical without becoming undocumented one-off behavior

### Success Metrics
- Epic 003 is clear enough to split into small stories without mixing in unrelated concerns
- Later feature stories can depend on configuration objects and repository-backed metadata instead of hardcoded dropdown values
- The event priority/category source and communication system metadata model are structurally clear before implementation starts
- A later story can define an explicit allowlist and flow for exporting/importing non-secret operational configuration after reset without reopening Epic 002 security concerns
