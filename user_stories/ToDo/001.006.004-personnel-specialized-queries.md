## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

Personnel tracking has three common, critical query patterns that occur frequently:

1. **View active personnel** - Default view shows only current deployments (active=1), not historical
2. **View person's history** - See all status changes for one person/unit over time
3. **Find overdue check-ins** - Safety-critical: which personnel have missed expected check-in times

These queries are used so frequently that they deserve dedicated methods rather than requiring operators to construct filter dicts every time. This improves:
- **Usability** - Simple method calls instead of filter dict construction
- **Performance** - Optimized queries for common cases
- **Safety** - Overdue alarms are critical for personnel safety

Without these convenience methods, every GUI view would need to know filter dict structure and build queries manually.

### User Value
- Quick access to active personnel (most common view)
- Easy historical tracking for each person/unit
- Automatic detection of overdue check-ins (safety)
- Simplified GUI implementation

## Goal (WHAT)

**What functionality are we delivering?**

Implement three convenience methods for common personnel queries in SQLiteEventLogRepository.

### Acceptance Criteria

- [ ] **Get Active Personnel Entries**:
  - Can retrieve only currently active personnel entries
  - Returns results ordered for current operational awareness use
  - Returns an empty list when no active personnel entries exist
- [ ] **Get Personnel History**:
  - Can retrieve the full recorded history for a specific person or unit
  - Includes both active and inactive entries
  - Returns results in recency-first order and an empty list when nothing matches
- [ ] **Get Overdue Alarms**:
  - Can retrieve personnel entries that represent overdue, enabled, and not-yet-triggered check-in alarms
  - Returns results ordered so the most urgent overdue cases appear first
  - Returns an empty list when no overdue alarms exist
- [ ] **Unit Tests Pass**:
  - Active-entry retrieval behavior is verified, including ordering and empty-result handling
  - Personnel-history behavior is verified, including mixed active/inactive history
  - Overdue-alarm behavior is verified, including eligibility rules and urgency ordering

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Implement in SQLiteEventLogRepository** - Replace NotImplementedError stubs
- **Use parameterized queries** - Prevent SQL injection
- **Use _row_to_personnel_entry helper** - Reuse from 001.006.001
- **Return empty list** - Not None, if no matches
- **Preserve defined sort behavior** - Active entries, history, and overdue alarms must return in the order required by their workflows

### Negative Constraints (MUST NOT)
- **Don't implement alarm notification system** - Just return overdue entries, notification is separate concern
- **Don't implement status update workflow** - GUI manages creating/updating entries
- **Don't duplicate general list/filter behavior here** - Those capabilities belong to `001.006.002`

### Soft Preferences (SHOULD)
- Keep these specialized query methods clearly distinct from the general filtered-read path
- Make each query's operational purpose obvious in tests and documentation
- Keep the query behavior aligned with the repository patterns already established in the other personnel stories

## Implementation Notes

### What's Needed

1. **Active Personnel Query Capability**
   - Provide a focused query path for current active personnel views without requiring callers to construct generic filters manually
   - Return results in the ordering most useful for current operational awareness

2. **Personnel History Query Capability**
   - Provide a dedicated query path for retrieving the full recorded history of one person or unit
   - Include both active and inactive records while preserving recency-first ordering

3. **Overdue Alarm Query Capability**
   - Provide a dedicated query path for overdue check-in review based on alarm-enabled, overdue, and not-yet-triggered conditions
   - Return results in urgency order so the most overdue cases are surfaced first

4. **Testing Capability**
   - Add focused tests for the three specialized query paths, including empty-result cases and their required ordering rules

### Technical Considerations
- These methods represent the highest-frequency personnel-specific query workflows and should remain simpler to call than the generic filtered-read path
- Overdue-alarm behavior depends on consistent repository time storage and a clear definition of which alarm states qualify as overdue
- Active-personnel and history queries should remain easy to distinguish from each other in both implementation and tests
- This story adds specialized convenience methods on top of the broader read/list behavior established earlier

## Context & Background

These convenience methods exist because personnel tracking has a few high-frequency workflows that are more clearly expressed as dedicated repository methods than as repeated generic filter construction.

This story covers:
- Active-personnel retrieval for current operational views
- Person/unit history retrieval for review and audit workflows
- Overdue-alarm retrieval for safety-focused monitoring

The trade-off is a slightly broader repository surface area in exchange for clearer, more intention-revealing calling code in the GUI and related workflows.

## Testing Strategy

### Key Test Scenarios

**get_active_personnel_entries tests**:
- [ ] Returns only entries where active=1
- [ ] Does not return entries where active=0
- [ ] Sorts by last_contact_time DESC
- [ ] Returns empty list if no active personnel

**get_personnel_history tests**:
- [ ] Returns all entries for specific "who" value
- [ ] Includes both active=1 and active=0 entries
- [ ] Sorts by logged_time DESC (most recent first)
- [ ] Returns empty list if no entries for that person
- [ ] Different "who" values return different results

**get_overdue_alarms tests**:
- [ ] Returns only entries where alarm_enabled=1
- [ ] Returns only entries where expected_checkin_time < now
- [ ] Returns only entries where alarm_triggered=0
- [ ] Does not return entries where expected time in future
- [ ] Does not return entries where alarm_triggered=1
- [ ] Sorts by expected_checkin_time ASC (most overdue first)
- [ ] Returns empty list if no overdue alarms
