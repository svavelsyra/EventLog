## Purpose (WHY) ⭐ MOST IMPORTANT

EventLog also needs to answer one of the most safety-critical operational questions: who is out, what is their current status, where are they, and when were they last heard from? If personnel tracking exists only as repository methods or raw database records without a usable operator workflow, the application still fails one of its central field purposes.

This epic exists to deliver the Personnel tab as a practical status-tracking workflow in the main GUI. The purpose is to ensure that EventLog can:
- let an operator create and review personnel status entries rapidly in the main tabbed UI
- show the current operational picture by default while still preserving full historical tracking behind it
- support the documented status-change pattern where new entries supersede old ones instead of overwriting operational history
- capture optional reminder-related fields as part of personnel logging without requiring the full reminder-notification system to live inside this epic
- connect view, presenter, validation, repository access, and preference-backed column/filter behavior into one coherent personnel workflow

The deeper reason is safety and accountability. Operators must be able to see who is deployed, whether contact is overdue, and how status has changed over time without manually reconstructing history or guessing which record is current.

### Success Criteria
- A complete Personnel tab workflow exists for normal Phase 1 use, from entering personnel status through seeing it in a current-status view
- The Personnel tab supports both current-state review and historical-tracking workflows without erasing or hiding operational history incorrectly
- The status-update flow creates the next personnel state cleanly instead of treating history as ordinary inline edits
- Reminder-related data can be captured and displayed as part of the personnel workflow, while the broader alarm/reminder notification behavior remains separately bounded
- The Personnel tab respects the MVP boundary, with view, presenter, validation, and repository responsibilities staying structurally clear
- Epic 008 stays focused on personnel-tab delivery, without absorbing a full notification engine, generic attachments, or unrelated communication/event work

## Goal (WHAT)

Define and deliver the Personnel tab as the third end-to-end GUI feature workflow in the EventLog application.

This epic covers the end-to-end contract for:
1. rendering the personnel entry form and personnel log table in the main application UI
2. creating, validating, saving, listing, filtering, and editing `PersonnelEntry` records through MVP interactions
3. supporting the current-status default view plus deliberate access to historical entries and person/unit history
4. supporting the status-update pattern that creates new active records while superseding prior ones
5. capturing and displaying reminder-related personnel fields while leaving broader alert/notification orchestration to a later epic
6. integrating preference-backed column configuration and related tab-level UI state where that contract already exists

- [ ] Personnel presenter contract for form interaction, validation handling, save/edit behavior, active/history queries, and table refresh behavior
- [ ] Personnel entry create/read/update/delete workflow integrated with repository methods from Epic 001 and secure startup assumptions from Epic 002

- [ ] Reminder-field capture and display contract for `alarm_enabled`, `expected_checkin_time`, and related personnel indicators without defining the whole notification engine here
- [ ] Column-configuration and tab-state integration that consumes the preference contract defined by Epic 005 without redefining it
- [ ] Explicit boundary between personnel-tab delivery and future concerns such as global alarm/reminder orchestration, attachments, and broader roster-management tooling

## Limitations (CONSTRAINTS & REQUIREMENTS)

**What boundaries must we work within? What must we avoid?**

### Hard Constraints (MUST)
- **Use MVP responsibilities cleanly** - The Personnel tab must follow the documented Tkinter View + Presenter pattern, keeping widget rendering in the view and coordination/validation flow in the presenter
- **Build on existing foundations instead of bypassing them** - Personnel data must flow through the repository/adapter path from Epic 001 and use the secured/configured runtime assumptions established by later foundational epics
- **Support the documented top/bottom personnel layout** - The tab must preserve the fast-entry form plus current-status/history review pattern already defined in the GUI design
- **Support current-state and history workflows** - The tab must make active entries easy to review by default while still allowing deliberate access to all entries and person/unit history
- **Support the status-change pattern cleanly** - Personnel status progression should create new entries and supersede old ones instead of quietly overwriting the operational trail
- **Capture reminder-related data as part of the form contract** - If reminder fields are present in the design, the personnel workflow must have a defined way to enter, persist, and display them even if later alerting behavior is handled elsewhere
- **Support practical table usage on small screens** - Sorting, truncation, scrollbars, and column visibility behavior must align with the netbook-sized UI constraints documented for the project
- **Consume preference-backed UI state where applicable** - Column configuration and related tab UI state should use the Epic 005 preference contract rather than inventing personnel-tab-specific persistence rules
- **Keep testing with delivery stories** - Later `008.x` stories should include their own presenter/gui/repository tests instead of creating a broad standalone testing story

### Negative Constraints (MUST NOT)
- **Do not absorb the global alarm/reminder workflow** - Periodic overdue detection, cross-tab attention behavior, acknowledgement orchestration, and broader reminder surfacing remain a later dedicated epic
- **Do not absorb generic file attachments or document handling** - Personnel attachments remain a later dedicated epic rather than part of the first personnel-tab workflow boundary
- **Do not absorb communication or event workflows** - Those remain separate feature epics with their own delivery boundaries
- **Do not turn the personnel tab into a full roster-management/admin system** - This epic is about operational tracking workflows, not broad personnel-master-data administration
- **Do not bypass repository/convenience-query intent with view-side filtering logic** - Views must not rebuild business/query rules that already belong in presenter or repository interactions
- **Do not treat inline edits as the only update model** - The personnel workflow must not collapse the separate historical status pattern into ordinary record editing everywhere
- **Do not hardcode demo/example values as runtime truth** - Demo status labels and example names are visual references only and must not become the implementation contract

### Soft Constraints (SHOULD)
- Prefer small follow-up stories that separate tab/view scaffolding, presenter save/load behavior, status-update/history handling, and reminder-field capture/display
- Keep Swedish UI labels and operator-facing terminology aligned with the design documents and demo reference
- Favor explicit current-state versus history controls so operators do not confuse the active picture with the historical record
- Keep reminder-field handling simple and reviewable before adding broader alerting or escalation behavior

## Context & Background

### Architectural Context
Epic 001 created the personnel-entry persistence path, including specialized convenience queries for active personnel, history, and overdue alarms. Epic 005 defined the preference boundary needed for column configuration and other per-tab UI persistence.

That means Epic 008 is the feature epic that turns the personnel data model into a usable operator workflow:
- repository-backed `PersonnelEntry` CRUD
- specialized current-state, history, and overdue-query support
- presenter/view coordination
- status-update and supersedes behavior in the GUI layer
- preference-backed column behavior
- practical personnel tracking on small screens

### Why Personnel Comes After Communication and Events
Personnel tracking is a core feature, but it carries extra workflow complexity:
- it has both current-state and historical views
- it relies on the status-update pattern instead of only simple entry editing
- it overlaps with reminder/alarm behavior that can easily expand beyond the basic tab scope

For that reason, it makes sense as a later tab-delivery epic after the Communication and Event tabs have already established reusable MVP and table/view patterns.

### Structural Boundary for Reminder Behavior
The personnel design includes reminder-related fields and overdue concepts. That is important operationally, but it creates a boundary question: how much reminder behavior belongs in the tab, and how much belongs in a separate alarm/reminder epic?

Epic 008 should keep the personnel-tab side of that contract clear:
- the tab can capture reminder fields and show reminder-related state
- the tab can display data that indicates reminder status
- the broader alert/notification engine, acknowledgement orchestration, or cross-tab reminder surfacing should remain separate so the personnel workflow does not absorb a whole second subsystem

### What This Epic Excludes
This epic is intentionally about **personnel-tab delivery**, not about:
- communication logging workflows
- event logging workflows
- full alarm/reminder orchestration across the application
- generic attachment/document workflows
- full personnel administration or roster-management features
- cross-application polish unrelated to the personnel tab's core workflow

### Key Risks
- **Boundary drift** - personnel delivery could easily absorb reminder-engine behavior, roster administration, or generic attachment scope if not held tightly
- **History-model confusion** - later implementation could blur the difference between editing a record and recording a new status state
- **Presenter bloat** - status-update, history filtering, reminder-field handling, and table formatting could collect into one class if story splitting is weak
- **Current-vs-history UX ambiguity** - operators may misunderstand which view shows the current picture unless the controls and defaults are explicit
- **Reminder-boundary leakage** - in-tab reminder display could accidentally become ad hoc global alarm orchestration if later stories do not preserve the boundary deliberately

### Success Metrics
- Epic 008 is clear enough to split into small stories without mixing in unrelated event, communication, attachment, or full reminder-engine scope
- A later implementation pass can deliver a genuinely usable personnel workflow rather than only a static tab mockup
- The personnel workflow preserves the current-state/history model instead of collapsing it into simplistic record editing
- The project gains a stable personnel-tab base for later reminder/alarm work without re-arguing the tab architecture
