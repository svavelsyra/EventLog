## Purpose (WHY) ⭐ MOST IMPORTANT

The original `006.003` scope turned out to contain two different concerns that are easier to review and implement separately:

1. the **operator-side** Communication form interaction that consumes runtime configuration and defines what the operator sees at runtime
2. the **consumer-driven configuration-source workflow** needed to support JSON import/export, starter-template generation, a seeded standard baseline, and later customization without redesigning the operator-side seam

Keeping those concerns together would make the story harder to review because it would mix operator workflow decisions with configuration-source and import/export cleanup decisions. This parent story now serves as the split reference for the smaller child stories that carry those separate concerns.

### User Value
- Reviewers get smaller child stories with cleaner boundaries.
- The operator-facing Communication form work can proceed without forcing import/export and source-cleanup work into the same patch.
- The consumer-driven configuration follow-up stays explicitly linked to the Communication workflow rather than becoming a vague cleanup task.

---

## Goal (WHAT)

Split the original `006.003` scope into two smaller child stories with explicit boundaries.

### Acceptance Criteria
- [ ] `006.003.001` defines the operator-facing configuration-driven Communication form interaction, including visible path depth, qualifier behavior, and runtime-seam consumption boundaries.
- [ ] `006.003.002` defines the configuration-source workflow needed to support Communication configuration JSON import/export, starter-template generation, seeded standard defaults, and UI import/apply behavior for customization.
- [ ] The split makes it explicit which concerns stay operator-facing and which concerns stay source/import-facing, without reopening full admin/config-management scope.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep the split small and purposeful** - Each child story should remain focused enough for one coherent review.
- **Preserve the user-confirmed boundary** - Operator-side runtime consumption and source/import cleanup must stay distinguishable after the split.
- **Keep the split linked to Communication-tab needs** - The follow-up should be framed as consumer-driven configuration-source cleanup discovered by the Communication workflow, not as a vague reopening of all Epic `003` scope.

### Negative Constraints (MUST NOT)
- **Do not treat the split as story completion** - This parent story should not be marked done or moved as part of this planning update alone.
- **Do not collapse the child stories back into one mixed implementation story** - The point of the split is cleaner review boundaries.

### Soft Preferences (SHOULD)
- Prefer child stories that keep the operator-side story implementable even if the source/import cleanup follows immediately after rather than in the same patch.
- Keep direct dependent `006.x` stories pointed at the child story that actually defines the operator-facing form contract.

## Context & Background

The split happened because the Communication tab is the first real consumer of the runtime communication-config seam, and that consumer work exposed two separate follow-up needs:
- operator-side config consumption that defines the Communication form flow
- source/import workflow so editable JSON, explicit import into the database, and a seeded standard baseline can support later setup/admin work cleanly

Keeping both concerns explicitly linked under `006.003` preserves that they were discovered while implementing the Communication workflow, even though they touch different layers.
