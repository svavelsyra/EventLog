## Purpose (WHY) ⭐ MOST IMPORTANT

Even after the Communication tab can save entries and react to runtime configuration, the workflow is still not operationally usable if the lower log view remains a crude dump of rows with no practical review behavior. Operators must be able to scan recent communications quickly, narrow the list with simple filters, and understand which row is currently selected without opening a detail or edit flow every time. If those table behaviors are left undefined, later correction and preference stories will have to build on unstable display rules and ad hoc selection behavior.

This story exists to establish the first practical log-table contract for the Communication tab. It defines how saved `CommunicationEntry` rows are formatted for table display, how a small set of simple filters narrows the visible results, how ordering/sorting behaves in the early slice, and how row selection is tracked without yet turning selection into a full edit/detail action. That creates a stable review surface before the edit/detail workflow and preference-backed column work arrive.

Without this story, later work risks:
- the Communication log view staying technically populated but operationally hard to scan
- detail/edit stories depending on unstable row formatting or ambiguous selected-row ownership
- filter logic emerging inconsistently between view, presenter, and repository layers
- column-preference work from later stories arriving before the table's base behavior is even settled

### User Value
- Operators can review recent Communication entries in a readable and predictable table.
- The Communication tab becomes more useful for daily log scanning before edit/detail features arrive.
- Later stories gain a stable table contract for selection, detail loading, and preference-backed column behavior.

---

## Goal (WHAT)

Add the first practical Communication log-table behavior on top of the existing view, presenter, and configuration-driven form work.

For this story, that means:
- formatting Communication entries into readable table rows for the lower log area
- supporting a small approved set of simple AND-based filters for narrowing results, including `Från`, `Till`, and a time-from/time-to range
- defining default ordering and lightweight user sorting behavior for the table, with the same sort interaction available on every visible column
- tracking passive row selection in a stable way that later stories can reuse for detail/edit flows
- allowing keyboard-driven row movement in the table so selection can be advanced without mouse-only interaction

This story is table-focused only. It does not yet deliver double-click detail views, edit/delete actions, edited-state presentation, advanced query behavior, or preference-backed column customization.

### Acceptance Criteria
- [ ] The Communication presenter/view renders saved `CommunicationEntry` rows into a readable table contract for the log area, including basic formatting such as display-ready time values, readable Communication labels, and truncated message content where appropriate for the table.
- [ ] The Communication log supports simple AND-based filtering for the approved Phase 1 filter set, including `Från`, `Till`, `Tid från`, and `Tid till`, while existing filter placeholders already present in the UI may remain in place without being treated as a reason to widen the guaranteed behavior contract for this story.
- [ ] `Tid från` and `Tid till` use the same exact typed datetime input contract already accepted by the Communication create-entry flow, specifically `YYYY-MM-DD HH:MM` and `YY-MM-DD HH:MM`, so operators can apply tighter time-range filtering without relying only on ambiguous shorthand across long-running logs.
- [ ] The table defaults to recent-first ordering by `Tid` and supports lightweight user sorting on every visible column through a consistent header-click toggle behavior, without requiring broader repository or preference redesign.
- [ ] The presenter/view maintains passive row-selection behavior with a stable selected-entry identity suitable for later detail/edit stories, preserves that selection across reload/filter/sort only when the same entry remains visible, clears it when the selected entry is no longer visible, and supports keyboard-driven row movement such as arrow-key navigation without turning selection into open/edit behavior, while focused tests cover formatting, filter, sorting, and selection behavior without absorbing edit/detail flows.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep this story table-only** - The work must focus on log-table display/review behavior rather than broadening into edit, detail, or config-management concerns.
- **Use simple AND-based filtering** - Phase 1 filtering should stay aligned with the documented simple filter model rather than expanding into advanced Boolean query logic.
- **Use a bounded time-range filter for time** - Time filtering in this story should use a simple from/to range rather than one-sided or advanced temporal query behavior.
- **Reuse the approved exact datetime typing contract for time-range filters** - `Tid från` and `Tid till` should accept the same exact typed datetime formats already supported by Communication entry creation (`YYYY-MM-DD HH:MM` and `YY-MM-DD HH:MM`) instead of inventing a separate filter-only format.
- **Preserve MVP ownership** - The presenter owns row formatting, filter/sort coordination, and selection-state decisions; the view owns widget rendering and user interaction readback.
- **Keep basic keyboard selection local to the table** - Keyboard row movement should extend the table-selection contract without prematurely turning table interaction into detail-opening or broader shortcut policy.
- **Respect the existing repository path** - Filtering/loading should build on the approved Communication-entry access path instead of inventing view-side data caches as the source of truth.
- **Make visible columns consistently sortable in this slice** - The early sort contract should stay simple and predictable by allowing the same sort interaction on every visible column instead of introducing column-specific exceptions.
- **Preserve existing filter placeholders unless a later story deliberately redesigns them** - If the current Communication UI already exposes placeholder filter controls beyond the guaranteed Phase 1 set, this story should not remove them merely to enforce a narrower contract.
- **Support practical small-screen behavior** - Formatting and column choices for this slice should keep the log readable on constrained screens, including message truncation where needed.
- **Include focused tests with the story** - Tests should cover representative formatting, filtering, sorting, reload, and selection cases.

### Negative Constraints (MUST NOT)
- **Do not absorb detail/edit/delete workflows** - Opening entries, loading them back into the form, and correction actions belong to `006.005`.
- **Do not absorb edited-state presentation yet** - Visible edited indicators should remain part of the later correction story.
- **Do not absorb preference-backed column customization** - Persisted column visibility/order/width behavior belongs to `006.006`.
- **Do not introduce advanced filtering/query-builder behavior** - This story should remain simple and reviewable.
- **Do not move filter or formatting decisions into the view** - The GUI should not become the owner of table business/display rules.
- **Do not turn row selection into implicit edit behavior** - Selection should remain passive in this slice.
- **Do not absorb detail-opening shortcuts here** - Opening the selected entry belongs with the detail workflow story rather than the table-selection story.
- **Do not redesign the create-entry time parsing contract just to satisfy table filtering** - Reuse the existing exact datetime input shape for filters rather than reopening the entry-time format decision here.

### Soft Preferences (SHOULD)
- Prefer a stable hidden row identity such as entry ID so later stories can reuse selection cleanly.
- Prefer table formatting that is readable and predictable before it is feature-rich.
- Keep the first sorting/filtering slice small enough that later detail and preference work can extend it without reworking the whole table contract.
- Prefer a stricter exact datetime filter path over ambiguous shorthand when operators need to review logs spanning long periods.

## Implementation Notes

### Suggested Approach
1. Extend the Communication presenter so it can map loaded `CommunicationEntry` objects into a simple table-row display contract.
2. Define or reuse a small filter-submission seam from the Communication view for the approved Phase 1 filters, explicitly including `Från`, `Till`, and a `Tid från` / `Tid till` range that reuses the existing exact datetime typing contract.
3. Implement simple default ordering plus lightweight user sorting behavior on every visible column without over-expanding repository responsibilities or introducing column-specific exceptions.
4. Add passive selected-row tracking by stable entry identity so reload/filter/sort behavior can preserve selection when that same entry remains visible and clear it when it does not, including basic keyboard row movement.
5. Add focused tests for formatting, filtering, sort toggling/default order, keyboard selection movement, and selection persistence/clearing.

### Affected Components
- Communication presenter under `src/gui/presenters/`
- Communication view seam under `src/gui/views/`
- Communication-entry repository access only as needed for existing simple filter support
- Focused presenter/view tests for table behavior

## Context & Background

This story comes after the first save/load presenter loop and the first operator-facing configuration-driven form interaction seam. At this point, the Communication tab needs a stable lower-table review contract before later stories add correction workflows or preference-backed column behavior.

The design documents already expect:
- a lower log table in the Communication tab
- readable time/method/message presentation
- simple AND-based filtering in Phase 1
- default latest-first review behavior
- exact typed datetime input already available in the create-entry flow for tighter operator control when needed

This story intentionally stops short of the edit/detail workflow so that selection and table behavior can be reviewed separately before those actions begin to depend on them.

Basic keyboard row movement belongs here because it is another way of driving selection, but explicit open behavior still belongs to the later detail workflow story.
