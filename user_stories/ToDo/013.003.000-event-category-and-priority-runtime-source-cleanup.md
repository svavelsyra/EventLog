# User Story: 013.003 - Event Category and Priority Runtime Source Cleanup

---

## Purpose (WHY) ⭐ MOST IMPORTANT

Epic `013` exists because event categories and priorities are clearly meant to be configuration-backed, but they should not stay mixed into the communication-focused scope of Epic `003`. If their authority remains ambiguous, later event validators, presenters, and tests will drift into mixed authority where some values are database-driven and others remain hardcoded defaults.

This story exists to remove that ambiguity before the Event workflow depends on it. Without it, later work could drift into:
- event priority lists hardcoded in presenters while categories come from the database
- validators and GUI filters disagreeing about which priorities/categories are valid
- default values such as `Normal` existing only as scattered string literals instead of inside one clear runtime contract
- multiple partially-correct "sources" surviving at once because no single runtime authority was explicitly named and enforced
- later epics re-opening the same source-of-truth question repeatedly

### User Value
- Event workflows get one clear authoritative source for valid categories and priorities.
- Validators, dropdowns, and filters can stay consistent.
- Reviewers can verify event-classification behavior without chasing multiple competing sources.

---

## Goal (WHAT)

Finalize and deliver the runtime source-of-truth contract for valid event categories and priorities so later event stories consume one coherent configuration model.

### Acceptance Criteria
- [ ] Epic `013` has one explicit authoritative runtime source for valid event categories and priorities, that source is named clearly in the affected docs/code seams, and any missing storage/support needed for it is added rather than left implied.
- [ ] Category and priority loading flows through the repository/core configuration path established by earlier `013.x` stories, so validators, presenters, and later event filters consume one shared runtime contract instead of parallel fallback lists or presenter-local authority.
- [ ] Default event-priority behavior, including the ordinary default such as `Normal`, plus active/inactive ordering/selection expectations, is documented and implemented through that runtime contract instead of scattered hardcoded fallback lists.
- [ ] Focused tests verify consistent category/priority retrieval, validation against the configured active values, and predictable behavior for inactive or unknown values.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Resolve priority ambiguity explicitly** - This story must end the unclear state where priorities might be partly hardcoded and partly configuration-driven.
- **Use the event-metadata pipeline** - Event metadata must be read through Epic `013` schema/repository/core loading rather than presenter-local lists.
- **Support later event validation and UI filtering** - The result must be practical for both validators and Event-tab dropdown/filter usage.
- **Keep default behavior explicit** - The ordinary default priority should be part of the contract, not an undocumented magic string.
- **Do not leave parallel authorities alive** - If earlier `013.x` stories or current code/docs already hint at a source, this story must converge on one final authority rather than preserving multiple semi-official paths.

### Negative Constraints (MUST NOT)
- **Do not absorb Event-tab GUI delivery** - The Event workflow itself belongs to Epic `007`.
- **Do not turn this into generic configuration-management UI** - This story clarifies the source of truth, not how operators broadly edit it.
- **Do not rely on temporary hardcoded fallbacks as the final design** - If a migration or bridge is needed, it must still converge to one clear authority.
- **Do not absorb report-template category logic** - Structured reports remain Epic `004` territory.
- **Do not reopen upstream schema work without cause** - If `013.001` already established the needed storage source, this story should finalize and consume it rather than creating a second competing structure.

### Soft Preferences (SHOULD)
- Prefer terminology and ordering that matches the project design docs and likely operator expectations.
- Keep room for inactive values and future category/priority expansion without changing the validation contract shape.
- Make it easy for later tests and presenters to ask for the active display list and the default priority through the same config object.

---

## Implementation Notes

### Suggested Approach
1. Confirm or add the storage required for event priorities if not already covered by `013.001`.
2. Wire category and priority reads through the repository and core config-loading seams.
3. Remove or avoid presenter/validator-local hardcoded lists.
4. Make the chosen authority explicit in the relevant docs/contracts so later stories do not re-open the question.
5. Add focused tests around active values, defaults, and invalid selections.

### Technical Considerations
- This story is intentionally narrower than the whole Event workflow: its job is to make the metadata source stable before Epic `007` consumes it.
- If implementation reveals category and priority cleanup are still too large together, this story is a valid candidate for later split into `013.003.xxx` children.
- Historical event entries should keep their stored category/priority values even if live configuration later changes.
- Any later event-metadata portability work is cleaner only after this story settles which event-metadata source is actually authoritative enough to export/import.

### Affected Components
- Operational-config schema/repository/core loading where event metadata lives
- Event validation support in the core layer
- Focused tests for category/priority source-of-truth behavior

---

## Context & Background

Communication configuration now lives in Epic `003`; this epic exists because event metadata is a different problem. Categories are table-shaped, priorities are referenced often, but the final authoritative runtime source still needs to be made singular and explicit.

This story closes that gap before Epic `007` turns event categories and priorities into user-facing dropdowns and filters.

---

## Testing Strategy

### Key Test Scenarios
- [ ] Category and priority lists come from one explicit operational-config source.
- [ ] Default event priority is obtainable through the shared runtime contract.
- [ ] Validators reject values outside the configured active set while tolerating historical stored values where later read/display rules require it.
- [ ] Inactive or removed category/priority rows do not force later callers to guess fallback behavior.

### Integration Points
- `013.002` provides the repository/runtime-loading path this story should finalize for event metadata.
- `007.003` will later consume these category/priority objects in the Event form and filters.
- `001.005` event persistence already stores `priority` and `category` as entry values, so this story must preserve that historical-storage model.
