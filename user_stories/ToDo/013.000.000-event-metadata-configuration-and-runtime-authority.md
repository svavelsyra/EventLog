## Purpose (WHY) ⭐ MOST IMPORTANT

Communication configuration turned out to be the structurally complex part of operational configuration, while event categories and priorities are a different, flatter problem. Keeping both domains inside one epic risks making communication work feel broader than it is and lets event-metadata questions block communication implementation unnecessarily.

This epic exists to give the application one clear runtime authority for event metadata after the communication-focused split of Epic `003`. The purpose is to ensure that EventLog can:
- define valid event categories and priorities through one shared operational runtime source
- establish explicit default-priority and active/inactive behavior without hardcoded presenter-local fallback lists
- let validators, presenters, and later filters consume the same event metadata contract
- preserve historical event-entry values even when live metadata changes later
- stay separate from communication-specific configuration so both domains remain reviewable and implementable in smaller slices

The deeper reason is scope cleanliness. Event metadata still matters, but it is not the same modeling problem as communication systems, recursive options, and qualifiers. Splitting them keeps both sides smaller and structurally cleaner.

### Success Criteria
- A clear runtime source exists for valid event categories and priorities
- Event category/priority rules come from one defined authority rather than mixed local fallbacks
- Event metadata can be loaded and consumed without depending on communication-specific structures
- Historical event entries remain readable even if current category/priority configuration changes later
- Epic `013` stays focused on event metadata authority rather than broad event-tab delivery or generic admin UI

---

## Goal (WHAT)

Define and deliver the configuration-data and runtime-authority contract for event metadata.

This epic covers the end-to-end contract for:
1. storing or otherwise defining event category/priority metadata through one approved runtime source
2. seeding initial default event metadata for Phase 1 use
3. loading event metadata into repository/core-facing runtime objects
4. establishing default-priority and active/inactive behavior
5. keeping event metadata ownership separate from both communication configuration and Event-tab GUI delivery

### Key Deliverables
- [ ] Clear source of truth for event categories and valid priorities
- [ ] Seed/default-data strategy for initial event metadata
- [ ] Repository/core access patterns for event-metadata reads and runtime loading
- [ ] Explicit default-priority contract
- [ ] Explicit boundary between event metadata authority and other concerns such as communication config, structured report templates, and Event-tab delivery

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep event metadata separate from communication configuration** - This epic must not drift back into communication systems/options/qualifier ownership.
- **Use one explicit runtime authority** - Categories and priorities must converge on one clear source rather than parallel local lists or fallback contracts.
- **Preserve historical event-entry integrity** - Saved event entries must continue storing their own values rather than depending on live metadata rows for meaning later.
- **Build on the secured database path** - Event metadata loading still assumes the underlying secure persistence/bootstrap path from Epic `002`.
- **Keep testing with delivery stories** - Later `013.x` stories should include their own tests rather than creating a broad standalone testing story.

### Negative Constraints (MUST NOT)
- **Do not absorb communication configuration** - Systems, recursive communication options, qualifiers, and communication quick-add/update work remain Epic `003` territory.
- **Do not absorb Event-tab GUI delivery** - Consuming categories/priorities in the Event tab belongs to Epic `007`.
- **Do not absorb structured report template lifecycle** - Template/runtime ownership remains Epic `004` territory.
- **Do not turn this into broad event-configuration administration UI** - The epic is about runtime authority first, not full management screens.
- **Do not leave priority/category authority ambiguous** - If earlier docs imply multiple sources, this epic must resolve that rather than preserve all of them.

### Soft Constraints (SHOULD)
- Prefer a flat, practical Phase 1 metadata model unless later user discussion explicitly approves more complexity.
- Keep terminology and ordering aligned across repository, core, and GUI consumers.
- Favor small follow-up stories that separate schema/seed, runtime loading, and final authority cleanup.
