## Purpose (WHY) ⭐ MOST IMPORTANT

After `014.001` establishes the correct Tk lifetime and startup hosting model, the project still needs a real visible shell before later feature epics can plug in safely. The point of this story is not to deliver any real tab workflow yet. The point is to give EventLog an honest top-level GUI shape so later communication, event, and personnel work attaches to a production-owned shell instead of continuing to assume one abstractly.

This story exists to make the application visible in the right structure early. Without it, later work risks:
- tab epics quietly inventing their own shell assumptions
- GUI architecture staying theoretical instead of being tested against a real window layout
- screen-space and layout problems surfacing too late, after multiple feature stories already depend on the wrong top-level shape
- placeholder/demo-style GUI code lingering because there is still no production shell seam to replace it

### User Value
- The operator can finally see a real EventLog main window after successful startup.
- Later tab stories gain a truthful production host for their widgets and presenters.
- Reviewers can evaluate the shell layout separately from tab-business behavior.

---

## Goal (WHAT)

Create the first visible production main-window shell after successful startup, with only the minimal structural widgets needed for later GUI work.

### Acceptance Criteria
- [ ] After successful startup through the app-shell lifecycle from `014.001`, the application shows a real main window instead of ending at the startup step.
- [ ] The main window contains the documented top-level shell structure for this phase: a toolbar area, a tab container/notebook area, and a status area owned by a production seam under `src/gui/`.
- [ ] The shell exposes placeholder hosts for Communication, Event, and Personnel tabs so later epics have stable attachment points, but those placeholders do not claim to implement the real workflows yet.
- [ ] Focused GUI tests verify that the visible shell appears after successful startup and that the expected top-level sections/tabs are present without requiring real tab workflow behavior.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep this story visible but structurally minimal** - The deliverable is the shell scaffold only, not the first real logging workflow.
- **Build on the app-shell lifetime from `014.001`** - The visible shell should attach to the application lifetime model already established there rather than reopening root ownership.
- **Align with the documented top-level GUI shape** - The shell should reflect the intended toolbar + notebook/tabs + status-area structure already described in the GUI architecture/design documents.
- **Expose stable tab hosts for later epics** - Communication, Event, and Personnel must have clear placeholder ownership points for Epics `006`, `007`, and `008`.
- **Keep the shell production-owned** - The visible shell should live under the real application GUI structure, not in demo or temporary prototype code.

### Negative Constraints (MUST NOT)
- **Do not implement real communication/event/personnel workflows here** - Forms, tables, filters, presenters, and CRUD behavior remain later epic work.
- **Do not absorb live status-bar logging behavior yet** - This story may create the status area visually, but richer logging/status behavior should remain later work unless a tiny enabling seam is unavoidable.
- **Do not absorb preference persistence or operator-context behavior** - Epic `005` remains responsible for persisted window/operator/UI-state behavior.
- **Do not wire final reset/close destructive behavior here** - Shell-level `Nollställ`, shutdown, and active-context cleanup integration belong to `014.003`.
- **Do not let placeholders grow into disguised half-features** - Placeholder content should identify later owners and validate structure only.
- **Do not use `Demo/demo_app.py` as implementation contract** - Demo ideas may inspire labels/layout direction loosely, but production structure must be driven by the documented architecture and current runtime seams.

### Soft Preferences (SHOULD)
- Prefer compact placeholder content that makes the future tab ownership obvious to reviewers.
- Favor a shell structure that can later accept real tab views/presenters without large widget-tree churn.
- Keep the first visible status area simple and neutral if richer behavior is not yet ready.

---

## Implementation Notes

### Suggested Approach
1. Add the production main-window shell view/controller seam under `src/gui/` after the app-shell startup handoff is complete.
2. Build only the top-level frames/widgets needed for this phase: toolbar area, notebook/tab container, and status area.
3. Add three placeholder tab hosts for Communication, Event, and Personnel.
4. Keep any visible toolbar/status elements structurally present but behavior-light unless a later story explicitly owns that interaction.
5. Add focused tests proving the shell appears and the tab placeholders exist after successful startup.

### Affected Components
- App-shell/main-window GUI modules under `src/gui/`
- `src/app.py` or the app-shell seam as needed for visible-shell handoff
- Focused GUI tests for shell rendering after startup
