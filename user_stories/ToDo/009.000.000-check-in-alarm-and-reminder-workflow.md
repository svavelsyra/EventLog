## Purpose (WHY) ⭐ MOST IMPORTANT

Capturing expected check-in times in the database is not enough by itself. If the application does not actively surface overdue personnel reminders at the right time, operators still have to manually scan the personnel list and remember who should have reported in. That undermines one of the main safety reasons for tracking personnel in the first place.

This epic exists to define and deliver the in-application reminder workflow that turns stored personnel reminder data into operationally useful alerts. The purpose is to ensure that EventLog can:
- detect when enabled personnel check-in reminders become overdue
- surface overdue reminders clearly enough that the operator does not need to discover them by chance
- support acknowledgement or clearing behavior so overdue reminders can transition out of the active-alert state deliberately
- keep reminder behavior inside the application's offline UI and persistence model instead of depending on external notification services
- separate the reminder engine from the baseline Personnel tab so the personnel workflow and the alerting workflow do not become one oversized epic

The deeper reason is safety under workload. In a busy command post, the operator cannot rely on memory alone to notice that a patrol or team failed to check in. The reminder workflow exists to reduce the chance that overdue contact is missed.

### Success Criteria
- EventLog can identify overdue personnel reminders based on stored expected check-in data and current runtime time
- Overdue reminders are surfaced clearly in the application without requiring manual ad hoc queries
- Operators have a defined way to acknowledge or clear reminder state so the same overdue case does not remain ambiguous forever
- Reminder behavior stays offline and local, using the existing repository/domain model rather than external services or platform-specific integrations
- Epic 009 stays focused on reminder/alert workflow, without absorbing the whole Personnel tab, communications, attachments, or broad task-management behavior
- The epic is clear enough to split into small stories for detection/refresh logic, UI surfacing, acknowledgement flow, and startup/recovery behavior

## Goal (WHAT)

Define and deliver the in-application check-in alarm and reminder workflow for overdue personnel tracking.

This epic covers the end-to-end contract for:
1. detecting overdue personnel reminders from repository-backed personnel data
2. refreshing and surfacing overdue reminder state within the GUI at practical intervals
3. presenting overdue reminder state clearly in the personnel workflow and, where appropriate, broader application attention surfaces
4. acknowledging or clearing reminder state through a deliberate presenter/repository flow
5. handling startup and refresh behavior when overdue reminders already exist when the application opens

### Key Deliverables
- [ ] Reminder detection contract built on personnel reminder data and overdue-query support from Epic 001
- [ ] Presenter/service contract for periodic or event-driven reminder refresh within the application's normal GUI loop
- [ ] In-application reminder surfacing contract for overdue personnel cases, including clear visual attention behavior and review entry points
- [ ] Reminder acknowledgement/clear workflow that updates stored alarm state deliberately instead of relying on transient UI-only dismissal
- [ ] Startup/resume behavior contract for already-overdue reminders present when the application launches or reconnects to an existing database
- [ ] Explicit boundary between reminder workflow and neighboring concerns such as baseline personnel-tab entry/editing, OS notifications, and general task scheduling

## Limitations (CONSTRAINTS & REQUIREMENTS)

**What boundaries must we work within? What must we avoid?**

### Hard Constraints (MUST)
- **Build on the personnel reminder data model already defined** - Overdue reminder behavior must use `alarm_enabled`, `expected_checkin_time`, `alarm_triggered`, and the dedicated overdue-query support already planned in the repository layer
- **Keep reminder behavior inside the application's offline GUI/runtime loop** - The workflow must operate locally in the Tkinter application without internet services or external schedulers
- **Use MVP responsibilities cleanly** - Detection/coordination logic belongs in presenters/services, while views display reminder state and collect acknowledgement actions
- **Support deliberate acknowledgement behavior** - The workflow must define how an overdue reminder transitions out of the active-alert state instead of leaving that state ambiguous
- **Handle already-overdue cases at startup** - The application must have an explicit rule for how reminders are surfaced when they are overdue before the GUI opens
- **Keep reminder surfacing readable on small screens** - Alert indicators, review entry points, and any reminder list/dialog behavior must respect the project's constrained-screen design requirements
- **Preserve local persistence as the source of truth** - Reminder status should not depend on transient in-memory flags alone when stored state such as `alarm_triggered` already exists
- **Keep testing with delivery stories** - Later `009.x` stories should include their own presenter/gui/repository tests instead of creating a broad standalone testing story

### Negative Constraints (MUST NOT)
- **Do not absorb the baseline Personnel tab workflow** - Normal personnel entry, history review, and status-update flows remain Epic 008 territory
- **Do not require OS-level notifications, system tray integration, or external messaging services** - Phase 1 reminder behavior should remain fully in-app and stdlib-compatible
- **Do not turn reminders into a generic task-management or calendar system** - This epic is about personnel check-in reminders, not arbitrary reminders for any project concern
- **Do not absorb communications, events, or file-attachment workflows** - Those remain separate epics with their own boundaries
- **Do not rely on view-local timer hacks without a defined contract** - Reminder refresh behavior must be explicit and testable instead of becoming scattered widget-level polling logic
- **Do not expand immediately into multi-stage escalation behavior** - Escalation chains, sound policies, or complex snooze models should not be assumed unless separately planned later

### Soft Constraints (SHOULD)
- Prefer small follow-up stories that separate overdue detection/refresh, UI surfacing, acknowledgement behavior, and startup/recovery handling
- Keep reminder attention behavior obvious without creating visual chaos in the main application window
- Favor in-app review paths that take the operator directly to the relevant personnel context instead of presenting disconnected alerts
- Leave room for later enhancements such as near-due warnings or richer escalation without requiring them in the first pass

## Context & Background

### Architectural Context
Epic 001 created the personnel data model and the repository-side overdue-query path needed for finding overdue reminders. Epic 008 defines the baseline Personnel tab workflow, including capture and display of reminder-related personnel fields. Epic 009 exists because those two pieces still do not answer the operational question of how overdue reminders are surfaced actively inside the application.

That means Epic 009 is the reminder-specific workflow layer that sits above the personnel persistence model and alongside the personnel GUI:
- repository-backed overdue-reminder queries
- presenter/service-driven refresh behavior
- in-application reminder surfacing
- acknowledgement/clear transitions
- startup/recovery handling for already-overdue reminders

### Why Reminder Behavior Deserves Its Own Epic
Reminder logic is tightly related to personnel tracking, but it is still a distinct subsystem:
- it needs recurring or event-driven refresh behavior
- it affects more than just the personnel-entry form
- it introduces acknowledgement state and alert visibility questions
- it can easily expand into sound, escalation, or broader notification scope if not bounded

Keeping it separate allows the Personnel tab to stay focused on status tracking while Epic 009 focuses on turning stored reminder data into actionable operator attention.

### Phase 1 Reminder Scope
The existing design already supports:
- storing expected check-in times
- finding overdue alarms in the repository layer
- showing overdue personnel rows or reminder indicators in the personnel context

Epic 009 should define how those pieces become an actual workflow rather than isolated features. That likely includes:
- when overdue queries run
- where overdue state is shown
- how the operator acknowledges the reminder
- how already-overdue entries appear after application startup

### What This Epic Excludes
This epic is intentionally about **check-in alarm and reminder workflow**, not about:
- baseline personnel-tab entry or status-update delivery
- general-purpose reminders for unrelated features
- operating-system notification integration
- complex escalation chains or alert policies
- communication, event, or attachment workflows

### Key Risks
- **Boundary drift** - reminder delivery could easily absorb full personnel-tab work, OS notifications, or generalized task-management scope if not held tightly
- **Refresh ambiguity** - later implementation must define clearly when overdue checks run so behavior is predictable and testable
- **State confusion** - `alarm_triggered`, acknowledgement, and overdue visual state could become inconsistent if the workflow contract is not explicit
- **Alert fatigue or clutter** - reminder surfacing must be visible enough to help without turning the UI into constant noise on small screens
- **Startup surprise behavior** - already-overdue reminders could feel inconsistent or alarming if the startup rule is not clearly defined before implementation

### Success Metrics
- Epic 009 is clear enough to split into small stories without mixing in unrelated personnel-entry, attachment, or platform-notification scope
- A later implementation pass can surface overdue reminders reliably without manual ad hoc querying
- Operators have a defined way to acknowledge reminder state instead of treating alerts as vague transient visuals
