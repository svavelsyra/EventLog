## Purpose (WHY) ⭐ MOST IMPORTANT

Once EventLog treats code as the authoritative source for bootstrap defaults and `config.ini.template` as a convenience/reference artifact, the project needs a dependable way to recreate that reference artifact from the application itself. Otherwise the template can become stale, get lost in deployment, or stop matching the real bootstrap contract even though the application still knows what the canonical shape should be.

This story exists to let the application regenerate a fresh reference `config.ini.template` from the approved bootstrap config contract. Without it, later work risks:
- a shipped `config.ini.template` drifting away from the real code-authoritative defaults
- operators or support personnel lacking a clean way to recover a readable reference template after files are lost or damaged
- documentation and deployment convenience depending entirely on a static checked-in artifact rather than the live contract the app already owns
- future config-help or recovery flows having no approved regeneration seam to reuse

### User Value
- Operators or support personnel can regenerate a fresh reference template when config files are missing or damaged.
- The project gets a cleaner way to keep the reference template aligned with the live bootstrap contract.
- Reviewers can validate template regeneration separately from the core bootstrap load/save behavior.

---

## Goal (WHAT)

Add an app-exposed way to generate a fresh canonical `config.ini.template` from the approved bootstrap config contract, while keeping that template a reference artifact rather than a second runtime authority.

### Acceptance Criteria
- [ ] The application exposes an approved operator-visible seam to generate or write a fresh `config.ini.template` from the current authoritative bootstrap config contract.
- [ ] The generated template reflects the approved config sections, keys, comments, and safe defaults needed for human reference without depending on a checked-in template file as the runtime source of truth.
- [ ] The generated template remains non-secret and reference-oriented, avoiding any requirement to dump sensitive runtime material or turn template generation into a general export/backup feature.
- [ ] Focused tests are required for template generation output shape and for keeping the generated template aligned with the code-authoritative bootstrap contract.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Build on the code-authoritative bootstrap contract** - This story must generate the template from the approved app-owned defaults/contracts rather than treating the checked-in file as primary.
- **Keep the output reference-oriented** - The generated file is a readable template/example, not a dump of all live runtime state.
- **Keep secrets out of the generated template** - No passwords, key material, remembered secret paths, or other sensitive content may be emitted.
- **Stay within local/offline behavior** - Template regeneration must work entirely offline with stdlib-only runtime constraints.
- **Keep the operator-visible entry seam small** - A simple menu action, command surface, or equivalent app-exposed seam is enough; this story does not require a broad settings console.

### Negative Constraints (MUST NOT)
- **Do not turn `config.ini.template` into runtime authority** - The template remains reference/convenience only.
- **Do not turn this into a general config export/import feature** - Broader portability or backup behavior remains outside this story.
- **Do not absorb the main bootstrap load/save contract** - That belongs to `005.001`.
- **Do not absorb database-backed preference persistence** - That remains with the other Epic `005` stories.

### Soft Preferences (SHOULD)
- Prefer output that is easy for humans to inspect and compare with `config.ini`.
- Keep the regeneration path reusable if later support tooling or recovery flows need it.
- Favor deterministic ordering of sections and keys so regenerated templates are stable and reviewable.

---

## Implementation Notes

### Suggested Approach
1. Reuse the code-authoritative bootstrap config contract defined by the earlier `005` stories.
2. Define a small app-exposed entry point for writing a fresh reference template.
3. Keep the emitted content canonical and non-secret rather than copying arbitrary live state.
4. Add focused tests for generated-template structure and alignment with the approved bootstrap contract.

### Affected Components
- App/config helpers under `src/`
- The app-exposed seam that triggers template regeneration
- Focused tests for generated-template output
