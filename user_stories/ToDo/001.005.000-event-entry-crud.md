## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

Event logging is the second core function of EventLog. Operational events, incidents, observations, and status changes must be persistently stored so that:

1. **Situational awareness** - Track what's happening operationally
2. **Historical record** - Complete timeline of events during operations
3. **Priority tracking** - Critical events can be filtered and highlighted
4. **Search and analysis** - Find specific events by category, priority, time, etc.

Without CRUD operations for EventEntry, operators cannot log operational events, making the application incomplete for field use.

### User Value
- Operators can log important operational events
- Events are permanently stored with priority and category
- Users can search historical events by multiple criteria
- Critical events can be tracked and reviewed

## Goal (WHAT)

**What functionality are we delivering?**

Implement all CRUD (Create, Read, Update, Delete) operations for EventEntry in SQLiteEventLogRepository. This includes basic retrieval, filtered queries by priority/category, and full-text search.

### Acceptance Criteria

**Create Operation**:
- [ ] Can persist new event entry to database
- [ ] Converts domain types to appropriate database storage types
- [ ] Returns database-assigned identifier
- [ ] Auto-sets timestamp if not provided
- [ ] Applies sensible defaults for optional fields

**Read Operations**:
- [ ] Can retrieve specific event entry by identifier
- [ ] Returns None when identifier not found
- [ ] Converts database row to EventEntry domain object
- [ ] Converts field types between database storage and domain representation (datetime, boolean, etc)

**List & Filter**:
- [ ] Can retrieve all entries when no filters provided
- [ ] Supports dynamic filtering by operator, priority, category, participant, and date range
- [ ] Results sorted by event time (most recent first)
- [ ] Returns empty list when no matches

**Search**:
- [ ] Can search entries by text in event description
- [ ] Search is case-insensitive
- [ ] Can combine search with filters
- [ ] Returns empty list when no matches

**Update Operation**:
- [ ] Can modify existing event entry by identifier
- [ ] Updates all modifiable fields
- [ ] Sets edited flag automatically
- [ ] Returns success status (true if updated, false if not found)
- [ ] Converts types properly

**Delete Operation**:
- [ ] Can permanently remove event entry from database
- [ ] Returns success status (true if deleted, false if not found)

**Helper Methods**:
- [ ] Private row-to-entity converter handles type conversions
- [ ] Handles NULL values appropriately
- [ ] Converts all database types to domain types

## Implementation Notes

### What's Needed

1. **Create Operation Capability**
   - Purpose: Persist new event entry to database storage
   - Input: EventEntry domain object
   - Output: Database-assigned identifier
   - Key considerations: Type conversions to storage format, sensible defaults for optional fields, auto-timestamp

2. **Read Operation Capability**
   - Purpose: Retrieve specific event entry from database
   - Input: Integer identifier
   - Output: EventEntry object if found, None if not found
   - Key considerations: Reuse row-to-entity conversion helper, type conversions from storage format

3. **List & Filter Capability**
   - Purpose: Retrieve multiple entries with optional filtering
   - Input: Optional filter criteria dictionary
   - Output: List of EventEntry objects (empty list if no matches)
   - Key considerations: Dynamic SQL query building, filter logic combines with AND, sort by event time descending, reuse row-to-entity conversion

4. **Search Capability**
   - Purpose: Find events containing specific text
   - Input: Search text string, optional filter criteria dictionary
   - Output: List of EventEntry objects matching criteria
   - Key considerations: Case-insensitive text matching, combine text search with filters, reuse row-to-entity conversion

5. **Update Operation Capability**
   - Purpose: Modify existing event entry in database
   - Input: EventEntry object with identifier set
   - Output: Boolean success status
   - Key considerations: Auto-set edited flag, preserve original logged timestamp, type conversions to storage format, check if row was affected

6. **Delete Operation Capability**
   - Purpose: Permanently remove event entry from database
   - Input: Integer identifier
   - Output: Boolean success status
   - Key considerations: Hard delete (row removed), check if row was affected

7. **Row-to-Entity Conversion Helper**
   - Purpose: Transform database row data into domain entity (reusable across read operations)
   - Input: Database query result row
   - Output: EventEntry domain object
   - Key considerations: Type conversions for datetime/boolean, NULL value handling

### Technical Considerations
- EventEntry has fewer database-specific fields than CommunicationEntry (no JSON storage)
- Priority defaults to 'Normal' if not provided
- Category can be NULL (optional field)
- Pattern is identical to CommunicationEntry CRUD (proven approach)

## Context & Background

This story follows the exact same pattern as CommunicationEntry CRUD (001.004). The structure is proven and maintainable:
- Create: Convert entity → SQL INSERT → return ID
- Read: SQL SELECT → convert row → return entity
- Update: Convert entity → SQL UPDATE → return bool
- Delete: SQL DELETE → return bool
- Helper: Row → entity conversion

**EventEntry is simpler than CommunicationEntry**:
- No JSON fields (system_capabilities)
- Fewer specialized fields
- Similar filtering needs
