## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

Personnel tracking is the third core function of EventLog. Tracking who is where, their status, and when they last checked in is critical for operational safety and situational awareness:

1. **Safety** - Know who is out on patrol/mission and when they should return
2. **Accountability** - Track all personnel and units at all times
3. **Check-in alerts** - Get notifications when someone is overdue
4. **Historical tracking** - Full history of status changes for each person/unit

Without CRUD operations for PersonnelEntry, operators cannot track personnel movements and status, creating a significant operational gap.

### User Value
- Operators can track who is where and what they're doing
- Overdue check-ins trigger alerts (safety net)
- Complete history of personnel status changes
- Quick view of all active personnel and their locations

## Goal (WHAT)

**What functionality are we delivering?**

Implement all CRUD operations for PersonnelEntry in SQLiteEventLogRepository. This includes basic CRUD, filtered queries, and specialized convenience methods for active entries, history, and overdue alarms.

### Acceptance Criteria

- [ ] **Create Operation**:
  - Can persist new personnel entries and apply the required default values for new records
  - Converts personnel data correctly between domain representation and database storage formats
  - Returns the database-assigned identifier after successful creation
- [ ] **Read Operations**:
  - Can retrieve a single personnel entry by identifier
  - Returns `None` when the requested entry does not exist
  - Reuses shared row-to-entity conversion logic for consistent type handling
- [ ] **List & Filter**:
  - Can retrieve all personnel entries when no filters are supplied
  - Supports filtering by relevant personnel attributes and time boundaries needed for operator workflows
  - Returns results in a consistent recency-first order and an empty list when nothing matches
- [ ] **Special Queries**:
  - Can retrieve active personnel entries for current operational awareness views
  - Can retrieve full personnel history for a specific person or unit
  - Can retrieve overdue alarm candidates based on enabled alarms, expected check-in time, and alarm state
- [ ] **Update Operation**:
  - Can modify an existing personnel entry by identifier
  - Automatically marks updated entries as edited while preserving the original logged timestamp
  - Returns a clear success/failure result when the target entry exists or is missing
- [ ] **Delete Operation**:
  - Can permanently remove a personnel entry from the database
  - Returns a clear success/failure result when the target entry exists or is missing
- [ ] **Helper Methods**:
  - Includes private row-to-entity conversion support for repository reuse
  - Handles optional values and database-to-domain type conversions correctly

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Implement in SQLiteEventLogRepository** - Replace NotImplementedError stubs from 001.003
- **Use parameterized queries** - Prevent SQL injection
- **Type conversions must be correct** - Preserve consistent datetime and boolean storage behavior across personnel operations
- **Handle NULL values** - Optional fields can be None in database
- **Set edited flag on update** - Always set edited=1 when updating
- **Preserve logged_time** - Never change logged_time on update
- **Validate alarm fields** - If alarm_enabled=1, expected_checkin_time must not be NULL (database constraint)

### Negative Constraints (MUST NOT)
- **Don't implement status update workflow yet** - GUI will handle marking old entry inactive
- **Don't implement supersedes relationship tracking** - Basic CRUD only, relationship management is GUI concern
- **Don't validate business rules** - That's Core layer responsibility
- **Don't implement soft deletes** - Hard delete for now
- **Don't add alarm notification system** - Just return overdue entries, notification is separate

### Soft Preferences (SHOULD)
- Follow repository patterns already established for CommunicationEntry and EventEntry
- Keep success/failure and error handling behavior consistent across repository operations
- Centralize row-to-entity conversion so personnel read paths stay consistent
- Keep specialized personnel query methods aligned with common operator workflows

## Implementation Notes

### What's Needed

1. **Create and Row-Mapping Capability**
   - Persist new personnel entries with the required default values for timestamps, active state, edit state, and alarm flags
   - Use a shared row-to-entity helper so stored values are converted consistently back into `PersonnelEntry` objects

2. **Read and Filter Capability**
   - Support single-entry retrieval plus multi-entry retrieval with optional filtering for common personnel review workflows
   - Keep ordering and empty-result behavior consistent with the repository conventions used for other entry types

3. **Specialized Personnel Query Capability**
   - Provide focused query paths for active personnel views, person/unit history, and overdue alarm review
   - Keep these convenience methods aligned with the operational scenarios described in the Purpose section

4. **Update and Delete Capability**
   - Support modifying and permanently removing personnel entries while preserving immutable history fields and reporting success cleanly
   - Keep write behavior consistent with other repository CRUD implementations

### Technical Considerations
- PersonnelEntry includes multiple time-oriented fields, so conversion behavior should stay consistent across create, read, update, and specialized query paths
- Active state and overdue-alarm access patterns are core to the personnel workflow, so query behavior should prioritize those use cases clearly
- Supersedes tracking remains stored data at this layer; relationship workflow handling stays outside this story
- This parent story defines the overall capability boundary, while the split sub-stories carry the more detailed implementation breakdown

## Context & Background

PersonnelEntry has the most complex querying needs of the three entities:
- **Active vs inactive** - Most views only show active=1 entries
- **Historical tracking** - Need to see all entries for one person
- **Alarm management** - Need to find overdue check-ins

These specialized query paths keep common operational lookups simple without forcing every caller to reconstruct the same filtering logic.

**Pattern consistency**:
- Follows the same repository CRUD structure as CommunicationEntry and EventEntry
- Adds personnel-specific query paths where operational workflows need them most
- Keeps generic filtering available for broader list views while preserving simpler entry points for common tasks

## Testing Strategy

### Key Test Scenarios

**Basic CRUD**:
- [ ] Create entry and retrieve it by ID (round-trip test)
- [ ] Create entry with all fields populated
- [ ] Create entry with minimal fields (only required: who, operator)
- [ ] Create entry with NULL optional fields
- [ ] Defaults apply correctly (active=1, edited=0, etc.)
- [ ] last_contact_time defaults to logged_time if not provided
- [ ] Retrieve non-existent ID returns None
- [ ] Update entry changes fields correctly
- [ ] Update sets edited=1 automatically
- [ ] Update preserves logged_time
- [ ] Update non-existent ID returns False
- [ ] Delete entry removes it from database
- [ ] Delete non-existent ID returns False

**Filtering**:
- [ ] Get all entries returns all entries
- [ ] Filter by who returns correct subset
- [ ] Filter by active returns only active or inactive entries
- [ ] Filter by date range returns correct subset

**Convenience Methods**:
- [ ] get_active_personnel_entries returns only active=1 entries
- [ ] get_active_personnel_entries sorts by last_contact_time
- [ ] get_personnel_history returns all entries for one person
- [ ] get_personnel_history includes both active and inactive
- [ ] get_personnel_history sorts by logged_time DESC
- [ ] get_overdue_alarms returns only alarm_enabled=1, past due, not triggered
- [ ] get_overdue_alarms sorts by expected_checkin_time ASC (most overdue first)
- [ ] get_overdue_alarms returns empty list if no overdue alarms

**Type Conversions**:
- [ ] Type conversions work correctly (datetime, boolean)
- [ ] NULL values handled properly
- [ ] Multiple datetime fields converted correctly
