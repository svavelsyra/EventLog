## Purpose (WHY) ⭐ MOST IMPORTANT

The operator-facing Communication form now works through the live database-backed runtime path, which is valuable, but the project still lacks the agreed minimum configuration-source workflow around that path. The application needs a good standard baseline for normal use, that baseline needs to be seeded into fresh databases, and the same baseline needs to be exportable as a real starter/template JSON. On top of that, operators still need a visible way to import a chosen JSON when they want to customize the setup before a fuller admin/setup UI exists.

This story exists to add that missing configuration-source workflow without redesigning the working operator-facing GUI path from `006.003.001`. The project needs an explicit JSON import/export seam plus generated template output so Communication configuration can be edited outside the database, imported into the database deliberately, and then consumed through the existing runtime authority path. That keeps the database as the live runtime source after import while also preserving a seeded standard setup that makes the app usable immediately for most normal cases. The standard baseline is expected to cover the agreed common defaults such as `RA180`, `Rakel`, `Motorola`, `Kurir`, and `Telefon`.

### User Value
- Reviewers get one coherent baseline rule: fresh databases, template export, and imported standard setup all describe the same standard Communication configuration.
- The project gains a deliberate JSON import/export seam that supports customization without making import mandatory for normal use.
- Later Communication stories can keep relying on the working DB-backed runtime path while future setup/admin work builds on explicit portable setup artifacts instead of hidden DB-only conventions.

---

## Goal (WHAT)

Add the missing Communication-configuration JSON import/export workflow, seeded standard baseline alignment, and starter-template generation while preserving the existing DB-backed runtime authority consumed by the Communication workflow.

For this story, that means:
- defining or reusing one explicit JSON contract for Communication configuration import/export
- supporting export of the current Communication configuration from the database into JSON
- supporting generation of a starter/template JSON file from the application, similar in spirit to the existing app-config template generation, where that template represents the same standard setup that fresh databases receive through seeding
- keeping fresh-database seeding of the standard setup as the normal baseline for ordinary use rather than treating import as mandatory
- keeping that standard seeded/template baseline aligned with the agreed common systems (`RA180`, `Rakel`, `Motorola`, `Kurir`, `Telefon`) and their approved default option/qualifier shapes
- supporting import of that JSON into the database so the existing runtime path continues to feed the Communication UI from the database after import
- exposing a visible UI path to load/apply a chosen Communication JSON when the user wants to customize beyond the standard baseline

This story is a **configuration-source cleanup and import/export** slice. It is not the admin/setup GUI itself and it is not a reason to redesign the operator-facing Communication form interaction already defined by `006.003.001`.

### Acceptance Criteria
- [x] The project exposes one explicit Communication configuration JSON import/export contract that can represent the runtime system/path/qualifier shape without forcing the Communication presenter/view to know about source-specific payload details.
- [x] The application can generate or export a Communication configuration JSON file that is suitable as an editable starter/template for later import, analogous in purpose to `config.ini.template`, and that template represents the same standard setup a fresh database receives by default.
- [x] Importing that JSON updates the database-owned Communication configuration so the existing DB → loader → presenter → GUI path continues to provide the expected systems/path/qualifier runtime behavior after import.
- [x] The application exposes a visible UI path for selecting and importing one Communication configuration JSON file into the current database when the user wants to customize beyond the standard baseline.
- [x] Focused tests cover seeded-baseline/template alignment, JSON round-trip/import behavior, and confirm that imported configuration still reaches the Communication workflow correctly.

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep the operator-facing runtime contract stable** - The presenter/view seam proven by `006.003.001` should not need redesign just because the editable source/import workflow is being corrected.
- **Keep the database as the live runtime authority after import** - JSON should be an import/export and template-generation format for this story, not a presenter-local runtime source that bypasses the repository/core loader path.
- **Keep the seeded standard setup as the normal baseline** - Fresh or recreated databases should remain immediately usable with the standard Communication setup even before any import occurs.
- **Keep the template and seeded baseline aligned** - The standard template output should describe the same standard setup the DB seeding creates so that exporting the template and importing it back reproduces the baseline cleanly.
- **Keep the cleanup consumer-driven** - Any backend/core/bootstrap change in this story must be justified by the Communication workflow's real needs rather than becoming a broad reopening of Epic `003`.
- **Allow small backend/core/bootstrap adjustments when truly needed** - This story may touch lower layers if that is what the import/export seam requires.
- **Support generated starter output** - The solution must allow the application to create a starter/template Communication configuration JSON without requiring the user to reverse-engineer the format manually.
- **Include focused tests with the story** - Tests should prove JSON round-trip/import behavior and stable runtime behavior after import.

### Negative Constraints (MUST NOT)
- **Do not redesign the operator-facing form behavior** - Visible field flow, path depth, and qualifier behavior remain owned by `006.003.001` unless a separate user decision changes them.
- **Do not absorb admin/setup authoring UI** - Guided settings/configuration-management UX remains outside this story.
- **Do not reopen all Epic `003` scope** - This is targeted import/export and source cleanup, not a broad backend rewrite.
- **Do not couple the Communication presenter/view directly to JSON files** - JSON import/export belongs below the presenter/view seam.
- **Do not require manual SQLite editing as the intended workflow** - Customization should happen through explicit export/edit/import rather than direct DB surgery.
- **Do not make JSON import mandatory for ordinary use** - The standard seeded setup should be good enough that normal users can start working without first importing a file.
- **Do not absorb later table/detail/template stories** - Keep this story about configuration source handling only.

### Soft Preferences (SHOULD)
- Prefer reusing the existing communication portability/import contract if it already matches the needed JSON shape closely enough.
- Prefer making template generation and import/export explicit and testable rather than hidden in ad hoc bootstrap behavior.
- Keep the patch small enough that reviewers can see exactly what changed in the source-of-truth workflow.

## Implementation Notes

### Suggested Approach
1. Reuse or define one explicit JSON shape for Communication configuration import/export.
2. Add or expose application-owned helpers that can:
   - export the current DB-backed Communication configuration to JSON
   - generate a starter/template Communication configuration JSON file
   - import a JSON file into the database-owned Communication configuration
 3. Align the standard seeded DB setup and the generated template output so they describe the same baseline systems/options/qualifiers.
 4. Add a visible UI path that lets the user choose and import one Communication JSON file into the current database.
 5. Verify that the presenter/view contract from `006.003.001` remains unchanged at the consumption boundary and that imported configuration still reaches the GUI through the existing runtime path.
 6. Add focused tests covering seeded baseline/template alignment, JSON export, template generation, JSON import, UI import wiring, and post-import runtime behavior.

### Affected Components
- Communication configuration import/export and template-generation seam
- Core/runtime configuration helpers under `src/core/` if needed
- Backend/repository/bootstrap wiring only as needed to support JSON import/export, seeded-baseline/template alignment, and UI import/apply flow
- Focused tests for JSON round-trip/import behavior and post-import runtime-source behavior

## Context & Background

This story exists because the Communication workflow is the first real consumer strong enough to reveal that the missing problem is no longer presenter access to runtime config; that part already works. The missing problem is the lack of an explicit editable source/import workflow for Communication configuration before a fuller admin/setup GUI exists.

The intended direction is:
- a good seeded standard setup for ordinary use
- template export that reproduces that same standard setup as editable JSON
- export of the current live setup when the user wants to customize from what they already have
- explicit import into the database when the user wants to customize beyond the baseline
- database remains the live runtime authority after import

That direction is a better fit for future admin-configurable Communication setup than either hiding everything in DB seeding alone or requiring JSON import before the app is useful.
