## Purpose (WHY) ⭐ MOST IMPORTANT

Epic `013` cannot become real if event metadata still exists only as scattered assumptions about categories, priorities, and default values. The application needs one durable storage/seed foundation for valid event metadata before later repository, core, and Event-tab work can consume it safely.

This story exists to establish that event-metadata storage and the first practical seed data for Phase 1 use. Without it, later work could drift into:
- presenter-local category/priority lists
- conflicting priority defaults in different layers
- unclear active versus inactive behavior for old versus new event entries
- repeated re-argument about where event metadata actually lives

### User Value
- A fresh database gets practical event categories and priorities immediately.
- Later stories can depend on explicit event metadata storage rather than implied defaults.
- Reviewers can verify one seed/source foundation before the Event tab consumes it.

---

## Goal (WHAT)

Create the schema/storage and first-run seed-data contract for event metadata.

### Acceptance Criteria
- [ ] The database schema or equivalent approved runtime source contains the event metadata storage needed by Epic `013`, covering categories, priorities, active/inactive state, ordering, and explicit default-priority support if that source requires durable storage.
- [ ] First-run or initial-schema seeding provides practical Phase 1 default event metadata with deterministic behavior that does not duplicate seed records on repeated startup/readiness paths.
- [ ] The event metadata contract preserves historical event-entry integrity by keeping saved event entries responsible for storing the category/priority values actually used at log time.
- [ ] Focused schema/repository tests verify that a new database receives the expected event-metadata records and that repeated initialization does not create conflicting duplicates.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep event metadata as data, not scattered constants** - The deliverable must create or extend a real runtime source rather than rely on presenter-local lists.
- **Seed for Phase 1 practicality** - A fresh database must start with usable default categories and priorities without broad admin UI work first.
- **Support active/inactive evolution** - Metadata rows should support non-destructive change where needed.
- **Make default-priority handling explicit** - The chosen default-priority source must be structurally clear here rather than left implicit for later stories.
- **Keep runtime metadata distinct from event history** - Old event entries must not rely on live metadata rows to remain meaningful.

### Negative Constraints (MUST NOT)
- **Do not absorb Event-tab GUI delivery** - Event form/filter consumption belongs to Epic `007`.
- **Do not absorb communication configuration** - Communication systems/options/qualifiers remain Epic `003` territory.
- **Do not absorb report templates** - Structured-report template/runtime ownership remains Epic `004` territory.
- **Do not require a full event-configuration admin screen** - This story establishes storage and seeding only.

### Soft Preferences (SHOULD)
- Prefer flat, practical Phase 1 metadata unless later user discussion explicitly approves more complexity.
- Keep naming aligned with later repository/core runtime objects.
- Favor deterministic insert-or-update seeding semantics over fragile one-off initialization.

## Implementation Notes

### Suggested Approach
1. Add the event-metadata storage required for categories, priorities, and default-priority behavior if it does not already exist in acceptable form.
2. Define the initial default rows and ordering.
3. Ensure initialization/seed execution can recognize already-seeded data safely.
4. Add focused tests around fresh initialization and rerun-safe seeding.

### Affected Components
- `src/db/schema/sqlite/initial_schema.sql`
- SQLite schema/bootstrap support and any event-metadata seed-data helpers
- Focused DB tests for event-metadata schema + seed initialization
