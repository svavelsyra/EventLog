## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

Communication logging is the primary function of EventLog. This story enables operators to persistently store radio messages, phone calls, and orders. The Create operation is the foundation - without it, no communications can be logged.

The row mapper is equally critical - it handles the complex type conversions between database and Python objects (datetime, JSON, boolean), enabling all other CRUD operations to work correctly.

### User Value
- Operators can log communications (stored permanently)
- Communications survive application restarts
- Type conversions handled correctly (datetime, JSON, boolean)
- Foundation for read/update/delete operations

---

## Goal (WHAT)

**What functionality are we delivering?**

Implement the row-to-entity mapper and the create operation for `CommunicationEntry` in the SQLite repository layer under the approved package layout.

### Acceptance Criteria

- [ ] **Row-to-Entity Conversion**:
  - Can convert database row data to CommunicationEntry domain object
  - Converts field types between database storage and domain representation (datetime, boolean, structured data, etc)
  - Handles NULL values for optional fields correctly
  - Reusable by read operations
- [ ] **Create Operation**:
  - Can persist new communication entry to database
  - Converts Python domain types to appropriate database storage types
  - Returns database-assigned identifier
  - Auto-sets timestamp to current time
- [ ] **Unit Tests Pass**:
  - Round-trip test validates conversions (create then retrieve)
  - Type conversions work bidirectionally
  - NULL handling verified

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Implement in the SQLite repository layer** - Replace the placeholder behavior using the approved repository package layout
- **Delegate low-level DB execution downward** - Repository handles mapping/query intent; adapter handles low-level execution/transaction mechanics
- **Use parameterized queries** - Prevent SQL injection (? placeholders)
- **Set logged_time automatically** - Use `datetime.now().isoformat()`
- **Preserve entry.id** - Don't modify the entry object passed in
- **Follow transaction boundary rules** - Insert persistence should respect the adapter/repository transaction behavior from 001.003

### Negative Constraints (MUST NOT)
- **Don't implement read operations yet** - That's story 001.004.002
- **Don't implement update/delete yet** - That's story 001.004.003
- **Don't validate business rules** - Repository layer only stores data
- **Don't implement search** - That's story 001.004.003

### Soft Preferences (SHOULD)
- Row mapper should be reusable by all read operations
- Clear variable names for type conversions
- Standard library JSON module for structured data serialization

## Context & Background

This story establishes the pattern for all entity CRUD operations:
- Row mapper handles type conversions (reusable by get/get_all/search)
- Create operation converts entity → SQL INSERT via adapter boundary → return ID
- Type conversion mappings:
  - Python datetime ↔ SQL TEXT (ISO 8601 format)
  - Python dict ↔ SQL TEXT (JSON format)
  - Python bool ↔ SQL INTEGER (0/1)
  - Python None ↔ SQL NULL

EventEntry and PersonnelEntry will follow this same pattern in their respective stories.
