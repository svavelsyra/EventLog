---

## Purpose (WHY) ⭐ MOST IMPORTANT

The storage decision is only safe if the code has one real source of truth for where app-generated data lives. Right now the project already has path-related seams such as `DEFAULT_CONFIG_PATH`, `resolve_managed_sqlite_database_path(...)`, and default log-path settings, but if each one keeps its own local idea of the runtime root, the application can still drift into mixed storage behavior.

This story exists to establish one shared app-local runtime data root under the application folder so later code no longer has to guess where generated state belongs.

### User Value
- Administrators and maintainers get one predictable root for generated app data.
- Later code changes do not need to re-decide where config, database, and logs belong.
- Moving the app folder preserves the runtime data set because the path contract is rooted in the app folder itself.

---

## Goal (WHAT)

Introduce a centralized `data/` root contract and shared path-resolution helpers for app-generated runtime data.

### Acceptance Criteria
- [x] One shared helper/constant contract defines the canonical app-local runtime data root under the application folder, using a `data/` directory.
- [x] The default runtime config path, managed SQLite database path, and default file-log path are derived from that shared root instead of being independently rooted at the app folder, process working directory, or user-profile locations.
- [x] Startup/bootstrap consumers use the shared path helpers for runtime-owned paths instead of duplicating raw path assembly such as direct `Path(...)/"config.ini"` logic in multiple places.
- [x] Focused tests cover app-folder-relative resolution and prove that moving the whole app folder changes the resolved runtime data paths consistently.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Keep this story on path ownership only** - The story is about the shared data-root contract and its direct runtime-path consumers, not broader startup behavior changes.
- **Use the application folder as the root reference** - The approved model is app-local storage under `data/`, not per-user OS folders.
- **Preserve current architecture boundaries** - Generic startup/config layers should consume shared path helpers; they should not each grow their own storage policy.
- **Cover the direct current path seams** - At minimum this story must address the live config path, managed SQLite database path, and default file-log path.
- **Include focused automated coverage** - Path ownership is easy to regress, so direct tests are required.

### Negative Constraints (MUST NOT)
- **Do not absorb documentation-wide cleanup** - broader docs/reference alignment belongs in `015.002`.
- **Do not redesign startup UI flows** - this story should not widen into presenter/view/controller changes unless they are a direct consequence of consuming the shared path contract.
- **Do not introduce a second optional storage root model** - this slice should lock in one default contract, not reopen the storage-location debate.
- **Do not leave duplicated legacy path constants as parallel authorities** once the shared helper exists.

### Soft Preferences (SHOULD)
- Prefer a small, readable helper set over many narrowly duplicated constants.
- Keep path names explicit enough that future generated artifacts can join the same root without ambiguity.

---

## Implementation Notes

### Suggested Approach
1. Identify the current app-owned runtime path seams in `src/app.py` and `src/config/app_config.py`.
2. Introduce a single app-local `data/` root helper/constant in the owning config/path module.
3. Route live config, managed database, and default log paths through that shared root.
4. Update the focused path/bootstrap tests so they assert the new app-local layout.

### Technical Considerations
- `DEFAULT_CONFIG_PATH` is currently rooted directly under the app folder and will need to become a consumer of the shared data-root contract.
- `resolve_managed_sqlite_database_path(...)` already provides a useful seam and should align with the same root instead of silently diverging from config/log defaults.
- The change should keep path resolution deterministic when the app folder is copied elsewhere.

### Affected Components
- `src/app.py`
- `src/config/app_config.py`
- Direct-dependent bootstrap/path tests under `tests/unit/`

## Context & Background

The product decision is already settled: generated EventLog data should stay with the app in a writable app-local location rather than going into one folder per Windows user. This story is the smallest code-facing step that turns that decision into a reusable contract.
