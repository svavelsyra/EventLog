## Purpose (WHY) ⭐ MOST IMPORTANT

A visible shell is not structurally complete until it owns the top-level destructive and exit lifecycle correctly. EventLog already has shared reset and cleanup contracts from the startup/bootstrap work. Once a real main window exists, the shell must reuse those contracts instead of inventing ad hoc GUI-local cleanup behavior. Otherwise the project would end up with one reset model at startup and a second, inconsistent one in the main window.

This story exists to connect the visible shell to the existing active-context invalidation and cleanup flow. Without it, later work risks:
- a toolbar `Nollställ` action that looks real but is disconnected from the shared reset contract
- close/shutdown behavior that leaves active persistence access or cleanup ownership unclear
- duplicate reset/cleanup rules in GUI code instead of one app-owned lifecycle path
- later feature stories inheriting a misleading shell that looks complete but still lacks correct lifecycle behavior

### User Value
- The shell gains realistic top-level lifecycle behavior instead of only visual structure.
- Reviewers can verify that the main window uses the same reset/cleanup contract already established by the app/startup layers.
- Later feature work can depend on a shell that already handles destructive and exit ownership correctly.

---

## Goal (WHAT)

Wire the minimal main-window shell into the existing active-context reset, close, and cleanup seams so top-level shell actions use the established app-owned lifecycle behavior.

### Acceptance Criteria
- [ ] Shell-level destructive reset entry points such as `Nollställ` invoke the existing shared active-context reset path (`run_active_context_reset` or an equivalent app-owned seam) instead of implementing GUI-local cleanup logic.
- [ ] Normal shell close/shutdown uses the existing active-context release/invalidation ownership clearly, so the main window does not leave repository lifetime ambiguous when the operator exits the application.
- [ ] The shell integrates caller-visible reset/cleanup outcomes in a coarse shell-appropriate way while preserving the current app-owned invalidation, backend-cleanup, and remembered-target-reset behavior.
- [ ] Focused unit tests cover shell-triggered reset/close lifecycle behavior, including correct use of existing app-owned cleanup callbacks and no duplicate cleanup path in GUI code.

---

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Reuse the existing reset/cleanup contract** - This story must build on the already-established app/startup reset seams rather than re-inventing deletion or repository cleanup behavior in the shell.
- **Keep destructive and normal exit ownership explicit** - Reset and close may differ in user-facing effect, but both must have clear lifecycle ownership above the widget layer.
- **Preserve invalidation-before-cleanup behavior where required** - Existing access-release and backend-cleanup ordering should remain intact when invoked through the shell.
- **Keep shell UI handling coarse and app-facing** - Shell code may decide how to trigger and summarize outcomes, but it should not absorb backend-specific cleanup policy.
- **Test the top-level lifecycle behavior directly** - This story should include focused tests proving that shell reset/close actions call the owning app-layer lifecycle seam.

### Negative Constraints (MUST NOT)
- **Do not absorb tab workflow behavior** - Communication, Event, and Personnel interactions remain separate epics.
- **Do not create a second reset implementation in GUI code** - The shell must not duplicate `run_active_context_reset` behavior or backend cleanup details.
- **Do not expand into preference persistence or operator-context management** - Epic `005` remains responsible for those concerns.
- **Do not add new backend cleanup policy here** - This story consumes the existing cleanup contract; it does not redefine what reset means.
- **Do not turn the shell story into broad status-bar polish** - Outcome display may be coarse, but richer live logging/status behavior remains later work.

### Soft Preferences (SHOULD)
- Prefer shell callback seams that stay easy to test without deep Tk widget coupling.
- Keep shell-owned outcome messaging neutral and simple until later GUI polish stories exist.
- Favor explicit separation between destructive reset actions and normal exit/close actions, even if both currently rely on shared underlying app-layer seams.

## Implementation Notes

### Suggested Approach
1. Use the visible shell surface from `014.002` as the owner of top-level reset/close entry points.
2. Route destructive reset actions into `run_active_context_reset` or a thin equivalent app-owned seam rather than duplicating cleanup logic.
3. Route normal close/shutdown through the existing active-context release ownership, keeping repository lifetime explicit.
4. Keep shell-visible outcome handling coarse and stable.
5. Add focused lifecycle tests for reset and close paths.

### Affected Components
- App-shell/main-window shell wiring under `src/gui/`
- `src/app.py` or the app-owned shell lifecycle seam used by the visible shell
- Focused unit tests for reset/close/cleanup behavior from the shell surface
