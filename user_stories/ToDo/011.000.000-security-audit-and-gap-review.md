## Purpose (WHY) ⭐ MOST IMPORTANT

EventLog is being designed for operational use in a security-sensitive environment, but a secure design on paper is not enough by itself. As the architecture, user stories, and implementation grow, assumptions can drift, local reset behavior can miss artifacts, sensitive data can leak into convenience features, and cross-cutting risks can hide in the seams between epics.

This epic exists to perform a deliberate security audit of the system as it exists at the time of review and identify meaningful security gaps before they become entrenched implementation habits. The purpose is to ensure that EventLog can:
- review its current architecture, design docs, stories, and implementation state against the intended security model
- identify concrete gaps, contradictions, or blind spots in reset behavior, secret handling, file handling, configuration portability, logging, and startup/bootstrap assumptions
- produce a prioritized set of findings that can inform later remediation stories without turning the audit itself into an unbounded implementation effort
- distinguish clearly between real security issues, accepted risks, and low-value "security theater"
- catch cross-epic risks early, especially where one epic assumes another will handle cleanup or protection responsibilities

The deeper reason is trustworthiness. In a system meant for operational logging under possible device loss or capture, a missed cleanup rule or a hidden data leak matters more than polished wording in a design document. The audit exists to check whether the system really matches its stated security intent.

### Success Criteria
- A defined audit scope exists that covers the current security-relevant parts of the system rather than only the encryption epic in isolation
- The audit produces concrete findings with rationale, severity, and recommended follow-up actions where needed
- The audit distinguishes design gaps, implementation gaps, documentation contradictions, and consciously accepted risks
- Reset and local-artifact handling are reviewed explicitly, including whether attachment directories and other non-database artifacts survive `Nollställ`
- The epic remains review-only and analysis-focused, without quietly turning into a large implementation or remediation epic
- The epic is clear enough to split into small review/reporting stories rather than one vague “check security” task

## Goal (WHAT)

Define and deliver a structured security audit of EventLog's current system state, producing actionable findings without implementing fixes directly inside this epic.

This epic covers the end-to-end contract for:
1. defining the audit scope and review criteria for security-relevant areas of the system
2. reviewing architecture, design docs, user stories, and current implementation for security gaps and contradictions
3. evaluating reset/destruction behavior and other local-artifact cleanup responsibilities across database, logs, configuration, attachments, and related files
4. documenting findings with severity, rationale, and suggested follow-up paths
5. producing a review output that can feed later security or cleanup stories without confusing audit work with implementation work

## Limitations (CONSTRAINTS & REQUIREMENTS)

**What boundaries must we work within? What must we avoid?**

### Hard Constraints (MUST)
- **Keep this epic review-only** - The audit may inspect docs, code, file layout, and current behavior, but it must not quietly absorb implementation work for discovered issues
- **Audit the system as it actually exists** - Review must consider the real current state of architecture docs, design docs, user stories, and implemented code rather than only idealized future design intentions
- **Cover cross-epic boundaries explicitly** - Security responsibility handoffs between epics such as secure bootstrap, reset cleanup, attachments, logs, and portability/export must be examined deliberately
- **Review reset/local-artifact cleanup explicitly** - `Nollställ`, secure deletion, and best-effort cleanup rules must be checked against all local artifacts that may exist, not only the database file
- **Distinguish findings by severity and confidence** - The audit should not present every concern as equally urgent or equally proven
- **Preserve the project's realistic threat model** - Findings should align with the project's offline, local, single-database, operational-use assumptions rather than importing irrelevant enterprise-web-app concerns
- **Produce actionable output** - Findings should be concrete enough that later remediation stories can be created from them without redoing the entire investigation
- **Keep testing/verification proportional to the review task** - Audit stories may inspect behavior or test targeted scenarios where needed, but they should not become a disguised feature implementation stream

### Negative Constraints (MUST NOT)
- **Do not implement fixes inside the audit epic** - If the audit finds a cleanup or secrecy problem, the result should be a documented finding and follow-up recommendation, not a stealth code change inside this epic
- **Do not turn the audit into a broad compliance exercise** - This epic is about practical system security gaps, not a generic checkbox audit against unrelated standards
- **Do not focus only on encryption** - Security review must not ignore logs, config, attachments, portability files, reset behavior, and other local artifacts just because encryption is more visible
- **Do not manufacture fake security requirements** - Avoid findings that add friction without meaningful protection under the project's actual threat model
- **Do not treat undecided future features as current vulnerabilities automatically** - The audit should note boundaries and future risks clearly, but it should not misclassify every planned later feature as a present implementation defect
- **Do not absorb ongoing maintenance forever** - This epic is a defined review pass, not a permanent catch-all security bucket

### Soft Constraints (SHOULD)
- Prefer small follow-up stories that separate audit scoping, document/code review, reset/artifact review, and findings/reporting
- Favor findings that point to specific artifacts, assumptions, or workflows instead of abstract “might be insecure” language
- Keep accepted-risk documentation explicit where the system intentionally chooses speed, offline simplicity, or best-effort deletion over stronger guarantees
- Leave room for periodic later audits without making this first audit epic carry every future review cycle by default

## Context & Background

### Architectural Context
Epic 002 defines the secure-bootstrap and encrypted-database direction. Epic 005 defines non-secret preference persistence boundaries. Epic 010 defines file attachments stored in the local filesystem with metadata in the database. The security architecture and design docs also define `Nollställ` as a best-effort cleanup path for security-sensitive local artifacts.

That creates several cross-cutting security questions that do not belong neatly inside one implementation epic:
- does reset/delete behavior cover every local artifact type the system now expects to create?
- do portability/export ideas risk leaking secrets or machine-specific security assumptions?
- do logs, config, and convenience helpers reveal more than intended?
- are attachment files, template assets, and other local files handled consistently with the intended security model?

### Why This Is Its Own Epic
Security review is valuable precisely because it is not the same thing as continuing implementation. If the audit is buried inside implementation work, gaps can be rationalized away or patched locally without checking whether the broader design is coherent.

This epic therefore exists to pause and inspect the current system boundary deliberately.

### Reset Gap Already Identified
A concrete example already exists: the current security and reset descriptions focus on the database, logs, config, and reachable key material, while the newly defined attachment epic introduces a separate filesystem attachment directory. That raises an explicit review question:
- if attachments exist on disk, are they also destroyed during `Nollställ` and related reset flows?

That kind of cross-epic gap is exactly what this audit epic should capture.

### What This Epic Excludes
This epic is intentionally about **security audit and gap review**, not about:
- implementing encryption/bootstrap features
- implementing reset cleanup fixes directly
- implementing attachment cleanup directly
- redesigning the threat model from scratch unless the audit shows a real contradiction
- ongoing permanent security maintenance as an always-open work bucket

### Key Risks
- **Scope vagueness** - “audit security” can become shapeless unless the finding model and review boundaries are explicit
- **Finding inflation** - the audit could over-report minor concerns if severity and confidence are not defined clearly
- **Boundary confusion** - review-only work can drift into stealth implementation unless the epic stays disciplined
- **Cross-epic blind spots** - attachment cleanup, portability files, logs, and reset behavior may be missed if the audit focuses too narrowly on database encryption
- **Stale conclusions** - if the audit happens too early or too late relative to implementation, findings may need explicit dating and context

### Success Metrics
- Epic 011 is clear enough to split into small audit stories instead of one vague review blob
- Later remediation stories can be created directly from the audit findings without redoing the investigation
- Reset/local-artifact handling, including attachment cleanup coverage, is reviewed explicitly rather than assumed
