## Purpose (WHY) ⭐ MOST IMPORTANT

**Why does this story exist?**

Operators need to correct personnel records, remove mistakes, and search historical personnel data quickly:

1. **Update** - Correct status, location, mission notes, or expected check-in information
2. **Delete** - Remove duplicate or erroneous personnel records
3. **Search** - Find personnel entries containing relevant names, statuses, locations, or notes

Without update and delete, personnel logs cannot be corrected when information changes or mistakes are made. Without search, finding relevant personnel records in larger histories becomes slow and error-prone.

### User Value
- Correct inaccurate personnel records
- Remove duplicate or mistaken entries
- Search personnel history by relevant text
- Maintain reliable personnel tracking data for operational awareness

---

## Goal (WHAT)

**What functionality are we delivering?**

Implement update, delete, and search operations for PersonnelEntry in SQLiteEventLogRepository.

### Acceptance Criteria

- [ ] **Update Operation**:
  - Can modify an existing personnel entry by identifier
  - Updates the relevant personnel fields while preserving immutable history fields
  - Automatically marks updated records as edited
  - Returns a clear success/failure result when the target record exists or is missing
- [ ] **Delete Operation**:
  - Can permanently remove a personnel entry from the database
  - Returns a clear success/failure result when the target record exists or is missing
  - Uses hard-delete behavior rather than a soft-delete workflow
- [ ] **Search Operation**:
  - Can search personnel entries by text across the relevant personnel text fields
  - Search is case-insensitive
  - Can combine text search with supported personnel filters
  - Returns an empty list when no matches are found
- [ ] **Unit Tests Pass**:
  - Update behavior is verified, including edited-flag and not-found handling
  - Delete behavior is verified, including not-found handling
  - Search behavior is verified, including case-insensitive matching and filter combination

## Limitations (CONSTRAINTS & REQUIREMENTS)

### Hard Constraints (MUST)
- **Implement in SQLiteEventLogRepository** - Replace NotImplementedError stubs
- **Use parameterized queries** - Prevent SQL injection
- **Follow established personnel repository patterns** - Keep write and search behavior consistent with sibling entity stories
- **Set edited flag on update** - Updated personnel records must be marked as edited automatically
- **Preserve logged_time** - Original logged timestamp remains immutable on update
- **Support case-insensitive search** - Search behavior must not depend on exact letter case

### Negative Constraints (MUST NOT)
- **Don't implement soft deletes** - Hard delete only for this phase
- **Don't implement specialized personnel queries here** - Those belong to `001.006.004`
- **Don't implement GUI-driven status workflow logic** - GUI manages broader active/inactive workflow decisions
- **Don't move business-rule validation into the DB layer unnecessarily** - Core-layer responsibilities stay outside this story except for repository-level persistence rules

### Soft Preferences (SHOULD)
- Keep update and delete behavior consistent with other repository write operations
- Reuse established personnel read/filter patterns where helpful for search behavior
- Keep search-field handling clear and easy to verify in tests
