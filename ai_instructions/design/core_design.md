# Core Design (AI)

**Domain Models & Business Logic**  
**Last Updated**: 2026-05-05

## Overview

**Design Decision**: Three specialized entities instead of one generic entity.

**Reasoning**: Different information types (communications, events, personnel) have different required fields and business rules. Specialized entities provide:
- Cleaner domain model (no unused fields)
- Better data integrity and validation
- Type-specific business logic

## CommunicationEntry Entity

Domain entity representing radio/phone/in-person communications and written orders.


### Fields

#### Identity & Timestamps

**id** (int, auto-increment)
- Primary key
- Auto-generated by database
- Immutable after creation

**event_time** (datetime)
- When the communication occurred
- Required
- Cannot be in the future (validation)
- User-editable (for late logging)

**logged_time** (datetime)
- When the operator entered it into system
- Required
- Auto-populated to now() if not provided
- Typically not user-editable

#### Communication Content

**message_content** (str, max 5000 characters)
- The actual message text
- Required, cannot be empty
- Multiline supported
- Maximum 5000 characters (validation enforced)
- **Reasoning**: Prevents abuse while allowing substantial content

**from_field** (str)
- Sender/source of communication
- Required, cannot be empty
- Free text entry
- **Reasoning**: Better empty than "unknown" garbage data

**to_field** (str, nullable)
- Recipient/destination
- Optional - broadcasts or unclear recipients
- Free text entry
- **Reasoning**: Better empty than "alla" garbage data

#### Communication Selection Path & Top-Level Qualifiers

**Design Philosophy - Neutral Tier Model**:
- Communication entries should no longer assume one universal radio-shaped hierarchy such as system → method → channel.
- The underlying selection structure may be recursive beneath the chosen communication system/way.
- The current operator-facing UI may still render the first few visible levels as `tier_1` / `tier_2` / `tier_3`, but that is a practical UI limit rather than a fixed domain-storage limit.
- This keeps the operator-facing experience stable while allowing the underlying structure to grow later without redesigning the model.

**communication_system** (str, nullable)
- Base communication way/system used for this entry
- Usually the first and most important operator selection
- Examples:
  - `RA180`
  - `Motorola`
  - `Rakel`
  - `Kurir`
  - `Spoken Word`
- **Historical storage**: The chosen base way/system must remain readable even if configuration changes later

**communication_path** (list/JSON, nullable)
- Ordered list of selected option nodes beneath the top-level communication system
- Each step records the operator-visible choice that was actually used at that time
- Example shapes:
  - `[ {"value": "2", "label": "Channel 2"} ]`
  - `[ {"value": "5", "label": "Company Net"}, {"value": "ALT", "label": "Alternate Route"} ]`
- Phase 1 may commonly use shallow paths even though the structure is allowed to be recursive
- **Historical storage**: The stored path should preserve what was selected even if later configuration labels or child options change

**Qualifier Model**:
- Some important communication properties should sit above the tier path as top-level qualifiers
- Phase 1 discussion currently centers on:
  - `encrypted` / `clear`
  - possibly `data` as a simple on/off operational qualifier for systems such as RA180
- Qualifiers are configuration-driven and can be overridden per system as:
  - selectable
  - forced to true
  - forced to false
  - hidden / not operator-editable

**communication_qualifiers** (dict/JSON, nullable)
- Stores the chosen top-level qualifier values for the entry
- Optional, but expected for systems that expose meaningful qualifiers
- Stored as JSON in database
- Example shapes:
  - RA180 common setup: `{"encrypted": true, "data": true}`
  - Motorola: `{"encrypted": false}`
  - Rakel: `{"encrypted": true}`
- **Historical storage**: The stored qualifier values should preserve what the operator logged at that time even if later system defaults or visibility rules change

**Selection Examples**:
- **RA180 common practical setup**: `communication_system="RA180"`, `communication_path=[{"value": "2", "label": "Channel 2"}]`, `communication_qualifiers={"encrypted": true, "data": true}`
- **Motorola clear radio**: `communication_system="Motorola"`, `communication_path=[{"value": "A", "label": "Channel A"}]`, `communication_qualifiers={"encrypted": false}`
- **Rakel simplified Phase 1**: `communication_system="Rakel"`, `communication_path=[{"value": "X", "label": "Channel X"}]`, `communication_qualifiers={"encrypted": true}`
- **Kurir**: `communication_system="Kurir"`, `communication_path=[]`, qualifier visibility defined by configuration for that system

#### Communication Configuration Rules (Settings)

**Phase 1 - Hardcoded/Seeded Defaults**:
- Seed configuration should define, per system:
  - child labels for the next visible selection level where needed
  - recursive option nodes beneath the system
  - which top-level qualifiers exist for that system
  - whether each qualifier is selectable, forced, or hidden
- This allows the GUI to render the configured communication shape dynamically without inventing local per-system logic.
- The current GUI may stop after three visible dropdown levels for practical reasons, but the underlying structure should not require redesign if a deeper configured path is approved later.

**Phase 1 anchor examples**:
- **RA180**:
  - uses channels
  - exposes top-level `encrypted`
  - likely exposes top-level `data`
  - real-world channel restrictions on voice/data remain domain context, not mandatory app enforcement in Phase 1
- **Motorola**:
  - uses channels
  - `encrypted` forced false / clear
- **Rakel**:
  - uses channels
  - simplified to `encrypted` forced true in Phase 1
- **Kurir**:
  - non-radio example included to prove the shared model is not radio-only
  - lower tiers may be unused in Phase 1

**Future Development**:
- Later stories may allow richer per-path qualifiers, more complex selector types, or deeper enforcement rules if operational value justifies them.
- The current design should stay open for that growth without forcing Phase 1 to solve every real-world nuance immediately.

#### Status & Metadata

**confirmed** (bool)
- Whether recipient confirmed receipt
- Required (defaults to False)
- Checkbox in UI

**operator** (str)
- Who logged this entry (person at terminal)
- Required
- Auto-prefilled from last operator used
- User can change

**edited** (bool)
- Whether entry was modified after initial save
- Required (defaults to False)
- Set to True when entry is edited

#### Optional Fields

**attachments** (list of attachment references, nullable)
- Supporting documents (scanned orders, photos, etc.)
- Optional
- Multiple files allowed
- Common use: email attachments, scanned written orders
- Phase 1 attachment content is stored inside the encrypted database boundary

### Business Rules

#### Validation Rules

**Event Time Validation**:
- Cannot be in the future
- Must be valid datetime
- Can be in the past (for late logging)

**Required Field Validation**:
- `message_content`: Must not be empty string, **maximum 5000 characters**
- `event_time`: Must be valid datetime
- `operator`: Must not be empty string

**Optional Field Validation** (can be empty/null):
- `from_field`: If provided, reasonable length
- `to_field`: If provided, reasonable length

**Selection Path Validation** (conditional):
- Configuration defines the recursive option tree beneath each communication system.
- `communication_system` should be one configured communication way/system.
- Each step in `communication_path` should correspond to a valid child option under the previously selected parent context.
- The current UI/runtime may enforce a practical visible depth limit, but the underlying validation should be based on configured parent/child relationships rather than a permanently fixed three-field schema.
- The core layer should validate structural fit against configuration, but Phase 1 should **not** try to enforce every real-world operational restriction such as all voice/data permissions per channel.

**Qualifier Validation**:
- `communication_qualifiers` must be valid JSON if present.
- Only qualifiers defined for the selected system should be accepted.
- If a qualifier is forced by the system configuration, the stored value should match the forced value.
- Hidden/uneditable qualifiers may still be stored with their configured forced value if later filtering/reporting depends on them.
- Phase 1 should keep qualifiers simple and boolean-like where practical.

**Length Limit Handling**:
- When `message_content` exceeds 5000 characters:
  - Validation raises `ValidationError` with clear message
  - GUI should show character count to user as they type
  - GUI may warn at 4900 characters (approaching limit)
  - **Do not silently truncate** - user must decide what to keep

**Attachment Validation**:
- File format must be in allowed list (txt, pdf, doc/docx, xls/xlsx, jpg, png, gif)
- File must exist and be readable
- Consider file size limits (TBD)

**Auto-Fill Rules**:
- `logged_time`: Set to now() if not provided
- `operator`: Prefill with last operator used
- `confirmed`: Default to False
- System-defined forced qualifiers may be auto-filled even when not shown in the GUI

## EventEntry Entity

Domain entity representing operational events, incidents, and status changes.

### Fields

#### Identity & Timestamps

**id** (int, auto-increment)
- Primary key
- Auto-generated by database
- Immutable after creation

**event_time** (datetime)
- When the event occurred
- Required
- Cannot be in the future (validation)
- User-editable (for late logging)

**logged_time** (datetime)
- When the operator entered it into system
- Required
- Auto-populated to now() if not provided
- Typically not user-editable

#### Event Content

**event_description** (str, max 5000 characters)
- What happened - the event description
- Required, cannot be empty
- Multiline supported
- Maximum 5000 characters (validation enforced)
- **Reasoning**: The actual event description is the point - this truly must exist

**whom** (str, nullable)
- Who is involved or affected by the event
- Optional - in fast situations, involvement might be unclear
- Free text entry
- Examples: "Pluton 2", "Grupp 3", "Kompani", specific callsign
- **Reasoning**: Better empty than "okänd" garbage data; free text allows flexibility

#### Event Classification

**priority** (str, enum)
- Event priority/severity level
- Optional (defaults to "Normal" if not specified)
- Values: "Low", "Normal", "High", "Critical"
- Used for filtering and visual indicators (optional color coding)
- Phase 1: Hardcoded values
- Phase 2+: User-configurable list
- **Reasoning**: In fast situations, logging the event is more important than categorizing priority

**category** (str, nullable)
- Event type/category tag
- Optional (can be empty)
- Single category per event
- Examples: "7S", "Kontakt", "Förflyttning", "Underhåll", "Rapport"
- Phase 1: Hardcoded common categories
- Phase 2+: User-manageable list
- **Reasoning**: Single category keeps it simple; can be added later during review

#### Structured Reports (Attachments)

**Design Decision**: Structured reports (like 7S) are separate entities attached to EventEntry.

**Reasoning**:
- Avoids field duplication (7S has time/location, EventEntry already has event_time)
- Extensible to future report formats (9-liner, SALUTE, custom templates)
- Keeps EventEntry clean and simple
- Report templates can evolve without changing EventEntry schema

**Relationship**:
- EventEntry has `attachments` list (can include files AND structured reports)
- Structured report is a separate entity with reference to parent EventEntry
- One EventEntry can have multiple structured reports (e.g., 7S + situation sketch)

**EventEntry Fields for Reports**:
- `event_description`: Operator's short memory note/summary (required)
  - Examples: "Fientlig patrull observerad", "Kontakt sektor Bravo"
  - Operator writes quick note for context
- `attachments`: List references to attached files AND structured reports
- When structured report attached, auto-generate summary for table view (see StructuredReport design below)

**See StructuredReport Entity** below for 7S report design.

#### Status & Metadata

**operator** (str)
- Who logged this entry (person at terminal)
- Required
- Auto-prefilled from last operator used
- User can change

**edited** (bool)
- Whether entry was modified after initial save
- Required (defaults to False)
- Set to True when entry is edited

#### Optional Fields

**attachments** (list of attachment references, nullable)
- Supporting documents (photos, reports, etc.)
- Optional
- Multiple files allowed
- Common use: incident photos, status reports
- Phase 1 attachment content is stored inside the encrypted database boundary

### Business Rules

#### Validation Rules

**Event Time Validation**:
- Cannot be in the future
- Must be valid datetime
- Can be in the past (for late logging)

**Required Field Validation**:
- `event_description`: Must not be empty string, **maximum 5000 characters**
- `event_time`: Must be valid datetime
- `operator`: Must not be empty string

**Optional Field Validation** (can be empty/null):
- `whom`: If provided, reasonable length
- `priority`: Defaults to "Normal" if not provided
- `category`: If provided, must be from configured categories (or free text in Phase 1)

**Optional Field Validation**:
- `category`: If provided, must be from configured categories list (or free text in Phase 1)
- `attachments`: Can include file attachments AND structured report references
  - File attachments: Same validation as CommunicationEntry
  - Structured reports: Validated by their own entity validators (see StructuredReport)

**Length Limit Handling**:
- When `event_description` exceeds 5000 characters:
  - Validation raises `ValidationError` with clear message
  - GUI should show character count to user as they type
  - GUI may warn at 4900 characters (approaching limit)
  - **Do not silently truncate** - user must decide what to keep

**Auto-Fill Rules**:
- `logged_time`: Set to now() if not provided
- `event_time`: Set to logged_time if not provided
- `operator`: Prefill with last operator used
- `priority`: Default to "Normal" if not specified
- `edited`: Default to False

## PersonnelEntry Entity

Domain entity representing personnel and group tracking - status, location, and check-in management.

### Fields

#### Identity & Timestamps

**id** (int, auto-increment)
- Primary key
- Auto-generated by database
- Immutable after creation

**logged_time** (datetime)
- When this status entry was created in system
- Required
- Auto-populated to now() if not provided
- Typically not user-editable

#### Personnel Identification & Status

**who** (str)
- Person name or group callsign
- Required, cannot be empty
- Free text entry with autocomplete from active personnel
- Examples: "Pluton 2", "Grupp A", "Sgt. Andersson", "GC", "GC-1"
- **Reasoning**: Must know WHO is being tracked; free text for flexibility with splits/merges

**status** (str, nullable)
- Current status description
- Optional - in fast situations, just knowing someone left might be enough
- Free text entry with autocomplete from existing statuses
- Examples: "Ute på patrull", "På väg tillbaka", "I skydd", "Återvänd"
- **Reasoning**: Can add proper status later during review; better empty than "okänd status"

**location** (str, nullable)
- Where they are (if known)
- Optional
- Free text entry
- Examples: "Punkt Alpha", "Sektor Bravo", "Grid 12345678"

**last_contact_time** (datetime, nullable)
- When we last heard from them
- Optional (defaults to logged_time if not specified)
- Can be in the past
- User-editable
- Important for tracking communication gaps
- **Reasoning**: Defaults to "now" if unknown; can be updated later

#### Mission & Notes

**mission_notes** (str, max 5000 characters)
- What they're doing, mission details, notes
- Optional
- Multiline supported
- Maximum 5000 characters (validation enforced)

#### Check-in Alarm (Optional)

**alarm_enabled** (bool)
- Whether a check-in reminder is set for this entry
- Required (defaults to False)
- Controls whether alarm fields are used

**expected_checkin_time** (datetime, nullable)
- When they should contact us (or we should contact them)
- Only populated if `alarm_enabled` = True
- Must be in the future (validation)

**alarm_triggered** (bool)
- Whether the alarm has been triggered/acknowledged
- Required (defaults to False)
- Set to True when operator acknowledges the alarm
- Used to clear tab notification

#### Historical Tracking

**active** (bool)
- Whether this is the current status for this entity
- Required (defaults to True)
- Only one entry per "who" should be active=True at a time
- Previous entries marked active=False when new status created

**supersedes** (str, nullable)
- Comma-separated list of entry IDs that this entry replaces
- Optional
- Examples: "123" (single), "123,124" (merge), null (new entity)
- Automatically set by "Update Status" button workflow
- Can be set/edited retroactively via "Set Relationship" dialog

#### Status & Metadata

**operator** (str)
- Who logged this entry (person at terminal)
- Required
- Auto-prefilled from last operator used
- User can change

**edited** (bool)
- Whether entry was modified after initial save
- Required (defaults to False)
- Set to True when entry is edited

### Business Rules

#### Validation Rules

**Required Field Validation**:
- `who`: Must not be empty string
- `logged_time`: Must be valid datetime
- `operator`: Must not be empty string

**Optional Field Validation** (can be empty/null):
- `status`: Reasonable length if provided; defaults to "" if not specified
- `location`: No specific validation (free text)
- `last_contact_time`: Defaults to logged_time if not specified
- `mission_notes`: If provided, maximum 5000 characters

**Alarm Validation** (conditional):
- If `alarm_enabled` = True:
  - `expected_checkin_time` must be set
  - `expected_checkin_time` must be in the future (at time of creation)
- If `alarm_enabled` = False:
  - `expected_checkin_time` must be null/empty

**Supersedes Validation**:
- If provided, must be comma-separated integers
- Each ID should exist in database (optional validation)
- IDs should refer to same "who" entity (recommended, not enforced)

**Active Flag Business Rule**:
- When creating new entry via "Update Status": automatically set previous entry to active=False
- Manual validation: Should ideally be only one active=True per unique "who" value (warning, not error)

**Length Limit Handling**:
- When `mission_notes` exceeds 5000 characters:
  - Validation raises `ValidationError` with clear message
  - GUI should show character count to user as they type
  - GUI may warn at 4900 characters (approaching limit)
  - **Do not silently truncate** - user must decide what to keep

**Auto-Fill Rules**:
- `logged_time`: Set to now() if not provided
- `operator`: Prefill with last operator used
- `last_contact_time`: Default to logged_time if not specified
- `status`: Default to "" (empty string) if not specified
- `active`: Default to True
- `alarm_enabled`: Default to False
- `alarm_triggered`: Default to False
- `edited`: Default to False

## StructuredReport Entity

Domain entity representing structured military report formats (7S, 9-liner, etc.) attached to EventEntry.

**Design Philosophy**:
- Separate entity from EventEntry (avoids field duplication)
- Extensible to multiple report formats
- Report data stored as JSON (flexible schema per template type)
- Auto-generates summary for EventEntry table display

### Fields

#### Identity & Relationship

**id** (int, auto-increment)
- Primary key
- Auto-generated by database
- Immutable after creation

**parent_event_id** (int)
- Foreign key to EventEntry that this report is attached to
- Required
- Immutable after creation (reports don't move between events)

**report_type** (str)
- Type of structured report
- Required
- Examples: "7S", "9-liner", "SALUTE", "Custom"
- Determines which template/fields are used
- Phase 1: "7S" only
- Phase 2+: Additional report types

**logged_time** (datetime)
- When the report was created/attached
- Required
- Auto-populated to now() if not provided

**operator** (str)
- Who created/attached this report
- Required
- Auto-prefilled from last operator used

#### Report Data

**report_data** (dict/JSON)
- Structured report fields stored as JSON
- Required
- Structure varies by report_type
- **For 7S report**:
  ```json
  {
    "s1_stund": "2026-04-18T14:30:00",  
    "s2_stalle": "Grid 12345678",
    "s3_styrka": "ca 20 personer",
    "s4_slag": "Soldater",
    "s5_sysselsattning": "Förflyttning österut",
    "s6_symbol": "Gröna uniformer",
    "s7_sagesman": "Grupp 2 förpost"
  }
  ```
- **Reasoning**: JSON allows different templates without schema changes

**auto_summary** (str, generated)
- Auto-generated text summary of report for table display
- Generated when report is saved
- Displayed in EventEntry log table
- **For 7S**: Formatted as comma-separated key info
  - Example: "14:30, Grid 12345678, ca 20 personer, Soldater, Förflyttning österut"
  - Or: "14:30, Grid XXX, 20 pers, Soldater, Förfl. öst, Gröna unif, Grp 2"
- **Template configuration** defines which fields to include in summary and format

#### Metadata

**edited** (bool)
- Whether report was modified after initial save
- Required (defaults to False)
- Set to True when report is edited

### Business Rules

#### Validation Rules

**Required Field Validation**:
- `parent_event_id`: Must reference existing EventEntry
- `report_type`: Must be from configured report types
- `report_data`: Must be valid JSON/dict
- `operator`: Must not be empty string

**Report Type Validation** (conditional):
- Configuration defines required/optional fields per report type
- **Phase 1 - 7S**:
  - All 7S fields (s1-s7) optional individually
  - GUI encourages filling all fields
  - s1_stund: If provided, must be valid datetime (can be past, not future)
  - s2-s7: Free text with reasonable length limits (500 chars each)
- **Phase 2+**: Additional report types with their own field requirements

**Auto-Summary Generation**:
- Triggered on save/update
- Uses template configuration to determine:
  - Which fields to include
  - Field order
  - Formatting (comma-separated, labeled, etc.)
- Stored in `auto_summary` field for quick table display

**Auto-Fill Rules**:
- `logged_time`: Set to now() if not provided
- `operator`: Prefill with last operator used
- `edited`: Default to False

## Configuration Management

### Communication System Configuration

**Configuration-driven system behavior**:
- Each communication system can be configured with:
  - Supported methods (what the system can do)
  - System capabilities (system-specific features/options)
  - Channel support (whether methods use channels/designations)
- Configuration stored in database
- GUI adapts based on configuration

**Phase 1 Configuration** (Hardcoded for your unit - Platoon level):
- **RA180 System**:
  - Supported methods: ["Radio", "Phone", "Data"]
  - Capabilities: transmission_mode (DART/Speech for Radio and Data), encryption (bool)
  - Channels: Radio method uses channels/designations
  - Single instance: One "RA180" entry (sufficient for platoon operations)
- **RA146 System**:
  - Supported methods: ["Radio"]
  - Capabilities: encryption (bool)
  - Channels: Radio method uses channels/designations
  - Single instance: One "RA146" entry
- **Simple methods** (no system):
  - "In Person", "Ordnance", "Other"
  - No capabilities, no channels

**Phase 2 Configuration** (Company level - Multiple instances):
- **Option A - Simple (Most common)**:
  - Keep single system entries (e.g., one "RA180")
  - Add more channels to handle all designations across physical radios
  - Channel designation identifies location/purpose
  - Specific radio instance not tracked (usually doesn't matter)
  
- **Option B - Detailed instance tracking**:
  - Multiple system entries: "RA180-1", "RA180-2", "RA180-Nord", "RA180-Syd"
  - Each instance has separate channel configurations
  - Use when: Tracking specific radio instance is operationally important
  - Trade-off: More configuration overhead

**Phase 3 Configuration** (Future - System type abstraction):
- Separate system type (e.g., "RA180") from instance identifier (e.g., "1", "Nord")
- System types define shared capabilities
- Instances inherit and can override (e.g., "RA180-2's phone broken")
- Reduces configuration duplication for multi-instance scenarios

**Channel-Level Capability Restrictions** (Acknowledged but not enforced):
- **Reality**: Some configurations restrict capabilities per channel (e.g., Ch1=DART only, Ch2=DART+Speech, Ch3=Speech only)
- **Design decision**: App does NOT enforce channel-specific capability restrictions
- **Reasoning**:
  - Operator knows operational constraints (training/doctrine)
  - App logs what operator actually used (captures reality)
  - Over-engineering: Too granular for practical benefit
  - If needed: Phase 3+ could add channel-capability mapping (low priority)

### User-Configurable Data

**Priorities** (for EventEntry):
- User maintains list in settings
- Default: ["Low", "Normal", "High", "Critical"]
- UI: Add/remove/reorder in settings dialog (Phase 2+)
- Phase 1: Hardcoded defaults

**Categories** (for EventEntry):
- User maintains list in settings
- Default: ["Kontakt", "Förflyttning", "Underhåll", "Rapport"]
- UI: Add/remove in settings dialog (Phase 2+)
- Phase 1: Hardcoded defaults
- Tags/labels style

### Settings Storage

**LastOperator**:
- Stored in database user_preferences
- Retrieved when creating new entry
- Updated when entry is saved

## Validation Module

**Separate validation logic for each entity type**:

### CommunicationEntryValidator
```python
from datetime import datetime

class CommunicationEntryValidator:
    @staticmethod
    def validate_event_time(dt: datetime) -> None:
        ...
    
    @staticmethod
    def validate_required_field(value: str, field_name: str, max_length: int | None = None) -> None:
        ...
    
    @staticmethod
```

## Error Handling

**ValidationError**:
- Custom exception for validation failures
- Contains field name and error message
- Caught by presenters for UI display
